# vision: --against local|cloud for acceptance tests

## the outcome world

### day-in-the-life

**before:**
```bash
rhx git.repo.test --what acceptance --mode apply           # runs against... what?
rhx git.repo.test --what acceptance --mode apply --locally  # runs locally... vs what?
```

**after:**
```bash
rhx git.repo.test --what acceptance --mode apply --against local --env prep
rhx git.repo.test --what acceptance --mode apply --against cloud --env prep
```

the caller knows exactly:
1. **where** the tests run (local server vs cloud server)
2. **which env** they're testing (prep, prod)

### before/after contrast

| before | after |
|--------|-------|
| `--locally` is cryptic to newcomers | `--against local\|cloud` is self-documenting |
| default is unclear (cloud? local?) | no default — must choose |
| `--env` optional | `--env` required for acceptance |
| fails silently if wrong env | fails fast with clear error |

### the "aha" moment

> "oh, so `--against local` means i run a local server and test against it, while `--against cloud` means tests hit the deployed prep/prod environment. and i have to specify `--env` so the test knows which credentials to use. that makes sense."

## user experience

### usecases

| usecase | contract |
|---------|----------|
| test local dev server (test creds) | `--what acceptance --against local --env test` |
| test local dev server (prep creds) | `--what acceptance --against local --env prep` |
| test deployed prep env | `--what acceptance --against cloud --env prep` |
| test deployed prod env | `--what acceptance --against cloud --env prod` |

### contract inputs & outputs

**inputs:**
- `--what acceptance` — selects acceptance tests (required)
- `--against local|cloud` — specifies test target (required for acceptance)
- `--env test|prep|prod` — specifies credential environment (required for acceptance)
  - note: `--env test --against cloud` is invalid (test is not deployed to cloud)

**outputs:**
- same as today: turtle vibes output, log paths, exit codes

### what it looks like

```bash
# develop locally: start server, run acceptance tests against it
npm run start:local &
rhx git.repo.test --what acceptance --against local --env prep --mode apply

# before deploy: verify against deployed prep
rhx git.repo.test --what acceptance --against cloud --env prep --mode apply

# smoke test prod after deploy
rhx git.repo.test --what acceptance --against cloud --env prod --mode apply
```

### timelines

1. developer writes code
2. runs unit + integration tests
3. starts local server
4. runs `--against local --env prep` to verify acceptance
5. pushes, CI runs acceptance `--against cloud --env prep`
6. merges to main
7. CI deploys to prod
8. CI runs acceptance `--against cloud --env prod` (smoke test)

## mental model

### how users would describe this to a friend

> "acceptance tests can run against a local server you spin up, or against the deployed cloud environment. you tell it which with `--against`. and you tell it which credentials to use with `--env`."

### analogies

- `--against local` = "test against localhost"
- `--against cloud` = "test against the real deployed service"
- like how playwright has `baseURL` pointing to localhost or deployed URL

### terms

| user term | our term | meaning |
|-----------|----------|---------|
| "local" | `--against local` | tests hit localhost server |
| "cloud", "deployed", "remote" | `--against cloud` | tests hit deployed service |
| "environment", "env" | `--env` | which credentials + config to use |

## evaluation

### how well does it solve the goals?

| goal | solved? |
|------|---------|
| replace cryptic `--locally` | yes — `--against local\|cloud` is explicit |
| make test target obvious | yes — must choose, no hidden default |
| clarify credential context | yes — `--env` required |

### pros

- **explicit is better than implicit** — no guessing what "default" means
- **self-documenting** — `--against cloud --env prep` reads clearly
- **pit of success** — can't forget to specify env
- **composable** — works with extant `--scope`, `--mode`, `--resnap`

### cons

- **breaking change** — extant `--locally` usage breaks (per wish: hardcut)
- **more typing** — `--against local --env prep` vs `--locally`
- **learning curve** — users must learn new flags

### edgecases and pit of success

| edgecase | handling |
|----------|----------|
| `--what acceptance` without `--against` | error: `--against local\|cloud required for acceptance` |
| `--what acceptance` without `--env` | error: `--env required for acceptance` |
| `--against local` with `--what unit` | error: `--against only valid with --what acceptance` |
| `--env test --against cloud` | error: `--env test is not deployed to cloud, use --against local` |
| `--locally` (old flag) | error: `--locally is deprecated, use --against local\|cloud` |

## what is awkward?

### awkward: `--against` naming

`--against` is clear but unusual. alternatives considered:
- `--target local|cloud` — "target" is overloaded (jest --testPathPattern)
- `--server local|cloud` — "server" implies implementation detail
- `--run-on local|cloud` — verbose
- `--against local|cloud` — ✓ reads naturally: "run tests against local"

verdict: `--against` is the best option despite being slightly unusual

### awkward: two required flags

requiring both `--against` and `--env` for acceptance adds friction. but:
- friction is intentional — forces clarity
- alternatives (inferring one from the other) are ambiguous
- explicit > implicit for acceptance tests (high stakes)

verdict: accept the friction, it's the right tradeoff

### awkward: hardcut breaking change

no backwards compatibility means extant commands break. but:
- per wish: hardcut is intentional
- `--locally` is confusing, best to remove entirely
- deprecation warning could ease transition (but wish says hardcut)

verdict: follow wish — hardcut, provide clear error if old flag used
