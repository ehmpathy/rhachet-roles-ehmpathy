# vision: cohesive "and then..." await transitions

---

## MANDATORY: commit-based freshness

> **the artifact we await MUST come AFTER the squash merge we just performed.**

this is the core behavioral requirement. everything else is polish.

### the rule

| after this merge... | we await... | freshness check |
|---------------------|-------------|-----------------|
| feat PR → main (M1) | release PR | `git merge-base --is-ancestor M1 <release-pr-head>` |
| release PR → main (M2) | tag | `git merge-base --is-ancestor M2 <tag-commit>` |

### why mandatory

without this check, we can pick up stale artifacts from prior runs:
- old release PR from yesterday's feat merge
- old tag from last week's release

commit ancestry is the only reliable way to know "this artifact is from THIS merge".

### how we get the merge commit

`gh pr view --json mergeCommit` returns the SHA of the commit created on main when the PR was merged.

> "If merged via a **squash**, `merge_commit_sha` represents the SHA of the squashed commit on the base branch."
> — [GitHub REST API: Pull Requests](https://docs.github.com/en/rest/pulls/pulls)

GraphQL equivalent (`mergeCommit` field):
> "The commit that was created when this pull request was merged."
> — [GitHub GraphQL Schema](https://docs.github.com/en/graphql/overview/public-schema)

## the outcome world

### current state

the "found immediately" case works correctly:

```
🫧 and then...

🌊 release: chore(release): v1.3.0
   ├─ 👌 all checks passed
   ├─ 🌴 automerge enabled [added]
   └─ 🥥 let's watch
      ├─ 💤 1 left, Xs in action, Xs watched
      └─ ✨ done! Xs in action, Xs watched
```

what's absent:
1. **commit-based freshness**: no check that artifact comes AFTER prior squash merge
2. **await with poll UI**: when artifact isn't found immediately, no poll feedback
3. **timeout diagnostics**: when artifact never appears, no workflow status lookup
4. **found after wait**: no `✨ found!` message to indicate successful await

### after

**await with poll (found after wait):**

```
🫧 and then...
   ├─ 💤 5s in await
   ├─ 💤 10s in await
   └─ ✨ found! after 15s

🌊 release: chore(release): v1.3.0
   ├─ 👌 all checks passed
   ├─ 🌴 automerge enabled [added]
   └─ 🥥 let's watch
      ├─ 💤 1 left, Xs in action, Xs watched
      └─ ✨ done! Xs in action, Xs watched
```

on timeout, the mechanic gets actionable diagnostics:

```
🫧 and then...
   ├─ 💤 5s in await
   ├─ 💤 10s in await
   └─ ⚓ release pr did not appear in 90s
      └─ 🔴 release-please
            ├─ https://github.com/owner/repo/actions/runs/12345
            └─ failed
```

### the "aha" moment

the mechanic realizes: "the await phase is now part of the same tree — i can see the progression, and when it fails, i immediately know where to look." no more uncertainty about whether the release PR is stale or if release-please crashed.

## user experience

### usecases & goals

| usecase | goal | outcome |
|---------|------|---------|
| happy path release | merge feature → release PR → tag → done | smooth flow with clear progress indicators |
| debug timeout | understand why artifact didn't appear | immediate link to release-please workflow with status |
| distinguish fresh vs stale | know if release PR is from THIS merge | commit-based freshness ensures we never pick up stale PRs |

### contract inputs & outputs

**the await transition:**

```bash
# emit_transition_await artifact_type artifact_display check_fn prior_merge_commit
#
# artifact_type: "release-pr" | "tag"
# artifact_display: display name for timeout message ("release pr" | "tag v1.3.0")
# check_fn: returns artifact JSON if found fresh, empty if not
# prior_merge_commit: SHA of the merge commit to check freshness against
#
# note: both artifact types display as `🌊 release:` in output
#       - release-pr: `🌊 release: chore(release): v1.3.0`
#       - tag: `🌊 release: v1.3.0`
```

**freshness check:**
```bash
# artifact is fresh if its commit is ahead of prior_merge_commit
git merge-base --is-ancestor $prior_merge_commit $artifact_commit
```

## mental model

### how users describe it

> "after my PR merges, git.release waits for release-please to create the release PR, then waits for the tag. if it doesn't show up, it tells me exactly which workflow failed."

### analogies

- **airport departures board**: you watch the status update from "scheduled" → "board" → "departed". the await phase is like the board refresh until your flight appears.
- **package tracker**: after handoff to carrier A, you wait for carrier B to scan it. the await is the gap between scans.

### terminology

| user says | we say |
|-----------|--------|
| "wait for the release PR" | "await transition to release-pr transport" |
| "the release is stuck" | "await timeout on artifact" |
| "did release-please run?" | "release-please workflow status" |

## evaluation

### how well does it solve the goals?

| goal | solution | score |
|------|----------|-------|
| cohesive tree output | all await lines use `├─` / `└─` connectors | ✓ solved |
| actionable timeouts | workflow status + link on timeout | ✓ solved |
| no stale artifacts | commit-based freshness via `git merge-base` | ✓ solved |
| unified logic | single `and_then_await` function for both transports | ✓ solved |

### pros

- consistent treestruct across all output states
- immediate diagnostics on failure (no separate lookup needed)
- commit-based freshness handles both new PRs and updated PRs
- single reusable await function reduces duplication

### cons

- commit-based check requires `gh pr view --json mergeCommit` call (extra API call)
- timeout diagnostics require workflow lookup (another API call on failure path)

### pit of success edgecases

| edgecase | how we handle it |
|----------|------------------|
| stale release PR from prior run | commit freshness check rejects it |
| release PR opened before feat merge, updated after | latest commit on PR is checked, not PR creation time |
| release-please workflow failed | timeout shows workflow status + link |
| release-please workflow still active | timeout shows "in_progress" status |
| tag created before we start to watch | commit freshness check validates it |

## what is awkward?

### the "found immediately" case

when artifact is found on first check, we emit:
```
🫧 and then...

🌊 release: chore(release): v1.3.0
   ├─ 👌 all checks passed
   ...
```

the blank line feels slightly awkward — but it's necessary to separate the transition bubble from the next transport. alternatives:
- `🫧 and then... ✨` (inline indicator) — but this breaks when we DO have wait lines
- omit the bubble entirely — but then the flow is less clear

the current design (blank line = immediate, sub-branches = waited) is the cleanest tradeoff.

### workflow name hardcoded

we assume "release-please" as the workflow name. if a repo uses a different name, timeout diagnostics will show "not found". this is acceptable for ehmpathy repos but worth the note.

### timeout message format

the wish shows:
```
└─ ⚓ release pr did not appear in 90s
```

but also:
```
└─ ⚓ tag v1.3.0 did not appear in 90s
```

the tag case includes the version, the release PR case doesn't. this is intentional — we know the expected tag version, but we don't know the expected release PR number. slightly asymmetric but accurate.
