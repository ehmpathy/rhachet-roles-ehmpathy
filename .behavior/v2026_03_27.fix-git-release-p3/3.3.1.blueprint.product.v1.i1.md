# blueprint: and_then_await with commit-based freshness

## summary

create a single reusable `and_then_await` operation that:
1. awaits artifacts (release-pr or tag) with commit-based freshness
2. emits poll UI with cumulative elapsed time
3. provides timeout diagnostics with release-please workflow status
4. replaces inline tag await logic and release-pr await

the operation enforces the MANDATORY requirement: artifacts MUST come AFTER the squash merge just performed.

## contracts

### and_then_await inputs

```bash
and_then_await artifact_type artifact_display prior_merge_commit
```

| arg | type | description |
|-----|------|-------------|
| artifact_type | `release-pr` or `tag` | which artifact to await |
| artifact_display | string | display name for timeout message |
| prior_merge_commit | sha | merge commit to check freshness against |

### and_then_await outputs

| outcome | exit code | stdout | AWAIT_RESULT |
|---------|-----------|--------|--------------|
| found immediately | 0 | `🫧 and then...` + blank | artifact json |
| found after wait | 0 | `🫧` + `💤` lines + `✨ found!` | artifact json |
| timeout | 2 | `🫧` + `💤` lines + `⚓` + `🔴` | empty |

### freshness check contract

```bash
# artifact is fresh if prior_merge_commit is ancestor of artifact_commit
git merge-base --is-ancestor $prior_merge_commit $artifact_commit
# exit 0 = fresh (is ancestor)
# exit 1 = stale (not ancestor)
```

## decomposition notes

### why single reusable operation

both await scenarios (release-pr and tag) share:
- poll loop with timeout
- freshness check via commit ancestry
- poll UI with elapsed time
- timeout diagnostics with workflow status

the only differences are:
- what artifact to look for (PR vs tag)
- how to retrieve it (gh pr list vs git fetch --tags)
- display name in timeout message

a single `and_then_await` with artifact_type parameter covers both cleanly.

### why not inline

inline logic at each call site would duplicate:
- poll loop structure
- timeout logic
- workflow status lookup
- output format

the extant tag await loop (lines 874-895) already shows this problem: no freshness check, no poll UI, no workflow diagnostics. a consolidated reusable operation fixes all gaps at once.
