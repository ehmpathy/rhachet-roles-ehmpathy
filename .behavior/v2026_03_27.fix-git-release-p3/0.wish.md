## architectural requirements

### core requirement: commit-based freshness

the "and then..." transition awaits the NEXT transport artifact. the commit on the next transport must be AHEAD OF the merge commit of the prior transport on main. this prevents stale artifacts from prior runs from being picked up.

- feature PR squash-merges to main (merge commit M1) в†’ await release PR whose head commit is ahead of M1
- release PR squash-merges to main (merge commit M2) в†’ await tag whose commit is ahead of M2

check: `git merge-base --is-ancestor M1 <next-transport-commit>`

must support squash merges: both the feature PR and the release PR are squash merged into main. use the squash merge commit on main (not the PR's head commit) as the baseline. get via `gh pr view --json mergeCommit`.

this handles both cases:
1. release PR opened after feat merge (new PR created by release-please)
2. release PR opened before but updated after feat merge (extant open PR updated by release-please with new commit)

### single reusable await operation

the `await_release_pr_after_merge` function and the inline tag await logic must be consolidated into a single reusable operation:

**file**: `src/domain.roles/mechanic/skills/git.release/git.release._.and_then_await.sh`

```bash
# and_then_await artifact_type artifact_name check_fn prior_merge_commit
#
# artifact_type: "release-pr" | "tag"
# artifact_name: display name ("release pr" or "tag v1.3.0")
# check_fn: function that returns artifact if found fresh (ahead of prior_merge_commit), empty if not
# prior_merge_commit: SHA of the squash merge commit to check freshness against
#
# freshness check: git merge-base --is-ancestor $prior_merge_commit $artifact_commit
#
# returns:
#   exit 0 + sets AWAIT_RESULT: artifact found
#   exit 2: timeout (constraint)
#
# output:
#   - found immediately: no sub-branches (blank line before next transport)
#   - found after wait: `в”њв”Ђ рџ’¤` lines + `в””в”Ђ вњЁ found!`
#   - timeout: `в””в”Ђ вљ“` + workflow status lookup
```

both call sites use the same operation:
1. release PR await after feature merge
2. tag await after release PR merge
