# 1.vision: git.branch.rebase lock refresh

## the outcome world

### before/after contrast

**before:**
```
rebase → conflict in pnpm-lock.yaml → take theirs → continue → push → ⛈️ CI fails → pnpm install → commit → push again
```

**after:**
```
rebase → conflict in pnpm-lock.yaml → take theirs → sees ⚡ suggestion → lock refresh → continue → CI passes ✓
```

### the "aha" moment

the value clicks when CI stops failed builds on lock file mismatches after rebases. the mechanic no longer needs to:
1. wait for CI to fail
2. diagnose the issue
3. run pnpm install manually
4. create another commit
5. push again

instead, the skill proactively suggests the fix and the lock is correct on first push.

## user experience

### usecases

| usecase | goal | contract |
|---------|------|----------|
| lock conflict settled | regenerate lock | `rhx git.branch.rebase lock refresh` |
| proactive suggestion | remind mechanic | output from `take` when lock file settled |

### contract inputs & outputs

```
rhx git.branch.rebase lock refresh

inputs:
  - (none) — detects package manager automatically

outputs:
  - regenerated lock file (staged)
  - or error if:
    - no rebase in progress
    - no lock file extant
    - pnpm/npm install fails
```

### example timeline

```
$ rhx git.branch.rebase take --whos theirs pnpm-lock.yaml
```

## mental model

### how users describe it

> "after I take a lock file in rebase, I refresh it so CI doesn't fail"

> "it's like regenerate the lock file but aware that you're mid-rebase"

### analogies

- **npm install after merge**: same concept, just aware of rebase state
- **lock file heal**: the lock was taken from one timeline, refresh heals it for the current timeline

### user terms vs our terms

| user says | we call it |
|-----------|------------|
| "regenerate the lock" | `lock refresh` |
| "fix the lock file" | `lock refresh` |
| "reinstall dependencies" | `lock refresh` |

## evaluation

### how well does it solve the goals?

| goal | solved? | notes |
|------|---------|-------|
| prevent CI failures | ✓ | lock regenerated before push |
| reduce manual steps | ✓ | one command vs multiple |
| proactive guidance | ✓ | `take` suggests refresh |

### pros

- eliminates common CI failure mode
- fits naturally in rebase flow
- proactive suggestion catches the case before it's a problem
- works with pnpm, npm, and yarn

### cons

- adds another step to rebase flow (though less than the alternative)
- install can be slow on large projects
- may change lock in ways beyond just the conflict resolution

### pit of success edgecases

| edgecase | action |
|----------|--------|
| no rebase in progress | error: "no rebase in progress" |
| no lock file extant | error: "no lock file found" |
| install fails | show error output, suggest manual fix |
| both pnpm-lock and package-lock | prefer pnpm-lock if pnpm available |
| yarn.lock extant | use yarn install |
| pnpm-lock extant but pnpm not installed | error: "pnpm not found, install pnpm or use npm" |
| yarn.lock extant but yarn not installed | error: "yarn not found, install yarn" |
| multiple lock files taken in single `take .` | show suggestion once (not per file) |

## open questions & assumptions

### assumptions

1. **lock refresh = pnpm/npm install**: regenerate lock via install
2. **stage after refresh**: the refreshed lock should be staged automatically
3. **detect package manager**: we can detect pnpm vs npm vs yarn from lock file presence
4. **rebase-only by design**: command intentionally scoped to rebase (wisher confirmed: merge support maybe later)

### decisions (from wisher)

1. **rebase-only for now**: `lock refresh` should NOT work outside of rebase. maybe in the future for merge conflicts, but not v1.
2. **suggest, not auto-refresh**: just suggest the command after `take`. auto-refresh is slow and a surprise. maybe in the future.
3. **support yarn.lock**: yes, support pnpm, npm, and yarn.

### research needed

- does `pnpm install` in rebase behave differently than normal?
- what flags ensure we regenerate lock without fetch of new versions?

## what is awkward?

### feels off

- **"lock refresh" vs "lock regenerate"**: refresh implies partial update, but we do a full regenerate. however "refresh" is shorter and feels more natural.

- **nested subcommand depth**: `git.branch.rebase lock refresh` is 4 levels deep. alternative: `git.branch.rebase refresh-lock` (2 levels) but breaks pattern of group.

### tradeoffs

| tradeoff | choice | why |
|----------|--------|-----|
| auto vs manual | manual with suggestion | safer, user controls when |
| nested vs flat | nested `lock refresh` | extensible (could add `lock check`) |
| pnpm only vs multi | multi (pnpm + npm + yarn) | wider applicability |

### fights mental model?

- users might expect `take` to handle this automatically — but auto-run install is slow and a surprise
- proactive suggestion is a good middle ground
