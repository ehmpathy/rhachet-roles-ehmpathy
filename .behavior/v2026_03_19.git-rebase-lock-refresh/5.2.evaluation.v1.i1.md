## divergence analysis

### divergence 1: operations inlined vs functions

**blueprint says:**
```
### new operations (in git.branch.rebase.lock.sh)
- `detect_lock_file()` — returns lock file path or empty
- `detect_package_manager()` — returns pnpm|npm|yarn
- `is_pm_installed()` — check if pm binary available
- `run_install()` — execute pm install
```

**implementation has:**
- lock file detection: inline if-elif chain (lines 118-131)
- package manager detection: merged with lock detection
- pm installed check: inline `command -v` (lines 147, 154)
- run install: inline case statement (lines 169-179)

**justification:**
blueprint noted "inline output functions in lock.sh (follows YAGNI — single consumer)". the same YAGNI principle applies to these operations:
1. each operation is used exactly once
2. to extract to a function would add indirection without benefit
3. the code is already readable with section headers
4. excessive abstraction for 2 call sites is not justified

**verdict:** justified divergence — YAGNI ✓

### divergence 2: output functions inline vs shared

**blueprint says:**
```
### turtle vibes output
- inline output functions in lock.sh (follows YAGNI — single consumer)
```

**implementation has:**
- output functions defined inline at lines 30-60
- not sourced from git.commit/output.sh

**justification:**
this matches the blueprint exactly. lock.sh is the only consumer, and the signatures differ slightly (no "commit" specific paths).

**verdict:** matches blueprint ✓
