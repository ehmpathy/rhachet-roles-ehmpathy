# 2.1.criteria.blackbox: git.branch.rebase lock refresh

## usecase.1 = refresh lock file after take

### episode: successful refresh

```
given a rebase in progress
  given a lock file was taken (pnpm-lock.yaml, package-lock.json, or yarn.lock)
    when `rhx git.branch.rebase lock refresh` is run
      then the lock file is regenerated via the correct package manager
        sothat the lock matches the current package.json state
      then the lock file is staged
        sothat `git rebase --continue` can proceed
      then output shows success with turtle vibes
        sothat the mechanic knows refresh is complete
```

### episode: error cases

```
given no rebase in progress
  when `rhx git.branch.rebase lock refresh` is run
    then error: "no rebase in progress"
      sothat the mechanic knows the command is rebase-specific

given a rebase in progress
  given no lock file extant
    when `rhx git.branch.rebase lock refresh` is run
      then error: "no lock file found"
        sothat the mechanic knows there is no lock to refresh

given a rebase in progress
  given pnpm-lock.yaml extant
    given pnpm not installed
      when `rhx git.branch.rebase lock refresh` is run
        then error: "pnpm not found, install pnpm or use npm"
          sothat the mechanic knows what tool is needed

given a rebase in progress
  given yarn.lock extant
    given yarn not installed
      when `rhx git.branch.rebase lock refresh` is run
        then error: "yarn not found, install yarn"
          sothat the mechanic knows what tool is needed

given a rebase in progress
  given a lock file extant
    when install fails
      then error output is shown
      then suggestion for manual fix is shown
        sothat the mechanic can diagnose and fix
```

### exchange: inputs and outputs

```
inputs:
  - (none) — detects package manager automatically

outputs:
  - success: regenerated lock file (staged)
  - error: descriptive message for each failure mode
```

## usecase.2 = proactive suggestion after take

### episode: suggestion shown

```
given a rebase in progress
  given a lock file has conflicts
    when `rhx git.branch.rebase take --whos theirs pnpm-lock.yaml` is run
      then the conflict is settled
      then under the settled file, a suggestion is shown:
        "lock taken, refresh it with: ⚡"
        "└─ rhx git.branch.rebase lock refresh"
        sothat the mechanic knows to refresh before continue

given a rebase in progress
  given multiple files have conflicts (including a lock file)
    when `rhx git.branch.rebase take --whos theirs .` is run
      then all conflicts are settled
      then the suggestion is shown once (not per lock file)
        sothat output is clean
```

### episode: no suggestion

```
given a rebase in progress
  given a non-lock file has conflicts
    when `rhx git.branch.rebase take --whos theirs some-other-file.ts` is run
      then the conflict is settled
      then no lock refresh suggestion is shown
        sothat mechanics are not confused by irrelevant suggestions
```

## usecase.3 = package manager detection

### exchange: detection rules

```
given pnpm-lock.yaml extant
  when refresh is run
    then pnpm install is used

given package-lock.json extant (no pnpm-lock.yaml)
  when refresh is run
    then npm install is used

given yarn.lock extant (no pnpm-lock.yaml, no package-lock.json)
  when refresh is run
    then yarn install is used

given both pnpm-lock.yaml and package-lock.json extant
  given pnpm is installed
    when refresh is run
      then pnpm install is used (pnpm preferred)

given both pnpm-lock.yaml and package-lock.json extant
  given pnpm is not installed
    when refresh is run
      then npm install is used (fallback)
```
