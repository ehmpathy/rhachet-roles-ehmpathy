# vision: git.branch.rebase take

## the outcome world

### before

during a rebase, a clone encounters a conflict in `pnpm-lock.yaml`:

```
🐢 hold up dude...
git.branch.rebase continue
   ├─ rebase
   │  └─ status: conflict
   ├─ conflict
   │  ├─ commit: abc123 feat(api): add validation
   │  └─ files
   │     └─ pnpm-lock.yaml
   └─ next steps
      ├─ settle conflicts in files above
      └─ rhx git.branch.rebase continue
```

clone thinks: "i just want the main branch's lock file. let me..."

```sh
git checkout --theirs pnpm-lock.yaml
```

**blocked.** permission denied. `git checkout` is destructive — it can wipe uncommitted work, switch branches, discard changes. no safe prefix extant for permissions.jsonc.

clone asks human for permission. human approves. flow interrupted.

next conflict: same thing. ask again. approve again. repeat 5 times.

### after

```sh
rhx git.branch.rebase take --whos theirs pnpm-lock.yaml
```

```
🐢 righteous!
git.branch.rebase take
   ├─ whos: theirs
   ├─ files
   │  └─ pnpm-lock.yaml ✓
   └─ done
```

pre-approved. no interruption. clone continues the rebase flow autonomously.

### the "aha" moment

> "i can settle lock file conflicts without asking permission every time!"

the value clicks when a rebase has 5+ commits and each one conflicts on `pnpm-lock.yaml`. instead of 5 permission prompts, zero.

## user experience

### usecases

| goal | command |
|------|---------|
| take theirs for lock file | `rhx git.branch.rebase take --whos theirs pnpm-lock.yaml` |
| take ours for config | `rhx git.branch.rebase take --whos ours .eslintrc.json` |
| take theirs for multiple files | `rhx git.branch.rebase take --whos theirs '*.lock' package.json` |
| take theirs for all conflicts | `rhx git.branch.rebase take --whos theirs .` |

### contract

```
rhx git.branch.rebase take --whos ours|theirs [paths...]

inputs:
  --whos     required   ours|theirs
  paths      required   file paths or globs (use . for all)

outputs:
  - settles conflicts for specified files
  - stages settled files
  - prints tree showing what was settled

guards:
  - must be in a rebase
  - specified files must have conflicts
  - cannot operate on non-conflicting files
```

### timeline

```
1. clone runs: rhx git.branch.rebase begin --mode apply
2. conflict: pnpm-lock.yaml
3. clone runs: rhx git.branch.rebase take --whos theirs pnpm-lock.yaml
4. clone runs: rhx git.branch.rebase continue
5. conflict: pnpm-lock.yaml (next commit)
6. clone runs: rhx git.branch.rebase take --whos theirs pnpm-lock.yaml
7. clone runs: rhx git.branch.rebase continue
8. complete
```

no permission prompts. smooth flow.

## mental model

### how would users describe this?

> "take settles conflicts by picking one side — ours or theirs — like `git checkout --ours/--theirs` but scoped to rebase conflicts only."

### analogies

| concept | analogy |
|---------|---------|
| `--whos theirs` | "accept incoming" in merge editors |
| `--whos ours` | "keep current" in merge editors |
| `take` | "settle by choosing a side" |

### terms: user vs internal

| user says | we say |
|-----------|--------|
| "take theirs" | `--whos theirs` |
| "accept incoming" | `--whos theirs` |
| "keep mine" | `--whos ours` |
| "keep current" | `--whos ours` |

### why `--whos` not `--who`?

- `--who` reads as "who did this?" (past tense, query)
- `--whos` reads as "whose version?" (possessive, selection)
- `--whos theirs` = "whose version? theirs."

## evaluation

### how well does it handle the goals?

| goal | handled? |
|------|---------|
| avoid permission prompts for conflict settlement | ✓ yes |
| scoped to safe operations only | ✓ yes (rebase + conflicting files) |
| consistent with extant skills | ✓ yes (same patterns as begin/continue/abort) |

### pros

- **permissionable**: single prefix `rhx git.branch.rebase take` covers all uses
- **safe**: guards prevent misuse outside rebase context
- **ergonomic**: fewer commands than manual `git checkout` + `git add`
- **composable**: fits naturally in the begin → take → continue → take → continue flow

### cons

- **learning curve**: users must learn `--whos` terminology
- **limited scope**: only works during rebases (not merges) — by design

### edgecases and pit of success

| edgecase | guard |
|----------|-------|
| no rebase in progress | error: "no rebase in progress" |
| file not in conflict | error: "file has no conflicts" |
| invalid --whos value | error: "whos must be ours or theirs" |
| glob matches non-conflicting files | only settle conflicting matches, warn about skipped |

## open questions & assumptions

### assumptions

1. **ours/theirs semantics**: in a rebase, "ours" = the commit we replay, "theirs" = the base branch (origin/main). this is git's semantics — counterintuitive but correct.

2. **auto-stage**: after `take`, files are staged via `git add`. (note: [research] needed to verify if `git checkout --theirs` auto-stages or requires explicit `git add`)

3. **no quota consumption**: `take` doesn't count as a "commit use" — it's conflict settlement, not commit creation.

4. **no stage permission**: `take` does not require stage permission — if you can rebase, you can settle conflicts.

5. **best-effort on partial failure**: if some files succeed and others fail, report success for each and error for failures. don't rollback successful takes.

## what is awkward?

### ours/theirs confusion

during a rebase:
- `ours` = the commit being replayed (your branch's commit)
- `theirs` = the base you're rebasing onto (origin/main)

this is **backwards** from merge intuition where "ours" = current branch.

**mitigation**: clear output showing what was taken:
```
🐢 righteous!
git.branch.rebase take
   ├─ whos: theirs (from origin/main)
   ├─ files
   │  └─ pnpm-lock.yaml ✓
   └─ done
```

### naming: `take` vs alternatives

| option | feel |
|--------|------|
| `take` | active, clear: "take their version" |
| `settle` | specific but passive |
| `accept` | passive, unclear direction |
| `checkout` | too close to dangerous git checkout |

`take` wins — active verb, clear intent, no collision with forbidden terms.

### naming: `--whos` vs `--from` vs `--side`

| option | example | feel |
|--------|---------|------|
| `--whos theirs` | "whose version? theirs" | clear possessive |
| `--from theirs` | "from theirs" | preposition, slightly odd |
| `--side theirs` | "side theirs" | gaming terminology, less intuitive |

`--whos` wins — reads naturally as a question-answer: "whose? theirs."

## summary

`git.branch.rebase take` fills the gap between conflict detection and settlement by providing a safe, permissionable way to choose sides. it enables autonomous rebase flows without per-file permission prompts.

```
rhx git.branch.rebase take --whos theirs pnpm-lock.yaml
```

simple. safe. seaturtle-approved. 🐢
