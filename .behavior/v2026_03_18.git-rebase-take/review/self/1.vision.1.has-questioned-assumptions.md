# self-review: has-questioned-assumptions

review of hidden assumptions in the vision for `git.branch.rebase take`.

---

## assumption 1: ours/theirs semantics in rebase

**what do we assume?**
- in rebase, "ours" = commit we replay, "theirs" = base branch

**what evidence supports this?**
- git documentation confirms this is git's semantics
- tested behavior in real rebases

**what if the opposite were true?**
- if ours/theirs were swapped, users would get wrong files
- but git's semantics are fixed — we can't change them

**did the wisher say this?**
- no — i assumed this based on git knowledge

**verdict**: ✅ holds — git's documented behavior, not our choice

---

## assumption 2: `take` should auto-stage files

**what do we assume?**
- after take, files should be staged automatically

**what evidence supports this?**
- `git checkout --ours/--theirs <file>` stages the file
- this is what git does natively

**what if the opposite were true?**
- if we didn't stage, clone must: `git add <file>` → more friction
- but maybe clone wants to review before stage?
- in rebase conflicts, you pick a side then continue — review happens before `take`, not after

**did the wisher say this?**
- no — implicit in "take their version"

**verdict**: ✅ holds — matches git behavior, reduces friction

---

## assumption 3: `take` doesn't consume quota

**what do we assume?**
- `take` is not a commit, so no quota decrement

**what evidence supports this?**
- `git checkout --theirs` doesn't commit
- commit happens in `continue`

**what if the opposite were true?**
- if `take` consumed quota, it would feel punitive
- "i just want to settle this conflict" shouldn't cost

**did the wisher say this?**
- no — i inferred this

**exceptions?**
- what if someone does 50 `take` operations?
- still reasonable — they're just settlement, not commits

**verdict**: ✅ holds — semantically correct

---

## assumption 4: file must have conflicts

**what do we assume?**
- `take` only works on files with conflicts

**what evidence supports this?**
- in rebase, only files with conflicts need settlement
- conflict-free files are already handled

**what if the opposite were true?**
- if we allowed `take` on any file in rebase...
- what would `take --whos theirs clean-file.ts` mean?
- in rebase, "theirs" version of a clean file is... already in the tree?
- unclear at best, unexpected behavior at worst

**did the wisher say this?**
- no — implied by context ("settle conflicts")

**verdict**: ✅ holds — semantic clarity

---

## assumption 5: only rebases, not merges

**what do we assume?**
- `take` only works in rebases, not merges

**what evidence supports this?**
- wish says "for rebases"
- ours/theirs semantics differ between rebase and merge

**what if the opposite were true?**
- if we supported merges, ours/theirs would mean different things
- rebase: ours = commit we replay, theirs = base
- merge: ours = current branch, theirs = branch we merge
- we'd have to explain which mode you're in

**did the wisher say this?**
- yes — wish specifically mentions "for rebases"

**verdict**: ✅ holds — explicitly scoped by wisher

---

## assumption 6: paths can be globs

**what do we assume?**
- `take --whos theirs '*.lock'` should work

**what evidence supports this?**
- wish shows single file, but globs are natural extension
- other skills support globs (cpsafe, mvsafe, rmsafe)

**what if the opposite were true?**
- if we only supported exact paths, clone must enumerate all conflicts
- "take theirs for all lock files" is a common need

**did the wisher say this?**
- wish shows `[paths|globs]` — yes, globs are expected

**verdict**: ✅ holds — explicitly in wish

---

## assumption 7: `.` means all files with conflicts

**what do we assume?**
- `take --whos theirs .` settles all conflicts

**what evidence supports this?**
- `.` is common shorthand for "all" in git commands
- `git add .` stages all

**what if the opposite were true?**
- if `.` meant "current directory only", we'd need `--all` flag
- but `.` from repo root = all files

**did the wisher say this?**
- no — i assumed this as convenient shorthand

**should we require explicit flag?**
- maybe `--all` is clearer than `.`?
- but `.` matches git's patterns and wish mentions `[paths|globs]`

**verdict**: 🟡 questionable — consider if `--all` is more explicit

**action**: keep `.` support, but document clearly. don't add `--all` to avoid bloat.
