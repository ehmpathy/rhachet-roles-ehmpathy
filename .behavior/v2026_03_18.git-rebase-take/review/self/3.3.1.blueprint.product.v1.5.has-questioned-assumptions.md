# review: has-questioned-assumptions (fifth pass)

## reviewed: git.branch.rebase.take blueprint

stepped away. came back. one more look at assumptions.

### assumption: output format consistency

**assumed:** take should emit turtle vibes tree like other git.branch.rebase skills

**holds because:**
- begin, continue, abort all use turtle vibes
- users expect consistent output
- tree format is scannable

### assumption: exit codes

**assumed:** exit 0 on success, non-zero on failure

**holds because:**
- unix convention
- enables composition with `&&`
- enables error detection in tests

### assumption: glob patterns work via bash expansion

**assumed:** user can pass `*.lock` and bash expands it

**holds because:**
- consistent with other shell commands
- no need to reimplement glob logic
- users know shell glob syntax

### assumption: partial success is acceptable

**assumed:** if 2/3 files settle, report success for 2, error for 1

**holds because:**
- each file settlement is independent
- user benefits from partial progress
- rollback would discard useful work
- consistent with `git add` behavior on multiple files

### what if we required all-or-none?

if we rolled back on any failure:
- user runs `take --whos theirs a.ts b.ts c.ts`
- a.ts and b.ts settle
- c.ts fails (deleted in rebase)
- rollback undoes a.ts and b.ts
- user must redo work

this is worse. partial progress is more valuable.

### verdict

all assumptions hold with evidence. the design is sound.
