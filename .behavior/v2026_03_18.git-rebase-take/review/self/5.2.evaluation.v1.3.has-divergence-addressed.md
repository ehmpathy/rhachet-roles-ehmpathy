## divergence 2: section names

### the divergence

blueprint: `print_tree_branch "files"` with mixed success/error indicators
implementation: four sections (settled, skipped, failed, not found)

### skeptical question

**is this laziness?** did i just not want to implement the mixed-indicator approach?

**answer**: no. i implemented what was more useful. the four-section approach came from a specific insight: a file can be in one of four states:
1. settled (conflict resolved, staged)
2. skipped (not in conflict)
3. failed (checkout or add error)

these are mutually exclusive. the blueprint's mixed approach would require inline indicators like "✓ file.ts" and "✗ other.ts" in a single "files" section.

**could this cause problems later?** no. the four sections are exhaustive — every input file ends up in exactly one section. no state is missed.

**would a skeptic accept this?** yes. the skeptic would ask "how do i quickly see which files failed?" the answer is "look at the failed section" rather than "scan the entire files list for ✗ markers."

### verdict

✓ backup is correct. the implementation is more scannable.

## divergence 3: exit code for partial

### the divergence

blueprint: "0 if all settled or partial success"
implementation: exit 1 if any failures or not-found

### skeptical question

**is this laziness?** did i just not want to implement partial success = exit 0?

**answer**: no. this was a deliberate choice based on criteria analysis.

the criteria (usecase.9) says: "exit code reflects partial success or failure"

the phrase "reflects failure" implies non-zero. if partial success returned 0, it would NOT reflect the failure portion.

**could this cause problems later?** consider the use case:

```sh
rhx git.branch.rebase take --whos theirs *.lock nonexistent.txt && rhx git.branch.rebase continue
```

with exit 0 on partial: the continue would run even though nonexistent.txt was not found.
with exit 1 on partial: the continue would not run — the user is alerted to the issue.

exit 1 is safer — it forces the user to acknowledge partial failure.

**would a skeptic accept this?** yes. the skeptic would ask "what happens if i pipe this into another command?" the answer is: exit 1 breaks the pipe, which is correct behavior for a partial failure.

### verdict

✓ backup is correct. repair would hide partial failures.

## divergence 4: glob expansion method

### the divergence

blueprint: "use bash glob on paths"
implementation: use `git ls-files "$pattern"`

### skeptical question

**is this laziness?** did i just use git ls-files because it was easier?

**answer**: no. bash glob expansion has a problem: it expands against the filesystem, not the git index.

consider: `.gitignore` contains `*.lock`. a file `temp.lock` exists on disk but is not tracked.

with bash glob: `*.lock` would expand to include `temp.lock`
with git ls-files: `*.lock` would only return tracked `.lock` files

in a rebase context, all conflict files are tracked. the index is the source of truth for what can be settled.

**could this cause problems later?** consider:

user runs: `rhx git.branch.rebase take --whos theirs *.log`

with bash glob: might try to settle untracked log files (fails silently or errors)
with git ls-files: only settles tracked log files in conflict

git ls-files is more correct.

**would a skeptic accept this?** yes. the skeptic would ask "what if i have gitignored files that match my pattern?" the answer is: they are correctly excluded because they cannot be in conflict.

### verdict

✓ backup is correct. git ls-files is semantically correct for this domain.
