### deep assumptions questioned

#### assumption: bash is the right choice

**question:** why bash and not typescript?

**holds because:**
- extant git.branch.rebase.*.sh skills are all bash
- bash has direct git command access without process spawn overhead
- consistency with dispatcher pattern
- no build step required for shell executables

#### assumption: best-effort partial failure is correct

**question:** should we rollback on any failure?

**holds because:**
- each file settlement is independent
- rollback would undo successful work
- user benefits from partial progress
- consistent with extant patterns (e.g., git add multiple files)

#### assumption: globs intersect with conflict files

**question:** what if user wants to take non-conflict files?

**holds because:**
- the skill is specifically for conflict settlement
- non-conflict files have no "ours" or "theirs" version to take
- guard prevents misuse with clear error message

#### assumption: no quota consumption

**question:** should take consume commit quota?

**holds because:**
- take does not create commits
- take only settles conflicts (stages files)
- commit quota is for actual commits
- consistent with how git.stage.add works

### what if the opposite were true?

**what if `take` should consume quota?**
- no. quota gates commit creation, not conflict resolution
- if `take` consumed quota, a 10-commit rebase with conflicts on each would exhaust quota before any commits were made
- this would make rebases unusable

**what if we should rollback on any failure?**
- no. consider: user runs `take --whos theirs a.ts b.ts c.ts`
- a.ts and b.ts succeed, c.ts fails (file was deleted in rebase)
- rollback would undo a.ts and b.ts progress
- user would have to redo work
- partial progress is more valuable than atomic failure

**what if globs should not filter to conflict files?**
- no. `git checkout --theirs README.md` on a non-conflict file does... what?
- git would error or do unexpected work
- filter to conflict files is the safe, correct behavior
