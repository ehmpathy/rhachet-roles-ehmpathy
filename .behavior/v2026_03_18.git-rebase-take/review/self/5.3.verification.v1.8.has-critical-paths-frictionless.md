## critical paths from vision

### path 1: take theirs for lock file in rebase

```
1. clone runs: rhx git.branch.rebase begin --mode apply
2. conflict: pnpm-lock.yaml
3. clone runs: rhx git.branch.rebase take --whos theirs pnpm-lock.yaml
4. clone runs: rhx git.branch.rebase continue
```

**verification**: test case1 exercises step 3. actual snapshot output:

```
🐢 righteous!

🐚 git.branch.rebase take
   ├─ whos: theirs
   ├─ settled
   │  └─ pnpm-lock.yaml ✓
   └─ done
```

**friction analysis**:
- **clarity**: "righteous!" indicates success immediately
- **actionable**: file shown with ✓ confirms what was settled
- **completeness**: "done" signals operation complete
- **exit code**: test verifies `expect(result.status).toBe(0)`

**friction check**: none. single command, clear output, no ambiguity.

### path 2: take theirs for all conflicts

```
rhx git.branch.rebase take --whos theirs .
```

**verification**: test case3 exercises this. actual snapshot output:

```
🐢 righteous!

🐚 git.branch.rebase take
   ├─ whos: theirs
   ├─ settled
   │  ├─ a.ts ✓
   │  ├─ b.ts ✓
   │  └─ c.ts ✓
   └─ done
```

**friction analysis**:
- **batch operation**: all three conflict files settled in one command
- **visibility**: each file listed individually with ✓
- **consistency**: same output structure as single file case

**friction check**: none. `.` means "all" — intuitive, no need to list each file.

### path 3: partial failure

```
rhx git.branch.rebase take --whos theirs a.ts b.ts nonexistent.ts
```

**verification**: test case11 exercises this. actual snapshot output:

```
🐢 mostly righteous...

🐚 git.branch.rebase take
   ├─ whos: theirs
   ├─ settled
   │  ├─ a.ts ✓
   ├─ not found
   │  └─ nonexistent.ts
   └─ done (with errors)
```

**friction analysis**:
- **header signals partial**: "mostly righteous..." (not "righteous!" or "hold up dude...")
- **settled section**: shows which files succeeded with ✓
- **not found section**: shows which files failed
- **footer signals partial**: "done (with errors)" not just "done"
- **exit code**: test verifies `expect(result.status).toBe(1)`

**friction check**: none. output clearly distinguishes success vs failure — user knows exactly what worked and what did not.
