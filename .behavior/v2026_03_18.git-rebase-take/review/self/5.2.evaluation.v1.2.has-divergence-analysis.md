## issue found: glob expansion method

the blueprint says "use bash glob on paths, then intersect with conflict files."

the implementation uses `git ls-files "$pattern"` to expand globs, not bash expansion.

**why this matters**: bash glob (`*.lock`) expands against the filesystem. git ls-files expands against the git index. these can differ for uncommitted files.

**is this a bug?**: no. in a rebase context, all conflict files are tracked (in the index). git ls-files is more correct — it respects gitignore and only returns tracked files.

**fix**: i should add this to the divergence table.

## why each resolution is correct

### three header variants → backup

the blueprint said "righteous! or hold up dude..." — two states: success and failure.

the implementation adds "mostly righteous..." for partial success.

**why backup is correct**:
- the criteria (usecase.9) explicitly defines partial success as a state
- the blueprint did not assign a header to this state
- the implementation fills the gap with a third header
- this is additive, not contradictory
- repair would mean a degraded UX with no header for partial state

### section names → backup

the blueprint said "files" as a single section with success/error per file.

the implementation uses four sections: settled, skipped, failed, not found.

**why backup is correct**:
- the four categories are mutually exclusive outcomes
- separation clarifies what happened to each file
- the blueprint's single "files" section would require inline disambiguation
- this is a presentation improvement, not a semantic change
- repair would collapse four clear categories into one mixed list

### exit code for partial → backup

the blueprint said "0 if all settled or partial success".

the implementation says "0 if all settled, 1 if any failures or not-found".

**why backup is correct**:
- the criteria (usecase.9) says "exit code reflects partial success or failure"
- "reflects failure" implies non-zero exit
- the blueprint was ambiguous between "partial success = success" and "partial success = partial"
- the implementation chose "partial = non-zero" which aligns with unix convention (0 = clean exit)
- repair would mean exit 0 on failure, which violates unix semantics

### glob expansion method → backup

the blueprint said "use bash glob on paths".

the implementation uses `git ls-files "$pattern"`.

**why backup is correct**:
- bash glob expands against filesystem (includes untracked, gitignored files)
- git ls-files expands against index (only tracked files)
- in a rebase context, all conflict files are tracked — the index is authoritative
- git ls-files is more correct for this domain
- repair would allow gitignored files as valid input

### test renumber and count → backup

tests were renumbered and count increased from 10 to 11.

**why backup is correct**:
- all blueprint scenarios are still covered
- case8 adds coverage for the WHOS-not-empty guard
- numbers are implementation detail, not contract
- repair would delete valid test coverage
