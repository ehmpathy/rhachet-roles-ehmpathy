## vision adherance: why it holds

### the core promise

**vision stated**: "take settles conflicts by selection of ours or theirs version"

**why the implementation delivers this**:

1. **line 183**: `git checkout "--$WHOS" "$file"` — this is the exact git primitive that replaces a file's content with the specified version during a rebase conflict.

2. **line 184**: `git add "$file"` — this stages the settled file, which is required for `git rebase --continue` to proceed.

the implementation is not a workaround or approximation. it uses the canonical git method for conflict settlement.

### the permission bypass promise

**vision stated**: "`git checkout --theirs` is destructive — this skill scopes to rebase conflicts only"

**why the implementation delivers this**:

1. **line 81**: `is_rebase_in_progress()` — guards against use outside rebase context. a clone cannot use `take` to checkout files when no rebase is active.

2. **lines 154-158**: files must be in `CONFLICT_FILES` (from `get_conflict_files()`) — guards against checkout of non-conflict files.

the skill wraps a destructive command (`git checkout`) with guards that restrict it to the safe context (rebase + conflict files only).

### the output format promise

**vision stated**: turtle header with tree format, ends with "done"

**why the implementation delivers this**:

1. **lines 204-209**: `print_turtle_header` selects vibe based on success/failure
2. **line 211**: `print_tree_start "git.branch.rebase take"` — exact match to other skills
3. **lines 215-268**: tree branches for settled/skipped/failed/not-found
4. **lines 271-276**: terminates with `done` or `done (with errors)`

the output structure matches the pattern in `git.branch.rebase.begin.sh` and `git.branch.rebase.continue.sh`.

## criteria adherance: why each usecase holds

### usecase.1 & usecase.2: take single file

**criteria**: file replaced with selected version, file staged, output shows success

**why it holds**:

- `git checkout --ours/--theirs <file>` replaces the file content. this is git's built-in behavior, not custom logic.
- `git add <file>` stages it. again, git primitive.
- the implementation wraps these primitives with error tracking and output formatting.

there is no custom file manipulation that could diverge from git semantics.

### usecase.3: glob pattern

**criteria**: all matched files in conflict are settled

**why it holds**:

- **line 141**: `git ls-files "$pattern"` — expands globs via git's own pattern matcher, which respects `.gitignore` and tracks only versioned files.
- **lines 154-158**: each match is checked against `CONFLICT_FILES` — only conflict files proceed to settlement.

the implementation cannot settle non-conflict files because of the intersection filter.

### usecase.4: dot for all

**criteria**: all conflict files settled

**why it holds**:

- **lines 132-138**: when pattern is ".", iteration reads directly from `$CONFLICT_FILES`
- no glob expansion needed — the source is already the authoritative list

the implementation uses the direct source (`get_conflict_files`) rather than a filter, which is both simpler and more correct.

### usecase.5: no rebase in progress

**criteria**: error with non-zero exit

**why it holds**:

- **line 81**: `is_rebase_in_progress()` returns false when neither `.git/rebase-merge` nor `.git/rebase-apply` extant
- **lines 82-86**: error message printed, `exit 1`

the guard runs before any mutation. no partial state possible.

### usecase.6: file not in conflict

**criteria**: error that file has no conflicts

**why it holds**:

- **lines 146-147**: if file extant but not in `CONFLICT_FILES`, added to `FILES_SKIPPED`
- **lines 229-240**: skipped files appear in output with "not in conflict" notice

the implementation chose "skipped" rather than "error" because a glob might match both conflict and non-conflict files. this is a UX improvement: inform without fail-fast on mixed input.

### usecase.7: invalid --whos value

**criteria**: error with non-zero exit

**why it holds**:

- **lines 96-101**: explicit string comparison against "ours" and "theirs"
- any other value triggers error and exit 1

no regex or partial matching. the guard is strict.

### usecase.8: no paths given

**criteria**: error with non-zero exit

**why it holds**:

- **lines 104-109**: check `${#PATHS[@]} -eq 0`
- empty array triggers error and exit 1

the guard runs before any file system operations.

### usecase.9: partial success

**criteria**: settled files shown, errors shown, exit reflects failure

**why it holds**:

- **lines 179-192**: each file tracked separately in `FILES_SUCCESS` or `FILES_FAILED`
- **lines 215-268**: each category has its own output section
- **lines 271-273**: if ANY file failed or not found, exit 1

the implementation does not short-circuit on first error. it processes all files, then reports.

### usecase.10: output format

**criteria**: turtle header, tree with whos, tree with files, ends with done

**why it holds**:

- the implementation uses shared output operations (`print_turtle_header`, `print_tree_start`) from the same source as other git.branch.rebase skills
- the tree structure is manually constructed to match the established pattern

no abstraction drift — same operations, same format.

## blueprint adherance: why codepaths match

### shared operations reuse

**blueprint stated**: reuse `is_rebase_in_progress()`, `get_conflict_files()` from operations.sh

**why this holds**:

- **line 28**: `source "$SKILL_DIR/git.branch.rebase.operations.sh"`
- **line 81**: calls `is_rebase_in_progress`
- **line 114**: calls `get_conflict_files`

the implementation does not duplicate these functions. it sources and calls them.

### glob expansion via git

**blueprint stated**: "use bash glob on paths, then intersect with conflict files"

**why this holds**:

- **line 141**: `git ls-files "$pattern"` — uses git's glob expansion, which is more robust than bash's native `shopt -s globstar`
- **lines 154-158**: intersection check via grep

the implementation chose `git ls-files` over bash glob. this is a safer choice: `git ls-files` only returns tracked files and respects repo boundaries.

### best-effort semantics

**blueprint stated**: "settle what can be settled, report errors for failures"

**why this holds**:

- **lines 182-192**: for-loop processes each file independently
- errors are tracked in `FILES_FAILED`, not used to abort
- exit code is computed AFTER all files processed

no `set -e` trap aborts the loop. errors are collected, not fatal.

## lessons for future implementations

1. **guard placement**: run all guards before any mutation. the implementation does this perfectly — no partial state on failure.

2. **best-effort loops**: track success/failure per item, then report at end. do not fail-fast in loops where partial success is acceptable.

3. **reuse operations**: source shared operations rather than copy. the implementation sources from `git.branch.rebase.operations.sh` rather than duplicating `is_rebase_in_progress()`.

4. **git primitives**: use git's built-in commands (`checkout`, `add`, `ls-files`) rather than custom file manipulation. the implementation is thin wrapper, not replacement.
