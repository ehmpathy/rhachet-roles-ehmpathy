# review: has-questioned-assumptions

## reviewed: git.branch.rebase.take blueprint

### assumption 1: ours/theirs semantics in rebase

**assumed:** in rebase, `ours` = base branch, `theirs` = commit in replay

**evidence:** confirmed via integration tests. test showed:
- `take --whos theirs` → file has feature branch content
- `take --whos ours` → file has main branch content

this matches git's documented behavior. tests prove correctness.

### assumption 2: git checkout auto-stages

**assumed:** `git checkout --theirs` does NOT auto-stage files

**evidence:** confirmed in implementation. explicit `git add` required after checkout.
tests verify files are staged after take operation.

### assumption 3: rebase detection via .git directories

**assumed:** rebase in progress detected via `.git/rebase-merge` or `.git/rebase-apply`

**evidence:** reused from extant `is_rebase_in_progress()` function in operations.sh.
this pattern works for both interactive and non-interactive rebases.
tests confirm detection works.

### assumption 4: glob expansion intersects with conflict files

**assumed:** need to intersect user-provided globs with actual conflict files

**evidence:** correct — user may specify `*.ts` but only some .ts files are in conflict.
implementation filters to only conflict files, skips non-conflict matches with notice.

### could a simpler approach work?

**evaluated:** wrap `git checkout --ours/--theirs` directly

**rejected:** cannot add safe permission for raw `git checkout` — too destructive.
subcommand scope is minimal overhead for safety guarantee.

### issues found: none

all assumptions validated via tests and extant patterns.
