## the deeper question

in pass two, i noted that take.sh uses raw echo for tree lines instead of output.sh functions.

but i moved past it too quickly. let me trace the exact gap.

---

## trace: what take.sh outputs

```
🐢 righteous!

🐚 git.branch.rebase take
   ├─ whos: theirs
   ├─ settled
   │  ├─ pnpm-lock.yaml ✓
   │  └─ yarn.lock ✓
   └─ done
```

three unique patterns:
1. `├─ settled` — a branch with children below it
2. `│  ├─ file ✓` — nested file with status marker
3. dynamic last-item detection (└─ vs ├─)

---

## trace: what output.sh provides

```bash
print_tree_branch "label" [is_last]     # outputs: ├─ label or └─ label
print_tree_leaf "key" "value" [prefix]  # outputs: ├─ key: value
print_nested_leaf "key" "value"         # outputs: └─ key: value
```

**gap**: no function for nested file lists with status markers.

- `print_tree_leaf` outputs `key: value` format (colon separator)
- take.sh needs `file ✓` format (space separator, status at end)
- take.sh needs dynamic `│  ` prefix for nested children

---

## should take.sh add functions to output.sh?

**no.** this would be premature abstraction.

reasons:
1. **single use**: only take.sh needs nested file lists with status markers
2. **rule.prefer.wet-over-dry**: wait for 3+ usages before extraction
3. **complexity**: a generic function would need parameters for prefix, separator, status position
4. **clarity**: inline echo is readable and self-evident in this context

---

## verification: no other skill uses this pattern

searched for file list output with status markers:
- cpsafe.sh: outputs `from → dest` (different pattern)
- mvsafe.sh: outputs `from → dest` (different pattern)
- rmsafe.sh: outputs file paths only (no status markers)
- git.commit.set.sh: uses print_tree_leaf for key: value pairs

no extant skill outputs nested file lists with per-file status markers.

---

## final verdict

**no duplicate mechanisms.**

the inline echo pattern in take.sh is acceptable because:
1. the nested file list pattern is unique to this skill
2. extant output.sh functions don't support this pattern
3. extraction would be premature abstraction (single use case)
4. the code is readable and follows the visual tree structure

if future skills need similar output, that's when to extract a shared function.
