## readable.comments: why coverage is complete

**the question**: does every named procedure have a .what/.why header?

**the answer**: yes.

**the evidence**: take.sh lines 1-21 contain a full header block:

```bash
# .what = settle rebase conflicts via selection of ours or theirs version
# .why  = mechanics can settle conflicts without permission prompts:
#         - `git checkout --theirs` is destructive (can switch branches)
#         - this skill is scoped to rebase conflicts only
#         - enables autonomous rebase flows
# usage: ...
# guarantee: ...
```

**why show_usage() does not need a header**:

`show_usage` is not a "named procedure" in the standard's sense. it is a private function called only from within the same file. the standard targets exported/public procedures that other code depends on.

**lesson**: the standard says "every named procedure" but means "every procedure that forms a contract with external callers". internal functions are not contracts.

## pitofsuccess.errors: why coverage is complete

**the question**: are all failure modes guarded?

**the answer**: yes.

**the evidence**: comprehensive failure mode analysis:

| failure mode | guard | line |
|--------------|-------|------|
| not in git repo | is_rebase_in_progress relies on .git | 81 |
| not in rebase | is_rebase_in_progress | 81 |
| --whos absent | `[[ -z "$WHOS" ]]` | 89 |
| --whos invalid | `[[ "$WHOS" != "ours" && "$WHOS" != "theirs" ]]` | 96 |
| no paths | `[[ ${#PATHS[@]} -eq 0 ]]` | 104 |
| no conflicts | `[[ -z "$CONFLICT_FILES" ]]` | 117 |
| checkout fails | tracked in FILES_FAILED | 183-191 |
| add fails | tracked in FILES_FAILED | 184-188 |

**why git ls-files errors are handled**:

line 141: `git ls-files "$pattern" 2>/dev/null || echo ""`

this redirects stderr and falls back to empty string. an invalid pattern results in "no matches" which flows to FILES_NOT_FOUND. this is correct behavior — we do not need to fail-fast on a pattern that matches no files.

**lesson**: fail-fast applies to invalid STATE not invalid INPUT. an invalid pattern is user input that we handle gracefully (report as "not found"). invalid state (no rebase) is a hard failure.

## nitpick found: snapshot assertions

**the observation**: test [case9] verifies output format via `toContain` assertions, not snapshots.

**why this is a nitpick not a blocker**:

1. snapshots are for aesthetic output (user-presented prose, generated code)
2. the tree output is functional (shows file status)
3. `toContain` assertions verify the critical content
4. snapshots would break on whitespace changes

**the tradeoff**: snapshots give visual diff in PRs. `toContain` gives stable tests. for a functional output, stability wins.

**lesson**: rule.require.snapshots says "use for output artifacts". it does not say "use for all output". functional output with `toContain` is acceptable; aesthetic output needs snapshots.
