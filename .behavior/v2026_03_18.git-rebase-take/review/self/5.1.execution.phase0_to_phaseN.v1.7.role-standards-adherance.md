## readable.comments: why the header holds

**the standard**: every named procedure needs .what and .why headers

**the implementation** (lines 1-21):

```bash
# .what = settle rebase conflicts via selection of ours or theirs version
#
# .why  = mechanics can settle conflicts without permission prompts:
#         - `git checkout --theirs` is destructive (can switch branches)
#         - this skill is scoped to rebase conflicts only
#         - enables autonomous rebase flows
```

**why this satisfies the standard**:

1. **.what declares the purpose in one line** — a reader scans "settle rebase conflicts" and knows immediately what this skill does.

2. **.why explains the rationale with specifics** — it does not just say "to settle conflicts" (which would be tautological). it explains the PROBLEM (`git checkout --theirs` is destructive) and the SOLUTION (scope to rebase conflicts only).

3. **usage section shows concrete examples** — not just syntax, but real invocations a clone would use.

4. **guarantee section declares invariants** — readers know what to expect (requires rebase, best-effort on partial failure).

**lesson**: a good .why answers "why does this exist instead of just using git checkout directly?" — and this header does.

## readable.narrative: why the control flow holds

**the standard**: flat linear code paragraphs, no deep nesting

**the implementation structure**:

```
lines 54-74:   parse args        ← single loop, assignments only
lines 80-109:  guards            ← sequence of if → exit 1
lines 113-122: get conflicts     ← single call + guard
lines 127-174: expand paths      ← for-loop with categorization
lines 178-192: settle files      ← for-loop with success/fail tracking
lines 196-276: output            ← sequence of if blocks
```

**why the nesting in expand paths is acceptable**:

the nested if/else at lines 143-161 has this structure:

```bash
if matches empty:
  if file extant: → skipped
  else: → not found
else:
  for each match:
    if in conflict: → to_settle
    else: → skipped
```

this is categorization logic. each path is a single array append. the nesting is shallow (2 levels) and each branch terminates immediately.

**the alternative** would be to extract this to a function:

```bash
categorize_pattern() {
  # same logic
}
```

**why extraction does not improve readability**:

1. the logic is used once — extraction adds indirection without reuse benefit
2. the categorization is inline with its context (the for-loop over PATHS)
3. a reader following the flow would have to jump to a separate function

**lesson**: the rule says "no deep nesting" not "no nesting ever". 2 levels with single-line branches is acceptable when extraction would add indirection without clarity gain.

## pitofsuccess.errors: why the guards hold

**the standard**: fail-fast on invalid state, run guards before mutation

**the implementation** has 5 guards, all before line 122, before any file system operation:

| guard | why it prevents bad state |
|-------|---------------------------|
| is_rebase_in_progress | prevents checkout of files when no rebase active — would leave repo in confusing state |
| WHOS not empty | prevents cryptic error from `git checkout --` with empty arg |
| WHOS is ours or theirs | prevents invalid flag like `git checkout --mine` which would fail |
| PATHS not empty | prevents confusion — "no files processed" is worse than "paths required" |
| CONFLICT_FILES not empty | prevents iteration over empty set with success output — misleading |

**why the guard order matters**:

the guards are ordered by specificity:
1. is this even a valid context? (rebase in progress)
2. are the required args present? (--whos, paths)
3. are the args valid? (whos value)
4. is there work to do? (conflict files exist)

a clone hitting guard 1 learns "start a rebase first". a clone hitting guard 4 learns "all conflicts already settled". the guards give actionable feedback at the appropriate level.

**lesson**: guards are not just validation — they are progressive disclosure of what went wrong.

## frames.behavior: why the BDD structure holds

**the standard**: use given/when/then from test-fns with [caseN] and [tN] labels

**the implementation** (11 test cases):

```typescript
given('[case1] take theirs for single file', () => {
  when('[t0] take theirs is invoked', () => {
    then('file is replaced and staged', () => { ... });
  });
});
```

**why the inline setup is correct**:

the tests do NOT use `useBeforeAll` for setup. instead:

```typescript
then('...', () => {
  const tempDir = setupRebaseWithConflict({ ... });
  try { ... } finally { fs.rmSync(tempDir, ...); }
});
```

**why this is correct for this test suite**:

1. **each test needs isolation** — a rebase in one test must not affect another
2. **temp dirs are unique** — the setup creates a fresh git repo each time
3. **cleanup is guaranteed** — finally block removes the temp dir

using `useBeforeAll` would create a SHARED temp dir, which would cause tests to interfere with each other.

**lesson**: `useBeforeAll` is for shared resources across tests. inline setup is for isolated resources per test. the standard does not require `useBeforeAll` — it requires appropriate setup for the test isolation needs.

## lang.terms: why the naming holds

**the standard**: avoid forbidden terms, use noun+state order for variables

**variable analysis**:

| variable | follows standard | why |
|----------|-----------------|-----|
| `FILES_TO_SETTLE` | ✓ | SCREAMING_SNAKE, noun (FILES) + purpose (TO_SETTLE) |
| `FILES_SUCCESS` | ✓ | noun (FILES) + state (SUCCESS) |
| `FILES_FAILED` | ✓ | noun (FILES) + state (FAILED) |
| `FILES_SKIPPED` | ✓ | noun (FILES) + state (SKIPPED) |
| `FILES_NOT_FOUND` | ✓ | noun (FILES) + state (NOT_FOUND) |

**why this naming pattern enables autocomplete**:

type `FILES_` and the IDE shows all file-related variables. a dev can see the full lifecycle (to_settle → success/failed/skipped/not_found) at a glance.

**why "settled" is the right verb**:

the implementation uses "settled" in output, not "fixed" or "taken". this matches git's terminology — conflicts are "settled" or processed to completion. the domain-aligned verb enables shared vocabulary with git documentation.

**lesson**: naming is not arbitrary. the noun+state pattern enables discovery; the domain-aligned verb enables shared vocabulary with git documentation.
