## critical assumption: using git checkout vs git restore

**what the vision assumes:**
- use `git checkout --ours/--theirs`

**alternative:**
- `git restore --ours/--theirs` is the modern equivalent
- `git restore` was added to split checkout's many responsibilities
- `git restore --source=HEAD --staged --worktree <file>` is more explicit

**which should we use?**
- `checkout` is older, more widely known
- `restore` is newer, cleaner semantics
- for conflict settlement: `git checkout --ours/--theirs` works and is documented

**verdict**: ✅ holds — checkout is fine, restore is alternative if needed

## critical assumption: partial failure behavior

**what the vision ignores:**
- glob matches 5 files
- 3 have conflicts
- 2 don't

**current behavior (implicit):**
- "only settle conflict matches, warn about skipped"

**but what about:**
- 3 files have conflicts
- `take` succeeds on 2
- `take` fails on 1 (e.g., file doesn't exist)

**should we:**
- abort entire operation on first failure? (atomic)
- continue and report failures at end? (best-effort)

**atomic pros:**
- predictable: all-succeed-or-none
- easier to reason about
- matches database transaction semantics

**best-effort pros:**
- progress is saved
- user can fix failed file and retry
- matches how clone thinks: "take these ones, deal with rest"

**verdict**: 🟡 needs decision — i lean toward best-effort with clear error report

## new assumption found: default to all files when no paths given

**what the vision says:**
```
paths      optional   file paths or globs (default: all files in conflict)
```

**what the wish says:**
```
git.branch.rebase take --whos ours|theirs [paths|globs]
```

**question: did the wisher mean "default to all" or "require paths"?**

- wish syntax `[paths|globs]` — brackets suggest optional
- but optional could mean "omit to see help" not "omit to apply to all"

**what if we default to all?**
- clone runs `rhx git.branch.rebase take --whos theirs` without paths
- all conflicts get theirs version
- this could be destructive if clone meant to take only one file

**what if we require explicit paths?**
- clone must always specify: `--whos theirs .` for all
- safer: explicit intent
- `.` is short enough to not be burdensome

**verdict**: 🔴 issue found — default to all is risky

**fix**: require explicit paths. if no paths given, show help. `.` for all is explicit enough.

**verdict**: ✅ holds — plan mode adds complexity without value for this use case

## re-examine: the `.` shorthand

**first pass noted**: 🟡 questionable

**deeper look:**
- `.` in current directory = all files in cwd
- from repo root, `.` = all files in repo
- but from subdirectory, `.` = only that subdirectory

**is this problematic?**
- no — matches git's behavior
- `git add .` from subdir = only that subdir
- consistent semantics

**verdict**: ✅ now holds — `.` is consistent with git patterns
