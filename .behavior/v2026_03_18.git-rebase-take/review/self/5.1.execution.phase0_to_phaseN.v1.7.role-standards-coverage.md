## potential absent patterns

**question**: are there patterns that SHOULD be present but are ABSENT?

### error wrapping pattern

**the standard** (from rule.prefer.helpful-error-wrap): wrap errors with context

**check**: does take.sh wrap errors?

no. errors are tracked in arrays (`FILES_FAILED+=("$file (checkout failed)")`) but not wrapped with HelpfulError.

**is this a gap?**

no. bash does not have access to HelpfulError (that is a TypeScript library). the parenthetical suffix `(checkout failed)` provides context in the bash idiom.

### pitofsuccess.procedures idempotency

**the standard** (from rule.require.idempotent-procedures): procedures should be safe to retry

**check**: is `take` idempotent?

yes. the operation `take --whos theirs file.ts` run twice:
1. first run: checks out file, adds to stage
2. second run: file is already staged, checkout is no-op, add is no-op

git checkout and git add are idempotent. the skill inherits their idempotency.
