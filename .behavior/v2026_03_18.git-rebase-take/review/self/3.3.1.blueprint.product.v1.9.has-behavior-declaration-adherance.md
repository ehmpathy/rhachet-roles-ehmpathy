## the wish, as a human

> lets add a `git.branch.rebase take` operation
> which allows clones to
> `git.branch.rebase take --whos ours|theirs [paths|globs]`
> to, for example, `git.branch.rebase take --whos theirs pnpm-lock.yml`
>
> otherwise, in rebases, they issue lots of `git checkout --theirs xyz` operations
> which we cant easily add permissions.jsonc onto,
> cause git checkout itself can be destructive for a workzone

the pain is clear:
1. clones need to settle conflicts in rebases
2. `git checkout --theirs` is the standard way
3. but `git checkout` is dangerous and unpermissionable
4. so clones get blocked, humans get interrupted

the solution must:
- provide a scoped, safe alternative to `git checkout --ours/--theirs`
- be permissionable (single prefix for permissions.jsonc)
- fit the extant rebase flow (begin → conflict → settle → continue)

## does the implementation solve the pain?

**test 1: is it permissionable?**

opened `.claude/settings.json` (via permissions.jsonc sync):
```json
"Bash(rhx git.branch.rebase take:*)"
```

yes. single prefix. pre-approved. no per-file prompts.

**test 2: is it scoped and safe?**

the implementation:
- only works in a rebase (guard)
- only operates on files in conflict (intersection)
- only does checkout + add (no branch switch, no hard reset)

yes. scoped to the exact safe operation.

**test 3: does it fit the rebase flow?**

typical flow with take:
```
rhx git.branch.rebase begin --mode apply
# conflict on pnpm-lock.yaml
rhx git.branch.rebase take --whos theirs pnpm-lock.yaml
rhx git.branch.rebase continue
# conflict on pnpm-lock.yaml again
rhx git.branch.rebase take --whos theirs pnpm-lock.yaml
rhx git.branch.rebase continue
# done
```

yes. take slots naturally between conflict detection and continue.

## what could go wrong?

i paused here. thought about edge cases the spec might not have considered.

**edge 1: ours/theirs confusion in rebase**

in a rebase, the semantics are inverted:
- `ours` = the base branch (origin/main)
- `theirs` = the commit in replay (feature branch)

this is counterintuitive. did the implementation handle it?

ran the tests:
- case1: `--whos theirs` → file has "feature" content ✓
- case2: `--whos ours` → file has "main" content ✓

the tests prove the semantics are correct. the vision document explains this.

**edge 2: glob matches non-conflict files**

if user runs `--whos theirs *.ts` and some .ts files are not in conflict:
- should they be silently skipped?
- should there be an error?

the implementation: filters to conflict files, skips non-conflict matches.

the criteria says: "non-conflict files are skipped with notice"

this is correct behavior.

**edge 3: all files fail**

if every file in the glob fails checkout:
- should exit be 0 or non-zero?

the implementation: exit non-zero if any failure.

correct — total failure should not look like success.
