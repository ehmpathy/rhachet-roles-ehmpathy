## the deeper question

## assumption 1: ours/theirs semantics

**what the code does**:
```bash
git checkout "--$WHOS" "$file"
```

**what this assumes**: git's ours/theirs semantics are stable.

**is this a backcompat concern?**: no. this is git's public API. git maintains backwards compat for us.

**why it holds**: we depend on git, not on internal git behavior.

## assumption 2: rebase state directories

**what the code does** (from operations.sh):
```bash
[[ -d "$GIT_DIR/rebase-merge" ]] || [[ -d "$GIT_DIR/rebase-apply" ]]
```

**what this assumes**: git stores rebase state in these directories.

**is this a backcompat concern?**: this could be. git could change internal state storage.

**trace**: these directories have been stable since git 2.0 (2014). git's own documentation references them.

**why it holds**: this is established git behavior, not a volatile target.

**what this assumes**: `--diff-filter=U` returns unmerged (conflicted) files.

**why it holds**: `U` filter is part of git's public API.
