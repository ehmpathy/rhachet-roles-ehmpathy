# research: audience reflection

## who is the audience?

### primary: mechanics (software engineers)

the direct users who invoke the skill to add or upgrade packages.

- **context**: work in production codebases with security requirements
- **constraints**: need audit trail for compliance, code review, and future maintenance
- **expertise**: familiar with npm/pnpm, comfortable with cli tools

### secondary: future maintainers

engineers who inherit the codebase and wonder "why is this package here?"

- **context**: onboard to new codebase, debug dependency issues
- **constraints**: limited time, need quick answers
- **expertise**: variable — from junior to senior

### tertiary: security/compliance reviewers

those who audit dependency decisions for security or regulatory reasons.

- **context**: review package additions for vulnerabilities, license compliance
- **constraints**: need evidence of due diligence
- **expertise**: security-focused, may not be hands-on engineers

## why do they care?

### mechanics (primary)

| motivation | detail |
|------------|--------|
| **pain relief** | "why did we add lodash?" — no more archaeology |
| **goal enable** | faster code review — reason is documented upfront |
| **frustration eliminate** | vulnerable packages blocked before they enter codebase |

### future maintainers (secondary)

| motivation | detail |
|------------|--------|
| **pain relief** | reason files answer "why is this here?" instantly |
| **goal enable** | evaluate if package still needed or can be removed |
| **frustration eliminate** | no need to grep git history or ask teammates |

### security/compliance (tertiary)

| motivation | detail |
|------------|--------|
| **pain relief** | audit trail proves security was checked |
| **goal enable** | compliance reports can cite reason files |
| **frustration eliminate** | no need to reconstruct decision rationale |

| future maintainers | medium | low | onboard time reduction |
| security/compliance | high | low | audit pass/fail |

## what do they care about?

### mechanics

- **success criteria**: install works, audit passes, reason persisted
- **acceptable tradeoffs**: extra friction (4 flags) for safety
- **reject tradeoffs**: broken install, unclear errors, lost reason files
- **delight**: clean turtle vibes output, clear success/failure
- **disappoint**: cryptic errors, silent failures, unexpected behavior

### future maintainers

- **success criteria**: reason file answers "why?" without context
- **acceptable tradeoffs**: terse reason is ok if informative
- **reject tradeoffs**: reason file empty or vague ("we needed it")
- **delight**: 3-section reason (why, complexity, alternatives)
- **disappoint**: absent reason file, unclear justification

### security/compliance

- **success criteria**: evidence of pre-install audit
- **acceptable tradeoffs**: some manual review still needed
- **reject tradeoffs**: vulnerable package installed without block
- **delight**: audit result in stdout, clear pass/fail
- **disappoint**: silent install of vulnerable package

## how will they experience it?

### mechanic journey

```
t0: mechanic wants to add zod
t1: invokes `rhx set.package.install --package zod --at 3.22.4 --for prod --reason @stdin`
t2: types reason via heredoc (or inline)
t3: skill validates flags
t4: skill checks security (bulk advisory)
    └─ if fail → blocked, mechanic sees vulnerabilities
t5: skill persists reason to .route/
t6: skill installs via pnpm/npm
t7: mechanic sees turtle vibes success output
t8: mechanic proceeds with development
```

### friction points

| point | friction | mitigation |
|-------|----------|------------|
| 4 required flags | more keystrokes than `npm install` | clear error messages guide completion |
| reason required | forces thought | 3-section template guides content |
| security check | adds latency (~1-2s) | bulk advisory is fast, no temp dir |
| @stdin heredoc | unfamiliar syntax | inline string fallback available |

### smooth experience

- clear, immediate feedback on flag validation
- fast security check (bulk advisory, not temp dir)
- turtle vibes output confirms success
- reason file location shown in output
- no ambiguous states — pass or fail clearly
