# research: rootcause reflection

## what is the symptom?

describe the observable problem or desired effect.

- **what is broken or absent?**
  - no audit trail for package additions — "why is this here?" unanswerable
  - no pre-install security check — vulnerable packages enter codebase
  - no version pin enforcement — @latest drift causes silent breaks

- **how does it manifest?**
  - 6 months later, nobody knows why lodash is in package.json
  - vulnerable packages discovered after deploy, not before
  - builds break due to transitive dependency changes

- **when does it occur?**
  - every `npm install $pkg` without ceremony
  - every code review that glosses over package.json changes
  - every security audit that reveals surprise dependencies

---

## 5 whys tree

### branch 1: no audit trail

1. why is there no audit trail? → mechanics use raw `npm install`
2. why do they use raw npm install? → no friction-free alternative that captures intent
3. why no alternative? → npm/pnpm don't require justification
4. why don't they require justification? → designed for speed, not compliance
5. why speed over compliance? → **package managers optimize for developer experience, not organizational memory**

### branch 2: vulnerable packages enter codebase

1. why do vulnerable packages enter? → no pre-install check
2. why no pre-install check? → npm audit runs after install, not before
3. why after? → npm needs lockfile to audit, lockfile created on install
4. why this order? → **npm audit architecture assumes dependencies already installed**
5. → bulk advisory endpoint exists but not integrated into install workflow

### branch 3: version drift

1. why do versions drift? → mechanics use `npm install foo` without version
2. why no version? → npm defaults to @latest when omitted
3. why is @latest default? → convenience for quick starts
4. why is drift a problem? → **transitive deps may change behavior between installs**
5. → lockfiles help but don't prevent the initial version ambiguity

## root cause summary

| layer | issue | is root cause? |
|-------|-------|----------------|
| 1 | mechanics use raw npm install | no — symptom |
| 2 | npm/pnpm lack justification flow | no — constraint |
| 3 | npm audit runs post-install | no — architecture |
| 4 | bulk advisory endpoint not integrated | no — absent integration |
| 5 | **package managers optimize for speed, not compliance** | **yes** |

secondary root cause: **npm audit architecture assumes installed deps**. bulk advisory endpoint exists to solve this, but remains obscure.

---

## recommendation

### address root cause

create a wrapper skill that:
1. requires explicit version (--at) to eliminate drift
2. checks bulk advisory before install to gate security
3. requires reason (--reason) to create audit trail
4. integrates with extant package managers (pnpm/npm) rather than replace

this addresses the root cause by adding compliance requirements onto tools that optimize for speed.

### why not address symptom

symptom treatment would be: "remind mechanics to document installs" or "run npm audit in CI". these are:
- easily bypassed (reminder ignored)
- post-hoc (CI catches after install, not before)
- no audit trail (CI blocks but doesn't capture "why")

root cause fix (the skill) is proactive and captures intent at decision time.
