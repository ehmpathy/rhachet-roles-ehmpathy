# vision: set.package.install + set.package.upgrade

## the outcome world

### day-in-the-life

mechanic needs to add `zod` for runtime validation. instead of blindly running `npm install zod`, they invoke:

```sh
rhx set.package.install \
  --package zod \
  --at 3.22.4 \
  --for prod \
  --reason @stdin <<'EOF'
## why this package?

zod provides runtime type validation with typescript inference.

## complexity abstracted

without zod, we'd manually write validation logic for every domain object input.
this abstracts ~50-100 lines per object into a single schema declaration.

## alternatives considered

- joi: older, no typescript inference, heavier
- yup: similar but less typescript-native
- io-ts: fp-ts dependency, steeper learning curve

zod chosen: modern, typescript-first, zero dependencies, active maintenance.
EOF
```

or inline for simple cases:

```sh
rhx set.package.install \
  --package uuid-fns \
  --at 1.0.0 \
  --for prod \
  --reason "uuid generation, replaces custom impl, lighter than uuid package"
```

the skill:
1. checks security via native `pnpm audit` / `npm audit` — fails fast if vulnerable
2. persists the reason to `.route/v2026_03_14.package.install/3.reason.for_zod.v1.i1.md`
3. installs via `pnpm add zod@3.22.4` (or npm if package-lock.json extant)

### before/after contrast

| before | after |
|--------|-------|
| mechanic runs `npm install` directly | mechanic uses `set.package.install` |
| no security check | security check required |
| no documentation of why | reason persisted in .route/ |
| version often omitted (@latest drift) | explicit version required |
| prod vs dev often wrong | --for prod\|prep enforced |

### the "aha" moment

six months later, someone asks "why do we have lodash?" — the answer is in `.route/v2025_09_12.package.install/3.reason.for_lodash.v1.i1.md`. no archaeology needed. no "i think someone added it for..." guesses.

## user experience

### usecases

1. **add production dependency** — install runtime code
2. **add development dependency** — install build/test tools
3. **audit trail** — understand why each package exists

### set.package.upgrade (in scope)

companion skill for version changes:
- `rhx set.package.upgrade --package zod --to 3.23.0`
- `rhx set.package.upgrade --package zod --to @latest`
- only checks: (1) security audit passes, (2) package already installed
- no new reason required — original install reason still applies
- allows downgrade to earlier version if later versions are broken

### contract inputs: set.package.install

| flag | required | values | purpose |
|------|----------|--------|---------|
| `--package` | yes | package name | what to install |
| `--at` | yes | version or @latest | pin version |
| `--for` | yes | prod \| prep | dependency type |
| `--reason` | yes | @stdin or `"inline text"` | justification (skill writes to .route/) |

### contract inputs: set.package.upgrade

| flag | required | values | purpose |
|------|----------|--------|---------|
| `--package` | yes | package name | what to upgrade |
| `--to` | yes | version or @latest | target version (can be lower) |

### contract outputs

1. security check result (pass/fail)
2. reason file at `.route/v{isodate}.package.install/3.reason.for_{package}.v1.i1.md`
3. updated package.json + lockfile

### timeline


## mental model

### how users describe it

> "it's like npm install but it makes you think first and leaves a paper trail"

### analogies

- **building permit**: you can't just add a room — explain why, get inspected
- **prescription**: doctor must document why this medicine, not just prescribe
- **git commit message**: you can't commit without explaining the change

### terms

| user says | we say |
|-----------|--------|
| "install" | set.package.install |
| "save" / "save-dev" | --for prod \| prep |
| "version" | --at |
| "why" | --reason |

## evaluation

### how well does it solve the goals?

| goal | solved? |
|------|---------|
| prevent vulnerable packages | yes — security check gate |
| document decisions | yes — reason persisted |
| enforce explicit versions | yes — --at required |
| symmetric with get.package.docs | yes — set.* naming |

### pros

- security-first: no install without check
- traceable: every package has documented reason
- deterministic: explicit version, no @latest drift
- pit of success: required flags prevent mistakes

### cons

- friction: more steps than raw npm install
- new workflow: different from familiar `npm install` pattern
- audit latency: TBD based on method chosen (see "undecided" section)

### edgecases & pit of success

| edgecase | handled |
|----------|---------|
| @latest passed as version | allowed — skill expands to actual version before install |
| npm registry unreachable | fail closed — no install without clean audit |
| reason too short | enforce 3 sections per wisher's questions |
| package already installed | use `set.package.upgrade` instead |

## open questions & assumptions

### [answered] — via self-review

| question | resolution |
|----------|------------|
| fail behavior on api error | fail closed — no install without clean check |
| reason input mechanism | @stdin or inline `"text"` — both write to .route/ |
| reason structure enforcement | enforce 3 sections per wisher's 3 questions |
| @latest behavior | expand to actual version, record in reason file |
| pnpm/npm detection | check for package-lock.json per wisher |
| --package flag syntax | named flag per rule.forbid.positional-args |
| sequence order | validate flags → security check → write reason → install |

### [research] — external research phase

| question | research needed |
|----------|-----------------|
| native audit without temp dir | can we pre-audit a package without temp dir install? (e.g., direct npm registry api, `npq` tool, or other) |

### [wisher] — answered by wisher

| question | wisher's answer |
|----------|-----------------|
| extant package behavior | separate skill `set.package.upgrade` with `--to @latest\|$semver`. only checks: (1) audit passes, (2) package already installed. no new reason required. |
| prod\|prep terminology | "prep" is intentional. wisher prefers it over "dev". |

### assumptions (validated)

1. .route/ directory pattern is acceptable — wisher explicit
2. friction is intentional — wisher wants deliberate installs
3. single package per call — implied by singular path in wish

---

## what is awkward?

### feels off

- **--for prod|prep**: "prep" is non-standard (npm/pnpm use "dev") but wisher intentionally prefers "prep"

### fights mental model

- users expect `npm install foo` — this requires 4 flags minimum
- "set" prefix feels database-y, not package-manager-y (but matches get.package.docs)

### uncomfortable tradeoffs

- **friction vs safety**: every install requires reason + security check. worth it for audit trail, but slows down exploration

### undecided

- **audit method**: temp dir install works but adds ~2-5s. need to research if lighter option exists (direct registry api? `npq`? other?). if free/simple alternative found, prefer that.

---

## summary

this skill transforms package installation from an impulsive `npm install` into a deliberate, documented, security-checked decision. the tradeoff is friction for traceability — appropriate for production codebases where "why is this here?" matters.
