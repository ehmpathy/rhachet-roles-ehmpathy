# research: external product access

## summary

this research identifies remote access requirements for `set.package.install` and `set.package.upgrade` skills. key findings:

1. **npm registry api** — packages can be queried via `https://registry.npmjs.org/{package}` without authentication
2. **npm bulk advisory endpoint** — vulnerabilities can be checked via `POST /-/npm/v1/security/advisories/bulk` without a temp dir install
3. **deps.dev api** — google's free vulnerability database requires no authentication
4. **osv.dev api** — open source vulnerabilities database, also free and unauthenticated

the bulk advisory endpoint is the lightest option for pre-install security audit.

---

## remote repositories required

### 1. npm registry

**purpose**: fetch package metadata, version info, latest version

**interface**: rest api at `https://registry.npmjs.org`

**authentication**: none required for public packages

### 2. npm advisory database

**purpose**: check package vulnerabilities before install

**interface**: `POST https://registry.npmjs.org/-/npm/v1/security/advisories/bulk`

**authentication**: none required

### 3. local filesystem

**purpose**: write reason files to `.route/`, detect `package-lock.json`

**interface**: standard filesystem operations

## convergence signals

### pre-install audit is possible
**sources [3], [9], [19], [22]** all confirm the bulk advisory endpoint accepts package+version without a full install.

### fail-closed is correct
**sources [12], [13], [23]** agree that security checks should block, not warn and proceed.

## conflict detection

### none found
all sources agree on fundamental approaches:
- registry api for metadata
- bulk advisory for vulnerabilities
- fail-closed on security issues

## anti-patterns

### [13] warns: do not auto-fix
`npm audit fix` can break applications by bumped transitive dependencies. the skill should block and report, not auto-remediate.

### [11] warns: do not abuse registry
no explicit rate limits, but excessive requests are unacceptable. single request per install is fine.

## implementation notes

### bulk advisory request format
```json
{
  "zod": ["3.22.4"],
  "lodash": ["4.17.21"]
}
```

### bulk advisory response format
```json
{
  "zod": [],
  "lodash": [
    {
      "id": 1234,
      "severity": "high",
      "title": "Prototype Pollution",
      "url": "https://npmjs.com/advisories/1234"
    }
  ]
}
```

empty array = no vulnerabilities = safe to install.

---

## recommendation

use npm bulk advisory endpoint for pre-install security audit:
1. no temp dir required
2. no external api dependency (snyk, aikido)
3. free and unauthenticated
4. same data source as `npm audit`

fallback order if primary fails:
1. npm bulk advisory (primary)
2. deps.dev api (fallback)
3. fail-closed (if both unreachable)
