# research: premortem reflection

## premortem: imagine we shipped and it failed

project forward: we shipped this, and it failed badly. why?

### most plausible failure scenarios

1. **npm bulk advisory endpoint deprecation** — npm changes or removes the bulk advisory endpoint without notice. skill breaks silently or starts false-negative (allow vulnerable packages).

2. **friction causes abandonment** — mechanics bypass the skill entirely, revert to raw `npm install` because 4 required flags feel tedious. audit trail benefit lost.

3. **reason files become graveyard** — reason files accumulate but nobody reads them. "why is this here?" still requires archaeology because file format too verbose or location forgotten.

---

## what obvious signs did we ignore?

look back from the imagined failure — what signs were present that we dismissed?

- npm audit endpoint undocumented in official docs — suggests internal API, subject to change
- "prep" terminology is non-standard — may confuse users who expect "dev"
- no extant skill uses 4 required flags — may set precedent that frustrates adoption
- .route/ directory pattern is new — may not be discoverable by future maintainers
- security checks add latency — impatient mechanics may skip the skill

---

## what assumptions could turn out wrong?

| assumption | what if wrong? | likelihood |
|------------|----------------|------------|
| bulk advisory endpoint stable | skill breaks, vulnerable packages allowed | medium |
| mechanics will accept friction | adoption fails, bypass via raw npm | medium |
| .route/ is discoverable | reason files ignored, benefit lost | low |
| pnpm is default in most repos | npm repos must pass package-lock.json detection every time | low |
| @latest expansion reliable | npm view fails, skill errors | low |
| reason file format is useful | too verbose or too terse for actual use | medium |

---

## what is the nightmare scenario?

**vulnerable package installed due to false-negative audit**

- bulk advisory endpoint returns `{}` (no vulnerabilities) due to API change or network issue
- skill interprets this as "safe" and proceeds with install
- vulnerable package with known CVE enters production codebase
- exploited in production, causes security incident
- audit trail shows skill "approved" the package
- skill becomes liability rather than safety net

### why this is worst

- defeats primary purpose (security gate)
- creates false confidence (mechanics trust the check)
- evidence (reason file) documents the failure
- recovery requires: identify all installed packages, re-audit manually, patch or remove

---

## what mitigations to consider?

| risk | mitigation | cost of mitigation |
|------|------------|--------------------|
| bulk advisory endpoint changes | monitor npm release notes; add fallback to `pnpm audit` / `npm audit` | medium — adds code path |
| friction causes abandonment | reduce required info; allow `--reason "terse text"` | low — already supported |
| false-negative audit | fail-closed on any unexpected response shape | low — already planned |
| reason files become graveyard | clear output shows file location; git hook reminder? | low |
| .route/ not discoverable | document in readme; add .route/readme.md | low |
| endpoint returns empty for unknown packages | verify package exists first via `npm view` | low — already needed for @latest |

---

## summary

primary risks: endpoint instability, friction-driven abandonment, false-negative audit. mitigations: fail-closed on unexpected responses, reduce friction where possible, verify package exists before audit. nightmare scenario (false-negative that leads to vulnerability) addressed by strict response validation.
