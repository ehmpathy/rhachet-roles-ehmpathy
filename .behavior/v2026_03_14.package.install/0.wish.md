wish =

we want to add a skill, similar to the get package docs readme skill

so that the mechanic can install packages

e.g., 

set.package.install (set prefix to be symmetric with the get.package.docs command)

which

1. requires flag of --for prod|prep (prod = production dependency, --prep = development dependency) and an --at flag for the specific version (@latest _is_ supported)

2. determinstically checks aikido.com or snyk or some good, free, security check library for whether that package is safe to use
    - fails fast if its not

3. requires the caller to explain why exactly we need a new package
    - how much complexity would it actually abstract away? 
    - could we avoid the need for that operation to begin with? and eliminate the need for that complexity?
    - which other packages were considered? why was this one decided to be the most modern, easiest to use, and with least dependencies?

we should require that it emits that explanation to

.route/v$isodate.package.install.$package/3.reason.v1.i1.md


---

then, if, and only if, this skill's executable detects

1. --for flag is set
2. --at flag is set (the specific version to use)
2. security check is clean
3. reason is declared

then go ahead and issue the `pnpm|npm install --save | --save-dev $package.name@version` command on their behalf
and choose pnpm by default, UNLESS there is explicitly a `package-lock.json` file in this repo, in which case unfortunately the repo uses npm still
