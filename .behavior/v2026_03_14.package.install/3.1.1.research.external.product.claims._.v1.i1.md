# research: external product claims

## summary

this research catalogs facts, assumptions, questions, and opinions from industry sources about npm package security, version management, and audit trails.

---

## claims by category

### assumptions [SUMP]

#### [SUMP-1] version pin provides reproducibility
**source**: [betterdev.blog - pin exact versions](https://betterdev.blog/pin-exact-dependency-versions/)
**date**: 2024
**credibility**: [practitioner]

> "Pinned dependencies means exact entries like 'foobar': '1.1.0' which means 'use only foobar version 1.1.0 and no other'."

**assumption**: exact versions guarantee reproducible builds across environments.

---

#### [SUMP-2] audit fatigue leads to ignored vulnerabilities
**source**: [aikido.dev - npm audit guide](https://www.aikido.dev/blog/npm-audit-guide)
**date**: 2025
**credibility**: [practitioner]

> "Audits need to be run manually or wired into scripts, and if you don't automate them, they pile up and create audit fatigue."

**assumption**: manual processes lead to neglect; automation is required.

---

#### [SUMP-3] 7-14 day delay prevents most supply chain attacks
**source**: [bastion.tech - npm supply chain 2026](https://bastion.tech/blog/npm-supply-chain-attacks-2026-saas-security-guide)
**date**: 2026
**credibility**: [practitioner]

> "A 7-14 day wait period before new packages or major updates would have prevented most 2025 attacks."

**assumption**: new package delay provides security buffer.

---

#### [SUMP-4] production build should exclude devdependencies
**source**: [betterstack - deps difference](https://betterstack.com/community/questions/nodejs-dependencies-devdependencies-difference/)
**date**: 2024
**credibility**: [practitioner]

> "When you deploy to production, you can run npm ci --production to install only dependencies, not devDependencies."

**assumption**: leaner production builds are safer and faster.

---

#### [SUMP-5] both package.json and lock file should be committed
**source**: [renovate docs](https://docs.renovatebot.com/dependency-pinning/)
**date**: 2024
**credibility**: [official]

> "Both the package.json file and the generated lock file should be committed to the repository."

**assumption**: version control of lock files is standard practice.

---

### questions [KHUE]

**date**: 2025
**credibility**: [practitioner]

> "npm audit catches known vulnerabilities but misses zero-days and malicious packages without CVEs."

---

#### [KHUE-2] how to handle transitive dependency vulnerabilities?

**credibility**: [practitioner]

---

#### [KHUE-3] should audit run before or after install?

**date**: 2024
**credibility**: [official]

> "npm audit automatically runs when you install a package."

**question**: is post-install audit sufficient, or should we pre-audit before any code enters the project?

---

#### [KHUE-4] what severity threshold should block install?

### opinions [OPIN]

#### [OPIN-1] npm audit fix is problematic
**source**: [cyberdesserts - npm audit fix not work](https://blog.cyberdesserts.com/npm-audit-fix-not-working/)
**date**: 2025
**credibility**: [blog]

> "npm audit fix can break your application by bumped transitive dependencies."

**opinion**: auto-fix is dangerous; manual review is safer.

---

#### [OPIN-2] pin dependencies even with lock file
**source**: [renovate docs](https://docs.renovatebot.com/dependency-pinning/)
**date**: 2024
**credibility**: [official]

> "The usual recommendation is to use a lock file regardless of whether you pin dependencies or not, and pin even if you have a lock file."

**opinion**: belt-and-suspenders approach provides extra safety.

---

#### [OPIN-3] CI should fail on high-severity vulnerabilities
**source**: [snyk blog - npm security best practices](https://snyk.io/blog/npm-security-best-practices/)
**date**: 2023
**credibility**: [practitioner]

> "Run npm audit before every deploy. Fail the build if critical vulnerabilities exist."

**opinion**: fail-fast in CI is preferable to runtime discovery.

---

#### [OPIN-4] disable scripts on install for safety
**source**: [owasp - npm security cheat sheet](https://cheatsheetseries.owasp.org/cheatsheets/NPM_Security_Cheat_Sheet.html)
**date**: 2024
**credibility**: [academic]

> "When you install packages make sure to add the --ignore-scripts suffix to disable the execution of any scripts by third-party packages."

**opinion**: postinstall scripts are a supply chain attack vector.

---

#### [OPIN-5] documentation prevents "why is this here?" questions
**source**: [medium - audited deps](https://medium.com/@ahmedrao609/i-audited-my-node-js-apps-dependencies-here-s-what-i-found-2f32a65b3aed)
**date**: 2024
**credibility**: [practitioner]

> "I audited my dependencies and found packages I didn't remember I added."

**opinion**: every package should have documented justification.

## convergence signals

### npm audit as first line of defense
**sources [FACT-1], [FACT-2], [OPIN-3]** converge: npm audit is standard and should run automatically, but may not be sufficient alone.

### version pin is standard practice
**sources [FACT-5], [SUMP-1], [OPIN-2]** converge: both lock files and exact version pins are recommended.

### documentation prevents future confusion
**sources [KHUE-5], [OPIN-5]** converge: audit trails capture decisions and prevent "why is this here?" questions.

---

## conflict detection

### severity threshold for block
**sources [OPIN-3]** suggest block only critical/high, while the vision implies block on any vulnerability. need to decide threshold.

### pre-install vs post-install audit
**[FACT-2]** states audit runs on install (post), but **[KHUE-3]** questions if pre-install is safer. vision implies pre-install via bulk advisory endpoint.

---

## anti-patterns

### [ANTI-1] do not use npm audit fix blindly
**sources [OPIN-1]** warn that auto-fix can break applications.

### [ANTI-2] do not skip audit in ci
**sources [SUMP-2]** warn that manual-only audits lead to audit fatigue and ignored vulnerabilities.

### [ANTI-3] do not trust postinstall scripts
**sources [OPIN-4]** warn that scripts are supply chain attack vectors.
