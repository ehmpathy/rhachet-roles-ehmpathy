# has-edgecase-coverage review (iteration 3): final edge case reflection

## what could go wrong that i have not yet considered?

### revisit: the playtest steps as a foreman would execute them

**step 7 (security block):** foreman runs install with lodash 4.17.4

question: what if lodash 4.17.4 has been patched since this playtest was written?

answer: CVE-2018-16487 and CVE-2019-1010266 are in lodash versions < 4.17.12. version 4.17.4 will remain vulnerable forever — it's a historical fact, not a patchable state.

**verdict:** safe — the playtest uses a genuinely and permanently vulnerable version.

---

### revisit: npm vs pnpm detection

step 11 tests npm detection (package-lock.json created first).

question: what if both package-lock.json AND pnpm-lock.yaml exist?

answer: the implementation checks `[[ -f package-lock.json ]]` first. if both exist, npm is used. this is correct — package-lock.json presence indicates npm is the primary manager.

question: should the playtest cover this dual-lock edge case?

answer: no — this is an implementation detail. the playtest covers the behavioral boundary (npm detection). the dual-lock scenario is a specific implementation choice, not a behavior change.

---

### revisit: @latest expansion

steps 3 and 6 test @latest.

question: what if npm registry is slow and @latest lookup times out?

answer: the skill uses `npm view $PACKAGE version` which has a default timeout. if it times out, the skill fails closed. this is acceptable — no install without version confirmation.

question: should the playtest cover timeout behavior?

answer: no — timeout is a network failure mode. the foreman cannot reliably reproduce this. the skill fails closed, which is the correct behavior.

## unusual but valid inputs reconsidered

### unicode in reason text

question: can reason text contain unicode (emojis, non-ascii)?

answer: reason is written via heredoc or bash string to a .md file. bash handles unicode fine. markdown renders unicode fine.

question: should the playtest cover this?

answer: no — this is not a boundary of the behavior. it's a property of bash/markdown that we rely on. if unicode broke, it would be a system failure, not a skill bug.

### very long package name

question: npm allows package names up to 214 characters. what happens with a very long name?

answer: the reason file path would be `.route/.../3.reason.for_{very_long_name}.v1.i1.md`. filesystem paths can be ~4096 chars on linux. 214 char package name is fine.

question: should the playtest cover this?

answer: no — this is within filesystem limits. not a behavioral edge case.
