> review for new mechanisms that duplicate extant functionality.

> - do we duplicate extant utilities or patterns?
> - could we reuse an extant component instead of a new one?

## new mechanisms in blueprint

### 1. set.package/output.sh

**extant equivalent**: git.commit/output.sh

**overlap**: both contain turtle vibes print functions (print_turtle_header, print_tree_start, etc.)

**question**: should we reuse git.commit/output.sh?

**analysis**:
- pro reuse: single source of truth, less duplication
- pro separate: skills self-contained, no coupling
- extant pattern: git.commit/output.sh is specific to git.commit (header says "for git.commit skills")

**verdict**: keep separate — follows WET principle (2 usages is acceptable, wait for 3+ before abstracting). if we later add more skills with turtle vibes, refactor to shared location.
