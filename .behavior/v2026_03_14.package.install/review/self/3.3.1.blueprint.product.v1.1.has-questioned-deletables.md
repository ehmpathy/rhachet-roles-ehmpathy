# self-review: has-questioned-deletables

## review guide

> try hard to delete before you optimize:
> for each component, ask:
> - can this be removed entirely?
> - if we deleted this and had to add it back, would we?
> - did we optimize a component that shouldn't exist?
> - what is the simplest version that works?

---

## components reviewed

### 1. set.package/ directory structure

**question**: can we use flat files instead of a subdirectory?

**analysis**: git.commit/ uses subdirectory pattern for related skills with shared operations. set.package has two skills (install, upgrade) that share operations (security check, version expand, package manager detect). subdirectory is justified by shared operations.

**verdict**: keep — matches extant pattern, justified by shared code.

---

### 2. output.sh

**question**: can we reuse git.commit/output.sh instead?

**analysis**: output.sh contains turtle vibes operations. git.commit/output.sh has identical functions. could symlink or copy.

**verdict**: keep separate — skills should be self-contained. if git.commit/output.sh changes, we don't want set.package to break unexpectedly.

---

### 3. set.package.operations.sh

**question**: can operations be inline in each skill?

**analysis**: 4 shared operations used by both skills. inline would duplicate ~100 lines. shared file is justified.

**verdict**: keep — prevents duplication, follows git.commit pattern.

---

### 4. _persist_reason() function

**question**: is reason persistence essential?

**analysis**: wisher explicitly requested reason documentation in `.route/`. this is core to the wish, not optional.

**verdict**: keep — explicitly requested.

---

### 5. @latest expansion

**question**: could we require explicit versions instead?

**analysis**: wisher's vision shows `--at @latest` as supported. users expect this convenience. expansion adds ~5 lines of code.

**verdict**: keep — explicitly in vision, minimal complexity.

---

### 6. integration tests (14 + 6 cases)

**question**: can we reduce test count?

**analysis**: each case tests a distinct criterion from blackbox criteria. no overlap. coverage maps 1:1 to requirements.

**verdict**: keep all — each tests distinct requirement.
