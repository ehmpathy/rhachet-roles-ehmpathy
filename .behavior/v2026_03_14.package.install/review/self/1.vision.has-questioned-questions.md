## why each [answered] question holds

### Q3: fail closed — why it holds

the wisher said "fails fast if its not" safe. "not safe" includes "unknown safety status" — if api is down, safety is unknown, therefore not confirmed safe, therefore fail fast. this is consistent with security-first principle. fail-open would defeat the entire gate purpose.

### Q4: @stdin mechanism — why it holds

rhachet skills use @stdin for multi-line input. git.commit.set uses `-m @stdin`. this is the established pattern. inline short reason is a convenience extension, not a deviation from pattern. stdin-first, inline-as-convenience matches the codebase conventions.

### Q5: enforce 3 sections — why it holds

wisher listed 3 specific questions:
1. how much complexity would it actually abstract away?
2. could we avoid the need for that operation?
3. which other packages were considered?

these are not suggestions — they are the content wisher wants documented. enforcing them ensures every package reason addresses the wisher's actual concerns.

### Q6: expand @latest — why it holds

the audit trail value comes from knowing *which version* was installed and *why*. if reason file says "@latest", six months later we don't know which version was actually installed. lockfile has the answer, but reason file should be self-contained. expand @latest to actual version at install time, record actual version.

### Q9: package-lock.json detection — why it holds

wisher explicitly said: "UNLESS there is explicitly a `package-lock.json` file in this repo, in which case unfortunately the repo uses npm still"

this is not an assumption — wisher specified the detection method. checking for package-lock.json presence is the correct implementation of wisher's words.

### Q10: --package flag — why it holds

rule.forbid.positional-args requires named arguments. `--package zod` is self-documented. matches `--at 3.22.4`, `--for prod`, `--reason @stdin` pattern. all flags are named, none positional.

### Q11: sequence order — why it holds

wisher said: "if, and only if, this skill's executable detects... security check is clean... then go ahead and issue the... command"

this implies: check conditions → then act. security check is a condition. reason is not a condition — wisher said "requires the caller to explain" but the gate is security check. order: validate all inputs → security check (gate) → persist reason → install.
