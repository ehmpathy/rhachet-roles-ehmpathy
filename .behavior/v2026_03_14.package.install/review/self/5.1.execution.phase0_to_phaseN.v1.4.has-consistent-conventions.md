# consistent conventions review (iteration 2): deeper reflection

## re-examination with fresh eyes

let me look at this again. what name choices did i make that might diverge?

### potential divergences examined

#### 1. `--at` vs `--version`

i chose `--at` for the version flag. is this consistent?

**extant patterns:**
- git.commit.set uses `--message` not `--msg`
- get.package.docs uses `--of` (short preposition)

**analysis:** `--at` is a preposition like `--of`. it reads naturally: "install zod at 3.22.4". `--version` would also work but is longer. the vision explicitly shows `--at`.

**verdict:** ✓ consistent with vision and preposition pattern

#### 2. `--for prod|prep` vs `--save|--save-dev`

npm uses `--save` and `--save-dev`. we use `--for prod|prep`.

**analysis:** vision explicitly requests `--for prod|prep`. "prep" is wisher's preferred term over "dev". this diverges from npm convention intentionally.

**verdict:** ✓ intentional divergence per wisher

#### 3. function prefix: `_` vs no prefix

**extant patterns in git.commit/operations.sh:**
- `get_behavioral_commits_on_branch` — no underscore
- `get_first_behavioral_commit_hash` — no underscore

**our pattern:**
- `_check_dependencies` — with underscore
- `_sanitize_package_name` — with underscore

**analysis:** git.commit.operations.sh doesn't use underscore prefix, but these are bash functions meant to be exported. our functions use underscore to signal "private to this module". both are valid bash conventions.

**verdict:** ✓ minor divergence, acceptable — underscore convention is more explicit about privacy

#### 4. directory name: `set.package/` vs `package/`

**analysis:** we chose `set.package/` to match skill names `set.package.install` and `set.package.upgrade`. alternative would be `package/` but that would be inconsistent with `git.commit/` pattern.

**verdict:** ✓ consistent with extant pattern

#### 5. test file location: collocated vs separate

**extant pattern:**
- tests in same directory as skills
- git.commit/*.integration.test.ts — not present
- i don't see extant integration tests for skills

**our pattern:**
- set.package/*.integration.test.ts — collocated

**analysis:** we introduced integration tests for skills. no extant pattern to follow. collocated tests follow rule.prefer.collocation (tests near code).

**verdict:** ✓ new pattern, follows general collocation principle

## conclusion
