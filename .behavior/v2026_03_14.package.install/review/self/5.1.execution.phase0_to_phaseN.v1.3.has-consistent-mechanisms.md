### why output.sh duplication holds

i reviewed git.commit/output.sh again. the functions have different behaviors:

| function | git.commit | set.package | why different |
|----------|------------|-------------|---------------|
| print_tree_leaf | 4 params (key, value, prefix, is_last) | 3 params (key, value, is_last) | git.commit needs deeper indent control |
| print_nested_leaf | 3 params (key, value, is_last) | 4 params (key, value, parent_is_last, is_last) | set.package needs parent-aware indent |

if we extracted to shared module:
1. would need conditional logic per caller
2. would couple git.commit and set.package
3. would add indirection for ~50 lines of code

**why it holds:** rule.prefer.wet-over-dry says wait for 3+ usages. we have 2. extraction would be premature abstraction.

### why package sanitization duplication holds

get.package.docs.sh line 90:
```bash
PACKAGE_SAFE="${PACKAGE//\//__}"
```

set.package/operations.sh `_sanitize_package_name()`:
```bash
echo "$package" | sed 's/\//__/g'
```

these are one-liners with identical purpose. could extract to shared utility, but:
1. adds file dependency
2. adds source statement
3. adds cognitive load
4. saves 1 line

**why it holds:** cost of extraction > benefit for one-liner.

## conclusion

no mechanism duplication issues. all duplication is either:
- intentional per WET principle (output.sh, 2 usages)
- trivial (package sanitization, 1 line)
- disjoint (operations.sh, different domains)
