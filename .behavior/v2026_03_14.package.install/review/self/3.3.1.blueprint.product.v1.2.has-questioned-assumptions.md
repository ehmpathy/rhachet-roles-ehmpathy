## review guide

>
> are there any hidden technical assumptions the junior made?
>
> for each assumption, ask:
> - what do we assume here without evidence?
> - what if the opposite were true?

## assumptions reviewed

### 1. npm bulk advisory endpoint stability

**assumption**: `POST /-/npm/v1/security/advisories/bulk` is stable and returns `{}` for safe packages.

**evidence**: research stone documented this endpoint from npm blog post and internal npm-audit usage.

**what if wrong**: endpoint could change, rate-limit, or require auth.

**verdict**: holds — endpoint is used by npm audit internally, well-documented. add error handle for unexpected responses.

---

### 2. npm view for @latest

**assumption**: `npm view $PACKAGE version` returns latest version reliably.

**evidence**: standard npm CLI behavior, documented.

**what if wrong**: package doesn't exist, network fails, scoped packages need different handle.

**verdict**: holds — standard pattern. blueprint already shows "fail closed" on curl/network errors. scoped packages work with npm view.

---

### 3. package-lock.json = npm

**assumption**: presence of package-lock.json means use npm.

**what if both lockfiles exist**: pnpm can coexist with npm lockfile at migration time.

**what if neither exists**: new project scenario.

**verdict**: holds per wisher's explicit wish. blueprint follows wisher's rule: "pnpm default, npm only if package-lock.json exists". if both exist, npm wins (stricter interpretation). if neither exists, pnpm default applies.

---

### 4. jq availability

**assumption**: jq is installed for JSON parse.

**evidence**: common tool, but not always present.

**alternative**: use node/npx for JSON parse instead? adds dependency on node (which is already present for npm).

**verdict**: ISSUE — should verify jq or use fallback. fix: add check for jq at start, fail with clear message if absent.

---

### 5. curl availability

**assumption**: curl is available for API calls.

**evidence**: nearly universal on *nix systems.

**alternative**: wget, or node-based fetch.

**verdict**: holds — curl is standard. if absent, fail with clear message. no need to add wget fallback for edge case.

---

### 6. scoped package names in file paths

**assumption**: package names work in file paths for reason files.

**example**: `@org/package` would create `.route/v.../3.reason.for_@org/package.v1.i1.md` — the `/` breaks path.

**verdict**: ISSUE — scoped packages break file paths. fix: replace `/` with `__` in file names (e.g., `3.reason.for_@org__package.v1.i1.md`).

---

### 7. version tag support

**assumption**: versions are semver or `@latest`.

**what about**: `@next`, `@beta`, `@canary` tags?

**verdict**: holds — `npm view $PACKAGE@next version` works for any tag. `@latest` is not special-cased, just passed through. the `_expand_version` function can handle any version string, not just `@latest`.

**refinement**: actually, blueprint shows special case for `@latest` only. should handle any `@*` tag.

**verdict**: ISSUE — `_expand_version` should expand any `@*` tag, not just `@latest`. fix: `if [[ "$VERSION" =~ ^@ ]]; then npm view "$PACKAGE$VERSION" version`.

---

### 8. empty response = safe

**assumption**: `{}` from bulk advisory means no vulnerabilities.

**evidence**: research stone documented this as the expected response format.

**verdict**: holds — per npm documentation, empty object means no advisories.

---

### 9. bash availability

**assumption**: bash is available.

**evidence**: all extant skills use bash.

**verdict**: holds — consistent with extant skills. non-issue.
