# has-edgecase-coverage review (iteration 2): deeper edge case analysis

## gaps from iteration 1

### gap 1: individual flag errors not tested

**review:** step 10 tests `--package` omitted only.

**consideration:** should we add steps for each flag?

**decision:** no — the playtest is for manual verification. if step 10 shows the error behavior pattern, the foreman can verify similar behavior for other flags. add 4 more steps (--at, --for, --reason, invalid --for) would be redundant.

**however:** i should verify step 10 is clear about what error to expect.

current step 10:
```sh
# omit --package
npx rhachet run --skill set.package.install \
  --at 1.0.0 \
  --for prod \
  --reason "test"
```

expected outcome:
- exit code 2
- error message mentions `--package is required`

**verdict:** sufficient — pattern is demonstrated, foreman can apply to other flags.

---

### gap 2: upgrade security failure not tested

**review:** set.package.upgrade uses the same `_check_security()` function as install.

**consideration:** should we add a step for upgrade + vulnerable version?

**decision:** no — this would test the same code path. step 7 already proves the security audit blocks vulnerable packages. the upgrade skill reuses this exact function.

**however:** let me verify the implementation actually shares the function.

checked: `set.package.upgrade.sh` sources `set.package.operations.sh` and calls `_check_security()`.

**verdict:** no gap — shared implementation means step 7 coverage applies to both skills.

---

### gap 3: scoped packages not tested

**review:** scoped packages like `@org/pkg` need path sanitization for the reason file.

**consideration:** should we add a step with a scoped package?

**decision:** no for playtest, yes for integration tests.

**reason:**
1. playtest is manual verification — a scoped package test requires the foreman to have a specific scoped package available
2. integration tests already cover `_sanitize_package_name()` which converts `@org/pkg` → `@org__pkg`
3. the happy path (step 1 with test-fns) proves the file write works; scoped packages are a variant

**verdict:** acceptable gap — covered by integration tests.

## additional edge cases considered

not applicable — bash skill runs synchronously, no concurrent access pattern.

impractical to test manually. skill will fail with filesystem error.

not tested. skill uses `jq` which will fail with parse error. acceptable — foreman would see jq error.

## why it holds (after deeper review)

1. **shared implementation reduces test surface** — security audit tested once, covers both skills
2. **pattern demonstration sufficient** — flag validation shown, foreman can verify variants
3. **scoped packages covered by unit tests** — not a playtest concern
4. **impractical edge cases acceptable** — disk full, malformed json are system failures

the playtest covers the behavioral edge cases. implementation edge cases are covered by integration tests.
