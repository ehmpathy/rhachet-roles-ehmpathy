## assumption 1: aikido/snyk has a free api tier

**what do we assume?** that a free security api exists with sufficient rate limits

**what evidence supports this?** none — this is pure assumption

**what if the opposite were true?**
- if no free tier: skill requires paid subscription or api key setup
- if rate limited: could block workflow at scale

**did wisher say this?** no — wisher said "aikido.com or snyk or some good, free, security check library"

**verdict**: ASSUMPTION TO VALIDATE. need to research which apis are free and their limits.

## deeper reflection: what did I add that wisher did not say?

upon re-read, several vision elements are my construction, not wisher's:

### invention 1: `--package` flag syntax

**what I wrote**: `--package zod`

**what wisher said**: "this skill's executable" and "$package.name@version"

**the gap**: wisher never specified the flag name. I invented `--package`.

**why it holds**: needs *some* way to specify package. `--package` is clear. but could be positional arg instead.

### invention 2: security check sequence

**what I wrote**: check security (step t2) → write reason (t3) → install (t4)

**what wisher said**: "if, and only if, this skill's executable detects... security check is clean... then go ahead"

**the gap**: wisher implies security check is a gate, but not when reason is persisted. I assumed reason is written after security passes.

**counterexample**: maybe reason should be written first (as intent), then security checked, so even failed installs have documented intent?

**why current holds**: makes sense to not persist reason for blocked packages. but worth questioning.

### invention 3: extant package behavior

**what I wrote**: "package already installed → warn but allow (version upgrade scenario)"

**what wisher said**: nothing about reinstalls or upgrades

**the gap**: I invented this behavior entirely

**verdict**: ASSUMPTION TO CLARIFY. should upgrades require new reason? should reinstalls be blocked?

### invention 4: the analogies

**what I wrote**: building permit, prescription, git commit message

**what wisher said**: nothing about analogies

**why it holds**: analogies aid comprehension. they are vision artifacts, not requirements.

### invention 5: minimum reason length

**what I wrote**: "minimum length or structure enforced? (open question)"

**what wisher said**: listed 3 specific questions to answer (complexity, avoidance, alternatives)

**the gap**: wisher was specific about *what* to explain, but I left enforcement as open question

**verdict**: ASSUMPTION. wisher's 3 questions could be enforced as required sections.

### invention 6: @latest expansion

**what I wrote**: "skill expands to actual version before install"

**what wisher said**: "@latest _is_ supported"

**the gap**: wisher said it's supported, but not what the skill does with it. I assumed it gets expanded to a pinned version. but maybe @latest is passed through literally?

**counterexample**: if @latest is passed through, lockfile pins it. but the reason file would say "@latest" not the actual version at install time.

**verdict**: ASSUMPTION TO CLARIFY. expand @latest to pin, or pass through?
