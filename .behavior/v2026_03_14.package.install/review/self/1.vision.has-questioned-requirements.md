# self-review: 1.vision.has-questioned-requirements

review of each requirement from the wish and vision.

---

## requirement 1: security check via external api (aikido/snyk)

**who said this?** the wish explicitly requires "deterministically checks aikido.com or snyk"

**what evidence supports it?** prevents known vulnerabilities from entry into codebase before install, not after

**what if we didn't do this?**
- vulnerable packages could be installed
- alternative: `npm audit` post-install catches issues but after the fact
- alternative: rely on ci/cd security scans

**could we achieve the goal simpler?**
- `npm audit` is built-in but runs post-install
- external api gives pre-install gate which is stronger

**question raised**: what if the api is down? vision says "fail closed" — no installs if api unreachable. is this too strict?

**verdict**: requirement holds. pre-install check is stronger than post-install. the "fail closed" behavior is correct for security.

## requirement 2: reason documentation

**who said this?** the wish requires explanation of why, complexity abstracted, alternatives considered

**what evidence supports it?** creates audit trail — "why is lodash here?" answerable in 6 months

**what if we didn't do this?** same as today — packages appear without explanation

**could we achieve simpler?** yes, but loses the core differentiator

**verdict**: requirement holds. this is the primary value proposition.

## requirement 3: --for prod|prep flag

**who said this?** the wish says "prod|prep"

**what evidence supports it?** separates production vs development dependencies

**what if we didn't?** all deps go to production deps (unsafe) or dev deps (broken at runtime)

**ISSUE FOUND**: "prep" is non-standard terminology. npm and pnpm use "dev" for development dependencies. the vision flags this as awkward but doesn't challenge the requirement.

**questions for wisher**:
- why "prep" instead of "dev"?
- is there a specific reason to diverge from npm/pnpm standard terminology?
- would `--for prod|dev` be acceptable?

**verdict**: requirement concept holds. terminology should be questioned with wisher.

---

## requirement 4: --at version flag required

**who said this?** the wish requires "--at flag for the specific version (@latest _is_ supported)"

**what evidence supports it?** prevents version drift, ensures determinism

**what if we didn't?** @latest becomes default, lockfiles drift unpredictably

**could we simplify?** allow omission with @latest as fallback? no — that defeats the purpose.

**verdict**: requirement holds. explicit version pins are important for reproducibility.

---

## requirement 5: .route/ storage path

**who said this?** wish specifies `.route/v$isodate.package.install.$package/3.reason.v1.i1.md`

**what evidence supports it?** persists documentation alongside behavioral route

**ISSUE FOUND**: the path pattern is specific but unexplained:
- what does `3.reason` represent? (step 3 in some process?)
- what is `v1.i1`? (version 1, iteration 1?)
- will there be other files like `3.1.research` or `3.2.alternatives`?

**question for wisher**: is this the complete path pattern or is there a larger .route structure implied?

**verdict**: requirement holds conceptually. path details need clarification.

---

## requirement 6: pnpm vs npm detection

**who said this?** wish says "choose pnpm by default, UNLESS there is explicitly a `package-lock.json` file"

**what evidence supports it?** repos use one or the other, detected by lockfile

**ISSUE FOUND**: the logic is inverted from what might be more intuitive:
- current: check for package-lock.json → npm
- alternative: check for pnpm-lock.yaml → pnpm, else npm

**could we simplify?** check for pnpm-lock.yaml positively rather than package-lock.json negatively. handles edge case where neither lockfile present.

**verdict**: requirement holds. detection method could be refined.
