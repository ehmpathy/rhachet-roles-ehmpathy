# vision: nice tests

## the outcome world

### day-in-the-life

**before**: you kick off `rhx git.repo.test --what integration --mode apply` on a big repo. your machine grinds to a halt. browser tabs freeze. slack notifications lag. you can't do any other work until the tests finish. you go get coffee.

**after**: same command. tests run in the background at lower priority. your machine stays responsive. you can browse docs, reply to messages, even run another repo's tests. the CPU is still working hard, but only when no other process wants it.

### the "aha" moment

> "wait, i can actually use my computer while tests run?"

the value clicks when you realize tests don't have to monopolize your machine. `nice -n 5` doesn't slow tests down — it just makes them yield to foreground work. on an idle machine, tests run at full speed. on a busy machine, tests politely step aside.

## user experience

### usecases

| goal | what happens |
|------|--------------|
| run big test suite | tests run at lower priority, machine stays usable |
| run tests + browse docs | browser gets CPU priority, tests use idle cycles |
| run tests on multiple repos | each test suite yields to the others and to your foreground work |
| CI/server context | nice is irrelevant (no interactive user), tests run at full speed anyway |

### contract inputs & outputs

**no change to contract.** the behavior is invisible to the user — same commands, same output, same exit codes. the only difference is system behavior: the test process yields CPU to other processes.

```sh
# usage unchanged
rhx git.repo.test --what unit --mode apply
rhx git.repo.test --what integration --mode apply
rhx git.repo.test --what all --mode apply
```

### timeline

1. user invokes `rhx git.repo.test --what integration --mode apply`
2. skill builds npm command as before
3. skill wraps execution in `nice -n 5 bash -c "..."`
4. tests run at lower priority (nice value 5)
5. output, logging, and exit codes work exactly as before

## evaluation

### how well does it solve the goals?

| goal | solved? |
|------|---------|
| machine doesn't lock up | yes — tests yield to foreground work |
| tests still run fast | yes — on idle CPU, full speed |
| no user action required | yes — automatic, no flags needed |
| works for all test types | yes — unit, integration, acceptance, all |

### pros

- **invisible improvement**: no new flags, no config, just better behavior
- **zero cost on idle machines**: if CPU is free, tests run at full speed
- **composable**: multiple nice processes share CPU gracefully
- **cross-platform**: `nice` works on linux and macos

### cons

- **windows**: `nice` doesn't exist on windows (but we're linux/mac focused)
- **not a hard limit**: tests can still use all cores, just at lower priority. for hard limits you'd need `cpulimit` or `--maxWorkers`
- **invisible**: users might not notice the improvement (is this a con?)

### edge cases & pit of success

| edge case | handling |
|-----------|----------|
| CI/server (no interactive user) | nice is irrelevant — tests run at full speed anyway |
| already nice'd shell | nice values stack (nice -n 5 inside nice -n 5 = nice -n 10), safe |
| timeout + nice | timeout wraps nice, both work correctly |
| windows | `nice` not available — need to handle gracefully (skip or warn?) |

## open questions & assumptions

### assumptions

1. users run tests on local machines where responsiveness matters
2. `nice` is available on all target systems (linux, macos)
3. nice value 5 is a good default (yields to foreground, but not overly deferential)

### questions (triaged)

1. **windows support?** [answered] — skip silently on windows. tests still work, just without priority benefit. warn/error adds noise without value.

2. **configurable nice value?** [wisher] — the wish doesn't ask for configurability. we'll use hardcoded `nice -n 5` unless wisher objects. escape hatch: direct `npm test` bypasses git.repo.test.

3. **CI detection?** [answered] — don't detect CI, always apply nice. nice in CI is harmless (no other processes to yield to). detection adds complexity without benefit.

### research at implementation time

- [research] confirm `nice` behavior on macos vs linux — both POSIX, likely identical, verify at implementation
- [research] confirm `nice` behavior inside docker containers — should work (shares host scheduler), verify at implementation

## groundwork

### external research

none — `nice` is a standard unix utility. from `man nice`:
> nice - run a program with modified scheduling priority

nice values range from -20 (highest priority) to +19 (lowest priority). default is 0. `nice -n 5` sets priority to 5 — the process yields to any process with a lower nice value (0-4).

### internal research

checked `git.repo.test.sh` at `src/domain.roles/mechanic/skills/git.repo.test/git.repo.test.sh`:

**where tests run (line 950-956)**:
```bash
if [[ -n "$TIMEOUT" ]]; then
  timeout "${TIMEOUT}s" bash -c "${env_prefix}${npm_cmd}" > "$temp_stdout" 2> "$temp_stderr" || exit_code=$?
else
  eval "${env_prefix}${npm_cmd}" > "$temp_stdout" 2> "$temp_stderr" || exit_code=$?
fi
```

**the fix**: wrap execution in `nice -n 5`:
```bash
if [[ -n "$TIMEOUT" ]]; then
  timeout "${TIMEOUT}s" nice -n 5 bash -c "${env_prefix}${npm_cmd}" > "$temp_stdout" 2> "$temp_stderr" || exit_code=$?
else
  nice -n 5 bash -c "${env_prefix}${npm_cmd}" > "$temp_stdout" 2> "$temp_stderr" || exit_code=$?
fi
```

**contracts to conform to**: none — this is internal behavior change, no contract changes.

**vocab**: "nice" is the unix term. we could say "run tests nicely" in docs/output.

**stdout patterns**: no output change required. the improvement is invisible.

## what is awkward?

### windows gap

`nice` doesn't exist on windows. options:
1. skip `nice` on windows (tests run at normal priority)
2. warn on windows that nice isn't available
3. error on windows (harsh)

**recommendation**: option 1 — skip silently on windows. tests still work, just without the priority benefit.

### invisible improvement

users might not notice the improvement. is this a problem? probably not — invisible improvements are good UX. but we could mention it in output:
```
🐢 cowabunga!

🐚 git.repo.test --what integration --mode apply
   ├─ nice: -n 10 (yields to foreground work)
   ├─ status: passed
   ...
```

**recommendation**: add a subtle line showing nice status. makes the behavior discoverable.

### nice value choice

why 10 and not 5 or 15?
- 10 is the default for `nice` without arguments
- 10 is "noticeably lower priority but not the lowest"
- 10 leaves room for even lower priority background work (nice -n 19)

seems fine. could make it configurable later if needed.
