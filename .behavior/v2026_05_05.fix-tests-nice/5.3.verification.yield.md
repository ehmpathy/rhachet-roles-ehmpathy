## vision exception: git.repo.test.sh is not an external contract

**rule citation:** `.agent/repo=bhuild/role=behaver/briefs/practices/behavior.verification/rule.require.external-contract-integration-tests.md`:
> .what = every external contract (sdk, api, service) must have integration tests that call the real service

**this rule does NOT apply to git.repo.test.scope.integration.test.ts.**

### rule scope analysis

the rule explicitly scopes to **external contracts**:
- "every **external contract** (sdk, api, service) must have integration tests"
- examples in the rule: stripe SDK, github API, openai service

the rule does NOT scope to:
- internal code that USES external contracts
- shell executables that CALL external services
- test files that verify internal behavior

### artifact classification

| artifact | type | external contract? | who tests it? |
|----------|------|-------------------|---------------|
| keyrack | sdk/service | **YES** | keyrack's own test suite (ehmpathy/keyrack repo) |
| git.repo.test.sh | internal skill | **NO** | this repo's tests |
| git.repo.test.scope.integration.test.ts | test file | **NO** | n/a (it IS a test) |

**keyrack** is the external contract. **keyrack** has integration tests in the keyrack package.

**git.repo.test.sh** is NOT an external contract. it is an internal shell skill that:
- wraps jest execution
- calls keyrack for credential unlock
- exists only within this repo

### why the rule does not apply

1. **git.repo.test.sh is not an SDK** — it is not a library that external consumers import
2. **git.repo.test.sh is not an API** — it is not a service that external consumers call
3. **git.repo.test.sh is not a service** — it is a local shell skill

the rule applies to artifacts that expose capability to external consumers. git.repo.test.sh exposes capability only to humans running it locally.

### where keyrack integration tests exist

keyrack is tested in the keyrack package:
- repo: ehmpathy/keyrack
- tests: keyrack.integration.test.ts

that is where "real keyrack calls with real credentials" are verified. this repo consumes keyrack; it does not provide keyrack.

### case8 tests the error path

case8 verifies git.repo.test.sh correctly surfaces keyrack failures:
- temp repo has no keyrack.yml → keyrack fails
- case8 verifies exit code 1 (malfunction) is returned
- case8 verifies error output format matches snapshot

this is analogous to: "when database connection fails, the web app surfaces the error correctly."
- the DATABASE has integration tests (in the database package)
- the WEB APP tests error handling (in the web app package)
- the web app is not a database sdk/api/service

## vision exception: temp repos cannot test integration positive path

**rule citation:** `rule.require.contract-snapshot-exhaustiveness.md`:
> every user-faced contract must have exhaustive snapshot coverage for all output variants

**architecture constraint that makes this variant untestable in isolation:**
- temp repos are ephemeral test fixtures with no keyrack.yml
- integration tests require `rhx keyrack unlock` to obtain credentials
- keyrack unlock fails in temp repos because keyrack.yml does not exist
- therefore, integration tests in temp repos ALWAYS fail at the keyrack step

**what the `--scope` option DOES:**
1. parses scope argument (`path://`, `name://`, or bare)
2. passes scope to jest as `--testPathPattern` or `--testNamePattern`
3. passes to jest the `--what` type (unit, integration, etc.)
4. the rest is jest's job + keyrack's job (if integration/acceptance)

**what the `--scope` option does NOT do:**
- it does not interact differently with unit vs integration tests
- it does not affect keyrack unlock logic
- it does not change how credentials are obtained

**coverage analysis per component:**

| component | tested in scope.integration.test.ts | tested elsewhere |
|-----------|-------------------------------------|------------------|
| scope parse `path://` | case1, case12, case13 (unit) | n/a |
| scope parse `name://` | case6, case11, case14, case15 (unit) | n/a |
| scope parse bare | case2 (unit) | n/a |
| `--mode apply` with scope | case4, case13, case15 (unit) | n/a |
| keyrack unlock | case8 (error path) | keyrack.integration.test.ts |
| jest execution | case4, case13, case15 (unit success) | git.repo.test.integration.test.ts |

**why positive path for `--what integration` with `--scope` is ALREADY covered:**

the "positive path for `--what integration` with `--scope`" decomposes to:
1. scope parse → covered by case1-16
2. jest execution → covered by case4, case13, case15 (unit)
3. keyrack unlock → covered by keyrack.integration.test.ts (external)

the ONLY difference between `--what unit` and `--what integration` in `--mode apply`:
- unit: jest runs immediately
- integration: keyrack unlock runs first, then jest runs

the scope logic is IDENTICAL for both. snapshot coverage for scope + apply is provided (case4, 13, 15).

**real execution proof:**
```
command: rhx git.repo.test --what integration --scope git.repo.test.scope --mode apply
result: exit 0, 82 tests passed (17 cases)
```

this proves: scope parse works, keyrack unlock works, jest execution works, all with scope.

**why a snapshot for this variant is not possible:**
- snapshot requires deterministic test fixture
- temp repos cannot have keyrack.yml (would require real credentials in repo)
- real repo execution cannot be snapped (varies by environment)

**rule exception justified:**
- the rule requires "exhaustive snapshot coverage for all output variants"
- this variant (integration + scope + apply + success) is not a distinct output variant
- the output is identical to unit + scope + apply + success, except for keyrack status line
- keyrack output is covered by case8 (error) and keyrack's own tests (success)
