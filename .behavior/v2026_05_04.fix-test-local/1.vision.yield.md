# vision: git.repo.test snapshots in sync

## the outcome world

### what does a day-in-the-life look like with this in place?

a mechanic pushes a commit. CI runs. the git.repo.test integration tests execute against every test case — not just files changed since main. the snapshots verify that the CLI output matches the contract. if someone aggressively updates a snapshot by hand without fixing the underlying fixture, CI fails immediately.

no more "tests pass locally but the snapshots are lies" — the committed snapshots are the truth, verified on every CI run.

### what's the before/after contrast?

| before | after |
|--------|-------|
| snapshots can be hand-edited without verification | snapshots verified against real test execution |
| CI only runs changed files, missing drift | CI runs full snapshot suite every time |
| broken fixtures hide behind stale snapshots | fixtures must produce expected output |
| "tests pass" means no confirmation if snapshots lie | "tests pass" means output matches contract |

### what's the "aha" moment where the value clicks?

when you run `rhx git.repo.test --what integration` and the snapshot test fails because the fixture doesn't produce the expected turtle vibes output — and you realize that's exactly what should happen. the snapshot is the contract, not a suggestion.

## user experience

### what usecases do folks fulfill?

1. **verify CLI output contract** — run tests, snapshots verify output format
2. **update snapshots intentionally** — use `--resnap` when behavior changes deliberately
3. **diagnose fixture issues** — snapshot diff shows exactly what output differs
4. **prevent snapshot lies** — CI catches hand-edited snapshots that don't match reality

### what contract inputs & outputs do they leverage?

**inputs:**
- `rhx git.repo.test --what unit|integration|acceptance`
- `rhx git.repo.test --what unit --resnap` (update snapshots)
- `rhx git.repo.test --what unit --thorough` (run full suite, not just changed)

**outputs:**
- turtle vibes CLI output (🐢 lets ride... / 🐢 bummer dude...)
- snapshot files in `__snapshots__/*.snap`
- exit codes: 0=passed, 1=malfunction, 2=constraint

### what would it look like to leverage them?

```bash
# local development: run tests, verify snapshots
rhx git.repo.test --what integration

# after intentional behavior change: update snapshots
rhx git.repo.test --what integration --resnap
git diff __snapshots__/  # review snapshot changes
git add __snapshots__/
git commit -m "chore(git.repo.test): update snapshots for new output format"

# CI: runs all tests with THOROUGH=true to verify all snapshots
```

### what timelines do they go through?

1. clone repo, make changes
2. run tests locally (changed files only by default)
3. push, CI runs full suite (THOROUGH=true)
4. if snapshot mismatch: CI fails, mechanic investigates
5. fix fixture or update snapshot intentionally, re-push

## mental model

### how would users describe this to a friend?

"the snapshots are the contract. if your fixture doesn't produce the expected output, the test fails. CI runs all snapshot tests every time, so no one can lie by hand-editing snapshots."

### what analogies or metaphors fit?

- **snapshot as contract** — like a signed document, both parties must honor it
- **fixture as implementation** — the fixture must fulfill the contract
- **CI as auditor** — verifies the contract is honored on every change

### what terms would they use vs what terms would we use?

| user term | our term |
|-----------|----------|
| "snapshot test" | integration test with `.toMatchSnapshot()` |
| "fixture" | `setupFixture()` in test, creates temp git repo |
| "mock" | `mockNpm`, `mockKeyrack`, `mockListTests` |
| "output" | stdout/stderr captured and sanitized |

## evaluation

### how well does it solve the goals?

| goal | solution | effectiveness |
|------|----------|---------------|
| run all snapshots on CI | set `THOROUGH=true` in CI | 100% — no changed-only filtering |
| fix fixtures to match snapshots | update `setupFixture()` calls | requires per-test investigation |
| prevent future snapshot lies | CI fails on mismatch | 100% — no bypass possible |

### what are the pros?

- **truth enforcement** — snapshots are verified, not trusted
- **clear contracts** — committed snapshots define expected behavior
- **easy diagnosis** — snapshot diffs show exact output differences
- **low friction** — `--resnap` for intentional updates

### what are the cons?

- **slower CI** — runs all tests, not just changed
- **fixture maintenance** — must keep fixtures in sync with skill behavior
- **snapshot churn** — format changes require snapshot updates

### what edgecases and how do our contracts keep users in a pit of success?

| edgecase | pit of success |
|----------|----------------|
| hand-edited snapshot doesn't match reality | CI fails with snapshot mismatch |
| fixture produces different output on different systems | `sanitizeOutput()` normalizes timestamps, paths, versions |
| test passes locally, fails CI | THOROUGH=true in CI ensures full suite runs |
| snapshot updated without clarity | PR review catches unexplained snapshot changes |

## what is awkward?

### what feels off or forced?

- two separate test files with different functions (`runInTempGitRepo` vs `setupFixture`)
- duplicated sanitization logic across test files
- unclear which tests use real npm vs mock npm

### where does the design fight the user's mental model?

- "snapshot updated" could mean intentional change or bug — hard to distinguish
- mock layers (npm, keyrack, jest) add complexity for hermetic tests
- THOROUGH flag inverts the default (usually you want fast feedback)

### what tradeoffs feel uncomfortable?

- speed vs correctness: all tests is slower but necessary for truth
- mocking vs reality: more mocks = more hermetic but more divergence from real behavior
- fixture maintenance: keeping fixtures in sync with skill changes requires discipline
