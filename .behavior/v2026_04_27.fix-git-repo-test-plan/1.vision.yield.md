# vision: git.repo.test --mode plan and uri scope format

## the outcome world

### day-in-the-life

**before**: mechanic runs `rhx git.repo.test --what unit --scope invoice --mode apply` and waits for tests to complete, only then to discover that the scope matched 47 files instead of the 3 they expected. 45 seconds wasted.

**after**: mechanic runs `rhx git.repo.test --what unit --scope invoice` and immediately sees (plan mode is default):
- exactly which files matched
- can adjust the scope before the full run

### before/after contrast

| aspect | before | after |
|--------|--------|-------|
| scope validation | run tests, wait, discover mismatch | preview instantly, adjust, then run |
| scope syntax | `path(foo)` / `name(foo)` | `path://foo` / `name://foo` |
| feedback loop | slow (test execution time) | instant (list only) |
| confidence | "hope this scope is right" | "i see exactly what will run" |

### the "aha" moment

> "oh, i can see all 47 files that matched before i spend a minute on them — let me narrow this to `name://should calculate total`"

## user experience

### usecases

1. **scope validation** — check what a scope pattern matches before test run
2. **test discovery** — explore what tests exist in a given area
3. **scope debug** — understand why a scope matched more/fewer files than expected

### contract

**inputs**:
```sh
rhx git.repo.test --what unit --scope 'path://invoice'
rhx git.repo.test --what unit --scope 'name://should calculate'
rhx git.repo.test --what integration --scope 'path://customer'
rhx git.repo.test --what unit --scope 'path://invoice' --mode apply  # execute tests
```

**outputs** (plan mode — default):
```
🐢 heres the wave...

🐚 git.repo.test --what unit --scope path://invoice

   ├─ files

   │  ├─ src/domain/invoice/validateLineItems.test.ts
   │  └─ src/domain/invoice/formatInvoice.test.ts
   └─ hint: run with --mode apply to execute
```

**note**: v1 shows files only. test name list is a v2 enhancement (requires file parse).

### uri format rationale

| format | pros | cons |
|--------|------|------|
| `path(foo)` | familiar to some | looks like function call, confuses shell |
| `path://foo` | uri-familiar, no shell escape needed | new pattern |

uri format aligns with:
- web urls: `https://example.com`
- file urls: `file:///path/to/file`
- custom schemes: `vscode://file/path`

## evaluation

### how well does it solve the goals?

| goal | satisfaction |

| see matched files before run | ✅ fully addressed |
| see matched test names | ✅ fully addressed |
| avoid wasted test runs | ✅ fully addressed |

### pros

- instant feedback on scope matches
- no wasted test execution time
- uri format is shell-safe (no parentheses to escape)
- consistent with other skills (`sedreplace --mode plan`)
- aligns with terraform/make conventions

### cons

- plan mode is now default, changes extant behavior (requires `--mode apply` to run tests)
- uri format differs from jest's native `--testNamePattern`
- need to maintain backwards compat with old paren format (at least temporarily)

### edgecases and pit-of-success

| edgecase | action |
|----------|--------|
| scope matches 0 files | show clear message: "no files matched" |
| scope matches 30+ files | truncate list after 30, show "(xyz more, abc total)" |
| invalid scope syntax | error with example of valid syntax |
| `--mode plan` with `--resnap` | warn: `--resnap` has no effect in plan mode |
| bare scope (no prefix) | default to path (backwards compat) |

## what is awkward?

### uri format requires adjustment

users familiar with `path(foo)` may find `path://foo` unexpected. however:
- paren format requires shell escape sometimes
- uri format is more universal
- we can support both formats in transition

### jest doesn't natively support uri format

jest uses `--testPathPatterns` and `--testNamePattern` — we wrap these with our own syntax. this abstraction may confuse users who know jest intimately.

**mitigation**: error message already shows the map:
```
🥥 did you know?
   ├─ --scope 'foo' filters by file path
   ├─ --scope 'path://foo' filters by file path (explicit)
   └─ --scope 'name://foo' filters by test name
```

### test name list complexity

`jest --listTests` only lists files, not individual test names. to show test names in `--mode plan`, we'd need to either:
1. parse test files for `describe()`/`it()` calls
2. run jest in a special mode that outputs test names without execution

this may be a v2 enhancement — v1 could just list files.

### name:// scope in plan mode shows path-matched files only

`jest --listTests` does not filter by test name — it only matches file paths. this means:

- `--scope 'path://invoice'` → shows files that match "invoice" in path ✅
- `--scope 'name://should calculate'` → shows ALL test files, not filtered by test name ⚠️

the `name://` filter only applies at test execution time (with `--mode apply`), not at plan time.

**mitigation**: output notes "name filter applies at runtime" when `name://` scope is used:
```
🐢 heres the wave...

🐚 git.repo.test --what unit --scope name://should calculate
   ├─ scope: name://should calculate
   │  └─ matched: 47 files (name filter applies at runtime)
   ├─ files
   │  ├─ src/domain/invoice/computeTotal.test.ts
   │  ├─ src/domain/invoice/validateLineItems.test.ts
   │  └─ ... (17 more, 47 total)
   └─ hint: run with --mode apply to execute
```

**alternative**: v2 could parse files to find which actually contain the test name. for v1, document the limitation.
