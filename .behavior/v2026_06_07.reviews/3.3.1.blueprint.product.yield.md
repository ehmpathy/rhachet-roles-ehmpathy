# blueprint: review.by skill + evals for rhachet-roles-ehmpathy

## summary

build the `rhx review.by --role mechanic` skill for `rhachet-roles-ehmpathy`:

1. **atomic rubric skills** — standalone `review.rubric=*.sh` skills callable directly
2. **review.by aggregator** — composes rubric skills, runs in parallel
3. **rubric definitions** — `rubrics.yml` in `briefs/reviews/` (domain knowledge, not skill config)
4. **eval infrastructure** — test harness in `.test/` subdirectory
5. **eval scenes** — minimal code cases for each rubric (positive + negative for all 8 rubrics)
6. **eval cases** — yaml case files with checks for each scene

**architecture**: atomic rubric skills are the composable primitive. `review.by.sh` is the aggregator. evals test rubric skills directly to isolate brain quality from dispatch correctness.

## codepath tree

### atomic rubric skill (leaf)

```
review.rubric=$SLUG.sh
├─ [+] review.rubric=$slug "$@"               # accept all args
│  │
│  ├─ [+] RULES=(hardcoded glob patterns)    # rule briefs for this focus
│  │
│  └─ [←] rhx review --rules $RULES "$@"     # delegate to bhrain, forward all args
```

each rubric skill is standalone: `rhx review.rubric=mech-failhides --paths 'src/**/*.ts'`

**passthrough:** `--mode`, `--diffs`, `--paths`, `--join`, etc. all forwarded via `"$@"` to `rhx review`. rubric skill only hardcodes `--rules`.

### review.by skill (aggregator)

```
review.by.sh
├─ [+] review.by --role $role [--for $slug] "$@"     # forward all args
│  │
│  ├─ [+] parse rubrics.yml                          # load focus list
│  │
│  ├─ if --for $slug:
│  │  └─ [←] review.rubric=$slug.sh "$@"              # forward all args to rubric skill
│  │
│  ├─ else:
│  │  ├─ [+] for each focus in rubrics.yml:
│  │  │  ├─ [←] review.rubric=$focus.sh "$@" > "$tmpdir/$focus.out" &
│  │  │  └─ pids+=($!)
│  │  │
│  │  ├─ [+] wait for all & collect exit codes:
│  │  │  └─ for pid in "${pids[@]}"; do wait "$pid"; codes+=($?); done
│  │  │
│  │  └─ [+] aggregate exit code:
│  │     └─ exit 2 if any code==2, exit 1 if any code==1, else 0
│  │
│  └─ [+] output via output.sh                       # cat tmpfiles, turtle vibes
```

**passthrough args:** `--mode`, `--diffs`, `--paths`, `--join` all forwarded via `"$@"` to rubric skills. rubric skills forward `"$@"` to `rhx review`. no arg parsing needed at aggregator level except `--for` and `--role`.

**exit code aggregation:** PIDs collected, `wait` per PID, highest severity wins (2 > 1 > 0).

**output isolation:** each focus writes to `$tmpdir/$focus.out`, concatenated after all complete.

```

├─ [+] rubrics: Rubric[]
│  ├─ slug: string                        # matches skill name

```

note: `rubrics.yml` is domain knowledge (what focuses exist), not skill config. rules are hardcoded in rubric skills to keep them atomic and self-contained.

### unit test — rubrics.yml.test.ts (sync validation)

validates that `rubrics.yml` entries are synchronized with actual skill files:

```typescript
describe('rubrics.yml', () => {
  given('[case1] all rubric slugs have corresponding skill files', () => {
    when('[t0] slugs are extracted from rubrics.yml', () => {
      then('each slug maps to a review.rubric=$slug.sh file', () => {
        const rubricsYml = yaml.parse(fs.readFileSync('rubrics.yml', 'utf-8'));
        const slugs = Object.keys(rubricsYml.rubrics);

        for (const slug of slugs) {
          const skillPath = path.join(__dirname, `review.rubric=${slug}.sh`);
          expect(fs.existsSync(skillPath)).toBe(true);
        }
      });
    });
  });

  given('[case2] all rubric skill files have yml entries', () => {
    when('[t0] skill files are discovered via glob', () => {
      then('each skill has an entry in rubrics.yml', () => {
        const skillFiles = glob.sync('review.rubric=*.sh', { cwd: __dirname });
        const rubricsYml = yaml.parse(fs.readFileSync('rubrics.yml', 'utf-8'));
        const ymlSlugs = Object.keys(rubricsYml.rubrics);

        for (const skillFile of skillFiles) {
          const slug = skillFile.match(/review\.focus=(.+)\.sh/)?.[1];
          expect(ymlSlugs).toContain(slug);
        }
      });
    });
  });
});
```

this catches sync failures at test time: forgot yml entry OR forgot skill file.

|----------|-----------------|-----------------|
| eval verdict | `verdict.pass`, `verdict.checks[].{slug,result}` | `verdict.summary`, `verdict.checks[].evidence` |
| review.by stdout (all clear) | full output | — |
| review.by stdout (findings) | full output | — |
| review.by stdout (--for slug) | full output | — |
| review.by stdout (--mode pull) | full output | — |
| review.by stdout (--help) | full output | — |
| review.by stderr (invalid role) | full output | — |
| review.by stderr (missing role) | full output | — |

note: `summary` and `evidence` are probabilistic (brain-generated freeform text) — they will drift even when behavior is correct. only snapshot stable/categorical fields.

**eval verdict snapshot projection (explicit code):**

```typescript
// in runReviewEval.ts assertions
then('verdict matches snapshot', () => {
  // project only stable fields — exclude probabilistic brain output
  const stableVerdict = {
    pass: verdict.pass,
    checks: verdict.checks.map(c => ({ slug: c.slug, result: c.result })),
  };
  expect(stableVerdict).toMatchSnapshot();
});
```

this prevents snapshot drift from brain-generated `summary` and `evidence` fields.

### confusion matrix

every focus needs both positive and negative cases:
- **positive** (expected=present): scenes with real violations
- **negative** (expected=absent): scenes with valid code

this prevents both false negatives (too lenient) and false positives (too strict).

### runReviewEval placement (WET-before-DRY)

`runReviewEval.ts` is placed in `mechanic/skills/review/.test/` even though it's role-agnostic.

**deliberate decision**: WET-before-DRY. when ergonomist or architect add evals, lift `runReviewEval.ts` to their common ancestor (e.g., `src/domain.roles/__shared__/`). until then, avoid premature abstraction.

do NOT import from mechanic's directory into other roles — that would be a scope leak. duplicate or lift when the second consumer appears.

### scene accumulation

scenes grow over time:
1. observe pattern in production
2. extract minimal scene
3. add eval case with checks
4. future brain changes must pass
