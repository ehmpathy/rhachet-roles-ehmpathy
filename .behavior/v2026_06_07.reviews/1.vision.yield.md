### the "aha" moment

the value clicks when:
- xai deprecates another model, we run the eval suite, compare verdicts, and pick the right brain in 10 minutes
- a new pattern emerges in production, we add it as a scene, and future brain changes must pass it
- we see a brain is too strict (false positives) or too lenient (false negatives) and have concrete cases to prove it

- we tune our rules and refs to produce better reviews — same brain, better verdicts — and evals prove the improvement
- we rightsize the brain per focus — cheaper/faster brains for simple reviews, more capable brains for nuanced ones — evals validate the choice

## eval architecture: generator + evaluator

### concept

```
scene (code scene from real patterns)
  ↓
generator (rhx review --rules '...' --paths ...)
  ↓
output.stdout
  ↓
evaluator (brain.choice.ask with zod schema)
  ↓
verdict: { pass: boolean, issues: string[] }
```

### the key insight

we do not test the review skill itself — we test whether a **specific brain** correctly applies **specific rules** to a **specific scene**.

the eval answers: "does this brain understand our rules well enough to catch what matters?"

### review focus skills (atomic, reusable)

each focus is a shell skill that dispatches to bhrain's `rhx review`:

```bash
# skills/review/review.focus=mech-failhides.sh
#!/usr/bin/env bash
# .what = review for hidden errors in catch blocks
# .why = detect failhide patterns that swallow errors

RULES=(
  '.agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.*.md'
  '.agent/repo=ehmpathy/role=mechanic/briefs/practices/code.test/pitofsuccess.errors/rule.*.md'
)

rhx review --rules "${RULES[*]}" "$@"
```

```bash
# skills/review/review.focus=mech-decode-friction.sh
#!/usr/bin/env bash
# .what = review for decode-friction in orchestrators
# .why = detect inline logic that should be named transforms

RULES=(
  '.agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/readable.narrative/rule.forbid.inline-decode-friction.md'
  '.agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/readable.narrative/rule.require.named-transformers.md'
)

rhx review --rules "${RULES[*]}" "$@"
```

these skills are usable standalone: `rhx review.focus=mech-failhides --paths 'src/**/*.ts'`

## confusion matrix: why it matters

### the mindset

**evals must test both directions.**

a brain that flags everything would pass all positive cases — but that's useless.
a brain that flags none would pass all negative cases — but that's also useless.

| case type | proves | without it |
|-----------|--------|------------|
| positive (expected=present) | brain catches real issues | brain could flag none and pass |
| negative (expected=absent) | brain avoids false alarms | brain could flag everything and pass |

**every eval suite needs both types of cases. this is not optional.**

### the problem we're solving

the original problem with `grok-build-0.1`:
> "the new one is over-pedantic and misunderstands the rules"

that's a **specificity problem** — too many false-positives. the brain flags valid code as violations.

a brain that only passes true-positive cases would pass if it flags everything — but that's useless. we need true-negative cases to prove it doesn't over-flag.

### the confusion matrix

| expected | observed | result | meaning |
|----------|----------|--------|---------|
| present | present | **true-positive** | generator correctly flagged issue |
| present | absent | **false-negative** | generator missed issue it should catch |
| absent | absent | **true-negative** | generator correctly did not flag |
| absent | present | **false-positive** | generator incorrectly flagged (false alarm) |

### sensitivity vs specificity

| metric | formula | measures | failure mode |
|--------|---------|----------|--------------|
| **sensitivity** | TP / (TP + FN) | does it catch real issues? | too lenient — misses violations |
| **specificity** | TN / (TN + FP) | does it avoid false alarms? | too strict — flags valid code |

a brain that passes all positive cases but fails negative cases is **too strict** (low specificity).
a brain that passes all negative cases but fails positive cases is **too lenient** (low sensitivity).

we need both.

### the eval balance

every focus needs both types of cases:

1. **positive cases** (expected=present): scenes with real violations
   - pass = true-positive (generator flags it)
   - fail = false-negative (generator misses it)

2. **negative cases** (expected=absent): scenes with valid code
   - pass = true-negative (generator stays quiet)
   - fail = false-positive (generator over-flags)

## eval cases (one file per case)

each eval case runs a focus skill against a scene, then asks an evaluator brain to judge the output.

### eval case format

each case specifies a list of **checks** — what the evaluator should verify.

**key insight**: each check looks for ONE SPECIFIC finding (or absence). a scene may produce multiple findings; each check verifies one of them.

```yaml
# eval.cases/mech-failhides.empty-catch.eval.yml
focus: mech-failhides
scene: eval.scenes/mech-failhides/empty-catch

evaluator:
  brain: anthropic/claude/sonnet-4.5

checks:
  - slug: swallowed-error
    description: should flag empty catch block as blocker
    expected: present
    severity: blocker
    reason: |
      the catch block is empty and swallows the error without:
      - logging it
      - rethrowing it
      - returning a meaningful value

      keywords: swallow, hidden, empty catch
```

```yaml
# eval.cases/mech-failhides.logged-rethrow.eval.yml
focus: mech-failhides
scene: eval.scenes/mech-failhides/logged-rethrow

evaluator:
  brain: anthropic/claude/sonnet-4.5

checks:
  - slug: no-false-flag
    description: should NOT flag logged rethrow
    expected: absent
    severity: blocker|nitpick  # should not be flagged at any severity
    reason: |
      the catch block logs and rethrows, which is valid:
      - catch (error) { log(error); throw error; }

      this should NOT be flagged as failhide
```

### severity matching

- `severity: blocker` — check for blocker only
- `severity: nitpick` — check for nitpick only
- `severity: blocker|nitpick` — check for either (any flag = match)

### multiple checks per case

a single scene can have multiple checks to verify both sensitivity and specificity:

```yaml
# scene has both an empty catch AND a valid rethrow
focus: mech-failhides
scene: eval.scenes/mech-failhides/mixed-handlers

checks:
  - slug: empty-catch-flagged
    description: should flag the empty catch block
    expected: present
    severity: blocker
    reason: |
      line 12 has an empty catch block that swallows errors
      keywords: empty catch, line 12

  - slug: valid-handler-clean
    description: should NOT flag the HelpfulError.wrap handler
    expected: absent
    severity: blocker|nitpick
    reason: |
      line 25 uses HelpfulError.wrap which is a valid error handler
      this should NOT be flagged at any severity
```

three checks = three separate verdicts. each check verifies one specific finding (or its absence).

### evaluator output schema

the evaluator brain outputs a structured verdict with confusion matrix terms:

```typescript
const schemaOfCheckResult = z.object({
  slug: z.string(),
  expected: z.enum(['present', 'absent']),
  observed: z.enum(['present', 'absent']),
  severityObserved: z.enum(['blocker', 'nitpick', 'none']),
  result: z.enum(['true-positive', 'false-positive', 'true-negative', 'false-negative']),
  evidence: z.string().nullable(),  // quote from generator output if found
});

const schemaOfEvalVerdict = z.object({
  checks: z.array(schemaOfCheckResult),
  pass: z.boolean(),  // true iff all results are TP or TN
  summary: z.string(),
});
```

**evaluator instructions**:

for each check:
1. read the `reason` field to understand what specific issue to look for
2. search the generator output for a finding that matches the reason
3. if `severity: blocker|nitpick`, match either severity level
4. set `observed: present` if found, `observed: absent` if not
5. compute result via confusion matrix

### pass criteria

```typescript
pass = checks.every(c =>
  c.result === 'true-positive' ||
  c.result === 'true-negative'
)
```

any `false-negative` or `false-positive` = fail

### reusable test runner

one test file per eval case, all use the same runner:

```typescript
// skills/review/.test/eval.cases/mech-failhides.empty-catch.eval.integration.test.ts
import { runReviewEval } from '../runReviewEval';

runReviewEval({ evalFile: __dirname + '/mech-failhides.empty-catch.eval.yml' });
```

### the runner

```typescript
// skills/review/.test/runReviewEval.ts
import { given, when, then, useThen, genTempDir } from 'test-fns';
import { genContextBrain } from 'rhachet';
import { execSync } from 'child_process';
import * as fs from 'fs';
import * as path from 'path';
import * as yaml from 'yaml';
import { z } from 'zod';

const schemaOfCheckResult = z.object({
  slug: z.string(),
  expected: z.enum(['present', 'absent']),
  observed: z.enum(['present', 'absent']),
  severityObserved: z.enum(['blocker', 'nitpick', 'none']),
  result: z.enum(['true-positive', 'false-positive', 'true-negative', 'false-negative']),
  evidence: z.string().nullable(),
});

const schemaOfEvalVerdict = z.object({
  checks: z.array(schemaOfCheckResult),
  pass: z.boolean(),
  summary: z.string(),
});

export const runReviewEval = (input: { evalFile: string }) => {
  const evalCase = yaml.parse(fs.readFileSync(input.evalFile, 'utf-8'));
  const sceneName = path.basename(evalCase.scene);

  describe(`review.eval: ${evalCase.focus}/${sceneName}`, () => {
    given(`[case1] ${sceneName}`, () => {
      const testDir = genTempDir({
        slug: sceneName,
        clone: path.join(__dirname, evalCase.scene),
        git: true,
      });

      when('[t0] review is run', () => {
        const generatorOutput = useThen('generator executes', () => {
          return execSync(
            `rhx review.focus=${evalCase.focus} --paths '${testDir}/**/*.ts' --mode push`,
            { encoding: 'utf-8' },
          );
        });

        const verdict = useThen('evaluator judges', async () => {
          const context = await genContextBrain({
            choice: { atom: evalCase.evaluator.brain },
          });

          const sceneContent = fs.readFileSync(`${testDir}/example.ts`, 'utf-8');
          const checksYaml = yaml.stringify(evalCase.checks);

          const result = await context.brain.choice.ask({
            role: { briefs: [] },
            prompt: `
analyze the review output and determine the result for each check.

## generator output (review results)
${generatorOutput}

## scene (code that was reviewed)
${sceneContent}

## checks to verify
${checksYaml}

for each check:
1. determine if the described issue was flagged in the generator output (observed: present|absent)
2. compare to expected (present|absent)
3. compute result via confusion matrix:
   - expected=present, observed=present → true-positive
   - expected=present, observed=absent → false-negative
   - expected=absent, observed=absent → true-negative
   - expected=absent, observed=present → false-positive
4. extract evidence (quote from generator output) if observed=present

pass = true iff all results are true-positive or true-negative
`,
            schema: { output: schemaOfEvalVerdict },
          });

          return result.output;
        });

        then('verdict is pass', () => {
          expect(verdict.pass).toBe(true);
        });

        then('no false-negatives', () => {
          const falseNegatives = verdict.checks.filter(c => c.result === 'false-negative');
          expect(falseNegatives).toEqual([]);
        });

        then('no false-positives', () => {
          const falsePositives = verdict.checks.filter(c => c.result === 'false-positive');
          expect(falsePositives).toEqual([]);
        });

        then('verdict matches snapshot', () => {
          expect(verdict).toMatchSnapshot();
        });
      });
    });
  });
};
```

## scene structure

scenes are minimal reproducible code cases from real patterns.

**this collection grows over time** — every time we encounter a new pattern in production that a review should catch (or should not flag), we add it as a scene.

```
skills/review/.test/eval.scenes/
├─ mech-failhides/
│  ├─ empty-catch/
│  │  └─ example.ts        # try { } catch (e) { }
│  ├─ logged-rethrow/
│  │  └─ example.ts        # try { } catch (e) { log(e); throw e; }
│  ├─ return-null/
│  │  └─ example.ts        # try { } catch (e) { return null; }
│  ├─ failfast-throw/
│  │  └─ example.ts        # if (!valid) throw new Error()
│  └─ failfast-silent-return/
│     └─ example.ts        # if (!valid) return
├─ mech-decode-friction/
│  ├─ inline-filter-map/
│  │  └─ example.ts
│  └─ named-transform/
│     └─ example.ts
└─ ... (more focuses)
```

### lifecycle: scene accumulation

1. **observe**: see a pattern in production code (good or bad)
2. **extract**: create minimal scene that reproduces it
3. **eval**: add test case with expected verdict
4. **guard**: future brain changes must pass all scenes

the scene collection is the ground truth. when brain providers change models, we run the eval suite and immediately see regressions.

## why genTempDir

- **isolation**: each test gets its own directory, no cross-test pollution
- **git support**: `git: true` initializes repo, enables `--diffs since-main`
- **cleanup**: auto-prunes directories older than 7 days
- **debuggable**: slug in directory name helps identify which test created it

## why brain as evaluator

- **structured output**: `brain.choice.ask({ schema })` enforces zod schema
- **auto-discovery**: `genContextBrain({ choice })` finds installed brains by slug
- **cost-effective**: `anthropic/claude/sonnet-4.5` balances cost and capability for evaluation
- **brain-agnostic**: swap evaluator slug to compare different judges

## evaluation

### how well does it solve the goals?

| goal | solution quality |
|------|------------------|
| detect brain regressions | ✓ scenes are ground truth |
| compare brains | ✓ swap slug, rerun evals |
| accumulate patterns | ✓ add scenes over time |
| fast feedback | ✓ 10 minutes vs weeks of manual test |

### pros

- **ground truth**: scenes encode what "correct" looks like
- **regression detection**: brain changes immediately show impact
- **institutional memory**: every pattern we encounter becomes a scene
- **brain-agnostic**: test any brain against same scenes

### cons

- **evaluator brain cost**: each eval case costs one brain call
- **scene maintenance**: must keep scenes current with rules
- **false confidence**: evaluator brain might agree with wrong verdicts

### edgecases

| edgecase | how we handle it |
|----------|------------------|
| scene dir absent | test fails with clear error |
| rules glob empty | generator output is empty, evaluator flags |
| evaluator disagrees with expected | snapshot diff shows regression |
| brain API down | test fails, retry later |
