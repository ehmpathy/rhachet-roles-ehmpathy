# blackbox criteria: review evals

## terms

| term | definition |
|------|------------|
| rubric | a .yml file with multiple focuses (e.g., `implementation.yml`) |
| focus | a named bundle of rules within a rubric |
| scene | minimal code case that demonstrates a pattern (good or bad) |
| eval case | focus + scene + checks = what to verify |
| check | one specific finding to verify (present or absent) |

---

## usecase.1 = run a single eval case

```
given an eval case file with focus, scene, and checks
  when the eval is executed
    then the focus skill runs against the scene
      sothat we get generator output
    then the evaluator brain judges the output
      sothat we get a structured verdict
    then each check has a result (TP/FP/TN/FN)
      sothat we know what passed and failed
    then pass = all results are TP or TN
      sothat any FP or FN = fail
```

## usecase.2 = verify sensitivity (catches real issues)

```
given a scene with a real violation
  given a check with expected=present
    when the eval is executed
      then observed=present (generator flagged it)
        sothat result=true-positive
      then the finding matches the reason description
        sothat we know it found the right issue

given a scene with a real violation
  given a check with expected=present
    when the generator misses it
      then observed=absent
        sothat result=false-negative
      then the eval fails
        sothat we catch leniency
```

## usecase.3 = verify specificity (avoids false alarms)

```
given a scene with valid code
  given a check with expected=absent
    when the eval is executed
      then observed=absent (generator stayed quiet)
        sothat result=true-negative
      then no blocker or nitpick for that pattern
        sothat valid code is not flagged

given a scene with valid code
  given a check with expected=absent
    when the generator flags it anyway
      then observed=present
        sothat result=false-positive
      then the eval fails
        sothat we catch over-pedantic brains
```

## usecase.4 = run all evals for a focus

```
given multiple eval cases for the same focus
  when all evals are executed
    then each case gets its own verdict
      sothat we see which scenes pass/fail
    then overall pass = all cases pass
      sothat one failure = focus has issues
```

## usecase.5 = compare brains via evals

```
given a set of eval cases
  given brain A and brain B
    when evals run with brain A as generator
      then we capture all verdicts
    when evals run with brain B as generator
      then we capture all verdicts
    then we can compare sensitivity and specificity
      sothat we pick the better brain
```

## usecase.6 = compare brains via eval verdicts

```
given the same eval cases
  given evaluator brain A and evaluator brain B
    when evals run with evaluator A
      then we capture verdicts
    when evals run with evaluator B
      then we capture verdicts
    then we can compare evaluator agreement
      sothat we know if evaluators are consistent
```

## usecase.7 = add a new scene from production

```
given a pattern observed in production
  when a minimal scene is created
    then the scene goes in eval.scenes/$focus/
      sothat it can be used by eval cases
    then an eval case is created with checks
      sothat the pattern is tested
    then future brain changes must pass it
      sothat regressions are caught
```

## usecase.8 = update snapshots after brain change

```
given a brain change (new model, new provider)
  when evals are run
    then some snapshots may differ
      sothat the change is visible in PR diff
    then reviewer can approve or reject
      sothat intentional changes are accepted
    when reviewer accepts
      then snapshots are updated
        sothat new behavior is the baseline
```

## usecase.9 = run evals via jest

```
given eval case files in eval.cases/
  when npm run test:integration -- --scope review.eval
    then all eval cases execute
      sothat we get test results
    then verdicts are snapshot tested
      sothat changes are visible in diffs
    then exit 0 = all pass, exit 1 = failures
      sothat CI can gate on evals
```

## usecase.10 = run evals for specific focus

```
given eval cases for multiple focuses
  when npm run test:integration -- --scope mech-failhides
    then only failhides eval cases run
      sothat feedback is fast
    then unrelated focuses are skipped
      sothat we focus on what matters
```

## exchange: eval case file format

### input: eval case YAML

```yaml
focus: mech-failhides
scene: eval.scenes/mech-failhides/empty-catch

evaluator:
  brain: anthropic/claude/sonnet-4.5

checks:
  - slug: swallowed-error
    description: should flag empty catch block as blocker
    expected: present
    severity: blocker
    reason: |
      the catch block is empty and swallows the error without:
      - emitting a log
      - rethrowing it
      - returning a value with context

      keywords: swallow, hidden, empty catch
```

### output: verdict JSON

```json
{
  "checks": [
    {
      "slug": "swallowed-error",
      "expected": "present",
      "observed": "present",
      "severityObserved": "blocker",
      "result": "true-positive",
      "evidence": "🔴 blocker: empty catch block swallows error at line 12"
    }
  ],
  "pass": true,
  "summary": "1/1 checks passed (1 TP, 0 FP, 0 TN, 0 FN)"
}
```

## boundary conditions

### severity combinations

| expected | severity | observed blocker | observed nitpick | result |
|----------|----------|------------------|------------------|--------|
| present | blocker | yes | - | TP |
| present | blocker | no | yes | FN (wrong severity) |
| present | blocker | no | no | FN |
| present | nitpick | - | yes | TP |
| present | nitpick | yes | - | FN (wrong severity) |
| present | blocker\|nitpick | yes | - | TP |
| present | blocker\|nitpick | - | yes | TP |
| absent | blocker | yes | - | FP |
| absent | blocker | no | yes | TN (nitpick ok) |
| absent | blocker | no | no | TN |
| absent | blocker\|nitpick | yes | - | FP |
| absent | blocker\|nitpick | - | yes | FP |
| absent | blocker\|nitpick | no | no | TN |

### edge cases

```
given scene directory does not exist
  when eval is executed
    then test fails with clear error
      sothat the problem is obvious

given rules glob matches zero files
  when focus skill runs
    then generator outputs empty findings
      sothat checks expected=present will FN

given evaluator brain API is down
  when eval is executed
    then test fails with network error
      sothat retry is possible later

given scene has no typescript files
  when focus skill runs
    then generator outputs no findings
      sothat checks expected=absent will TN

given check reason is vague
  when evaluator judges
    then evaluator may match wrong finding
      sothat reasons must be specific

given multiple findings match one check
  when evaluator judges
    then any match = observed=present
      sothat we don't miss findings
```

## exit codes

| code | scenario |
|------|----------|
| 0 | all evals pass (all TP or TN) |
| 1 | test framework error (jest failure) |
| 2 | eval failures (FP or FN detected) |

## confusion matrix reference

| expected | observed | result | meaning |
|----------|----------|--------|---------|
| present | present | TP | correctly flagged |
| present | absent | FN | missed issue |
| absent | absent | TN | correctly quiet |
| absent | present | FP | false alarm |

pass = all TP or TN
fail = any FP or FN
