# 1.vision: fix declapract.upgrade for missing variables and pnpm issues

## the outcome world

### what does a day-in-the-life look like with this in place?

**before**: a bot runs `rhx declapract.upgrade exec` on an old repo. declapract throws an error about missing variables (e.g., `slackReleaseWebHook` was added to practices last month, but this repo's declapract.use.yml was last touched 2 years ago). the bot considers this a "blocker" and escalates to human. human sighs, finds a recent repo, copies the variables, re-runs. 20 minutes wasted.

**after**: the bot sees the missing variable error, recognizes it as a resolvable constraint (not a blocker), looks up recent repos in the org via `git.repo.get`, finds a declapract.use.yml with the latest variables, updates the local config, and continues. human never interrupted.

### what's the before/after contrast?

| aspect | before | after |
|--------|--------|-------|
| missing variables | blocker, escalate to human | constraint, handle via lookup |
| pnpm version issues | confusion about pnpm-workspace.yml | clear instruction: use 10.24.0 |
| bot autonomy | limited by false blockers | can handle common issues |
| upgrade success rate | ~50% without human help | ~90% without human help |

### what's the "aha" moment where the value clicks?

when a bot successfully upgrades 5 repos in a row without human intervention, because each time it hit a missing variable error, it just looked up the latest pattern from a sibling repo.

## user experience

### what usecases do folks fulfill? what goals?

1. **bulk repo upgrades**: upgrade all repos in an org to latest practices
2. **single repo maintenance**: keep a repo in sync with latest best practices
3. **new repo onboarding**: inherit current variables from sibling repos

### what contract inputs & outputs do they leverage?

**inputs**:
- `rhx declapract.upgrade init` — creates route, binds to branch
- `rhx declapract.upgrade exec` — runs the upgrade

**outputs**:
- upgraded declapract.use.yml with latest variables
- upgraded package.json with correct pnpm version
- applied best practices

### what would it look like to leverage them?

```sh
# bot hits missing variable error during upgrade
# error: variable 'slackReleaseWebHook' is required but not provided

# bot follows stone instructions:
# 1. look up recent repos
rhx git.repo.get repos --repos 'ehmpathy/*' | head -20

# 2. get declapract.use.yml from a recently updated repo
rhx git.repo.get lines --in ehmpathy/domain-objects --paths 'declapract.use.yml'

# 3. copy missing variables, re-run
rhx declapract.upgrade exec
```

### what timelines do they go through?

1. init route (~10s)
2. run upgrade exec
3. if missing variable error: lookup + update (~30s additional)
4. if pnpm error: fix packageManager field (~10s additional)
5. continue with standard upgrade flow

## evaluation

### how well does it solve the goals?

| goal | solution quality |
|------|------------------|
| reduce human interruptions | strong — most errors become self-addressable |
| increase upgrade success rate | strong — common blockers addressed |
| maintain simplicity | good — just adds instructions to extant stones |

### what are the pros? the cons?

**pros**:
- minimal code change (just stone content updates)
- teaches bots a generalizable pattern (lookup sibling repos)
- no new dependencies or infrastructure

**cons**:
- requires sibling repos to be present and up-to-date
- lookup adds ~30s to failure recovery
- bots might copy variables that are repo-specific (should filter)

### what edgecases exist and how do our contracts keep users in a pit of success?

| edgecase | mitigation |
|----------|------------|
| no sibling repos with declapract | instruction to escalate to human |
| sibling repos also outdated | instruction to check 3-5 repos |
| org has no local clones | instruction to use github api fallback |
| pnpm-workspace.yml committed | no recovery needed, just don't add more |
| pnpm-workspace.yml uncommitted | instruction to remove it |

### variables: safe to copy vs repo-specific

based on review, here's initial guidance (to be validated with human):

| variable | safe to copy? | why |
|----------|---------------|-----|
| `organizationName` | yes | same across org |
| `projectName` | no | unique per repo |
| `infrastructureNamespaceId` | no | unique per repo/env |
| `slackReleaseWebHook` | no | unique per repo/team |

**rule**: when lookup of peer repos, only copy variables that are org-wide. always set `projectName` to current repo name.

## implementation

add recovery instructions to `1.upgrade.invoke.stone` that teach bots how to self-recover from two common problems:

### 1. absent variables error

add a section like:

```
## on absent variables error

if declapract throws an error about absent variables (e.g., "variable 'X' is required"):

1. this is a resolvable constraint, not a blocker
2. lookup recent repos: `rhx git.repo.get repos --repos 'ORG/*' | head -20`
3. get declapract.use.yml from 3-5 recent repos until current config found:
   `rhx git.repo.get lines --in ORG/REPO --paths 'declapract.use.yml'`
4. match useCase field (e.g., npm-package, lambda-service)
5. copy only org-wide variables (organizationName)
6. do NOT copy repo-specific variables (projectName, infrastructureNamespaceId, slackReleaseWebHook)
7. always set projectName to current repo name
8. re-run: `rhx declapract.upgrade exec`
9. if 3-5 repos checked and none have current config, escalate to human
```

### 2. pnpm version issues

add a section like:

```
## on pnpm issues

if pnpm-workspace.yml errors or pnpm version issues:

1. set packageManager in package.json to "pnpm@10.24.0"
2. if pnpm-workspace.yml exists but is uncommitted (check via git status), remove it
3. re-run: `rhx declapract.upgrade exec`
```

these instructions turn "blockers" into recoverable constraints with clear steps.

## what is awkward?

### what feels off or forced?

1. **copying variables blindly** — some variables like `infrastructureNamespaceId` are repo-specific. we should probably instruct bots to copy only "common" variables.

2. **pnpm version pinning** — feels like a workaround for a pnpm design decision. ideally declapract-typescript-ehmpathy would handle this, not the upgrade instructions.

3. **lookup as error recovery** — ideally declapract would auto-suggest missing variables or have a `--init-missing` flag. we're working around tool limitations.

### where does the design fight the user's mental model?

the user (bot) expects `declapract.upgrade exec` to "just work." having to do a multi-step lookup feels like extra work. but it's better than escalating to human.

### what tradeoffs feel uncomfortable?

**autonomy vs correctness**: letting bots copy variables from sibling repos could lead to incorrect values being copied. but the alternative (always escalate) is worse for throughput.
