# self-review: has-questioned-assumptions (round 2)

## deeper review of technical assumptions

### assumption: "rsync -a preserves permissions"
**what we assumed**: rsync with -a flag copies executable bits from source to dist
**evidence**: tested — build passed after chmod +x on source files
**what if opposite?**: if rsync didn't preserve, we'd need post-rsync chmod in build
**verdict**: holds — rsync -a includes --perms by default

### assumption: "subcommand router via case statement is idiomatic"
**what we assumed**: bash case statement is the right pattern for subcommand dispatch
**evidence**: common pattern in shell scripts (git, npm, docker all use it)
**alternatives**: could use getopt, but overkill for 2 subcommands
**verdict**: holds — minimal and conventional

### assumption: "exit code 2 for prereq failures"
**what we assumed**: exit 2 distinguishes prereq failures from other errors
**evidence**:
- exit 0 = success
- exit 1 = argument error
- exit 2 = prereq failure (declapract.use.yml absent, pnpm absent)
**verdict**: holds — clear exit code semantics, matches tests

### assumption: "fnm use will succeed"
**what we assumed**: fnm is available and .nvmrc exists
**evidence**: this is an ehmpathy repo convention — all repos have .nvmrc
**exception**: if fnm not installed, exec fails with clear error (fnm: command not found)
**verdict**: holds — acceptable failure mode

### assumption: "source command works for handler dispatch"
**what we assumed**: using `source` to load init.sh/exec.sh is safe and correct
**what if opposite?**: could use function definitions instead of sourced files
**evidence**:
- source allows clean separation of files
- variables from parent skill (SKILL_DIR, PROJECT_ROOT) are available
- common pattern in bash skill composition
**simpler approach?**: could inline all code in single file — but reduces readability
**verdict**: holds — clean file separation, proven pattern

### assumption: "templates are the right place for stones"
**what we assumed**: templates/ directory under skill contains route templates
**what if opposite?**: could put templates in a shared location
**evidence**:
- stones are skill-specific, not shared
- colocation with skill code aids maintenance
- follows pattern of other skills (init.declapract.upgrade was the same)
**verdict**: holds — skill-local templates are appropriate

### assumption: "SKIP_PNPM_CHECK env var is correct test escape"
**what we assumed**: test can bypass pnpm check via environment variable
**what if opposite?**: could require pnpm in all tests
**evidence**:
- CI environments may not have pnpm in PATH during test
- test validates prereq check logic, not pnpm availability
- similar pattern: SKIP_ROUTE_BIND for rhachet availability
**verdict**: holds — necessary for test isolation

## conclusion

after deeper examination:
- no hidden flaws in technical assumptions
- each assumption has evidence or explicit tradeoff
- design is minimal for requirements
