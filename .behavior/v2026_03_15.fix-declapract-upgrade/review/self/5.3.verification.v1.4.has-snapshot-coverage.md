# self-review: has-snapshot-coverage (round 4)

## question my assumptions

the guide says:
> "for CLI: is stdout/stderr captured?"

I have stdout captured. is stderr captured?

### stderr check

in case1, the skill outputs to stdout, not stderr. error cases (case2, 4, 5, 8, 9) output to stderr.

should error cases have snapshots?

### error case analysis

| case | output | snapshot useful? |
|------|--------|------------------|
| case2 | "error: not a declapract repo" | no — simple, assertion checks it |
| case4 | "error: not a declapract repo" | no — same as case2 |
| case5 | "error: pnpm not found" | no — simple, assertion checks it |
| case8 | "error: invalid subcommand..." | no — simple, assertion checks it |
| case9 | "usage: ..." | maybe — but usage is static text |

### decision

error messages are:
1. short (one line)
2. static (no dynamic content)
3. verified by assertions

snapshots would add noise without value.

## conclusion

snapshot coverage is correct:
- **init success**: yes (user-faced, dynamic)
- **errors**: no (short, static, assertions sufficient)

the gate holds.
