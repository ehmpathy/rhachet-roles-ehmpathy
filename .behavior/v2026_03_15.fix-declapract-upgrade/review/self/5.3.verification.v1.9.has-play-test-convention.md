## what is a journey test?

a journey test tests a feature from the user's perspective — from input to output, through the full flow. it answers: "does this behavior work as the user expects?"

the guide's preferred convention:
- `feature.play.test.ts` — journey test
- `feature.play.integration.test.ts` — journey test that needs integration runner

## what does this behavior have?

the behavior adds `declapract.upgrade.integration.test.ts`.

let me look at what it tests:

|------|---------------|
| case1 | init: creates route, outputs turtle vibes, no .sh files |
| case2 | init: fails with "not a declapract repo" |
| case3 | exec: starts upgrade flow |
| case4 | exec: fails with "not a declapract repo" |

| case8 | invalid subcommand shows error |

each case tests the skill from user input to user output. they are journey tests.

## does this behavior need `.play.`?

the guide says:
> "if not supported, is the fallback convention used?"

let me check the repo convention:

```
skills/**/*.play.*.ts    → 0 files
skills/**/*.test.ts      → 15 files
```

no skill in this repo uses `.play.`. all use `.integration.test.ts`.

the convention is established: `.integration.test.ts` for skill journey tests.

## why this convention works

skills in this repo are shell executables. they have one mode of test:
1. run the skill in a temp repo
2. check the output and file effects

there's no distinction between "unit" and "integration" or "play" tests for shell-based skills. the test either runs the skill or it doesn't.

`.integration.test.ts` accurately describes what's tested: the skill integrated with a filesystem context.

## should I introduce `.play.` for this behavior?

no. reasons:

1. **consistency**: 14 other skills use `.integration.test.ts`
2. **clarity**: `.integration.test.ts` describes the test type accurately
3. **overhead**: a new convention requires update to all 14 other files
4. **value**: `.play.` adds no clarity over `.integration.` for shell-based skills

## what I learned

the `.play.` convention is useful when a repo has multiple test types and needs to distinguish journey tests from unit tests. but when a repo has one test type (integration tests of shell-based skills), the extant convention suffices.

conventions serve understanding. when the extant convention serves understanding, follow it.

## conclusion

the journey tests exist and are located correctly. they use the repo's fallback convention (`.integration.test.ts`). the convention serves understanding.
