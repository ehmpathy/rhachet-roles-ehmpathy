# self-review: has-divergence-addressed (round 2)

## divergence 1: filediff markers ([~] vs [+]/[-])

### the divergence

blueprint used `[~] rename` markers to indicate files that would be renamed:
- `init.declapract.upgrade.sh → declapract.upgrade.sh`
- `init.declapract.upgrade/ → declapract.upgrade/`

actual git status shows separate add and delete operations, not renames.

### resolution: backup

**rationale:**
1. git does not track renames — it infers them via content similarity
2. our files had content changes (new subcommand router, new init.sh, new exec.sh)
3. with significant content changes, git sees add+delete, not rename
4. document as add+delete matches what `git status` reports
5. this is accurate to reality, not a shortcut

### skeptic challenge

**q: is this truly an improvement, or just laziness?**

a: improvement. blueprint's [~] markers were conceptual plan notation. to document what git actually did is more accurate. a future reviewer can see exactly what happened via git log.

**q: did we just not want to do the work the blueprint required?**

a: the blueprint did not require actual git renames. it described the conceptual transformation. the implementation achieved the goal (skill renamed) via the standard add+delete pattern.

**q: could this divergence cause problems later?**

a: no. git history is clean. `git log --follow` can still trace file history if needed. the functional outcome is identical.

**verdict:** backup is valid.

**q: did we just not want to do the work the blueprint required?**
