## divergence 1: filediff markers

### divergence statement
blueprint declared `[~] rename` markers for files that moved from `init.declapract.upgrade/` to `declapract.upgrade/`. actual git shows these as separate `[+] add` and `[-] delete` operations.

### resolution: backup

### rationale
1. **git behavior is deterministic.** git does not track renames explicitly. it computes similarity between deleted and added files. when content differs significantly (as it does here — new subcommand router, new files), git cannot infer a rename.

2. **we did not control this.** the blueprint assumed git would see renames. git did not. this is git's algorithm, not a choice we made.

3. **the documentation is accurate.** the evaluation's filediff tree shows what git actually reports. this is correct.

### skeptic questions

**q: could you have used `git mv` to create a rename?**

a: `git mv` only helps when content stays the same. we changed content significantly. `git mv` would not produce a different result.

**q: is the blueprint wrong, or is the implementation wrong?**

a: neither is wrong. the blueprint expressed intent (conceptual rename). the implementation achieved intent (new skill with same purpose). git's view (add+delete) is a third perspective that neither contradicts.

**q: should the blueprint be updated for next time?**

a: optionally. future blueprints could note that git may not see renames when content changes. but this is a minor documentation improvement, not a defect.

### verdict: backup is valid

## divergence 2: codepath markers

### divergence statement
blueprint used `[←] reused` markers for codepaths taken from prior implementation. actual evaluation used `[+] created` markers.

### resolution: backup

### rationale
1. **the code was not copied.** I wrote new code in new files. the logic is similar to 1.upgrade.invoke.sh, but the implementation is fresh.

2. **`[←]` was design guidance.** the blueprint said "take this logic from here" as a design hint. it did not require literal copy-paste.

3. **`[+]` is more accurate.** new files contain new code. future readers understand this better than `[←]` which implies transplant.

### skeptic questions

**q: did you actually write new code, or did you copy?**

a: I wrote new code. compare exec.sh to the old 1.upgrade.invoke.sh:
- different structure (prereq checks separated)
- different output messages (turtle vibes integrated)
- same upgrade commands (fnm, pnpm add, declapract apply, etc.)

the upgrade commands are the same because they are the correct commands to run. but the rest of the code is new.

**q: should you have literally copied to preserve `[←]` accuracy?**

a: no. literal copy would have been worse. the new structure is cleaner.

### verdict: backup is valid

## divergence 3: test case count

### divergence statement
blueprint declared 8 test cases. actual implementation has 9 test cases.

### resolution: backup

### rationale
1. **case9 tests a real error path.** when user runs `rhx declapract.upgrade` without `init` or `exec`, the skill should error with a helpful message. this path was not in the blueprint.

2. **more tests are better.** this is additive coverage, not scope creep. we did not skip any blueprint tests.

3. **blueprint cannot anticipate all edge cases.** the blueprint author focused on the main flows. implementation revealed an edge case worth a test.

### skeptic questions

**q: is this scope creep disguised as a test?**

a: no. the test covers error behavior that was already implemented. the error path exists in declapract.upgrade.sh lines 82-89. the test validates it works.

**q: should we delete case9 to match the blueprint?**

a: no. to delete test coverage to match a document is backwards. tests verify behavior; documents describe intent.

### verdict: backup is valid
