## edge cases NOT covered

### `--help` flag

the skill supports `--help` but the playtest doesn't verify it. this is a low-priority edge case because:
1. `--help` is standard CLI behavior
2. the automated tests don't test it either
3. output is obvious from the command

**appropriate gap**

### double exec

the playtest doesn't explicitly run exec twice. but:
1. exec is idempotent by design (declapract apply is idempotent)
2. each exec run is independent
3. there's no state to corrupt

**appropriate gap**

### pnpm not found

the playtest bypasses pnpm check because the sandbox has no pnpm. the automated tests (case5) verify this error path with:
```typescript
expect(result.status).toEqual(2);
expect(result.stdout).toContain('pnpm not found');
```

**covered by automated tests**

## what could go wrong that's NOT covered?

I'll brainstorm failure modes:

1. **route directory permission error**: what if `.route/` can't be created?
   - this is a filesystem error, not a skill behavior
   - OS error messages would surface
   - **not skill-specific, appropriate gap**

2. **concurrent init calls**: what if two processes init simultaneously?
   - findsert semantics handle this (upsert to same route)
   - race condition could create partial state, but skill doesn't prevent this
   - **low probability, appropriate gap**

3. **exec in wrong directory**: what if user runs exec outside repo?
   - no declapract.use.yml → error message (test 4, 5)
   - **covered**

4. **exec with corrupt declapract.use.yml**: what if yml is malformed?
   - skill checks file existence, not validity
   - declapract itself would fail with parse error
   - **external tool behavior, appropriate gap**

## conclusion

edge cases are well covered:
- error paths: tests 4, 5, 6, 7
- idempotency: tests 8, 9
- boundaries: tests 1, 8, 9

appropriate gaps exist for:
- `--help` (low priority)
- double exec (deterministic)
- pnpm not found (automated tests)
- filesystem errors (OS level)

the gate holds.
