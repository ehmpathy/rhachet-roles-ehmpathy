# self-review: has-questioned-deletables

## reviewed components

### 1. declapract.upgrade.sh (router)
**can it be removed?** no — required entry point for subcommand dispatch
**simplest version?** yes — parses subcommand, sources handler

### 2. declapract.upgrade/init.sh
**can it be removed?** no — creates route structure for workflow
**simplest version?** yes — copies stones/guards, binds route, outputs vibes

### 3. declapract.upgrade/exec.sh
**can it be removed?** no — contains the upgrade logic that was the whole point
**simplest version?** yes — runs fnm, pnpm add, declapract apply, pnpm install, pnpm fix

### 4. declapract.upgrade/output.sh
**can it be removed?** considered merge into router, but kept separate for reuse by both init.sh and exec.sh
**simplest version?** yes — print functions only

### 5. templates/*.stone and templates/*.guard
**can any be removed?** no — these define the workflow steps for the route
**simplest version?** yes — each stone is minimal instruction set

### 6. templates/1.upgrade.invoke.sh (DELETED)
**removed entirely** — this was the key deletion. upgrade logic now lives in exec.sh within the skill, not copied to the route.

## conclusion

the simplest version that works:
- skill with init/exec subcommands
- init creates route with workflow stones (no executable code)
- exec runs upgrade directly from skill

no further deletions identified. the design is minimal.
