# self-review: has-consistent-conventions (round 7)

## actual code comparison

read the actual implementation files and compared to extant skills.

### observation 1: dispatch pattern differs

**git.commit.uses.sh:**
```sh
exec "$SCRIPT_DIR/git.commit.uses.local.sh" "${ARGS[@]}"
```
uses `exec` to replace current process with subcommand.

**declapract.upgrade.sh:**
```sh
source "$SKILL_DIR/declapract.upgrade/init.sh"
```
uses `source` to run subcommand in current shell.

**is this an inconsistency?**
- `exec` replaces process — good for standalone subcommands
- `source` runs in current shell — good when subcommands need shared state

**why we use source:**
- init.sh and exec.sh need `SKILL_DIR` variable
- init.sh and exec.sh need output functions from output.sh
- sourced files are not standalone-invokable

**verdict:** different pattern is justified. `source` is correct for our use case.

### observation 2: variable name differs

**git.commit.uses.sh:** uses `SCRIPT_DIR`
**declapract.upgrade.sh:** uses `SKILL_DIR`

**note:** both mean the same. `SKILL_DIR` is more descriptive for skills.
**verdict:** acceptable variation. no change needed.

### observation 3: header format is consistent

both skills have identical header structure:
```
######################################################################
# .what = ...
# .why = ...
# usage: ...
# guarantee: ...
######################################################################
```

**verdict:** consistent

### observation 4: argument parse

both skills handle rhachet passthrough args (`--skill`, `--repo`, `--role`).

**verdict:** consistent

## conclusion

round 7 confirms: the dispatch pattern difference (`exec` vs `source`) is justified by different requirements. all other conventions align.
