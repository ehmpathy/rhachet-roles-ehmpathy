# self-review: has-questioned-requirements

## requirement 1: rename `init.declapract.upgrade` → `declapract.upgrade`

**who said this was needed?**
the wisher, in `0.wish.md`.

**what evidence supports this?**
- current name `init.declapract.upgrade` implies a one-shot init operation
- new name `declapract.upgrade` with subcommands (`init`, `exec`) better matches the multi-phase workflow
- mirrors familiar patterns: `terraform init/apply`, `git init/commit`

**what if we didn't do this?**
- `init.declapract.upgrade` would remain as-is
- we'd need a separate skill for `exec`, like `exec.declapract.upgrade` — awkward
- or we'd overload `init.declapract.upgrade` with a `--exec` flag — also awkward

**is the scope appropriate?**
yes — this is a straightforward rename with subcommand pattern. minimal scope.

**verdict: requirement holds** ✓

## requirement 3: move exec logic from route to skill

**who said this was needed?**
the wisher, with security rationale:
> "that way, we dont have to manage the chmod of invoke.upgrade.sh and eliminate the risk that if a malintent robot updates invoke.upgrade.sh it could have undesired privs"

**what evidence supports this?**
- currently, `1.upgrade.invoke.sh` is copied to `.route/` with `chmod 555`
- `.route/` is mutable (not version-controlled)
- a compromised agent could theoretically modify the template before copy
- the chmod 555 is a mitigation, but not a guarantee

**what if we didn't do this?**
- we'd continue to manage chmod
- the theoretical attack surface remains
- not a critical vulnerability, but a code smell

**could we achieve this simpler?**
considered alternatives:
1. **keep `.sh` in route but don't chmod** — worse, now writable
2. **don't have exec at all** — then the skill is useless
3. **inline the exec logic in the skill** — this is what we propose ✓

**is the scope appropriate?**
yes — move 15 lines of bash from a template to the skill. minimal change.

**verdict: requirement holds** ✓

## questioned: do we even need the route?

this wasn't in the wish, but let me question it.

**why do we have a route at all?**
- routes track progress through multi-stone workflows
- the declapract upgrade has multiple phases: upgrade → detect hazards → repair → reflect
- without a route, there's no structured workflow

**what if we just ran `declapract.upgrade` as a one-shot?**
- we'd lose the structured workflow
- humans would have to remember what to do next
- no guardrails, no reflection prompts

**verdict: route is needed** ✓

## questioned: is `exec` the right verb?

alternatives:
- `apply` — but declapract already uses `apply`, confuses the user
- `run` — too generic
- `upgrade` — `declapract.upgrade upgrade` is awkward
- `invoke` — matches the old filename, but wordy

**verdict: `exec` is the right choice** ✓
