# self-review: has-questioned-assumptions

## assumption 1: skill code is more secure than route code

**what do we assume here?**
we assume that code in the skill (version-controlled, in `src/`) is safer than code copied to `.route/` (mutable, local).

**what evidence supports this?**
- skills go through PR review before merge
- skills are published via npm, with version history
- `.route/` is gitignored, mutable, local-only
- a malicious actor with repo write access could modify route files undetected

**what if the opposite were true?**
if a malicious actor had commit access, they could also modify the skill source. however:
- skill changes go through CI/CD and PR review
- route changes are local and unreviewed
- the delta is "reviewed vs unreviewed", not "impossible vs possible"

**verdict: assumption holds** ✓

## assumption 2: users understand the init/exec pattern

**what do we assume here?**
we assume users will intuitively understand that `init` sets up and `exec` runs.

**what evidence supports this?**
- terraform uses `init`/`plan`/`apply`
- git uses `init`/`add`/`commit`
- npm uses `init`/`install`
- the pattern is ubiquitous in CLI tools

**what if the opposite were true?**
some users might expect `declapract.upgrade` to just... upgrade. one command.

**counterexample:**
`npm upgrade` is one command. it doesn't have `npm upgrade init` and `npm upgrade exec`.

**is this a problem?**
the wish explicitly asks for init/exec separation. the wisher wants:
1. route creation (for workflow progress)
2. upgrade execution (the actual work)

these are distinct operations. one command could do both, but then we lose granular control.

**mitigation:**
- if user runs `exec` without `init`, we fail with a clear message
- good error messages teach the pattern

**verdict: assumption holds, with mitigation** ✓

## assumption 3: the route is needed for declapract upgrades

**what do we assume here?**
we assume declapract upgrades benefit from structured workflows (detect hazards, repair defects, reflect).

**did the wisher say this?**
no — the wisher didn't mention the route at all. the wish focused on rename and security.

**what if we removed the route?**
- no structured workflow
- no guardrails
- but also: simpler, one command

**what evidence supports the route?**
the current skill creates a route with multiple stones:
- `1.upgrade.invoke`
- `2.detect.hazards`
- `3.1.repair.test.defects`
- `3.2.reflect.test.defects`
- `3.3.repair.cicd.defects`
- `3.4.reflect.cicd.defects`

this suggests declapract upgrades are multi-phase operations that benefit from structured progress.

**verdict: assumption holds** ✓ (we keep the route)

## assumption 4: `exec` is the best verb

**what do we assume here?**
we assume `exec` is better than alternatives like `apply`, `run`, `upgrade`.

**did the wisher say this?**
no — the wisher said "invocable via `declapract.upgrade exec`", but this might have been an example, not a mandate.

**what if we used a different verb?**
| verb | issues |
|------|--------|
| `apply` | conflicts with `declapract apply` |
| `run` | too generic, doesn't convey "upgrade" |
| `upgrade` | `declapract.upgrade upgrade` is redundant |
| `do` | too casual |
| `invoke` | matches old filename, wordy |
| `exec` | explicit, no conflicts |

**verdict: assumption holds** ✓

## assumption 5: chmod 555 is insufficient security

**what do we assume here?**
we assume `chmod 555` (read+execute, no write) is not a strong security guarantee.

**what evidence supports this?**
- root can override any chmod
- a process with sufficient privileges can modify the file
- symlink attacks could bypass chmod
- the file is copied before chmod, so the template could be tainted

**what if chmod 555 were sufficient?**
then we could keep the current design. but:
- we'd still have to manage chmod (complexity)
- we'd still have the attack surface (template modification)
- the design would still feel like a workaround

**verdict: assumption holds** ✓

all assumptions hold. no hidden issues found.
