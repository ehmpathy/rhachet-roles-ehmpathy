# self-review: has-questioned-assumptions

## assumptions reviewed

### 1. "skill execution is safer than route execution"
**assumption**: code in versioned skill is safer than code copied to mutable route
**evidence**:
- skill code is git-tracked and reviewed before merge
- route files are generated at runtime and could be modified
- chmod 555 is a guardrail, not a guarantee
**verdict**: holds — this is the core motivation for the refactor

### 2. "init/exec pattern is intuitive"
**assumption**: users understand terraform/git/npm analogs
**evidence**:
- terraform has `init` and `apply`
- git has `init` and `commit`
- npm has `init` and `install`
**verdict**: holds — common pattern in developer tools

### 3. "exec should work standalone (no init required)"
**assumption**: users may want quick one-off upgrades without ceremony
**evidence**:
- wisher confirmed this explicitly
- route is optional — only needed for workflow structure
**verdict**: holds — confirmed by requirement

### 4. "route should not contain .sh files"
**assumption**: executable code should not be copied to mutable directories
**evidence**:
- this was the explicit wish
- eliminates chmod management
- reduces attack surface
**verdict**: holds — core requirement

### 5. "templates should contain only stones and guards"
**assumption**: templates are declarations, not executables
**evidence**:
- stones describe workflow steps
- guards define completion criteria
- executables belong in skill code
**verdict**: holds — clear separation of concerns

## simpler approach considered

could we eliminate init entirely? no — the route provides:
- workflow structure for mechanic to follow
- branch bind for route.drive
- progress artifacts for each step

the current design is minimal for requirements.
