## refocus: what is edge case coverage?

edge case coverage means: can the playtest reveal unexpected behavior at the boundaries?

the playtest is for humans to run byhand. it should surface issues that automated tests might miss.

## fresh look: walk through each playtest step

### test 1: init happy path

**action**: `npx rhachet run --skill declapract.upgrade init`

**what could go wrong?**
- skill not built → error would surface
- skill not linked → error would surface
- output doesn't match expectation → human would notice

**edge case covered?** yes — this is the baseline.

### test 2: route contents

**action**: `ls -la .route/*declapract.upgrade*/`

**what could go wrong?**
- .sh files present → human would see them
- stone files absent → human would see the gap
- extra unexpected files → human would notice

**edge case covered?** yes — visual inspection catches anomalies.

### test 3: exec with prereq bypass

**action**: `SKIP_PNPM_CHECK=1 npx rhachet run --skill declapract.upgrade exec`

**what could go wrong?**
- prereq check doesn't work → automated tests catch this
- output doesn't show start message → human would notice

**edge case considered but not covered**: what if SKIP_PNPM_CHECK doesn't work?

**assessment**: the env var is a test convenience, not a production feature. if it didn't work, exec would fail with "pnpm not found". the playtest would still reveal the behavior.

### test 4-5: error on absent config

**action**: remove declapract.use.yml, run init/exec

**what could go wrong?**
- wrong error message → human would see
- wrong exit code → human could check with `echo $?`
- skill succeeds anyway → obvious failure

**edge case covered?** yes.

### test 6-7: invalid/absent subcommand

**action**: run with bad or no subcommand

**what could go wrong?**
- unhelpful error message → human would notice
- skill crashes instead of clean error → human would notice

**edge case covered?** yes.

### test 8: init idempotency

**action**: run init twice, count routes

**what could go wrong?**
- duplicate routes created → count would change
- second init fails → human would see error
- route contents differ → not explicitly checked

**gap identified**: the playtest doesn't check if route contents are identical after second init.

**assessment**: this is a minor gap. the route is overwritten, not appended. the structure is deterministic. but a paranoid human could add:
```sh
diff -r .route/*declapract.upgrade*/ /tmp/first-route/
```

**verdict**: acceptable gap for a byhand test. the key behavior (no duplicate) is verified.

### test 9: exec standalone

**action**: remove .route, run exec

**what could go wrong?**
- exec requires route → error would surface
- exec creates route accidentally → human would see with `ls .route/`

**edge case covered?** yes — the verify step explicitly checks `.route/` absence.

## what edges are NOT covered?

| edge | why not covered | acceptable? |
|------|-----------------|-------------|
| pnpm not found | sandbox lacks pnpm | yes — automated tests cover |
| fnm not found | sandbox lacks fnm | yes — real env would surface |
| route bind failure | sandbox lacks rhachet route.bind | yes — env var bypasses |
| concurrent init | hard to test byhand | yes — low probability |
| partial route state | would require interrupt | yes — recovery is manual |

## what I learned

edge cases for a CLI skill fall into categories:

1. **input validation**: tests 4-7 cover bad inputs
2. **idempotency**: tests 8-9 cover repeated calls
3. **output verification**: tests 1-2 cover expected output
4. **error paths**: tests 4-7 cover expected errors

the playtest covers all categories. gaps exist where sandbox limitations prevent the test (pnpm, fnm) or where the edge is impractical to test byhand (concurrency).

## conclusion

the playtest covers edge cases within sandbox constraints:
- input validation: yes
- idempotency: yes
- output verification: yes
- error paths: yes

gaps exist for external tool dependencies (pnpm, fnm) which are covered by automated tests.

the gate holds.
