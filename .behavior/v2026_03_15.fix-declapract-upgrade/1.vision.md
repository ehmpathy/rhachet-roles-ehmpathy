# vision: declapract.upgrade

## the outcome world

### before: tangled executables in mutable routes

a mechanic runs `rhx init.declapract.upgrade`. the skill:
1. copies `1.upgrade.invoke.sh` into `.route/v2026_03_15.declapract.upgrade/`
2. sets `chmod 555` to make it executable but not writable
3. the route.drive system expects the mechanic to run the copied shell file

**the friction:**
- the `.sh` file lives in mutable territory (`.route/`)
- `chmod 555` is a guardrail, but not a guarantee
- a malicious actor could modify the template before copy
- a clever exploit could bypass chmod and alter the copied file
- we're managing file permissions when we could just... not copy the file

### after: skill-based execution, route-based tracking

a mechanic runs `rhx declapract.upgrade init`. the skill:
1. creates `.route/v2026_03_15.declapract.upgrade/` with stone/guard files only
2. binds route to current branch

later, the mechanic runs `rhx declapract.upgrade exec`. the skill:
1. runs the upgrade logic directly from the skill code
2. no copied `.sh` files, no chmod dance

**the aha moment:**
> "wait, we don't need to copy executable code into the route. the skill IS the executable. the route just tracks progress."

### day-in-the-life contrast

| aspect | before | after |
|--------|--------|-------|
| init command | `rhx init.declapract.upgrade` | `rhx declapract.upgrade init` |
| exec command | `./1.upgrade.invoke.sh` (in route) | `rhx declapract.upgrade exec` |
| executable location | copied to mutable `.route/` | stays in immutable skill |
| chmod management | required (555) | eliminated |
| attack surface | template → copy → execute | skill → execute |

## user experience

### usecases

1. **init a declapract upgrade route**
   ```sh
   rhx declapract.upgrade init
   ```
   creates route, binds to branch, outputs turtle vibes.

2. **execute the upgrade**
   ```sh
   rhx declapract.upgrade exec
   ```
   runs: fnm use → pnpm upgrade → declapract apply → pnpm install → pnpm fix

3. **check progress (via route.drive)**
   ```sh
   rhx route.drive
   ```
   guides through stones: detect hazards → repair test defects → reflect → etc.

### contract inputs & outputs

**`declapract.upgrade init`**
- input: none (reads `declapract.use.yml` to verify repo)
- output: route directory with stones/guards, branch binding

**`declapract.upgrade exec`**
- input: none (runs in cwd)
- output: upgraded packages, applied practices, fixed lint/format

### timeline

```
t0: human creates branch for upgrade
t1: rhx declapract.upgrade init     ← creates route
t2: rhx declapract.upgrade exec     ← runs upgrade
t3: rhx route.drive                 ← guides through stones
t4: tests pass, pr merged
```

## mental model

### how users describe it to a friend

> "declapract.upgrade is a two-phase skill: `init` sets up the tracking route, `exec` runs the actual upgrade. the route tracks your progress through fixing whatever breaks."

### analogies

- **terraform plan/apply**: `init` is like `plan` (sets up), `exec` is like `apply` (does the work)
- **git init/commit**: `init` creates the structure, `exec` creates the change
- **npm init/install**: `init` sets up package.json, `install` does the work

### terminology

| user might say | we say |
|----------------|--------|
| "start the upgrade" | `declapract.upgrade init` |
| "run the upgrade" | `declapract.upgrade exec` |
| "the upgrade steps" | stones in the route |
| "what's left to do" | `route.drive` |

## evaluation

### how well does it solve the goals?

| goal | solved? |
|------|---------|
| eliminate chmod management | yes - no .sh files copied |
| reduce attack surface | yes - exec logic stays in skill |
| cleaner mental model | yes - init/exec pattern familiar |
| backwards compat | partial - command name changes |

### pros

- **security**: executable code never leaves version-controlled skill
- **simplicity**: no chmod dance, no template copying for shell files
- **clarity**: init/exec pattern mirrors terraform, git, npm
- **consistency**: skill name `declapract.upgrade` matches domain

### cons

- **breaking change**: `init.declapract.upgrade` → `declapract.upgrade init`
- **two commands**: user runs `init` then `exec` (vs one command that sets up everything)

### edgecases & pit of success

| edgecase | handling |
|----------|----------|
| run `exec` without `init` | works — exec is standalone, runs upgrade directly |
| run `init` twice | idempotent - overwrites route (findsert behavior) |
| run `exec` twice | idempotent - declapract apply is idempotent |
| no `declapract.use.yml` | fail with clear message at init/exec time |
| no pnpm | fail with clear message at exec time |

## open questions & assumptions

### assumptions

1. **skill execution is safer than route execution** - we assume the skill code is reviewed and trusted, while route files are mutable
2. **init/exec pattern is intuitive** - users understand terraform/git/npm analogs
3. **route still needed** - we keep the route for progress tracking (stones/guards)

### questions (triaged)

1. **[answered]** should we keep any `.sh` template in the route, or eliminate all of them?
   → no `.sh` files in the route. all shell files stay in the skill.

2. **[answered]** should `exec` require the route to exist, or should it work standalone?
   → `exec` works standalone. no route required. (wisher confirmed)

3. **[answered]** should we add a `--skip-init` flag to `exec` for quick one-off upgrades?
   → no. since `exec` works standalone, no flag needed. (wisher confirmed)

4. **[answered]** should `init` auto-run `exec`?
   → no. separates setup from action, gives user control.

5. **[answered]** should `exec` auto-init if route absent?
   → no. explicit is better than implicit.

6. **[answered]** should we add a `plan` subcommand to preview the upgrade?
   → no. upgrades are deterministic. uncertainty is in what breaks after exec, not in what exec does.

## what is awkward?

### the double-command dance

before: `rhx init.declapract.upgrade` then `./1.upgrade.invoke.sh`
after: `rhx declapract.upgrade init` then `rhx declapract.upgrade exec`

both are two commands. the after version is marginally better (both are `rhx` commands), but the user still runs two things.

**potential future improvement**: could `init` automatically run `exec`? or should `exec` automatically `init` if no route exists?

### the route still has stones for "invoke"

the route includes `1.upgrade.invoke.stone` which says "run the upgrade". but now the upgrade is `rhx declapract.upgrade exec`, not a file in the route.

**resolution**: update `1.upgrade.invoke.stone` to instruct:
```
run: rhx declapract.upgrade exec
```

the stone becomes a pointer to the skill, not a wrapper around a copied executable.

### naming: is `exec` the right verb?

alternatives considered:
- `apply` - but that's declapract's verb, might confuse
- `run` - generic, fine
- `exec` - explicit about execution
- `upgrade` - but that's the noun, `declapract.upgrade upgrade` is awkward

`exec` feels right - it's explicit and doesn't collide with declapract terminology.
