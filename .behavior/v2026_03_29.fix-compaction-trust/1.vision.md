# vision: trust but verify

## the outcome world

### before

mechanic resumes from compaction. summary says "CI fails because of orphan processes." mechanic dives in, writes fixes for orphan processes, commits, pushes, waits for CI. still fails. tries again. 20+ commits later, 3+ hours burned, human finally asks: "did you actually verify that's the problem?"

turns out: obsolete snapshot file. one deletion. done.

the summary was wrong. mechanic trusted it blindly. never verified.

### after

mechanic resumes from compaction. sees the same summary: "CI fails because of orphan processes."

but now there's a brief in memory: **trust but verify — don't even trust yourself.**

mechanic pauses. runs `rhx show.gh.test.errors`. sees the actual error: obsolete snapshot. deletes file. CI green. done in 5 minutes.

### the "aha" moment

> "wait — the summary was just a claim. i should have checked."

the value clicks when the mechanic realizes: compaction summaries are not ground truth. they're inherited conclusions from a prior self who may have been wrong.

## user experience

### usecases

| goal | action |
|------|--------|
| resume from compaction | read summary, then verify claims before you act |
| diagnose CI failure | `rhx show.gh.test.errors` — not trust prior diagnosis |
| understand file state | read the file — not trust "file contains X" |
| check PR status | `gh pr view` — not trust "PR is ready" |

### timelines

**without trust-but-verify:**
1. resume → read summary → act on claims → fail → retry → fail → ... → finally verify → fix

**with trust-but-verify:**
1. resume → read summary → verify claims → fix

the loop collapses. verification upfront prevents the retry spiral.

### contract

the brief teaches:
- **pattern:** claim → verify → act
- **antipattern:** claim → act
- **mantra:** trust but verify — don't even trust yourself

the optional hook reminds:
- emits a nudge via `PostCompact` hook (fires after compaction, before Claude responds)
- lists claim types that may be stale
- points to the brief

## mental model

### how users describe it

> "before i act on what the summary says, i check if it's still true."

> "i don't trust my own notes from yesterday. why would i trust a summary?"

### analogies

- **pilot checklist:** pilots don't trust memory. they verify instruments before takeoff.
- **git pull before push:** you don't assume your branch is current. you check.
- **measure twice, cut once:** carpenters verify before they commit.

### terms

| user says | we say |
|-----------|--------|
| "check first" | verify claims |
| "summary lies" | inherited conclusions may be stale |
| "don't assume" | trust but verify |

## evaluation

### how well does it solve the goals?

| goal | solution |
|------|----------|
| prevent blind trust in summaries | brief establishes verification pattern |
| reduce wasted effort | verification upfront prevents retry loops |
| institutionalize the lesson | brief persists across sessions |

### pros

- low-cost intervention (one brief, optional hook)
- applies broadly (all inherited claims, not just compaction)
- aligns with extant patterns (verify before act)

### cons

- requires mechanic discipline (brief is guidance, not enforcement)
- hook may be noisy if compaction is frequent
- doesn't prevent all misdiagnosis (just encourages verification)

### edgecases

| edgecase | pit of success |
|----------|----------------|
| claim is actually correct | verification confirms it quickly |
| verification is expensive | brief suggests cheap checks first (error logs, quick file reads) before expensive checks (full test suite) |
| no obvious verification | prompt mechanic to ask human |

## open questions & assumptions

### assumptions

- mechanics read briefs and internalize patterns
- compaction summaries are the primary source of inherited claims
- verification is cheap relative to action on wrong claims

### questions (triaged)

**[answered]** — resolved via review + research:
- hook is optional (wish says so explicitly)
- hook scope is compaction-only (wish says "on resume from compaction, emit:")
- **how to detect compaction:** use `PostCompact` hook (available in Claude Code v2.1.79+). fires after compaction completes but before Claude responds. no approximation needed.
- brief applies to all inherited state, not just compaction (wish says "don't even trust yourself")
- brief location is `src/domain.roles/mechanic/briefs/practices/work.flow/` (builds to `.agent/repo=ehmpathy/...`)
- observability is out of scope for initial implementation

## what is awkward?

### what feels off

- **scope creep:** the principle applies beyond compaction (your own conclusions can also be wrong). this is a feature, not a bug — the brief scopes broadly.

### where design fights mental model

- mechanics may feel "i should trust my prior self." the brief challenges this — your prior self is also fallible.

### uncomfortable tradeoffs

- **verification overhead:** every claim requires verification. this is the right tradeoff (verification is cheaper than action on wrong claims), but it adds friction.
- **hook noise:** if compaction is frequent, the reminder may become ignored. mitigation: keep the message concise.
