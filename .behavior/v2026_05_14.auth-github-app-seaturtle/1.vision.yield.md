# vision: auth-github-app-seaturtle

## the outcome world

### day-in-the-life

a developer at `acme-corp` wants to use ehmpathy's mechanic role in their repo. today:
- they install rhachet-roles-ehmpathy
- mechanic can read code, run tests, write code
- but mechanic can't push commits, create PRs, or watch CI — because that requires github auth
- to give mechanic github access, acme-corp would need to either:
  - create a machine user (costs a seat: $4-21/month)
  - share a human's PAT (security risk, tied to individual)

with seaturtle github app:
- acme-corp installs the "ehmpathy seaturtle" app from github marketplace (free)
- app grants mechanic narrow permissions: PRs, actions, contents
- mechanic can now push commits as `seaturtle[bot]`, create PRs, watch CI, enable automerge
- no user seat required, no shared credentials

### before/after contrast

| before | after |
|--------|-------|
| mechanic can't push | mechanic pushes as seaturtle[bot] |
| need machine user ($$$) | free github app install |
| PAT tied to human | app token tied to org |
| broad permissions (all repos) | scoped to selected repos |
| permanent token | short-lived JWT tokens |

### the "aha" moment

> "wait, i can just install an app and my AI assistant can create PRs? and it doesn't cost a seat?"

the value clicks when developers realize github apps are free and provide exactly the narrow permissions needed.

## user experience

### usecases

1. **push commits**: mechanic commits code as seaturtle[bot] with human co-author credit
2. **create PRs**: mechanic creates PRs when pushed to feature branches
3. **watch CI**: mechanic monitors workflow runs, surfaces failures
4. **enable automerge**: mechanic enables automerge on PRs once CI passes
5. **read cross-repo**: mechanic reads files from other org repos for research

### contract inputs & outputs

**install (one-time)**
```
1. org admin visits github.com/apps/ehmpathy-seaturtle
2. clicks "Install"
3. selects organization
4. chooses repos: "All" or "Selected"
5. done
```

**runtime (via keyrack)**
```
mechanic skill needs github access
  → keyrack.get({ key: 'EHMPATHY_SEATURTLE_GITHUB_TOKEN' })
  → keyrack uses EPHEMERAL_VIA_GITHUB_APP mech adapter
  → mech adapter parses JSON (appId, privateKey, installationId)
  → @octokit/auth-app generates installation access token (ghs_*)
  → token valid ~1 hour, keyrack auto-fetches fresh on next get
```

note: keyrack natively supports github app token generation via `EPHEMERAL_VIA_GITHUB_APP` mechanism. credentials stored as JSON; mech adapter handles JWT and token exchange.

### timeline

1. **minute 0**: org admin installs app (30 seconds)
2. **minute 1**: developer runs `rhx keyrack unlock --owner ehmpath` (unlocks credential vault)
3. **minute 2**: mechanic can push, create PRs, watch CI

note: keyrack auto-fetches fresh tokens on each `keyrack.get()` call. human re-runs unlock only if vault session expires (~9 hours).

## mental model

### how users describe it

> "it's like add a bot collaborator to your repo, but you don't pay for a seat"

> "github apps are free service accounts with fine-grained permissions"

### analogies

- **slack app**: you install slack apps to give bots access to channels. same idea — install github app to give mechanic access to repos.
- **oauth app**: like "sign in with google" but for repo access. one click, scoped permissions.

### terminology

| user term | our term | note |
|-----------|----------|------|
| "the bot" | seaturtle[bot] | the commit author identity |
| "install the app" | app installation | grant app access to repos |
| "token" | installation access token | short-lived credential |

## evaluation

### pros

- **free**: no user seat cost
- **secure**: short-lived tokens, narrow permissions, audit log
- **scoped**: per-repo or per-org installation
- **maintainable**: credentials managed centrally by ehmpath
- **identity**: commits attributed to seaturtle[bot], not a human

### cons

- **trust**: org must trust ehmpathy's app (we control the private key)
- **dependency**: if ehmpath's app is compromised, all installs affected
- **rate limits**: shared across all installations (but github app limits are generous)

### edgecases and pit-of-success

| edgecase | pit-of-success |
|----------|----------------|
| token expired | keyrack auto-fetches fresh token on next get (if vault unlocked) |
| vault session expired | human re-runs `rhx keyrack unlock --owner ehmpath` |
| app not installed | clear error: "seaturtle app not installed on this repo" |
| insufficient permissions | error shows which permission is absent |
| rate limited | gh cli retries with backoff |

## what is awkward?

### trust model

the awkward part: org admins must trust ehmpathy with write access to their repos. we hold the private key. if we're compromised, their repos are affected.

mitigation: narrow permissions (no admin, no secrets), audit logs, open source app declaration.

### credential flow

the keyrack → 1password → mech adapter → jwt → installation token chain is abstracted but has multiple steps:
1. human runs `rhx keyrack unlock --owner ehmpath` (once per ~9 hour session)
2. keyrack.get() handles JWT generation and token exchange automatically

this is simpler than manual shell functions — keyrack handles the complexity internally.

mitigation: clear error messages from keyrack, `rhx keyrack status` to diagnose issues.

### name: slug confirmed

github app slug: `seaturtle-by-ehmpathy` (verified available, matches `beaver-by-bhuild` pattern).

not awkward for users — they see it once on install.
