# blackbox criteria: auth-github-app-seaturtle

## usecase.1 = org admin installs seaturtle app

```
given an org admin who wants mechanic to have github access
  when they visit github.com/apps/seaturtle-by-ehmpathy
    then they see the app page with description and permissions
      sothat they understand what access they grant
    then they can click "Install"
      sothat they can proceed with installation

  when they click "Install" and select their organization
    then they are prompted to choose repository access ("All" or "Selected")
      sothat they control scope of mechanic's access
    then installation completes
      sothat the app is now installed on their org

given an org without the app installed
  when mechanic attempts github operations
    then operations fail with clear error: "seaturtle app not installed on this repo"
      sothat the developer knows to install the app
```

## usecase.2 = developer unlocks keyrack for mechanic

```
given a developer in a repo with rhachet-roles-ehmpathy installed
  when they run `rhx keyrack unlock --owner ehmpath`
    then they are prompted for 1password authentication
      sothat credentials are securely retrieved
    then keyrack session is unlocked (~9 hours)
      sothat mechanic can access credentials for work session

given keyrack session has expired
  when mechanic attempts github operations
    then operations fail with clear error: "vault session expired"
      sothat the developer knows to re-run unlock
```

## usecase.3 = mechanic pushes commits as seaturtle[bot]

```
given mechanic has permission to commit (via git.commit.uses)
  when mechanic runs `git.commit.set --mode apply --push`
    then commit is authored as seaturtle[bot]
      sothat commits are attributed to the app, not a human
    then commit includes Co-authored-by trailer with human's identity
      sothat human gets credit for delegated work
    then push succeeds to origin
      sothat changes are available for PR

given mechanic lacks permission (git.commit.uses quota = 0)
  when mechanic attempts to commit
    then operation fails with clear error: "no commit quota left"
      sothat mechanic knows to request permission from human
```

## usecase.4 = mechanic creates and manages PRs

```
given mechanic pushed commits to a feature branch
  when mechanic runs `git.commit.push` with --push flag
    then PR is created (or found if extant)
      sothat changes are ready for review
    then PR author shows as seaturtle[bot]
      sothat PR is attributed to the app

given a PR with passed CI
  when mechanic runs `git.release --mode apply`
    then automerge is enabled
      sothat PR merges automatically when ready
    then mechanic watches CI status
      sothat failures are surfaced promptly
```

## usecase.5 = mechanic watches CI and surfaces failures

```
given a PR with active CI
  when mechanic runs `git.release --watch`
    then mechanic monitors workflow runs until completion
      sothat status is known without human poll
    then on failure, mechanic surfaces error details
      sothat developer can diagnose issues

given CI fails with test errors
  when mechanic runs `show.gh.test.errors`
    then failed test output is retrieved and displayed
      sothat developer sees errors without exit to browser
```

## usecase.6 = mechanic reads cross-repo files for research

```
given mechanic needs to reference code in another repo
  when mechanic runs `git.repo.get lines --in owner/repo --paths 'path/to/file'`
    then file contents are returned (if app has access to that repo)
      sothat mechanic can research patterns across the org
    then if app lacks access, clear error is returned
      sothat developer knows to adjust app installation scope
```

## usecase.7 = token expires and auto-refreshes

```
given mechanic's token is near expiration (~55 minutes)
  when mechanic's skill calls keyrack.get()
    then keyrack auto-generates a fresh installation token
      sothat operations don't fail mid-session
    then new token is valid for ~1 hour
      sothat work can continue uninterrupted

given keyrack vault is locked
  when mechanic's skill calls keyrack.get()
    then operation fails with clear error: "vault is locked"
      sothat developer knows to run unlock
```

## usecase.8 = insufficient permissions

```
given the app is installed but lacks a required permission
  when mechanic attempts an operation that needs that permission
    then operation fails with clear error with absent permission named
      sothat developer knows to adjust app permissions

given rate limit is exceeded
  when mechanic attempts github operations
    then gh cli retries with backoff
      sothat transient limits don't cause immediate failure
    then if limit persists, operation fails with rate limit error
      sothat developer knows to wait
```
