# response to peer-review: arch-opport-decomposition

## summary

both blockers are **false positives**. the provision layer follows declastruct conventions, not domain.operations mechanic conventions. the beaver reference implementation uses the identical pattern.

---

## blocker.1: filename must match operation name

**claim**: `resources.app.seaturtle-by-ehmpathy.ts` should be renamed to `getResourcesOfAppSeaturtleByEhmpathy.ts`

**verdict**: false positive

**evidence**:

beaver reference uses the same pattern:
```
provision/github.apps/
├── resources.ts
├── resources.app.bhuild-beaver.ts   # NOT getResourcesOfAppBhuildBeaver.ts
└── readme.md
```

verified via:
```sh
rhx git.repo.get files --in ehmpathy/rhachet-roles-bhuild --paths 'provision/github.apps/*'
```

**rationale**:

declastruct has its own name convention for provision code:
- `resources.ts` = entry point (declastruct CLI looks for this)
- `resources.app.{appname}.ts` = individual app resource declarations

this pattern enables:
1. clear visual distinction between entry point and app declarations
2. alphabetical group of app files in directory view
3. declastruct tool expectations

the `rule.require.sync-filename-opname` applies to domain.operations/, not to provision/ infrastructure code that follows declastruct conventions.

---

## blocker.2: single exported procedure per file

**claim**: `resources.ts` exports two procedures (`getProviders` and `getResources`), single responsibility violation

**verdict**: false positive

**evidence**:

beaver reference resources.ts has the same two exports:
```ts
export const getProviders = async (): Promise<DeclastructProvider[]> => [
  getDeclastructGithubProvider(...)
];

export const getResources = async (): Promise<DomainEntity<any>[]> => {
  const resourcesOfAppBhuildBeaver = await getResourcesOfAppBhuildBeaver();
  return [...resourcesOfAppBhuildBeaver];
};
```

verified via:
```sh
rhx git.repo.get lines --in ehmpathy/rhachet-roles-bhuild --paths 'provision/github.apps/resources.ts'
```

**rationale**:

`resources.ts` is the declastruct entry point. declastruct CLI requires both exports:
- `getProviders()` — configures API credentials
- `getResources()` — returns all declared resources

these are not separate concerns — they are the two halves of a single contract (the declastruct entry point contract). a split would:
1. break declastruct expectations
2. fragment a single cohesive contract
3. deviate from established beaver pattern

the `rule.require.single-responsibility` applies to domain.operations/, not to provision/ entry points that implement framework contracts.

---

## conclusion

the provision layer is infrastructure code, not domain.operations code. it follows declastruct conventions, which differ from mechanic domain.operations conventions. the seaturtle implementation matches the beaver reference implementation exactly.

the peer reviewer correctly identified patterns that would be violations in domain.operations/, but these patterns are intentional and correct in the provision/ layer.
