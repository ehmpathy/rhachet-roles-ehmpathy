# vision: git.repo.test skill for lint enforcement

## the outcome world

### before

mechanic finishes work. onStop hook runs lint. 500 lines of eslint output flood the terminal. mechanic says "done!" and exits. human reviews PR, sees lint failures in CI, sighs.

the hook ran but didn't *enforce*. no exit code 2 = brain didn't know it had to fix anything. all that output = tokens burned, no action taken.

### after

mechanic finishes work. onStop hook runs `rhx git.repo.test --what lint`. terminal shows:

```
🐢 bummer dude...

🐚 git.repo.test --what lint
   ├─ status: failed
   ├─ defects: 7
   ├─ log: .log/role=mechanic/skill=git.repo.test/2026-04-07T14-32-01Z.stderr.log
   └─ tip: try `npm run fix` then rerun, or Read the log path above for details
```

exit code 2. brain sees constraint error. mechanic runs the fix, retries, passes. PR lands clean.

### the "aha" moment

"oh — the hook actually *blocks* me now. and i can see the log if i need details, but the summary tells me what i need to know."

## user experience

### usecases

| who | goal | command |
|-----|------|---------|
| mechanic (hook) | ensure lint passes before stop | `rhx git.repo.test --what lint` |
| mechanic (manual) | check lint status | `rhx git.repo.test --what lint` |
| human | verify mechanic's work | `rhx git.repo.test --what lint` |
| future | run other tests | `rhx git.repo.test --what types` / `--what unit` |

### contract

```
rhx git.repo.test --what <lint|types|unit|integration|format> [--when hook.onStop]

inputs:
  --what      required. which test to run.
  --when      optional. context hint (e.g., hook.onStop).

outputs:
  skill stdout:   turtle vibes summary (status, defect count, log path, hint)
  skill stderr:   empty
  exit code:      0 = pass, 2 = constraint (defects found)

  logs:           .log/role=mechanic/skill=git.repo.test/{isotime}.{stdout,stderr}.log
                  (captures npm stdout + npm stderr — the raw lint output)

  side effects:   findserts .gitignore in log dir (with self-ignore)
```

### timeline

1. hook triggers `rhx git.repo.test --what lint`
2. skill runs `npm run test:lint` internally
3. raw output captured to timestamped log files
4. skill emits summary to stdout
5. skill exits with appropriate code (0 or 2)
6. brain sees exit code, acts accordingly

## mental model

### how users describe it

"it's like a bouncer at the door — checks your lint before you leave. if you fail, you can't exit until you fix it. but it doesn't yell the whole rule book at you, just tells you there's a problem and where to look."

### analogies

| concept | analogy |
|---------|---------|
| summary vs full output | airport departure board vs flight manifest |
| log file | detailed receipt you can check later |
| exit code 2 | red light at intersection |

### terminology

| user says | we say |
|-----------|--------|
| "lint check" | `git.repo.test --what lint` |
| "it failed" | constraint error, exit 2 |
| "what broke?" | Read the log file |
| "fix it" | `npm run fix` |

## evaluation

### how well does it solve the goals?

| goal | solved? |
|------|---------|
| emit exit code 2 on failure | ✓ forces brain to address |
| reduce token waste | ✓ summary only, details in log |
| consistent vibes | ✓ turtle treestruct output |
| actionable output | ✓ defect count + log path + hint |

### pros

- **token efficient**: ~10 lines of output vs 500
- **enforceable**: exit code 2 = brain must act
- **debuggable**: full log available when needed
- **extensible**: `--what` pattern scales to types, unit, format
- **consistent**: same vibes as sedreplace, git.commit.set, etc.

### cons

- **indirection**: must open log file for details
- **new skill**: another thing to learn
- **log accumulation**: `.log/` dir grows over time (mitigated by gitignore)

### pit of success

| edgecase | handling |
|----------|----------|
| lint passes | exit 0, show success summary |
| lint fails | exit 2, show failure summary + log path |
| npm command fails | exit 1 (malfunction), show error |
| no package.json | exit 1, explain constraint |

## open questions & assumptions

### assumptions

- `npm run test:lint` is the canonical lint command
- exit code 2 is the correct semantic for "constraint, human must fix"
- isotime format like `2026-04-07T14-32-01Z` (colons replaced for filesystem)

### wisher decisions

1. **`--what all`**: future — defer to later iteration
2. **log retention**: not needed — skill findserts `.gitignore` with self-ignore in `.log/role=mechanic/skill=git.repo.test/`
3. **skill name**: keep `git.repo.test` — follows `git.repo.get` pattern
4. **inline errors**: never — always use log file, never inline output

### external research needed

- none required — this builds on extant patterns (show.gh.test.errors, git.commit.set)

## what is awkward?

### resolved concerns

- `git.repo.test` name — initially felt off since it runs npm, not git
  - **resolution**: `git.repo.*` means "the git repo" (this repo, cwd), not "a data repo" or "some other repo" — tests the current git repo

### tradeoffs

| tradeoff | choice | why |
|----------|--------|-----|
| summary vs detail | summary in stdout, detail in log | token efficiency wins |
| single command vs separate | single `--what` flag | extensibility, discoverability |
| log in repo vs temp | repo `.log/` dir | persistence, reviewability |

### mental model friction

- users might expect `npm run test:lint` to just work
  - answer: it does, but the skill wraps it for enforcement + vibes
- users might look for lint output in terminal
  - answer: summary shows path to full log
