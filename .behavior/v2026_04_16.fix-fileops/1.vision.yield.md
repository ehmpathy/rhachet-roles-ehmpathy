# vision: bracket-safe file operations

## the outcome world

### before

```bash
# file extant with brackets in name
cat 'briefs/ref.permit-requires-local-contractor.[ref].md'
# works - contents displayed

# mvsafe fails silently
rhx mvsafe 'briefs/ref.permit-requires-local-contractor.[ref].md' 'briefs/new-name.[ref].md'
# 🐢 crickets...
# files: 0
# moved: (none)

# user confused, works around it
cat 'briefs/old.[ref].md' | rhx teesafe 'briefs/new.[ref].md'
rm 'briefs/old.[ref].md'
```

### after

```bash
# option 1: --literal flag
rhx mvsafe --literal 'briefs/ref.permit-requires-local-contractor.[ref].md' 'briefs/new-name.[ref].md'
# 🐢 sweet
# files: 1
# moved: briefs/ref....[ref].md -> briefs/new-name.[ref].md

# option 2: escape brackets
rhx mvsafe 'briefs/ref.permit-requires-local-contractor.\[ref\].md' 'briefs/new-name.\[ref\].md'
# 🐢 sweet
# files: 1
```

### aha moment

"use `--literal` or escape with backslash" - clear, explicit, documented.

## the fix: `--literal` flag

### root cause

`is_glob_pattern()` treats `[` as a glob trigger, so `file.[ref].md` becomes a glob pattern where `[ref]` matches "r", "e", or "f".

### solution

Add `--literal` flag to all affected skills: **mvsafe**, **rmsafe**, **cpsafe**, **globsafe**.

When `--literal` is passed:
- path is treated as exact string
- no glob expansion occurs
- brackets are literal characters

```bash
# before (fails)
rhx mvsafe 'file.[ref].md' 'newname.[ref].md'
# 🐢 crickets... files: 0

# after - option 1: --literal flag
rhx mvsafe --literal 'file.[ref].md' 'newname.[ref].md'
# 🐢 sweet... files: 1

# after - option 2: escape brackets
rhx mvsafe 'file.\[ref\].md' 'newname.\[ref\].md'
# 🐢 sweet... files: 1
```

### why `--literal` flag (not change default)

- preserves backward compatibility
- explicit intent is clearer than magic heuristics
- power users keep `[abc]` character class globs
- matches user mental model: "I want this exact path"

### documentation requirements

Each skill must update:
1. **skill header** - add `--literal` to usage examples, show `\[` escape syntax
2. **skill help** (`--help` output) - document the flag and escape syntax
3. **briefs** - add examples with `--literal` flag and `\[` escapes
4. **permissions.jsonc** - add example for bracket paths
5. **crickets output** - when files=0 and path contains `[`, show tip about `--literal`
   ```jsonc
   // example: allow mvsafe with literal bracket paths
   "rhx mvsafe --literal 'src/**/*.[ref].md'",
   // example: allow mvsafe with escaped brackets
   "rhx mvsafe 'src/**/*.\\[ref\\].md'"
   ```

## user experience

### mvsafe/rmsafe/cpsafe

| goal | input | output |
|------|-------|--------|
| rename file with brackets (flag) | `mvsafe --literal 'a.[ref].md' 'b.[ref].md'` | file moved |
| rename file with brackets (escape) | `mvsafe 'a.\[ref\].md' 'b.\[ref\].md'` | file moved |
| delete file with brackets | `rmsafe --literal 'old.[ref].md'` | file removed |
| copy file with brackets | `cpsafe --literal 'a.[ref].md' 'b.[ref].md'` | file copied |
| glob move (unchanged) | `mvsafe 'src/*.ts' 'dest/'` | files moved |

### globsafe

| goal | input | output |
|------|-------|--------|
| find literal brackets (flag) | `globsafe --pattern '**/*.[ref].md' --literal` | files found |
| find literal brackets (escape) | `globsafe --pattern '**/*.\[ref\].md'` | files found |
| normal glob (unchanged) | `globsafe --pattern '**/*.ts'` | files found |

## mental model

### how users describe it

- "use `--literal` when my filename has brackets"
- "or escape the brackets with backslash"

### analogy

like `grep -F` (fixed string) vs `grep -E` (regex):
- default: pattern characters have special meaning
- `--literal`: pattern is exact match

## evaluation

### how well does it solve the goals?

well - users have clear options (`--literal` or `\[`) with good documentation.

### pros

- preserves backward compatibility completely
- explicit intent via flag or escape
- well-documented with examples in headers, help, briefs, permissions

### cons

- users must remember to use `--literal` or escape brackets
- two ways to solve same problem (flag vs escape) could confuse

### edgecases and pit of success

| edgecase | behavior |
|----------|----------|
| `file.[ref].md` in mvsafe | use `--literal` flag or escape as `file.\[ref\].md` |
| `file[abc].md` glob intent | works as before (character class) |
| literal `[` in globsafe | use `--literal` flag or escape as `\[` |

## what is awkward?

### forced tradeoffs

- `--literal` flag adds cognitive load for users with bracket filenames
- two solutions (flag vs escape) means more documentation

### mitigation

- clear examples in skill headers and help text
- smart error hints when zero matches and path contains `[`:

```bash
rhx mvsafe 'file.[ref].md' 'newname.[ref].md'
# 🐢 crickets...
# 
# 🐚 mvsafe
#    ├─ from: file.[ref].md
#    ├─ into: newname.[ref].md
#    ├─ files: 0
#    └─ moved: (none)
#
# 🥥 did you know?
#    ├─ path contains `[` which is a glob character
#    ├─ to treat `[` as literal, use either:
#    │  ├─ --literal flag: rhx mvsafe --literal 'file.[ref].md' 'newname.[ref].md'
#    │  └─ escape syntax: rhx mvsafe 'file.\[ref\].md' 'newname.\[ref\].md'
#    └─ see: rhx mvsafe --help
```
