# research.reflection.product.audience: git.repo.test

reflection on who benefits from the git.repo.test skill extension.

---

## who is the audience?

### primary: mechanics (claude clones)

ai agents enrolled as `repo=ehmpathy/role=mechanic` who work on code within ehmpathy repos.

characteristics:
- work in terminal via claude code
- have context window limits (~128k tokens)
- need to run tests frequently as part of code iteration
- unfamiliar with npm/jest/vitest quirks on each repo
- cannot remember conventions between sessions
- operate under permission rules that limit raw bash access

### secondary: humans who delegate to mechanics

developers who invoke mechanics to work on their repos.

characteristics:
- grant commit quota via `git.commit.uses`
- review mechanic output before merge
- expect mechanics to run tests correctly without supervision
- frustrated when mechanics struggle with test commands

### tertiary: future mechanics on new ehmpathy repos

agents on repos that adopt ehmpathy conventions but haven't been encountered before.

characteristics:
- rely on skill to "just work" on any conformant repo
- expect consistent behavior across repos
- need helpful errors when conventions are not met

---

## why do they care?

### primary: mechanics

| pain | how this relieves it |
|------|---------------------|
| "how do I run tests on this repo?" | one command works on all ehmpathy repos |
| "did I use the right flags?" | skill handles --testPathPattern, RESNAP, etc. |
| "why did my test hang?" | auto-keyrack unlock before integration tests |
| "my context is full of jest spam" | summary only, full output to log file |
| "where's the test output?" | log path shown in every result |
| "tests passed but I'm not sure" | clear turtle vibes with test counts |

### secondary: humans

| pain | how this relieves it |
|------|---------------------|
| "mechanic doesn't know how to run tests" | pre-approved skill just works |
| "mechanic ran tests in background" | foreground brief teaches correct usage |
| "mechanic can't figure out jest" | skill abstracts npm/jest quirks |
| "tests failed but mechanic didn't notice" | fail-fast with clear output |

### tertiary: future mechanics

| pain | how this relieves it |
|------|---------------------|
| "this repo has different conventions" | helpful error explains expected commands |
| "keyrack unlock failed" | error shows hint to check setup |

## how much do they care?

| audience | priority | urgency | stakes |
|----------|----------|---------|--------|
| mechanics (primary) | critical | high | blocks all test-related work |
| humans (secondary) | high | medium | affects delegation confidence |
| future mechanics (tertiary) | medium | low | future-proofs the skill |

### stakes detail

**mechanics**: cannot complete code tasks without tests. every bug fix, every feature, every refactor requires test verification. blocked mechanics = blocked work.

**humans**: low trust in mechanics who can't run tests. forces humans to intervene and run tests manually, which defeats the purpose of delegation.

**future mechanics**: skill should work on new repos without modification. otherwise, requires custom briefs per repo.

## what do they care about?

### mechanics

| criteria | delight | disappointment |
|----------|---------|----------------|
| works on first try | no fumble, just runs | "command not found" or wrong flags |
| saves context | summary only | raw jest output fills context window |
| shows what happened | test counts, time, log paths | "tests failed" with no detail |
| handles keyrack | auto-unlock | "credential locked" mid-test |
| scope filter | --scope just works | have to remember --testPathPattern |

### humans

| criteria | delight | disappointment |
|----------|---------|----------------|
| mechanic runs tests | confident delegation | "I had to run tests myself" |
| output is clear | can verify at a glance | have to dig through raw output |
| failures surface | fail-fast with exit 2 | silent pass when tests skipped |

## how will they experience it?

### mechanic journey

1. **receive task**: "fix the broken test in getUserById"
2. **discover skill**: see brief or run `rhx git.repo.test --help`
3. **run targeted test**: `rhx git.repo.test --what unit --scope getUserById`
4. **see spinner**: `🐢 ... (12s)` shows progress
5. **see result**: turtle vibes summary with test counts
6. **read log if needed**: log path in output
7. **iterate**: fix code, rerun test
8. **verify broader**: `rhx git.repo.test --what unit` for full suite
9. **complete task**: tests pass, move to commit

### touchpoints

| touchpoint | friction risk | smooth experience |
|------------|---------------|-------------------|
| first run | "what flags do I use?" | `--what unit` is intuitive |
| spinner | "is it stuck?" | elapsed time shown |
| keyrack | "why locked?" | auto-unlock with status |
| scope | "how do I filter?" | `--scope pattern` just works |
| resnap | "how do I update?" | `--resnap` sets RESNAP=true |
| failure | "what went wrong?" | log path in output |
| no tests | "why empty?" | "no tests matched scope" hint |

### friction points

1. **absent command**: repo lacks `test:unit` → fail fast with helpful error
2. **keyrack not setup**: mechanic on new repo → error shows hint
3. **wrong scope**: pattern matches no files → exit 2 with hint
4. **integration hangs**: external service down → timeout + log capture

---

## summary

the git.repo.test extension serves mechanics who need to run tests as part of their code iteration loop. the key value is:

1. **one command** that works across all ehmpathy repos
2. **summary output** that preserves context window
3. **auto-keyrack** that handles credential unlocks
4. **fail-fast** behavior with helpful hints
5. **log capture** for diagnosis when needed

humans benefit indirectly by mechanics they can trust to run tests correctly without intervention.
