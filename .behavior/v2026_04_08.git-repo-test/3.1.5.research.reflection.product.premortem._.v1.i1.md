# research.reflection.product.premortem: git.repo.test

premortem analysis: how could this fail?

---

## premortem: imagine we shipped and it failed

project forward: we shipped this, and it failed badly. why?

### most plausible failure scenarios

1. **jest output parser breaks on edge cases**
   - jest output format varies between versions and configurations
   - parser relies on regex that may not match all output formats
   - "no tests found" detection fails, mechanic thinks tests passed

2. **keyrack unlock fails silently**
   - mechanic assumes credentials are available
   - integration tests fail with cryptic auth errors
   - mechanic doesn't know to check keyrack setup

3. **scope flag doesn't work as expected**
   - `--testPathPattern` behavior differs between jest 29 and 30
   - mechanic's pattern matches no files but exit code is 0
   - mechanic thinks tests passed when they didn't run

4. **log files accumulate without cleanup**
   - every test run creates new log files
   - disk fills up over time
   - no rotation or cleanup mechanism

5. **tests hang indefinitely**
   - external service down, test waits forever
   - skill has no timeout
   - mechanic's session stuck

---

## what obvious signs did we ignore?

as we look back from the imagined failure — what signs were present that we dismissed?

- **jest 30 has incompatible flag**: `--testPathPattern` → `--testPathPatterns`. we noted it but may not handle it.

- **jest exit code varies by version**: research showed mixed info on whether jest exits 0 or 1 when no tests found. we assumed one behavior.

- **RESNAP is ehmpathy convention, not jest native**: research noted this. if repo doesn't handle RESNAP env var, snapshots won't update.

- **keyrack may not be configured**: new mechanics on new repos may not have keyrack setup. auto-unlock assumes it exists.

- **log directory may not be writable**: skill assumes it can create `.log/` but repo may have unusual permissions.

## what assumptions could turn out wrong?

| assumption | what if wrong? | likelihood |
|------------|----------------|------------|
| jest output format is parseable | parser fails, wrong stats shown | medium |
| repos have `test:unit`, `test:integration`, etc. | command not found, exit 2 | low (documented convention) |
| keyrack unlock is idempotent | error on already-unlocked | low (tested) |
| `--testPathPattern` works on all jest versions | jest 30 requires `--testPathPatterns` | medium |
| RESNAP env var is handled by jest config | snapshots don't update | medium |
| mechanics read output and notice failures | mechanic ignores exit code | low (exit code is programmatic) |
| log directory is writable | permission denied, skill fails | low |
| tests complete in reasonable time | hang forever | medium |

## what is the nightmare scenario?

**mechanic thinks tests passed when they didn't run.**

scenario:
1. mechanic runs `rhx git.repo.test --what unit --scope nonExistent`
2. jest finds no tests that match pattern
3. jest exits 0 (some versions do this)
4. skill shows "passed" with 0 tests
5. mechanic commits code without actual test verification
6. broken code merges to main
7. production bug discovered days later

this is the worst outcome because:
- it silently undermines the entire purpose of the skill
- it creates false confidence in code quality
- it's hard to detect until damage is done

---

## what mitigations to consider?

| risk | mitigation | cost of mitigation |
|------|------------|--------------------|
| parser fails on edge cases | snapshot test output formats, add fallback | medium (more tests) |
| jest 30 incompatible flag | detect jest version, use correct flag | medium (version detection) |
| no tests matched | use `--passWithNoTests` check or parse "0 total" as failure | low |
| log files accumulate | add brief about periodic cleanup, or auto-rotate | low |
| tests hang | add timeout flag to npm command | low |
| RESNAP not handled | document clearly in help, fail if no snapshots updated | low |
| keyrack not configured | helpful error with setup instructions | already planned |
| exit code 0 with no tests | always check test count, not just exit code | low |

### priority mitigations

1. **always check test count**: if suites=0 or tests=0, treat as constraint error regardless of exit code
2. **detect jest version**: check for jest 30+ and use `--testPathPatterns` if needed
3. **timeout**: add `--testTimeout` or wrapper timeout to prevent infinite hang
4. **document RESNAP**: make clear in help that repo must handle RESNAP env var

---

## summary

the highest risk is **silent false-positive** — mechanic thinks tests passed when they didn't run. mitigate by:

1. check test count, not just exit code
2. detect "no tests found" in output
3. detect jest 30 incompatible change
4. add timeout to prevent hangs

lower risks (log accumulation, keyrack setup) can be addressed with documentation and helpful errors.
