# research.reflection.product.rootcause: git.repo.test

root cause analysis: why do mechanics struggle with test runs?

---

## symptom

mechanics fumble when asked to run tests. they:
- use wrong npm commands
- forget jest flags
- run tests in background
- lose context to raw jest output
- forget keyrack unlock for integration tests
- struggle to find test failures in verbose output

---

## 5 whys

### branch 1: wrong commands

**why do mechanics use wrong npm commands?**
→ because npm command names vary across repos

**why do npm command names vary?**
→ because there's no enforced convention

**why is there no enforced convention?**
→ because repos evolved independently

**why does this matter for mechanics?**
→ because mechanics can't remember conventions between sessions

**root cause 1**: mechanics lack persistent memory of repo-specific commands. a skill that abstracts the variation would solve this.

### branch 2: context waste

**why do mechanics waste context on test output?**
→ because `npm run test` streams verbose jest output to terminal

**why is verbose output a problem?**
→ because jest outputs full stack traces, file paths, diffs for every failure

**why does this matter?**
→ because mechanic context windows fill with test output instead of code

**why can't mechanics filter the output?**
→ because there's no skill that captures output and returns summary only

**root cause 2**: no mechanism exists to capture full output to file while emitting summary to terminal.

### branch 3: credential failures

**why do integration tests fail mysteriously?**
→ because keyrack credentials are locked

**why are credentials locked?**
→ because mechanics don't know to unlock keyrack before tests

**why don't mechanics know?**
→ because the convention isn't documented in a machine-readable way

**why does this matter?**
→ because integration tests fail with cryptic auth errors, not "unlock keyrack"

**root cause 3**: no skill auto-unlocks keyrack before integration/acceptance tests.

---

## root causes summary

| root cause | description | fix |
|------------|-------------|-----|
| 1 | mechanics lack memory of repo conventions | skill abstracts npm command variations |
| 2 | no output capture with summary | skill captures to log, emits summary |
| 3 | no auto-keyrack unlock | skill unlocks keyrack for integration/acceptance |

---

## recommendation

address all three root causes via git.repo.test skill extension:

1. **abstract commands**: `--what unit|integration|acceptance` maps to correct npm command
2. **capture output**: full output to `.log/`, summary only to terminal
3. **auto-unlock**: keyrack unlock before integration/acceptance tests

this single skill extension addresses the root causes, not just the symptoms.

---

## validation

after implementation, verify:
- mechanic runs `rhx git.repo.test --what unit` without permission prompts
- mechanic sees test counts, not raw jest output
- mechanic finds full output in `.log/` when needed
- integration tests auto-unlock keyrack without mechanic intervention
- mechanics stop fumbling with npm/jest quirks
