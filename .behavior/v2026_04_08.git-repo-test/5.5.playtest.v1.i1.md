# playtest: git.repo.test skill extension

## prerequisites

- **run in rhachet-roles-ehmpathy repo** — this playtest is designed for this repo
- repo with build complete — run `npm run build` so dist/ has current skill code
- keyrack credentials for ehmpath/test — run `rhx keyrack unlock --owner ehmpath --env test` to verify
- check available test commands — run `npm run` to see which test:* commands exist

## sandbox

all playtest steps target extant repo state. no file creation needed — the skill outputs to `.log/` which is gitignored.

---

## happy paths

### 1. run unit tests

**command:**
```bash
rhx git.repo.test --what unit
```

**expected outcome:**
- turtle vibes header (cowabunga or bummer dude)
- status: passed or failed
- stats section with suites, tests (passed/failed/skipped), time
- log section with stdout/stderr paths in `.log/role=mechanic/skill=git.repo.test/what=unit/`
- exit 0 on pass, exit 2 on fail

**pass criteria:**
- output contains `🐚 git.repo.test --what unit`
- output contains `status: passed` or `status: failed`
- output contains `suites:` line
- output contains `tests:` line with passed/failed/skipped counts
- log paths are namespaced with `what=unit` (e.g., `.log/role=mechanic/skill=git.repo.test/what=unit/2026-04-08T14-23-01Z.stdout.log`)

---

### 2. run integration tests (with keyrack)

**command:**
```bash
rhx git.repo.test --what integration
```

**expected outcome:**
- keyrack unlock happens automatically
- output shows `keyrack: unlocked ehmpath/test`
- tests run and show summary
- log paths namespaced with `what=integration`

**pass criteria:**
- output contains `keyrack: unlocked ehmpath/test`
- output contains stats section
- log paths contain `what=integration`

---

### 3. run scoped tests

**command:**
```bash
rhx git.repo.test --what unit --scope getRole
```

**expected outcome:**
- only tests in paths that match `getRole` run
- stats show fewer suites than happy path 1 (compare: note suite count from step 1)
- output otherwise identical to full run

**pass criteria:**
- suite count is less than happy path 1 suite count
- output contains the scope pattern in command echo

---

### 4. run lint

**command:**
```bash
rhx git.repo.test --what lint
```

**expected outcome:**
- same output structure as extant behavior
- no keyrack unlock (lint does not need credentials)
- status passed or failed with defect count
- log paths namespaced with `what=lint`

**pass criteria:**
- no `keyrack:` line in output
- output contains `status: passed` or `status: failed`
- log paths contain `what=lint`

---

### 5. run all test types

**command:**
```bash
rhx git.repo.test --what all
```

**expected outcome:**
- runs lint, unit, integration, acceptance in sequence
- each type emits a status line as it completes
- keyrack unlocks once before integration
- shows total duration across all types
- log paths namespaced per type
- stops on first failure (fail-fast): if lint fails, unit/integration/acceptance do not run

**pass criteria:**
- output contains lines for each type (lint, unit, integration, acceptance)
- output contains `total:` line with aggregate duration
- log section shows paths for each type

## edgey paths

### E1. scope matches no tests

**command:**
```bash
rhx git.repo.test --what unit --scope nonexistent_pattern_xyz
```

**expected behavior:**
- exit 2 (constraint error)
- output shows `status: constraint`
- output shows `error: no tests matched scope 'nonexistent_pattern_xyz'`
- hint about scope pattern

---

### E2. absent test command

**note:** first check `npm run` to see if test:acceptance exists. if it exists in this repo, skip this edge case — it can only be tested in a repo without that command.

**command (in repo without test:acceptance):**
```bash
rhx git.repo.test --what acceptance
```

**expected behavior:**
- exit 2 (constraint error)
- output shows `status: constraint`
- output shows `error: no 'test:acceptance' command in package.json`
- hint about ehmpathy convention

---

### E3. keyrack locked

**note:** this edge case tests keyrack failure behavior. to reproduce:
- option 1: run `rhx keyrack lock --owner ehmpath --env test` then run the command below
- option 2: skip this edge case if you cannot reliably create a locked state

**command:**
```bash
rhx git.repo.test --what integration
```

**expected behavior:**
- skill attempts unlock
- if unlock fails, shows error from keyrack
- exit 1 (malfunction)

---

### E4. pass raw args

**command:**
```bash
rhx git.repo.test --what unit -- --verbose
```

**expected behavior:**
- `--verbose` passed to jest
- jest output may be more detailed
- skill summary unchanged

---

## pass/fail criteria

### ✓ pass if:

1. all happy paths show correct output structure
2. turtle vibes header present (cowabunga/bummer dude)
3. stats section parses correctly with suites/tests/time
4. log paths are namespaced by test type and present on both success and failure
5. keyrack auto-unlocks for integration/acceptance
6. --what all runs all types in sequence with progressive output
7. edge cases fail gracefully with helpful error messages
8. exit codes are semantic: 0=pass, 1=malfunction, 2=constraint
9. tip line present on test failure (e.g., "tip: Read the log...")

### ✗ fail if:

1. output lacks turtle vibes structure
2. stats section malformed or absent
3. log paths not namespaced by test type
4. keyrack not unlocked for integration/acceptance
5. --what all does not show progressive output
6. edge cases crash instead of graceful error
7. exit codes are wrong (e.g., exit 0 on test failure)
