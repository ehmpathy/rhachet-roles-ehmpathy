research on claims relevant to the git.repo.test skill extension.

## 1. test runner performance and developer experience

### [FACT] vitest is faster than jest

> "vitest delivers measurable performance improvements: 5.6x faster cold starts, 28x faster watch mode, and 64% ci pipeline reduction in real production scenarios." [1]

> "in a production monorepo migration, cold starts dropped from 214 seconds to 38 seconds, while watch mode re-runs went from 8.4 seconds to 0.3 seconds." [1]

sources [1], [2], [3] converge on vitest speed advantage.

### [FACT] watch mode speed changes developer behavior

> "the watch mode speed difference is transformational: when tests run in 200 milliseconds instead of 4 seconds, developers stop batching test runs and start running them continuously." [2]

### [OPIN] vitest has better developer experience

> "vitest overtook jest in developer satisfaction scores in the state of js 2024 survey." [1]

> "for teams using react with vite, solidjs, vue, or svelte, vitest delivers an outstanding developer experience and significantly faster feedback loops." [1]

### [SUMP] jest remains dominant for legacy/react native

> "jest remains a reliable and powerful solution especially in legacy systems or react native apps, and its ecosystem maturity and long history make it a safe choice for many large or older codebases." [1]

## 2. test output verbosity

### [FACT] verbose output aids debugging

> "during development, developers can use verbose output to debug code by running specific tests, which provides insights into the behavior of code and test cases, making it easier to fix problems." [4]

### [OPIN] summary mode preferred in ci

> "summary mode keeps output minimal, providing only the total number of tests executed along with a general pass or fail result, which is particularly useful for a quick overview when running multiple tests in a continuous integration setting." [4]

### [SUMP] production should minimize output

> "in production environments, minimizing unnecessary output helps reduce clutter and improve efficiency." [4]

### [OPIN] selectable verbosity is best

> "a recommended approach is to provide selectable verbosity levels so consumers can choose the detail they need." [4]

## 3. ai/llm context window constraints

### [FACT] context windows have finite token limits

> "an ai context window is the amount of information, measured in tokens, that a large language model (llm) can process at one time, functioning as the model's short-term memory." [5]

> "as a rough guide, 1,000 tokens equals roughly 750 words, or about 40 lines of code." [5]

### [FACT] performance degrades as context fills

> "model performance does not degrade at a cliff — it degrades gradually, and instructions from early in the conversation get progressively less likely to be followed as the window fills." [5]

### [OPIN] context chunking is essential

> "the simplest approach is to break your code into smaller, logical pieces, or 'chunks.' instead of feeding the ai an entire service, you might provide a single file, class, or even just a function." [5]

### [OPIN] summarization reduces token count

> "if your coding agent offers summarisation features, use them to condense lengthy conversations or code reviews, which helps to reduce the token count and keep the context manageable." [5]

**implication for git.repo.test**: raw jest output wastes tokens. summary-only output preserves context for actual work.

## 4. snapshot testing controversy

### [OPIN] snapshots are fragile

> "the most common drawback is the fragility of snapshot tests, leading to the necessity of constant updates of the 'golden standard' snapshot." [6]

### [OPIN] snapshots lack intent

> "snapshot tests lack (or at least, fail to encourage) expressing the author's intent as to what the code does, and there's no context that tells us why something changed." [6]

### [OPIN] snapshots can be valuable when used correctly

> "despite criticisms, there is value in snapshot testing if you use them effectively, particularly when writing tools for developers to ensure good error or warning messages are logged." [7]

### [SUMP] one company removed all snapshots

> "one major company removed or replaced nearly every extant snapshot test after six months of use, finding that snapshots are more trouble than they're worth, especially across large and/or fluid teams." [6]

**sources conflict**: [6], [8] argue against snapshots; [7], [9] argue they have value when used properly.

### [OPIN] code review mitigates snapshot downsides

> "practices such as treating snapshots as code that is subjected to code reviews, and the usage of detailed textual descriptions can mitigate the mentioned downsides." [9]

## 5. test artifact management

### [FACT] ci platforms delete artifacts after retention period

> "most ci/cd platforms (github actions, gitlab ci, circleci) automatically delete test artifacts after a retention period - typically 30-90 days." [10]

### [OPIN] local logs address retention problem

> "test artifacts — reports, screenshots, logs, coverage data — are the outputs generated every time your tests run. they're essential for debugging failures and tracking test health over time." [10]

**implication for git.repo.test**: storing logs in `.log/` within the repo provides immediate access without ci artifact lookup.

## 6. test isolation and parallel execution

### [FACT] jest runs test files in parallel, tests within files sequentially

> "when running jest tests in parallel, test files can execute concurrently, while tests within the same file run sequentially." [11]

### [OPIN] shared state causes parallel test failures

> "avoid shared state across tests, as shared state can cause unexpected interactions when tests run in parallel." [11]

### [KHUE] how should integration tests handle database isolation?

several approaches cited:
> "create a separate schema for each worker and configure them to read and write only to their assigned schema." [11]

> "using jest_worker_id, you can ensure you are using a unique database when making connections." [11]

### [FACT] faster tests improve development velocity

> "unit tests help in driving development and providing feedback quickly, even as fast as 3 or 4 minutes." [12]

### [OPIN] speed is paramount

## 8. fail-fast testing

### [FACT] fail-fast reduces debugging time

> "a fail-fast system is one that immediately reports any condition likely to indicate a failure and is designed to stop normal operation rather than attempt to continue a possibly flawed process." [13]

> "finding the cause of a failure is easier in a fail-fast system because it reports the failure with as much information as possible as close to the time of failure as possible." [13]

### [FACT] early detection reduces cost

> "the earlier you catch a defect, the cheaper it is to fix, as bugs caught in the production phase are more expensive to address than those caught during development." [13]

### [OPIN] fail-fast should stop test execution

> "in software testing, the fail-fast approach stops test execution immediately upon encountering a critical failure to catch bugs early and save time." [14]

**implication for git.repo.test**: detecting "no tests matched" early and failing with helpful error aligns with fail-fast principle.

## 10. test runner conventions

### [SUMP] npm test commands use colon convention

> "the colon (`:`) in command names like `test:unit` and `test:integration` is a common convention to group related commands." [16]

### [SUMP] ehmpathy repos use test:unit, test:integration, test:acceptance

## 11. keyrack and credential management

### [KHUE] should tests auto-unlock credentials?

**arguments for**:
- reduces friction for developers
- ensures integration tests can run
- credentials are short-lived anyway

**arguments against**:
> "ci systems only need access to secrets required for testing functionality, not every secret your application uses." [17]

**decision**: auto-unlock for convenience, with visible status in output.

---

## 12. foreground vs background execution

### [KHUE] should tests run in foreground or background?

**arguments for foreground**:
- immediate feedback
- easier to interrupt
- output visible in real-time (for debugging)

**arguments for background**:
- can continue other work
- parallel workflows

**decision per wish**: tests must run in foreground. summary-only output mitigates noise concern.
