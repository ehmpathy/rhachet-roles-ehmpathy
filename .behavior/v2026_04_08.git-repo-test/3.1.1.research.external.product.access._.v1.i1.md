# research.external.product.access: git.repo.test

research on remote access required for the git.repo.test skill extension.

---

## summary

the git.repo.test skill interacts with the following external interfaces:

1. **jest cli** — test runner with specific cli flags and output format
2. **npm scripts** — package.json test command conventions
3. **filesystem** — log capture to `.log/` directory
4. **keyrack** — credential management for integration/acceptance tests
5. **vitest cli** — alternative test runner (future compatibility)

---

## 1. jest cli interface

### 1.1 testPathPattern flag

jest provides `--testPathPattern` to filter tests by file path pattern.

> "a regexp pattern string that is tested against all tests paths before executing the test." [1]

> "the –testPathPattern facilitates the running of tests in specific directories. npm test -- --testPathPattern='./src/components' this command executes tests located in the specified path, minimizing the time spent on irrelevant tests." [1]

**breaking change in jest 30**: the flag was renamed.

> "if you're using jest 30 or later, note that there was a breaking change: the --testPathPattern flag is now --testPathPatterns." [1]

### 1.2 updateSnapshot flag

jest provides `-u` or `--updateSnapshot` to update snapshots.

> "the `--updateSnapshot` flag (with alias `-u`) is used to re-record every snapshot that fails during a test run." [5]

> "as of jest 20, snapshots in jest are not automatically written when jest is run in a ci system without explicitly passing `--updateSnapshot`." [5]

**note**: the RESNAP environment variable is an ehmpathy convention, not a jest native feature. it must be translated to `--updateSnapshot` or handled by the test configuration.

### 1.3 no tests found behavior

jest exits with code 1 when no tests are found.

> "when jest finds no tests, it exits with code 1. this happens when jest cannot locate any test files matching its configured patterns." [3]

> "you can run jest with the `--passWithNoTests` flag to exit with code 0 instead." [3]

**sources conflict**: some issues report jest exits 0 when no tests found [8], while others confirm exit 1 [3]. behavior varies by version.

### 1.4 listTests for validation

jest can list tests without running them.

> "you can use jest's built-in `--listTests` flag to see which files it detects by running `npx jest --listTests`." [17]

> "you can configure your ci pipeline to run `jest --listTests` and fail if no tests are detected." [17]

### 1.5 forceExit and detectOpenHandles

for hanging tests, jest provides diagnostic flags.

> "the `--detectOpenHandles` flag attempts to collect and print open handles preventing jest from exiting cleanly." [13]

> "the `--forceExit` option forces jest to exit immediately after all tests have completed, regardless of whether there are any lingering asynchronous operations." [13]

> "it is advised to tear down external resources after each test to make sure jest can shut down cleanly. using `--forceExit` is not recommended." [13]

### 1.6 testTimeout configuration

jest timeout can be configured via cli.

> "the `--testTimeout` flag sets the default timeout of a test in milliseconds, with a default value of 5000." [18]

> "cli options take precedence over values from the configuration." [18]

## 2. jest output format

### 2.1 summary output structure

jest outputs a summary in this format:

```
Test Suites: 1 passed, 1 total
Tests:       2 passed, 2 total
Snapshots:   0 total
Time:        1.234s
```

### 2.2 parsing considerations

> "jest outputs a summary line like 'Test Suites: 1 passed, 1 total' and similar information about tests." [19]

regex pattern for parsing:
```
Test Suites:\s*(\d+)\s*(passed|failed),\s*(\d+)\s*total
Tests:\s*(\d+)\s*(passed|failed|skipped),\s*(\d+)\s*total
```

### 2.3 json reporters

for structured output, json reporters are available.

> "a json test results processor for jest that outputs test results in json format to a file called test-results.json." [2]

> "jest-ctrf-json-reporter creates test reports following the ctrf standard, ensuring the generation of uniform json test reports." [2]

**convergence**: sources [2], [20] agree that json reporters provide more reliable parsing than console output.

## 3. npm scripts interface

### 3.1 passthrough arguments

npm uses `--` to pass arguments to underlying commands.

> "the conventional mechanism to pass arguments to npm scripts involves using the double dash (--) separator, which distinguishes arguments meant for npm itself from those intended for the script command." [4]

> "everything after `--` is passed as arguments to the command, for example `npm run lint -- --fix src/`." [4]

> "arguments will only be passed to the script specified after npm run and not to any pre or post script." [4]

### 3.2 test script conventions

> "a common convention includes separating test scripts into 'test:unit' and 'test:integration' in the package.json scripts section." [16]

> "the colon (`:`) in script names like `test:unit` and `test:integration` is a common convention to group related scripts." [16]

## 4. stdout/stderr capture

### 4.1 separate capture with tee

> "send standard error and standard output to the console and to separate files using process substitution: `(echo 'ok'; echo 'error' 1>&2) 2> >(tee err.log) > >(tee out.log)`" [6]

> "the tee command reads from standard input and outputs what it reads to the console and to a given file." [6]

### 4.2 process substitution

> "process substitution `>(…)` establishes a fifo and makes it available to tee for listening, then employs file redirection `>` to send the command's stdout to the fifo." [6]

### 4.3 ci/cd best practices

> "many test runners allow to capture stderr and show it only as part of the failure in the end, which would allow to see all the relevant error information in the end of the runner logs instead of hunting for the stderr output somewhere in the middle of the log." [12]

**convergence**: sources [6], [7], [12] agree that separate capture of stdout/stderr improves debugability.

## 5. credential management for tests

### 5.1 environment variables limitations

> "while environment variables have traditionally been viewed as a convenient and portable way of providing secrets to applications, they should no longer be the default method for providing sensitive values." [14]

> "environment variables are global, meaning access to them can't easily be limited to a specific process – any process running in the user space can access, edit or delete them." [14]

### 5.2 secrets management best practices

> "always separate application configuration from secrets and credentials, with secrets either completely injected to the application from an external resource as an ephemeral and short-lived access." [14]

> "ci systems only need access to secrets required for testing functionality, not every secret your application uses." [14]

### 5.3 keyrack approach

keyrack (ehmpathy internal) provides credential management that:
- unlocks credentials on demand (`rhx keyrack unlock --owner ehmpath --env test`)
- credentials are ephemeral (expire after ~9 hours)
- idempotent unlock (already unlocked = no error)

this aligns with best practices from [14], [15].

## 6. vitest cli (future compatibility)

### 6.1 test filtering

> "you can use the cli to filter test files by name (e.g., `$ vitest basic`), or use the `-t, --testNamePattern <pattern>` option to filter tests by full name." [9]

> "you can pass an additional argument as the filter of the test files to run, which will only execute test files that contain certain text in their paths." [9]

### 6.2 differences from jest

| feature | jest | vitest |
|---------|------|--------|
| path filter | `--testPathPattern` (v29) / `--testPathPatterns` (v30) | positional arg or config |
| name filter | `--testNamePattern` | `-t, --testNamePattern` |
| snapshot update | `-u, --updateSnapshot` | `-u, --update` |

**note**: vitest is mostly compatible but has subtle differences in flag names.

## 7. exit code semantics

### 7.1 jest exit codes

> "jest returns an exit code on test failure, and the standard behavior is: exit code 0 indicates success when all tests pass, exit code 1 indicates failure when tests fail." [10]

**sources conflict on edge cases**:

> "jest v20 had a problem where it would exit with code 0 when tests failed." [10]

> "when the --coverage flag is passed, jest exits with 0 even if threshold requirements are not met." [10]

### 7.2 skill exit code mapping

based on research and ehmpathy conventions:

| jest exit | condition | skill exit | meaning |
|-----------|-----------|------------|---------|
| 0 | tests pass | 0 | success |
| 1 | tests fail | 2 | constraint (tests failed) |
| 1 | no tests found | 2 | constraint (scope matched no tests) |
| non-zero | jest error | 1 | malfunction |

## 8. anti-patterns

### 8.1 live streaming raw output

> "during test execution, any output sent to stdout and stderr is captured." [12]

live streaming jest output to terminal wastes context tokens. capture to log file, emit summary only.

### 8.2 using forceExit as default

> "using `--forceExit` is not recommended, as it's a workaround rather than fixing the underlying issue." [13]

### 8.3 hardcoding credentials

> "hardcoding secrets like usernames, passwords, or tokens in your codebase can expose them to unauthorized access." [14]

use keyrack unlock instead of hardcoded env vars.

### 8.4 ignoring no-tests-matched

> "when attempting to run tests against a directory with no valid tests, jest exits with code 1, which is intended as an error condition." [3]

fail fast with helpful error instead of letting jest run with empty test set.

- [3] says jest exits 1 when no tests found
- [8] (same issue) reports jest exits 0 in some cases
- resolution: behavior varies by jest version; explicitly check with --listTests

**sources [1] and [21] note breaking change**:
- jest 29: `--testPathPattern`
- jest 30: `--testPathPatterns` (plural)
- resolution: detect jest version or use config file instead of cli flag
