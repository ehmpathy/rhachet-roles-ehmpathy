## user experience

### usecases

| goal | command |
|------|---------|
| run all unit tests | `rhx git.repo.test --what unit` |
| run all test types | `rhx git.repo.test --what all` |
| run specific test file | `rhx git.repo.test --what unit --scope getUserById` |
| run tests by pattern | `rhx git.repo.test --what integration --scope "customer.*"` |
| update snapshots | `rhx git.repo.test --what unit --resnap` |
| run thorough (full suite) | `rhx git.repo.test --what unit --thorough` |
| run acceptance tests | `rhx git.repo.test --what acceptance` |
| run lint (extant) | `rhx git.repo.test --what lint` |

### contract

```
rhx git.repo.test --what <type> [--scope <pattern>] [--resnap] [--thorough]

--what      lint | unit | integration | acceptance | all    (required)
--scope     file path pattern (passed to jest --testPathPattern)  (optional)
--resnap    update snapshots (sets RESNAP=true)       (optional)
--thorough  run full suite (sets THOROUGH=true)       (optional)
```

note: for test name filter or other jest flags, use `--` to pass raw args:
```
rhx git.repo.test --what unit -- --testNamePattern="should return"
```

### timeline

1. **clone receives task**: "fix the broken test in getUserById"
2. **clone runs targeted test**: `rhx git.repo.test --what unit --scope getUserById`
3. **test fails**: clone reads log path from output, diagnoses issue
4. **clone fixes code**: edits the file
5. **clone reruns test**: same command, test passes
6. **clone runs broader tests**: `rhx git.repo.test --what unit` to ensure no regressions

## evaluation

### how well does it solve the goals?

| goal | solved? | how |
|------|---------|-----|
| auto unlock keyrack | ✓ | skill calls `rhx keyrack unlock --owner ehmpath --env test` before integration/acceptance (skipped for unit/lint) |
| auto run npm correctly | ✓ | skill knows `npm run test:unit`, `test:integration`, etc. |
| auto pass scope correctly | ✓ | skill translates `--scope` to jest's `--testPathPattern` |
| capture full results to log | ✓ | full output to .log/ on both success and failure |
| tell clone where logs are | ✓ | log path in every output |
| save context tokens | ✓ | summary only, no live stream — clone sees counts not raw output |
| fail fast on no match | ✓ | detect empty test set, exit 2 with helpful error |
| never run in background | ✓ | brief documents foreground requirement |
| run all test types at once | ✓ | `--what all` runs lint → unit → integration → acceptance in sequence |
| run thorough full suite | ✓ | `--thorough` sets THOROUGH=true to skip impact analysis |
| namespaced logs | ✓ | each test type logs to `.log/.../what=${TYPE}/` to avoid overlap |

### pros

- **one command**: no npm/jest/vitest quirk fumble
- **context-efficient**: summary only, no raw jest output in context window
- **auto-permission**: pre-approved bash patterns
- **auto-keyrack**: integration tests just work
- **full logs**: always captured for diagnosis when needed
- **turtle vibes**: consistent, scannable output with test counts
- **scope filter**: easy to target specific tests
- **fail fast**: no tests matched = immediate error, not ambiguous jest output

### cons

- **another abstraction**: clones must learn `rhx git.repo.test` vs raw npm
- **repo conventions**: assumes `npm run test:unit`, `test:integration`, etc. exist
- **keyrack dependency**: integration tests require keyrack setup

### pit of success

| edgecase | how we handle |
|----------|---------------|
| no `test:unit` command | fail fast with helpful error |
| keyrack locked | auto-unlock `ehmpath/test` before test |
| scope matches no tests | fail fast with helpful error (exit 2, constraint) |
| resnap on lint | ignore (lint has no snapshots) |
| --what all first type fails | stop immediately, don't run remaining types |
| logs from different types | namespaced paths prevent overlap |

## open questions & assumptions

### assumptions

1. repos follow `npm run test:unit`, `test:integration`, `test:acceptance` convention
2. keyrack credentials are under `ehmpath` owner, `test` env
3. jest or vitest is the test runner (not mocha, tap, etc.)
4. `RESNAP=true` triggers snapshot update
5. jest output can be parsed to extract: test count, pass/fail/skip counts, time, suite count

### resolved decisions

1. **scope semantics**: `--scope` = file path pattern only (passed to `--testPathPattern`). for test name filter, use `--` to pass raw jest args.

2. **keyrack env**: always `test` env. if acceptance needs `prep`, that's a future enhancement.

3. **output behavior**: summary only to terminal, full output to log file. no live stream — saves context tokens. clone sees test counts, not raw jest output.

4. **no tests matched**: fail fast with exit 2 (constraint error), don't let jest run with empty test set.

### questions answered [answered]

1. **test command existence**: handled by fail-fast with helpful error. skill checks at runtime, fails gracefully if command absent.

2. **keyrack unlock command**: `rhx keyrack unlock --owner ehmpath --env test`. documented in `howto.keyrack.[lesson].md`.

### questions deferred for research [research]

1. **vitest support**: how does vitest handle `--testPathPattern`? same as jest?
   - implement jest first, add vitest support later if needed
   - research vitest docs when that enhancement is requested

2. **jest output format**: what's the exact format of jest summary output?
   - need to parse: Tests, Suites, Time, passed/failed/skipped counts
   - verify with `npm run test:unit` output in a real repo

3. **detect no tests matched**: how does jest signal "no tests matched"?
   - option A: check jest exit code
   - option B: parse output for "No tests found" message
   - option C: use `npx jest --listTests` first to check

## what is awkward?

### `--what` is verbose

`--what unit` is 11 characters. could be `--unit` (6 chars) or positional `unit` (4 chars).

however, `--what` is explicit and matches other skills. keep it.

### scope is file path only

`--scope` means file path pattern only. this is simpler and more common.

for test name filter, users pass raw args via `--`:
```
rhx git.repo.test --what unit -- --testNamePattern="should return"
```

this keeps the skill interface simple while still supporting advanced use cases.

### log path is always shown

even on quick success, we show the log path. might feel noisy.

however, consistency matters more than terseness. clones always know where to look. keep it.

### keyrack unlock is implicit

clone doesn't explicitly ask for keyrack unlock. could be surprising.

however, the alternative (clone must remember to unlock first) is worse. auto-unlock with visible output is the right tradeoff.
