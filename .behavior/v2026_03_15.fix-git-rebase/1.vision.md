# 1.vision: git.branch.rebase dispatcher skill

## the outcome world

### before

```sh
# mechanic must know exact skill names
rhx git.branch.rebase.begin --mode plan
rhx git.branch.rebase.continue
rhx git.branch.rebase.abort

# three separate permission entries needed
"Bash(npx rhachet run --skill git.branch.rebase.begin:*)"
"Bash(npx rhachet run --skill git.branch.rebase.continue:*)"
"Bash(npx rhachet run --skill git.branch.rebase.abort:*)"
```

### after

```sh
# consistent command pattern with subcommand
rhx git.branch.rebase begin --mode plan
rhx git.branch.rebase continue
rhx git.branch.rebase abort

# single permission entry with prefix match
"Bash(npx rhachet run --skill git.branch.rebase:*)"
```

### the "aha" moment

the pattern is similar to git's mental model:
- `git rebase --continue` becomes `rhx git.branch.rebase continue`
- `git rebase --abort` becomes `rhx git.branch.rebase abort`

note: native git uses flags (`--continue`), this skill uses subcommands (`continue`).
the translation is simple but not zero-cost.

## user experience

### usecases

| goal | command |
|------|---------|
| start rebase onto main | `rhx git.branch.rebase begin` |
| preview rebase changes | `rhx git.branch.rebase begin --mode plan` |
| execute rebase | `rhx git.branch.rebase begin --mode apply` |
| continue after conflicts settled | `rhx git.branch.rebase continue` |
| abandon rebase | `rhx git.branch.rebase abort` |
| get help | `rhx git.branch.rebase help` or `rhx git.branch.rebase --help` |

### contract

```
git.branch.rebase <subcommand> [options]

subcommands:
  begin     start rebase onto origin/main
  continue  continue rebase after conflicts settled
  abort     abandon rebase, restore to pre-rebase state
  help      show usage and available subcommands

options:
  --mode plan|apply    execution mode (begin only, default: plan)
  --help               show help
```

### timeline

```
1. mechanic: rhx git.branch.rebase begin --mode plan
   → previews: 3 commits ahead, 2 commits behind

2. mechanic: rhx git.branch.rebase begin --mode apply
   → conflicts on src/auth.ts

3. mechanic: settles conflicts in src/auth.ts, stages changes

4. mechanic: rhx git.branch.rebase continue
   → rebase completes, ready to push

5. mechanic: rhx git.commit.push --mode apply
   → pushes rebased branch
```

## mental model

### how users describe it

> "git.branch.rebase is like git rebase but safer - it checks permissions, shows plan first, and guides you through conflicts"

### analogies

- **traffic controller**: git.branch.rebase directs you to begin/continue/abort based on state
- **state machine**: enter via begin, progress via continue, exit via abort or completion

### terms

| user says | skill uses |
|-----------|------------|
| "rebase" | `git.branch.rebase begin` |
| "continue the rebase" | `git.branch.rebase continue` |
| "cancel" / "nevermind" | `git.branch.rebase abort` |
| "what's happening?" | `git.branch.rebase begin --mode plan` (state shown in error if mid-rebase) |

## evaluation

### pros

- **new pattern**: establishes dispatcher pattern (git.commit skills are invoked directly, this adds unified entry)
- **discoverability**: `rhx git.branch.rebase --help` shows all subcommands
- **permissions**: single prefix rule covers all subcommands
- **mental model**: similar to native git command structure

### cons

- **migration**: extant invocations using `git.branch.rebase.begin` continue to work via rhachet resolution

### pit of success

| edgecase | protection |
|----------|------------|
| begin while rebase in progress | error: "rebase already in progress", suggests continue/abort |
| continue with no rebase | error: "no rebase in progress" |
| abort with no rebase | error: "no rebase in progress" |
| unknown subcommand | error: "unknown subcommand", shows valid options |
| no subcommand | error: "subcommand required", shows help |

## what is awkward?

### potential friction

- **name choice**: why `git.branch.rebase` not `git.commit.rebase`?
  - considered: `git.commit.rebase` for consistency with `git.commit.push`
  - analysis: rebase operates on branches, not commits. commits get rewritten as side effect.
  - decision: `git.branch.rebase` as own namespace
  - rationale: major operation with own state machine (begin/continue/abort), doesn't fit under `git.commit.*`

- **help flag position**: `rhx git.branch.rebase --help` vs `rhx git.branch.rebase begin --help`
  - resolution: both work - main shows subcommands, subcommand shows its options

### design tradeoffs

- **explicit dispatcher**: adds one more file to maintain, but simplifies permissions and discoverability
- **exec vs source**: using `exec` means single process, cleaner but loses parent context (acceptable here)
