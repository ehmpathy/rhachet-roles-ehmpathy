# review.self: 1.vision.1.has-questioned-requirements

## requirement 1: explicit git.branch.rebase.sh dispatcher

**who said this was needed?**
the wish: "git.branch.rebase should have its own .sh skill and should dispatch to the subskills"

**what evidence supports this?**
- git.commit.uses.sh follows this pattern - dispatches to .local.sh and .global.sh
- centralized help text improves discoverability
- state-aware route can suggest continue/abort when appropriate

**what if we didn't do this?**
rhachet already resolves `rhx git.branch.rebase begin` → `git.branch.rebase.begin.sh` implicitly.
without explicit dispatcher:
- `rhx git.branch.rebase --help` would fail (no git.branch.rebase.sh found)
- no centralized error handler for unknown subcommands
- no state-aware suggestions

**verdict: holds**
the explicit dispatcher adds genuine value beyond implicit resolution.

## requirement 2: space-separated subcommands (rhx git.branch.rebase begin)

**who said this was needed?**
i inferred this from the wish and the native git mental model.

**what evidence supports this?**
- native git uses `git rebase --continue`, `git rebase --abort`
- current skills already show this pattern in their usage docs:
  - `rhx git.branch.rebase begin` in git.branch.rebase.begin.sh line 9

**what if we didn't do this?**
users would invoke `rhx git.branch.rebase.begin` (with dots) which also works.
but the vision claims both should work.

**is there a simpler way?**
the dispatcher receives first positional arg and maps it to subskill.
this is the simplest approach - same as git.commit.uses.sh.

**verdict: holds**
space separation matches native git and is already documented in subskills.

## requirement 3: single permission entry

**who said this was needed?**
i inferred this as a benefit.

**what evidence supports this?**
- current state requires 3 permission entries (begin, continue, abort)
- prefix match `git.branch.rebase:*` would cover all

**what if we didn't do this?**
permissions file would be more verbose. not a blocker, just less elegant.

**could we achieve the goal in a simpler way?**
yes - even without dispatcher, `git.branch.rebase:*` prefix would work if rhachet resolves `git.branch.rebase.begin` correctly.

**verdict: holds but not critical**
this is a nice-to-have benefit, not the core value prop.

## requirement 4: state-aware suggestions

**who said this was needed?**
i inferred this from the pit of success section.

**what evidence supports this?**
- git.branch.rebase.begin.sh already checks `is_rebase_in_progress` and suggests continue/abort
- to centralize this in dispatcher reduces duplication

**what if we didn't do this?**
subskills already handle their own guards. dispatcher could just dispatch.

**verdict: scope creep - remove from vision**
the wish asks for dispatch, not state-aware route. keep it simple.
each subskill already handles its own guards.

---

## changes made

1. **removed state-aware route from scope**
   - original vision implied dispatcher would be smart about state
   - simpler: dispatcher just dispatches, subskills handle guards

2. **clarified dispatcher value prop**
   - primary value: centralized help text, unknown subcommand handler
   - not: state-aware route (subskills do this)

---

## what holds

- explicit `git.branch.rebase.sh` dispatcher adds value (help, error handler)
- space-separated subcommands match mental model
- permission simplification is a nice benefit
- exec-based dispatch to subskills is the right pattern
