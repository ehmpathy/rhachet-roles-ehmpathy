## verdict

**snapshots not applicable to dispatcher, but present in subskills.**

rationale:
1. dispatcher tests verify dispatch behavior (is subskill invoked?)
2. subskill tests verify output format (22 snapshot assertions)
3. user sees subskill output, not dispatcher output

---

## reflection

snapshots are valuable to verify user-visible output. the dispatcher is invisible to users — they see subskill output. snapshot coverage at the subskill level is the right place.

### why this holds

1. **dispatcher is transparent**: users invoke `rhx git.branch.rebase begin`, see output from `git.branch.rebase.begin.sh`
2. **subskills have snapshots**: 22 snapshot assertions verify user-visible output
3. **dispatcher tests verify behavior**: dispatch correctness, not output format
4. **separation of concerns**: dispatcher owns route dispatch, subskills own presentation

### hostile reviewer check

| potential violation | found? | evidence |
|---------------------|--------|----------|
| user output not snapshotted | no | subskills have 22 snapshots |
| dispatcher hides output | no | dispatcher uses exec (transparent) |
| help output not verified | no | case1-2 use toContain for help text |
| error output not verified | no | case3-4 use toContain for error text |

snapshot coverage is at the correct layer (subskills).
