# vision: abstraction for readability, not just reuse

## the outcome world

### what does a day-in-the-life look like?

**before:**
a mechanic reads an orchestration function and hits opaque logic. they pause, trace backwards, mentally simulate the transform. 30 seconds lost. mental energy spent. for a robot: tokens burned, datacenter cycles wasted.

**after:**
the same logic is behind a named operation. no pause. the name tells the story. the orchestration reads like a book.

### three examples

#### example 1: structured string parse

```ts
// before — requires decode: what's at position 0? what's the format?
org: slug.split('.')[0]!

// after — intent is clear from the name
org: asKeyrackKeyOrg({ slug })
```

#### example 2: complicated boolean

```ts
// before — requires trace: what conditions make this true?
const eligible = user.age >= 18 && user.verified && !user.suspended && user.subscription !== 'free';

// after — intent is clear from the name
const eligible = isEligibleForPremiumFeatures({ user });
```

#### example 3: multi-step transformation

```ts
// before — requires simulation: what does this pipeline produce?
const activeUserEmails = users
  .filter(u => u.status === 'active' && u.emailVerified)
  .map(u => u.email.toLowerCase())
  .sort();

// after — intent is clear from the name
const activeUserEmails = getActiveUserEmails({ users });
```

### the pattern

the "decode-required logic" (array access, reduce, regex, conditionals with fallbacks) moves into transforms. orchestrators become pure narrative.

| category | before | after |
|----------|--------|-------|
| string parse | `slug.split('.')[0]!` | `asKeyrackKeyOrg({ slug })` |
| date extract | `new Date(ts).toJSON().split('T')[0]` | `asIsoDate({ from: date })` |
| aggregate | `items.reduce((s, i) => s + i.amount, 0)` | `computeTotal({ items })` |
| slug convert | `str.toLowerCase().replace(/\s+/g, '-')` | `asSlug({ from: str })` |
| pipeline | `.filter(...).map(...).sort()` | `getActiveUserEmails({ users })` |
| boolean | `a && b \|\| c && !d` | `isEligibleForDiscount({ order })` |
| etc | *any decode-required logic* | *named transform* |

### the "aha" moment

> "oh — abstraction isn't just about reuse. it's about making code readable at the *call site*."

the insight clicks when you realize:
- the orchestrator is where humans and robots spend reading time
- machine code in orchestrators forces readers to decode implementation
- named operations let readers understand intent immediately
- **readability abstraction pays dividends on every read, not just reuse**

## user experience

### what usecases do folks fulfill?

1. **writing orchestrators** — compose transforms into workflows that read like prose
2. **reading orchestrators** — understand intent immediately without decoding machine code
3. **reviewing prs** — verify logic at the intent level, not implementation level
4. **onboarding** — new team members grasp flows without deep implementation knowledge
5. **debugging** — trace through narrative flow, not opaque transformations

### what contract inputs & outputs do they leverage?

**transforms:**
```ts
// input: explicit shape
// output: semantic type
// implementation: decode-required logic lives HERE

const asKeyrackKeyOrg = (input: { slug: string }): string => {
  return input.slug.split('.')[0]!;
};

const isEligibleForPremiumFeatures = (input: { user: User }): boolean => {
  return input.user.age >= 18 && input.user.verified && !input.user.suspended && input.user.subscription !== 'free';
};

const getActiveUserEmails = (input: { users: User[] }): string[] => {
  return input.users
    .filter(u => u.status === 'active' && u.emailVerified)
    .map(u => u.email.toLowerCase())
    .sort();
};
```

**orchestrators:**
```ts
// orchestrators contain NO decode-required logic
// they read as narrative: subject → verb → object
// they use get/set/gen patterns

const setUserNotifications = async (input: { users: User[], user: User }, context: Context) => {
  const emails = getActiveUserEmails({ users: input.users });
  const canAccess = isEligibleForPremiumFeatures({ user: input.user });

  if (canAccess) await context.sdkEmail.sendBulk({ to: emails, subject: 'Update' });
  return { sent: canAccess ? emails.length : 0 };
};
```

### what would it look like to leverage them?

**pattern: extract decode-required logic into transforms**

when you see this in an orchestrator:
```ts
// decode-required: reader must simulate the reduce
const total = items.reduce((sum, item) => sum + item.amount, 0);
```

extract to:
```ts
// domain.operations/invoice/computeInvoiceTotal.ts
export const computeInvoiceTotal = (input: { items: LineItem[] }): number => {
  return input.items.reduce((sum, item) => sum + item.amount, 0);
};
```

then orchestrator becomes:
```ts
// no decode required: name tells the story
const total = computeInvoiceTotal({ items });
```

### what timelines do they go through?

1. **recognize** — spot machine code in orchestrator (array access, regex, string ops)
2. **name** — find the domain term for what this machine code produces
3. **extract** — create transform with semantic name
4. **replace** — swap machine code for operation call
5. **verify** — orchestrator now reads as narrative

## mental model

### how would users describe this to a friend?

> "we keep the 'how' in transforms and the 'what' in orchestrators. orchestrators should read like a story, not like assembly code."

> "if you have to decode what a line does, it belongs in a named operation."

### what analogies or metaphors fit?

**the book metaphor:**
- transforms are **vocabulary** — words with precise meaning
- orchestrators are **sentences** — composed of vocabulary to tell a story
- machine code in orchestrators is like **spelling out words letter by letter** mid-sentence

nobody wants to read "the c-a-t sat on the m-a-t" when they could read "the cat sat on the mat."

same energy as `slug.split('.')[0]!` vs `asKeyrackKeyOrg({ slug })`.

**the compiler metaphor:**
- transforms are the **instruction set** — atomic, well-defined
- orchestrators are **high-level code** — expressive, readable
- machine code in orchestrators is like **inline assembly** — breaks the abstraction

### what terms would they use vs what terms would we use?

| user says | we say |
|-----------|--------|
| "helper function" | "transform" |
| "extract a function" | "abstract for readability" |
| "too much detail here" | "machine code in orchestrator" |
| "make it readable" | "orchestrators as narrative" |

## evaluation

### how well does it solve the goals?

| goal | solved? |
|------|---------|
| reduce cognitive load on readers | yes — intent visible at call site |
| reduce tokens spent by robots | yes — name comprehension vs decode |
| create narrative flow | yes — orchestrators read as prose |
| prevent machine code creep | partially — requires discipline + review |

### pros

1. **immediate comprehension** — readers understand intent without decoding
2. **token efficiency** — robots spend fewer tokens grasping orchestrators
3. **reviewability** — prs discuss intent, not implementation details
4. **testability** — transforms are pure, easy to unit test
5. **evolvability** — change implementation without touching orchestrators

### cons

1. **more files** — each transform is a file
2. **naming burden** — must find good domain names
3. **judgment required** — "is this machine code?" needs calibration
4. **potential over-abstraction** — could extract too eagerly

### how do our contracts keep users in a pit of success?

1. **simple test**: "do i have to decode this to understand it?" — if yes, extract
2. **practical heuristic**: if it's not named by us in this repo or from an ehmpathy package, wrap it
3. **dual-level guidance**: architect briefs explain the structural principle, mechanic briefs provide implementation examples
4. **name patterns**: defer to `rule.require.get-set-gen-verbs` for extant name conventions

## the rule: dual-level structure

this rule lives at two levels:

### architect level (structural principle)

**the concept:** orchestrators compose, transforms compute.

- **orchestrators** = domain operations that compose other operations into workflows
- **transforms** = domain operations that perform actual computation/transformation

**rule.require.orchestrators-as-narrative:**
> orchestrators must read as narrative — each line tells *what* happens, not *how*

**rule.forbid.decode-friction-in-orchestrators:**
> orchestrators must not contain logic that requires mental simulation to understand

### mechanic level (implementation guidance)

**the heuristic:** "do i have to decode this to understand what it produces?"

- yes → extract to a named transform
- no → leave inline

this is NOT about specific categories. there is no enumerable list of "forbidden patterns." complexity that requires decode can come from anywhere.

**the test:** if you cannot immediately understand what a line produces by reading it, the implementation belongs in a transform.

**practical heuristic:** if the operation isn't named by us in this repo or from an ehmpathy package, it probably isn't named readably enough. language primitives and third-party apis are optimized for generality, not domain clarity. wrap them in domain-named transforms.

**note:** the examples in this vision (string splits, reduces, regex) are *illustrative*, not prescriptive. the rule applies to *any* logic that requires mental simulation to understand — regardless of what language constructs it uses.

### the two grains (architect definition)

| grain | role | contains | examples |
|-------|------|----------|----------|
| **transform** | compute | decode-required logic | `asKeyrackKeyOrg`, `isEligibleForPremiumFeatures`, `getActiveUserEmails` |
| **orchestrator** | compose | named operation calls only | `genInvoice`, `setCustomer`, `setUserNotifications` |

transforms do the work. orchestrators tell the story.

## what is awkward?

### what feels off or forced?

1. **terminology: "decode-required logic"** — we've moved away from "machine code" to "decode-required logic" which is clearer, but still requires calibration per reader.

2. **single-use extraction** — this explicitly contradicts `rule.prefer.wet-over-dry` which says wait for 3+ usages. reconciliation: readability abstraction is a *different category* than reuse abstraction (see proposed reconciliation below).

3. **subjectivity** — "do i have to decode this?" is inherently subjective. different readers have different thresholds. the heuristic "if it's not from this repo or an ehmpathy package, wrap it" helps calibrate.

### where does the design fight the user's mental model?

1. **"this creates too many files"** — users trained on "don't over-abstract" will resist single-use extractions
2. **"but it's just one line"** — line count isn't the metric; decode-cost is

**action required:** audit and update all briefs that touch abstraction to be coherent with this rule:
- `rule.prefer.wet-over-dry` — add exception: readability abstraction triggers immediately, reuse abstraction waits for 3+
- any brief that discourages single-use extraction — clarify that decode-cost overrides file count concerns
- ensure no brief contradicts "abstract for readability even if single-use"

the reconciliation: wet-over-dry applies to *reuse* abstraction (wait for 3+), but readability abstraction triggers immediately on decode-cost.

**todo:** create a handoff brief for bhuild repo to add a review rule that enforces readability abstraction in code reviews (detect decode-friction in orchestrators).

### what tradeoffs feel uncomfortable?

1. **file proliferation vs readability** — worth it, but feels heavy
2. **naming everything vs inline clarity** — sometimes `arr[0]` feels clearer than a named operation (we need to acknowledge when inline IS clearer)
3. **immediate abstraction vs emerging patterns** — reconciled below

## proposed reconciliation: two types of abstraction

| type | trigger | timing | example |
|------|---------|--------|---------|
| **readability abstraction** | decode-cost at call site | immediate | `slug.split('.')[0]` → `asKeyrackKeyOrg({ slug })` |
| **reuse abstraction** | duplication across contexts | wait for 3+ | 3 places call same logic → extract shared operation |

this resolves the tension:
- extract for readability even if single-use
- don't extract for reuse until pattern emerges
- a transform can later gain reuse value, but readability is the first driver

### the heuristic

> **readability abstraction:** "do i have to decode this to understand it?"
> - yes → extract immediately, even if single-use
> - no → leave inline

> **reuse abstraction:** "have i written this 3+ times?"
> - yes → extract for reuse
> - no → keep it wet, wait for pattern

both can apply to the same code. readability is the first gate.
