# blueprint: init.declapract.upgrade implementation

## summary

transform `declapract.upgrade.sh` into `init.declapract.upgrade.sh` that:
1. instantiates a rigid route at `.route/v$isodate.declapract.upgrade/`
2. binds the route to the current branch

the current `declapract.upgrade.sh` logic moves into `1.upgrade.invoke.sh` — an executable emitted to the route directory. the stone just tells the driver to run that executable.

## templates

### 1.upgrade.invoke.sh

the executable that performs the upgrade. chmod 555 (executable, not writable).

```bash
#!/usr/bin/env bash
######################################################################
# .what = upgrade declapract and apply latest best practices
#
# .why  = declapract manages development best practices and tooling
#         configuration across projects, ensuring consistency and
#         enabling easy upgrades to latest standards.
#
# guarantee:
#   ✔ uses fnm for node version
#   ✔ uses pnpm (not npm)
#   ✔ upgrades to latest declapract packages
#   ✔ applies practices twice (ensures idempotency)
#   ✔ reinstalls dependencies after application
#   ✔ fail-fast on any error
######################################################################

set -euo pipefail

echo "🐢 upgrade time!"

echo "📦 ensure node version"
fnm use

echo ""
echo "📦 upgrade declapract packages"
pnpm add declapract@latest declapract-typescript-ehmpathy@latest --save-dev

echo ""
echo "✨ apply best practices"
npx declapract apply && npx declapract apply

echo ""
echo "📦 reinstall dependencies"
pnpm install

echo ""
echo "🔧 auto fix"
pnpm fix

echo ""
echo "😎 easy part done, now for the fun part"
```

### 2.detect.hazards.stone

purpose: scan diffs for backward incompat changes

```markdown
# 2.detect.hazards

## purpose

scan the changes from declapract upgrade to detect hazards that may break tests, cicd, or deployments.

## instructions

1. review git diff for all changes:
   \`\`\`sh
   git diff
   \`\`\`

2. identify hazards in these categories:
   - **test hazards**: changes that may break unit, integration, or acceptance tests
   - **cicd hazards**: changes that may break github actions workflows
   - **path hazards**: renamed or moved files that may break imports
   - **config hazards**: changed configs that may affect behavior

3. for each hazard, document:
   - what changed
   - why it may break
   - how to verify (test command or manual check)

## emit

emit `2.detect.hazards.v1.i1.md` with the hazard list.
```

### 2.detect.hazards.guard

```yaml
reviews:

        - cicd hazards you overlooked
        - path hazards you overlooked
        - config hazards you overlooked

    - slug: has-validated-hazards
      say: |
        which of these are mistaken? are any not actual issues?

        for each hazard listed, prove or disprove via tests or logic.

### 3.1.repair.test.defects.stone

# 3.1.repair.test.defects

## purpose

run tests until all pass. document each defect with root cause.

## instructions

1. run all tests:
   \`\`\`sh
   npm run test
   \`\`\`

2. for each failure:
   - identify the defect
   - determine root cause (why did declapract upgrade cause this?)

- when root cause found → add to inventory

- why: root cause (why it arose from upgrade)
- fix: how it was fixed
```
