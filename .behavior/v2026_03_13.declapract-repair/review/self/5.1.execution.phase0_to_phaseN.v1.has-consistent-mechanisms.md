## why it holds: lessons for future implementors

### lesson 1: pattern reuse over code reuse in shell

shell scripts lack an import system. you cannot `import { print_turtle_header } from './output.sh'`. sourcing a file executes all its top-level code, which causes side effects.

**the pattern:**
- copy the structure and function signatures
- adapt for local context
- keep the vocabulary consistent (turtle vibes, tree structure)

**why this is not duplication:**
- duplication = same code solving same problem
- adaptation = same pattern solving related problem
- our output.sh solves "route instantiation output" not "git commit output"

### lesson 2: use extant infrastructure skills

the blueprint prescribed `npx rhachet run --repo bhrain --skill route.bind.set`. we invoked exactly that.

**the pattern:**
- search for extant skills that do what you need
- invoke them via npx rhachet run
- do not reimplement route binding logic

**why this matters:**
- route binding involves internal rhachet state
- reimplementing it would create drift between behaviors
- infrastructure skills are maintained centrally

### lesson 3: consistent argument parsing

every mechanic skill skips `--skill`, `--repo`, `--role`. this is not by accident.

**the pattern:**
```bash
--skill|--repo|--role)
  shift 2
  ;;
```

**why this matters:**
- rhachet passes these args to all skills
- skills must not fail on unknown args
- consistent handling prevents confusing error messages

### lesson 4: fail-fast over silent failure

we use `set -euo pipefail` and explicit prerequisite checks.

**the pattern:**
- check prerequisites early
- exit with clear error message
- use exit code 2 for prerequisite failures

**why this matters:**
- silent failures waste debugging time
- clear messages tell users exactly what to do
- exit codes enable scripting (`|| handle_error`)
