## requirements questioned

### requirement 1: the invocation becomes `rhx init.declapract.upgrade`

**who said this was needed?**
the wisher, through three iterations of clarification.

**what evidence supports this requirement?**
1. first: "invoke `rhx declapract.upgrade`"
2. then: "we replace declapract.upgrade completely"
3. finally: "lets call it via `rhx init.declapract.upgrade` actually"

**what if we didn't do this?**
users would invoke a generic `route.init` command and not understand that this skill is the entry point. the `init.` prefix signals that this skill initializes something — in this case, a route.

**is the scope correct?**
yes. the invocation is the contract. it must be precise.

---

### requirement 2: the skill replaces the current declapract.upgrade.sh entirely

**who said this was needed?**
the wisher: "we replace declapract.upgrade completely, to instead instantiate the prescribed route"

**what evidence supports this requirement?**
the extant skill is a one-shot command. the new skill must be a route-instantiator. these are incompatible behaviors — one must replace the other.

**what if we didn't do this?**
two skills would exist: the old one-shot and the new route-based. confusion would ensue. which to use? when? why?

**is the scope correct?**
yes. clean replacement is simpler than coexistence.

---

### requirement 3: route lives under `.route/`

**who said this was needed?**
the wish (line 12) and wisher confirmation: "route lives under `.route/`"

**what evidence supports this requirement?**
the wish specifies `.route/v$isodate.declapract.upgrade/`. the wisher confirmed explicitly.

**what if we didn't do this?**
if we used `.behavior/`, it would conflict with behavior definitions. routes are instances, behaviors are definitions. separation is correct.

**is the scope correct?**
yes. `.route/` for route instances, `.behavior/` for behavior definitions.

---

### requirement 4: reflect stones use the `approved?` judge

**who said this was needed?**
the wish (lines 112-113) and wisher confirmation: "they must use the `approved?` judge"

**what evidence supports this requirement?**
checking this behavior's guards, I see: `judges: - npx rhachet run --repo bhrain --skill route.stone.judge --mechanism approved?`

**what if we didn't do this?**
reflect stones would pass via self-review only. human validation would be skipped. the feedback loop to infrastructure would lack human oversight.

**is the scope correct?**
yes. reflect stones are the critical checkpoints where a human must approve the classification of defects.

---

### requirement 5: always use pnpm

**who said this was needed?**
the wish (line 41): "ensure to do it via pnpm, not npm"
the wisher confirmation: "it should use `pnpm` always"

**what evidence supports this requirement?**
explicit in wish and confirmed by wisher.

**what if we didn't do this?**
repos with pnpm lockfiles would have inconsistent state. npm would create `package-lock.json` alongside `pnpm-lock.yaml`.

**could we achieve this simpler?**
no. the wish is clear. always pnpm.

---

### requirement 6: route declaration via .stone and .guard files

**who said this was needed?**
the wisher: "the .v1.i1.md's are output artifacts that the driver emits. the stone and guard files declare the route."

**what evidence supports this requirement?**
checking this behavior route, I see: `1.vision.stone`, `1.vision.guard`, etc. these are the route declaration. the `.md` files are artifacts.

**what if we didn't do this?**
the vision would conflate declaration with artifacts. readers would not understand what is static (stones/guards) vs dynamic (artifacts).

**is the scope correct?**
yes. clear separation.

---

### requirement 7: require zero test failures even if they predate the upgrade

**who said this was needed?**
the wish (line 93): "it must take initiative and fix all defects, even if they existed before this pr"

**what evidence supports this requirement?**
explicit in wish.

**what if we didn't do this?**
mechanics could blame pre-extant failures on the upgrade. root cause analysis would be incomplete. the clean-slate goal would fail.

**is the scope too large?**
perhaps. but the friction is intentional. the upgrade is an opportunity to clean house.

---

### requirement 8: two reflect stones (test + cicd) rather than one

**who said this was needed?**
the wish explicitly lists separate stones: `3.2.reflect.test.defects` and `3.4.reflect.cicd.defects`

**what evidence supports this requirement?**
test repairs happen before push (t3). cicd repairs happen after push (t5). they're temporally separate. each needs its own human approval gate.

**could we achieve this simpler?**
consolidation would mean waiting until everything is done. the feedback loop would be delayed. separation allows progressive validation.
