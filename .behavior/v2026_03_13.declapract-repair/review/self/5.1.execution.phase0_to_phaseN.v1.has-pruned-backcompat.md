### behavior break: skill rename

the old `declapract.upgrade.sh` was deleted and replaced with `init.declapract.upgrade.sh`.

- **wisher requested this?** yes, blueprint filediff tree line 17: `[~] declapract.upgrade.sh → init.declapract.upgrade.sh (rename + rewrite)`
- **backwards compat shim added?** no
- **should a shim exist?** no - per rule.forbid.backwards-compat-hacks, we delete rather than shim

### code that might look like backwards compat but isn't

1. **`|| true` on route.bind.set**
   - looks like: fail-safe fallback
   - actually: allows skill to work when rhachet not installed (test isolation)
   - not compat: doesn't preserve old behavior, enables new behavior in more contexts

2. **SKIP_ROUTE_BIND env var**
   - looks like: escape hatch
   - actually: test isolation mechanism
   - not compat: new code, not to preserve old behavior

### why no backwards compat shim for the rename?

the rename from `declapract.upgrade.sh` to `init.declapract.upgrade.sh` could have been shimmed via:

```bash
# hypothetical shim at declapract.upgrade.sh
#!/usr/bin/env bash
echo "DEPRECATED: use init.declapract.upgrade instead"
exec "$(dirname "$0")/init.declapract.upgrade.sh" "$@"
```

**why this shim was not added:**

1. **rule.forbid.backwards-compat-hacks applies**: the brief states "we delete rather than shim" — shims create maintenance burden and hide the migration path

2. **no external callers**: `declapract.upgrade.sh` was only invoked via `rhx declapract.upgrade` which routes through rhachet's skill discovery. the skill registration will point to the new name after build+link.

3. **blueprint explicitly prescribed rename**: the filediff tree shows `[~] declapract.upgrade.sh → init.declapract.upgrade.sh (rename + rewrite)` — the wisher requested this break, not a gradual migration.

4. **init.* naming convention**: the new name follows the `init.*` pattern used by other route-instantiation skills (e.g., `init.behavior` in rhachet-roles-bhuild). this is a semantic improvement, not just a rename.

5. **clean break is safer**: a shim could cause confusion if someone invokes the old name and expects old behavior but gets new behavior. better to fail loudly (skill not found) than silently redirect.

### why the other code patterns are not backwards compat

**`|| true` on route.bind.set:**
- this is defensive coding for the current behavior, not preservation of old behavior
- old skill (`declapract.upgrade.sh`) did not call route.bind.set at all
- therefore, making route.bind.set fail-safe cannot be backwards compat — there's no old behavior to be compatible with

**SKIP_ROUTE_BIND env var:**
- this is a test-time mechanism, not a runtime compatibility layer
- it exists because integration tests run in temp directories without rhachet installed
- again, old skill never called route.bind.set, so this cannot be compat for old behavior

**cp -f flag:**
- this ensures idempotent file copy over chmod 555 files
- old skill did not create any files, so there's no prior behavior to be compatible with
- the -f flag is required for the new findsert semantics, not for old behavior

### conclusion: no backwards compat code present

implementation has no backwards compat code. the one behavior break (rename) was explicitly requested and correctly implemented as a clean break rather than a deprecated shim.
