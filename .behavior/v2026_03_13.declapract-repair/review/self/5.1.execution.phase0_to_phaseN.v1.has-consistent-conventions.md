### divergence found: print_tree_branch signature

**git.commit/output.sh (line 31-40):**
```bash
# print tree branch (has children)
# usage: print_tree_branch "commit" [is_last]
print_tree_branch() {
  local label="$1"
  local is_last="${2:-false}"
  ...
}
```

**init.declapract.upgrade/output.sh (line 30-41):**
```bash
# print tree branch (has children)
# usage: print_tree_branch "route" ".route/v2026_03_13.declapract.upgrade/ ✨" [is_last]
print_tree_branch() {
  local key="$1"
  local value="$2"
  local is_last="${3:-false}"
  ...
}
```

**divergence:** same function name, different signatures
- git.commit: `print_tree_branch(label, is_last)` — label only
- init.declapract.upgrade: `print_tree_branch(key, value, is_last)` — key + value

**why this happened:**
- git.commit branches display just labels: `├─ commit`
- init.declapract.upgrade branches display key-value pairs: `├─ route: .route/v2026_03_13.declapract.upgrade/ ✨`

**is this acceptable?**

analysis: the two output.sh files serve different skills and are not shared. shell scripts lack an import system, so each skill adapts the pattern for its own needs. the function names match the pattern (`print_tree_*`) but the signatures differ based on output needs.

**verdict:** divergence is acceptable because:
1. output.sh files are skill-local, not shared
2. pattern reuse (not code reuse) is the shell convention
3. the semantic meaning is preserved (both print tree structure)
4. callers only ever see one version (within their skill)

**alternative considered:** could have named the function `print_tree_branch_kv` to distinguish
**why rejected:** adds complexity for no user benefit — callers are skill-internal

**lesson:** when adapting shell patterns, document signature differences to prevent future confusion if patterns are ever unified.

### new terms introduced

#### print_coconut function

**searched for:** output footer patterns in turtle vibes

**found in git.commit/output.sh:**
- `print_turtle_header` — header with emoji
- `print_tree_*` — tree structure
- `print_tip` — dim tip text
- `print_instruction` — post-tree instructions

**our new term:** `print_coconut` — footer with bound route info

**why introduced:**
- no extant footer pattern for route instantiation
- git.commit uses `print_instruction` for post-action guidance
- route instantiation needs different footer: bound route info
- "coconut" follows turtle vibes vocabulary (tropical, beach)

**why consistent:**
- follows `print_{element}` pattern
- turtle vibes vocabulary is open to extension
- semantically distinct: coconut = completion marker

#### print_tree_item function

**searched for:** tree item patterns

**found in git.commit/output.sh:**
- `print_tree_branch` — takes label only
- `print_tree_leaf` — takes key + value

**our addition:** `print_tree_item` — takes label, outputs plain tree item

**why introduced:**
- needed a label-only tree item (no key-value pair)
- `print_tree_branch` was semantically wrong (we have children)
- `print_tree_leaf` requires value parameter

**alternative considered:** could have used `print_tree_branch` with empty value

**why our choice is better:**
- clearer intent: "item" vs "branch" vs "leaf"
- no null/empty value hack
- self-documenting

#### print_error function

**searched for:** error output patterns

**found in git.commit/output.sh:** `print_tree_error` — error within tree

**our addition:** `print_error` — standalone error with turtle vibes ("bummer dude")

**why different:**
- `print_tree_error` outputs within a tree structure
- our errors happen before tree is printed (prerequisites fail)
- standalone error needs its own header

**verdict:** not a convention divergence — different use case.

## lessons for future implementors

### lesson 1: init.* prefix signals route instantiation skills

the `init.` prefix signals a skill that creates a structured directory. examples:
- `init.behavior` → creates `.behavior/`
- `init.research` → creates `.research/`
- `init.declapract.upgrade` → creates `.route/`

do not use `init.` for skills that modify extant structure.

### lesson 2: route paths follow behavior paths

routes (`.route/`) mirror behavior paths (`.behavior/`):
- both use `v{date}.{slug}/` name convention
- both contain `.stone` and `.guard` files
- the difference is semantic: behaviors are for product construction, routes are for process workflows

### lesson 3: stone numbers indicate order

- single number (1, 2) for major phases
- decimal numbers (3.1, 3.2) for sub-phases within a major phase
- this matches behavior route conventions

### lesson 4: test files are collocated

test files live next to their source files with the pattern `{name}.integration.test.ts`. this enables:
- easy discovery of tests
- clear ownership
- automatic inclusion in test runs

### lesson 5: output.sh is pattern reuse, not code reuse

shell scripts lack an import system. when you create `{skill}/output.sh`, you are reusing the pattern from `git.commit/output.sh`, not sharing code.

this means:
- function signatures can differ to match skill needs
- semantic meaning should be preserved (both print turtle vibes output)
- document divergences so future unification is possible

when diverging:
1. keep function names in the same vocabulary (`print_tree_*`, `print_turtle_*`)
2. document why the signature differs
3. consider if a new name would prevent confusion (`print_tree_branch_kv`)
4. accept divergence if skill-local and callers never see both versions

### lesson 6: honest convention review catches real issues

this review initially claimed "no divergence found" — which was wrong. the `print_tree_branch` signature divergence was real. honest review:
1. compares actual code, not just names
2. documents divergences even if acceptable
3. explains why divergence is acceptable (or not)
4. creates lessons from the finding
