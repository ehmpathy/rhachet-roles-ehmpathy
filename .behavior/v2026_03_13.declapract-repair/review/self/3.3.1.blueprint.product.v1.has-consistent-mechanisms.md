**decision**: inline prereq checks is the extant pattern. **not a duplication concern**.

### question: does bhuild's init.behavior share any mechanisms we should reuse?

examined `init.behavior.sh` in rhachet-roles-bhuild:
- it uses typescript, not bash
- it calls `route.bind.set` (same as our blueprint)
- it uses the same stone/guard file format

**conclusion**: the mechanisms are consistent. init.behavior handles behavior routes; init.declapract.upgrade handles declapract upgrade routes. different domains, same patterns.

**verdict**: no mechanism to extract or share. **holds**.

---

### question: should we use a shared template engine?

the blueprint proposes heredoc templates. examined extant bash skills:
- git.commit.set.sh: inline strings
- git.commit.push.sh: inline strings
- declapract.upgrade.sh: no template, direct commands

no skill uses a template engine. all use heredocs or inline strings.

**verdict**: heredoc templates are consistent with extant pattern. **holds**.

---

### question: could prerequisite checks be shared?

compared prerequisites across skills:
- git.commit.set.sh: checks commit quota
- git.commit.push.sh: checks push permission
- declapract.upgrade.sh (current): no prerequisite checks
- init.declapract.upgrade (proposed): checks declapract.use.yml, pnpm

prerequisites differ per skill. sharing would require:
- parameterized check function
- extra complexity for no benefit
- each skill knows its own prerequisites

**verdict**: inline prerequisite checks are appropriate. **holds**.
