# vision: declapract.upgrade as a rigid thought route

## the outcome world

### day-in-the-life with this in place

a mechanic begins to upgrade a repo to latest practices. instead of a one-shot command and hope, they invoke:

```sh
rhx init.declapract.upgrade
```

this skill **replaces** the current implementation entirely. it:
1. instantiates `.route/v$isodate.declapract.upgrade/` with stones and guards
2. binds the route to the current branch
3. route.drive then shepherds execution through each phase:

1. **invoke** — run the upgrade, document any infrastructure defects
2. **detect** — scan diffs for changes that break tests, cicd, deployments
3. **repair tests** — fix all test failures, document root causes
4. **reflect tests** — classify defects: repo quirk vs practice bug vs adoption candidate
5. **repair cicd** — push, watch, fix, repeat until green
6. **reflect cicd** — same classification, feed learnings back to infrastructure

each defect found gets documented with:
- what broke
- how it was fixed
- why it arose (root cause)
- where the fix belongs (this repo? declapract-typescript-ehmpathy? declapract core?)

### before/after contrast

| before | after |
|--------|-------|
| run command, hope it works | rigid route with checkpoints |
| defects fixed ad-hoc, forgotten | defects documented, root-caused, fed back |
| same issues recur across repos | infrastructure evolves, issues vanish |
| "it broke" | "it broke X because Y, fix is Z" |
| mechanic improvises | mechanic follows proven path |

### the "aha" moment

the value clicks when a mechanic finishes a declapract upgrade and realizes:
- every defect they hit is now documented
- their reflection feeds directly into prevention of the same defect for the next repo
- the route prevented them from a skip of a critical check
- the guards caught issues they would have missed

**route structure:**

the skill instantiates `.route/v$isodate.declapract.upgrade/` with two types of files:

**route declaration** (static, instantiated by skill):
- `.stone` files — declare what the driver should do at each step
- `.guard` files — declare how to guard passage (self-reviews, judges)

```
.route/v2026_03_13.declapract.upgrade/
├── 1.upgrade.invoke.stone
├── 2.detect.hazards.stone
├── 2.detect.hazards.guard
├── 3.1.repair.test.defects.stone
├── 3.1.repair.test.defects.guard
├── 3.2.reflect.test.defects.stone
├── 3.2.reflect.test.defects.guard
├── 3.3.repair.cicd.defects.stone
├── 3.3.repair.cicd.defects.guard
├── 3.4.reflect.cicd.defects.stone
└── 3.4.reflect.cicd.defects.guard
```

**route artifacts** (dynamic, emitted by driver as it executes):
- `.v1.i1.md` files — artifacts produced at each stone

```
.route/v2026_03_13.declapract.upgrade/
├── 1.upgrade.invoke.v1.i1.md    — execution log, any infra defects
├── 2.detect.hazards.v1.i1.md    — identified changes that break
├── 3.1.repair.test.defects.v1.i1.md   — test fixes with root causes
├── 3.2.reflect.test.defects.v1.i1.md  — classifications and feedback
├── 3.3.repair.cicd.defects.v1.i1.md   — cicd fixes with root causes
└── 3.4.reflect.cicd.defects.v1.i1.md  — classifications and feedback
```

### timeline

```
t0: invoke `rhx init.declapract.upgrade`
    └─ route instantiated, bound to branch

t1: pass 1.upgrade.invoke
    └─ declapract upgrade executed
    └─ infra defects documented (if any)

t2: pass 2.detect.hazards
    └─ changes that break identified
    └─ guard validates completeness

t3: pass 3.1.repair.test.defects
    └─ all tests green
    └─ defects documented with root causes

    └─ defects classified

t5: pass 3.3.repair.cicd.defects
    └─ cicd green
    └─ defects documented with root causes

t6: pass 3.4.reflect.cicd.defects (human approval required)
    └─ defects classified
    └─ upgrade complete
```

## mental model

### how users describe this to a friend

> "it's like a guided checklist for upgrade of repos to latest practices, except the checklist also makes you document why it broke so we can fix the root cause and never hit that issue again."

### analogies & metaphors

- **flight checklist**: pilots don't improvise pre-flight checks. neither should mechanics improvise practice upgrades.
- **post-incident review**: every defect gets the same treatment as a production incident — root cause, fix, prevention.
- **kata**: a practiced sequence of moves that becomes second nature.

### terms

| we say | users say |
|--------|-----------|
| stone | step / checkpoint |
| guard | gate / validation |
| rigid route | guided workflow |
| reflect | retrospective / feedback |

## evaluation

### how well does it solve the goals?

| goal | solution quality |
|------|------------------|
| safe migration | ✅ guards prevent skipped steps |
| documented defects | ✅ stones require artifacts |
| feedback to infra | ✅ reflect stones classify defects |
| teachable process | ✅ route is self-documented |
| prevent recurrence | ✅ defects fed back to declapract repos |

### pros

- eliminates ad-hoc improvisation
- creates institutional memory of defects
- guards catch missed issues
- reflect stones drive infrastructure improvement
- mechanics learn the process by route traversal

### cons

- more ceremony than "just run the command"
- requires human approval at reflect gates
- route must be maintained as process evolves

### edgecases & pit of success

| edgecase | how we handle it |
|----------|------------------|
| upgrade succeeds with no defects | route still produces artifacts that document success |
| test failures present before upgrade | guard requires zero failures, mechanic must fix all |
| cicd flaky test | mechanic must diagnose and document, not skip |
| infra defect blocks upgrade | handoff document emitted for infrastructure team |

3. cicd is accessible via `git release --watch`
4. mechanics can request `rhx git.commit.uses allow` from human

## what is awkward?

### what feels off or forced?

- **two reflect stones** (test + cicd) — could they be one? but they happen at different times and need different human approvals
- **require zero test failures** even if they predate the upgrade — necessary for clean slate, but could frustrate mechanics on messy repos
- ~~**pnpm vs npm** — wish specifies pnpm, but many repos use npm~~ **[answered]** always use pnpm per wisher confirmation

### where does the design fight the user's mental model?

- users expect "run upgrade, fix what breaks" — the route adds structure that may feel like overhead until they see the feedback loop pay off
- "reflect" is a new concept — users may not immediately see why they need to classify defects

### tradeoffs that feel uncomfortable

| tradeoff | discomfort |
|----------|------------|
| more steps = safer but slower | mechanics may resent ceremony |
| human approval gates = quality but friction | slows autonomous operation |
| document all = learn but tedious | write time adds up |

## summary

the declapract.upgrade rigid route transforms a one-shot command into a system that learns. every upgrade becomes an opportunity to:
1. catch issues early via guards
2. document defects with root causes
3. classify defects as repo quirks vs practice bugs
4. feed improvements back to declapract infrastructure

the friction is intentional — it forces the discipline that prevents the same defect from the next repo.
