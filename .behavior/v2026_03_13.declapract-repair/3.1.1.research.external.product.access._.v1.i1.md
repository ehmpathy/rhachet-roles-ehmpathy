# research: external product access for init.declapract.upgrade

## summary

this research documents the remote repositories, APIs, and external systems required to implement the `init.declapract.upgrade` skill, along with their contracts, interfaces, and best practices.

---

## 1. npm registry access via pnpm

### what is accessed

the npm public registry at `registry.npmjs.org` for package installation and version resolution.

### contracts and interfaces

pnpm uses the npm registry API to fetch package metadata and tarballs. the primary commands are:
- `pnpm install` — installs dependencies per lockfile
- `pnpm add <pkg> --save-dev` — adds packages as devDependencies

### best practices

**[1] pnpm official documentation — pnpm install**
- source: https://pnpm.io/cli/install
- date: 2025 (current)
- credibility: [official]
> "In CI environments, pnpm doesn't generate a lockfile and fails to install if the lockfile is out of sync with the manifest or if no lockfile is present."

**[2] pnpm blog — pnpm in 2025**
- source: https://pnpm.io/blog/2025/12/29/pnpm-in-2025
- date: 2025-12-29
- credibility: [official]
> "In pnpm v10.0, pnpm moved to 'Security by Default.' The package manager no longer runs preinstall or postinstall operations by default, eliminating a massive class of supply chain attack vectors."

**[3] pnpm blog — security by default**
- source: https://pnpm.io/blog/2025/12/29/pnpm-in-2025
- date: 2025-12-29
- credibility: [official]
> "pnpm introduced minimumReleaseAge to block 'zero-day' releases (packages younger than 24 hours), giving the community time to flag malicious updates."

**[4] pnpm official documentation — pnpm add**
- source: https://pnpm.io/cli/add
- date: 2025 (current)
- credibility: [official]
> "pnpm allows you to install specified packages as devDependencies using the --save-dev flag."

### convergence signal

sources [1], [2], [3], [4] all from pnpm official documentation converge on: use `--frozen-lockfile` in CI, disable lifecycle operations by default for security.

## 2. npm registry security and supply chain protection

### what is accessed

npm package metadata and content from registry; potential exposure to supply chain attacks.

### best practices and anti-patterns

**[5] GitHub blog — securing npm supply chain**
- source: https://github.blog/security/supply-chain-security/our-plan-for-a-more-secure-npm-supply-chain/
- date: 2025
- credibility: [official]
> "Trusted publishing, a recommended security capability, was added to npm in July 2025 and removes the need to securely manage an API token in the build system."

**[6] CISA alert — npm supply chain compromise**
- source: https://www.cisa.gov/news-events/alerts/2025/09/23/widespread-supply-chain-compromise-impacting-npm-ecosystem
- date: 2025-09-23
- credibility: [official]
> "A supply chain attack on the npm registry, uncovered September 8, 2025, compromised 18 widely used packages through a phishing campaign targeting open source project maintainers."

**[7] Endor Labs — defending against npm attacks**
- source: https://www.endorlabs.com/learn/how-to-defend-against-npm-software-supply-chain-attacks
- date: 2025
- credibility: [practitioner]
> "Enforce integrity checks and pin versions in CI, and require lockfiles (package-lock.json, pnpm-lock.yaml) to be used in CI environments."

**[8] OWASP — npm security cheat sheet**
- source: https://cheatsheetseries.owasp.org/cheatsheets/NPM_Security_Cheat_Sheet.html
- date: 2025 (current)
- credibility: [official]
> "Any dependency or transitive dependency can execute arbitrary code at install time. A recommended safe-by-default approach is to disable all lifecycle operations globally."

### anti-patterns

sources [5], [6], [7], [8] warn against:
- running lifecycle operations without explicit allow-listing
- installing packages without lockfile integrity checks
- upgrading immediately to new versions without review period

**[10] declapract npm package documentation**
- source: https://www.npmjs.com/package/declapract
- date: 2025 (current)
- credibility: [official]
> "The apply command fixes all files which have failed to adhere to any of the project's declared practices and have an automatic fix available."

```bash

npx declapract apply
npx declapract apply  # run twice for idempotency
```

## 6. idempotent operations

### what is accessed

conceptual pattern for safe retry behavior across all remote operations.

### best practices

**[20] Splunk — idempotent design**
- source: https://www.splunk.com/en_us/blog/learn/idempotent-design.html
- date: 2025
- credibility: [practitioner]
> "Idempotence refers to an operation that can be applied multiple times without changing the outcome beyond the initial application."

**[21] HackerNoon — idempotence principle**
- source: https://hackernoon.com/the-idempotence-principle-in-software-architecture
- date: 2025
- credibility: [practitioner]
> "Use Unique Identifiers: Attach a unique ID (idempotency key) to each request to track and prevent duplicate processing."

**[22] Temporal — idempotency for durable systems**
- source: https://temporal.io/blog/idempotency-and-durable-execution
- date: 2025
- credibility: [practitioner]
> "Design for Idempotency from the Start: It's much easier to design for idempotency from the beginning than to add it later."

### convergence signal

## 8. npx security considerations

### what is accessed

npx runner for executing packages without global installation.

### anti-patterns and risks

**[26] Snyk — npm security best practices**
- source: https://snyk.io/articles/npm-security-best-practices-shai-hulud-attack/
- date: 2025
- credibility: [practitioner]
> "npx introduces potential security risks by allowing developers to execute code directly from the internet, opening the door for malicious actors to execute harmful code on your system."

**[27] Socket — npx glossary**
- source: https://socket.dev/glossary/npx-npm-package-runner
- date: 2025
- credibility: [practitioner]
> "When using npx, you're implicitly trusting that the package you're executing is safe, which is often based on the package's popularity or standing in the open-source community."

### mitigation

for `init.declapract.upgrade`, npx is used only for well-known, maintained packages (`declapract`) that are explicitly added as devDependencies first. this mitigates the "execute from internet" risk since the package is locked in pnpm-lock.yaml.
