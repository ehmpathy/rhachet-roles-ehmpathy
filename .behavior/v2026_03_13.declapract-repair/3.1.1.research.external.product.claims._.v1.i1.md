## 3. root cause documentation and post-mortems

### claims

**[FACT] post-mortems should capture root causes for prevention**

**[7] Google SRE Book — Postmortem Culture**
- source: https://sre.google/sre-book/postmortem-culture/
- date: 2025 (current)
- credibility: [official]
> "The goals of a postmortem are to understand all contributing root causes, document the incident for future reference and pattern discovery, and enact effective preventative actions."

**[FACT] blameless culture enables better learning**

**[8] Atlassian — Postmortems**
- source: https://www.atlassian.com/incident-management/handbook/postmortems
- date: 2025 (current)
- credibility: [official]
> "The goal of a postmortem is not to blame individuals, but to learn from mistakes and improve systems and processes. It's about fostering a culture of learning and continuous improvement."

**[OPIN] five whys may oversimplify systemic issues**

**[9] Pragmatic Engineer — Postmortem Best Practices**
- source: https://blog.pragmaticengineer.com/postmortem-best-practices/
- date: 2025
- credibility: [practitioner]
> "A root cause analysis or the Five Whys approach can be appealing because they simplify a post-incident effort, but they will not address the deeper systemic issues within your system."

**[FACT] post-mortems should be timely**

**[10] Datadog — Incident Postmortem Process**
- source: https://www.datadoghq.com/blog/incident-postmortem-process-best-practices/
- date: 2025
- credibility: [practitioner]
> "Best practice is to run the postmortem within 48 hours of incident resolution. The closer to the event, the fresher the memory and the more accurate the timeline."

## 8. backwards compatibility and upgrade testing

### claims

**[FACT] breaking changes have two categories**

**[23] Endor Labs — Breaking Changes**
- source: https://www.endorlabs.com/learn/breaking-changes-breaking-trust
- date: 2025
- credibility: [practitioner]
> "Breaking changes in software are modifications that disrupt compatibility with previous versions, and there are broadly two categories: syntactic changes and behavioral changes."

**[FACT] deprecation is preferable to deletion**

**[24] Zuplo — API Backwards Compatibility**
- source: https://zuplo.com/learning-center/api-versioning-backward-compatibility-best-practices
- date: 2025
- credibility: [practitioner]
> "Instead of deleting an endpoint, you should deprecate them, which means continuing to support them for a fixed period of time."

**[FACT] compatibility testing covers multiple aspects**

**[25] Qodex.ai — Backward Compatibility Testing**
- source: https://qodex.ai/blog/backward-compatibility-testing
- date: 2025
- credibility: [practitioner]
> "During backward compatibility testing, testers assess various aspects such as data migration, system configurations, functional behavior, user interfaces, performance, security measures, and API integrations."

**[SUMP] two-step migrations reduce risk**

**[26] Stack Overflow Blog — Backwards Compatibility**
- source: https://stackoverflow.blog/2020/05/13/ensuring-backwards-compatibility-in-distributed-systems/
- date: 2020
- credibility: [practitioner]
> "You can perform breaking changes in two steps: first, update a system so it works with both the old and new way of integrating, then allow time for external systems to switch."

## conflict detection

**sources [9] and [7] partially conflict on root cause analysis:**
- [7] (Google SRE) says postmortems should "understand all contributing root causes"
- [9] (Pragmatic Engineer) says "Five Whys approach... will not address deeper systemic issues"

**synthesis:** both are valid — five whys helps find immediate causes, but systemic analysis requires broader socio-technical approaches. the route should document both immediate fixes and systemic patterns.
