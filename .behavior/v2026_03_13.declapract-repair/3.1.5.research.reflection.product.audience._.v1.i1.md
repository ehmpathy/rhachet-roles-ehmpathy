# research: audience analysis for init.declapract.upgrade

## who is the audience?

### primary: mechanics who upgrade repos to latest practices

**role**: software engineer who uses the rhachet mechanic role
**context**: has a repo with declapract.use.yml, wants to upgrade to latest best practices
**constraints**: must preserve repo functionality, must not break tests or cicd, limited time for manual debug

### secondary: declapract maintainers

**role**: maintainers of declapract-typescript-ehmpathy and declapract core repos
**context**: receive feedback about practice bugs and adoption candidates from upgrade reflections
**constraints**: need clear root cause analysis to prioritize fixes

### tertiary: future mechanics on the same repo

**role**: any mechanic who runs declapract upgrade on this repo in the future
**context**: benefits from defects already documented and prevented
**constraints**: may not have context on prior upgrade issues

---

## why do they care?

### mechanics

| pain | how this relieves it |
|------|----------------------|
| upgrades break things unpredictably | rigid route ensures all hazards detected |
| defects found ad-hoc, forgotten | every defect documented with root cause |
| same issues hit repo after repo | reflect stones feed learnings back to infrastructure |
| "it broke" with no explanation | route requires "broke X because Y, fix is Z" |

### declapract maintainers

| pain | how this relieves it |
|------|----------------------|
| unaware of practice bugs until reported | reflect artifacts surface issues systematically |
| unclear which issues are common | defects classified as repo quirk vs practice bug |
| hard to prioritize fixes | root cause analysis with citations to exact practices |

### future mechanics

| pain | how this relieves it |
|------|----------------------|
| hit same issues as predecessors | infrastructure evolves, issues disappear |
| no institutional memory of upgrades | route artifacts document what broke and why |

---

## how much do they care?

| audience | priority | urgency | stakes |
|----------|----------|---------|--------|
| mechanics | high | immediate | blocked from ship if upgrade breaks repo |
| declapract maintainers | medium | deferred | accumulates technical debt if not addressed |
| future mechanics | low | future | benefits compound over time |

---

## what do they care about?

### mechanics judge success by

- **zero test failures** after upgrade completes
- **zero cicd failures** after push
- **clear documentation** of what changed and why
- **time to completion** — route should not add excessive overhead

### tradeoffs they would accept

- ✅ more upfront ceremony for safer migrations
- ✅ human approval gates for quality feedback
- ❌ would reject: route that takes longer than ad-hoc debug would have

### what would delight vs disappoint

| delight | disappoint |
|---------|------------|
| finish upgrade with every defect documented | hit same defect twice |
| reflection artifacts useful for future | artifacts become busywork |
| guards catch issues they would have missed | guards block on false positives |

---

## how will they experience it?

### mechanic journey

1. **invoke** — `rhx init.declapract.upgrade` — route instantiates
2. **pass 1.upgrade.invoke** — run declapract upgrade, see immediate failures
3. **pass 2.detect.hazards** — review diffs for backward incompat changes
4. **pass 3.1.repair.test.defects** — fix tests, document root causes
5. **pass 3.2.reflect.test.defects** — classify defects, request approval
6. **pass 3.3.repair.cicd.defects** — push, watch, fix, document
7. **pass 3.4.reflect.cicd.defects** — classify, request approval, done

### friction points

| point | friction | mitigation |
|-------|----------|------------|
| human approval gates | blocks autonomous operation | only 2 gates (reflect stones), not every step |
| documentation overhead | feels tedious | guards validate completeness, not style |
| prerequisite failures | blocks before start | clear error messages with install instructions |

### smooth experience looks like

- invoke once, route guides every step
- clear turtle vibes output at each step
- guards catch what they would have missed
- reflection artifacts become natural habit
- upgrade complete with confidence, not hope
