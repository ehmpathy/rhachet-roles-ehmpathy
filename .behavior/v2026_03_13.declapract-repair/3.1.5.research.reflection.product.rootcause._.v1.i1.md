# research: root cause analysis for init.declapract.upgrade

## what is the symptom?

### observable problem

the current `declapract.upgrade.sh` is a one-shot command that:
- runs the upgrade and hopes it works
- does not document defects found
- does not classify why defects arose
- does not feed learnings back to infrastructure
- uses npm instead of pnpm

### how it manifests

- same issues recur across repos
- mechanics improvise fixes ad-hoc
- no institutional memory of upgrades
- declapract maintainers unaware of practice bugs

### when it occurs

every time a repo needs to upgrade to latest practices

---

## 5 whys tree

### branch 1: defects recur across repos

1. **why do same defects recur?** → because mechanics fix ad-hoc without documentation
2. **why no documentation?** → because current skill does not require it
3. **why does current skill not require it?** → because it was built as one-shot command, not guided workflow
4. **why one-shot instead of workflow?** → because rigid routes did not exist when skill was created
5. **why?** → **[root cause]** infrastructure (rigid routes) is now available but skill not upgraded to use it

### branch 2: declapract maintainers unaware of practice bugs

1. **why unaware?** → because defects not reported systematically
2. **why no systematic report?** → because mechanics fix and move on
3. **why fix and move on?** → because no structured reflection step
4. **why no reflection step?** → because current skill ends after apply
5. **why?** → **[root cause]** no feedback loop built into upgrade process

### branch 3: upgrades break things unpredictably

1. **why unpredictable?** → because hazards not detected before repair
2. **why no detection?** → because no scan step for backward incompat changes
3. **why no scan step?** → because current skill goes straight to "run and hope"
4. **why run and hope?** → because skill lacks structured phases
5. **why?** → **[root cause]** skill not designed as multi-phase rigid route

---

## root cause summary

| layer | issue | is root cause? |
|-------|-------|----------------|
| 1 | defects not documented | no — symptom |
| 2 | no documentation requirement | no — proximate cause |
| 3 | skill is one-shot command | no — design decision |
| 4 | rigid routes unavailable when built | no — historical constraint |
| 5 | skill not upgraded to use rigid routes | **yes** |

secondary root causes:
- no feedback loop to declapract repos → **yes**
- no hazard detection phase → **yes**

---

## recommendation

### address root cause: transform to rigid route

the skill must be transformed from one-shot command to rigid thought route with:

1. **route instantiation** — `.route/v$isodate.declapract.upgrade/`
2. **route bind** — tie to current branch for route.drive
3. **multi-phase structure** — stones for invoke, detect, repair, reflect
4. **guard gates** — enforce documentation and human approval
5. **feedback loop** — reflect stones classify defects for infrastructure

this addresses all three root causes:
- skill upgraded to use rigid routes ✅
- feedback loop built into reflect stones ✅
- hazard detection built into 2.detect.hazards stone ✅

### symptom treatment not appropriate

there is no justification for symptom treatment (e.g., "just add a readme about document defects") because:
- the infrastructure (rigid routes) now exists
- the effort to transform is bounded
- the benefit compounds across all future upgrades
