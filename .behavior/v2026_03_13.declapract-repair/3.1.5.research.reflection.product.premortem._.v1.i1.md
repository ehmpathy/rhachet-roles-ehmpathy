# research: premortem for init.declapract.upgrade

## premortem: imagine we shipped and it failed

we shipped init.declapract.upgrade and it failed badly. why?

### failure scenario 1: overhead drove abandonment

mechanics found the rigid route too slow compared to ad-hoc debug. after a few upgrades, they started to skip steps, ignore guards, or not use the skill at all. the documentation became stale, the feedback loop broke.

### failure scenario 2: reflect gates created friction without value

human approval gates at reflect stones became rubber stamps. humans approved without read of the classifications. the feedback to declapract repos was low quality and ignored. the ceremony added time without benefit.

### failure scenario 3: route broke on edge cases

some repos had unusual structures (monorepos, workspace setups, custom cicd). the route assumed standard structure and failed in ways that blocked progress. mechanics had to abandon the route mid-way.

---

## what obvious signs did we ignore?

- mechanics already complain about ceremony in other tools
- human approval gates slow autonomous operation
- declapract repos are maintained by a small team with limited bandwidth
- not all repos have github actions cicd

---

## what assumptions could turn out wrong?

| assumption | what if wrong? | likelihood |
|------------|----------------|------------|
| mechanics have pnpm available | skill fails immediately; need install instructions | low — ehmpathy repos standardize on pnpm |
| mechanics have fnm available | stone fails at invoke; need install instructions | medium — some may use nvm instead |
| cicd is github actions | 3.3.repair.cicd.defects stone fails; need alternative paths | medium — some repos use other cicd |
| declapract apply is idempotent | double apply causes issues; need single apply | low — documented as idempotent |
| reflect artifacts are read by humans | artifacts become write-only busywork | medium — need to monitor adoption |
| guards catch real issues | guards produce false positives; mechanics lose trust | medium — need to tune guard criteria |

---

## what is the nightmare scenario?

**the skill becomes shelfware**: mechanics try it once, find it slower than ad-hoc debug, and never use it again. the route files accumulate in repos without traverse. no feedback reaches declapract repos. worse, we maintain the skill without benefit.

what would make us deeply regret this:
- mechanics actively avoid the skill
- route artifacts are empty or copy-paste garbage
- human approval becomes "click approve without read"
- the skill introduces bugs (e.g., breaks repos that worked before)

what would be hardest to recover from:
- loss of trust — once mechanics believe the skill is overhead, hard to change that perception
- stale documentation — if we stop maintain of the route, it becomes a hazard

---

## what mitigations to consider?

| risk | mitigation | cost of mitigation |
|------|------------|--------------------|
| overhead drives abandonment | ensure route is faster than ad-hoc debug for typical upgrades | design effort — measure time-to-completion |
| reflect gates become rubber stamps | make reflect artifacts actionable — link to declapract issues | integration work — may need gh cli automation |
| route breaks on edge cases | document scope assumptions; fail fast with clear errors | documentation + guard clauses |
| fnm not available | check fnm at 1.upgrade.invoke, fail with install instructions | already in design |
| cicd not github actions | document github actions assumption; allow manual cicd verification | documentation |
| guards produce false positives | self-review questions are specific and falsifiable | careful guard design |
| skill introduces bugs | comprehensive integration tests; plan mode before apply | test coverage |

### priority mitigations

1. **time-to-completion metric** — track how long upgrades take with route vs ad-hoc; ensure route is competitive
2. **actionable reflect artifacts** — consider automated issue creation on declapract repos (future enhancement)
3. **clear scope documentation** — state assumptions upfront (github actions, pnpm, fnm)
4. **comprehensive tests** — cover happy path, prerequisite failures, edge cases
