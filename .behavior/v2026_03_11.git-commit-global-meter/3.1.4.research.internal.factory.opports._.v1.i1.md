# research: factory opportunities

## the constraint

**THE bottleneck**: manual verification of stdout format.

when stdout format changes (e.g., turtle phrases, tree structure), I must:
1. run the skill manually
2. eyeball the output
3. verify it matches the vision's expected format
4. update snapshots

this is slow and error-prone. a mismatch between vision and implementation is easy to miss.

---

## flow and queues

| queue | wait time | controllable? |
|-------|-----------|---------------|
| `npm run test:integration` | ~10s | yes |
| snapshot review | manual | yes |
| CI pipeline | ~2min | no |

**bottleneck**: CI is the external queue, but it's acceptable for this scope.

---

## waste

| waste type | example | impact |
|------------|---------|--------|
| repeated setup | temp dir creation with git init | low (genTempDir handles it) |
| partial work | none — tests are atomic | none |
| manual steps | stdout format verification | medium |

**main waste**: manual comparison of vision stdout examples against actual output.

---

## small batches

**can verify incrementally**: yes
- each command (`block`, `allow`, `get --global`) can be tested independently
- tests are isolated and fast (~100ms each)
- smallest verifiable unit: single command invocation

---

## autonomy

**full autonomy**: yes
- no external teams needed
- no approvals required for local filesystem operations
- all dependencies are local (bash, jq, filesystem)

---

## density

| phase | time | % of total |
|-------|------|------------|
| test setup | 50ms | 5% |
| test execution | 50ms | 5% |
| assertion | 10ms | 1% |
| jest overhead | 890ms | 89% |

**insight**: jest startup dominates. for this scope, acceptable. for future, could consider watch mode to keep jest active.

---

## systems view

### opportunity: genTempDir with global storage

if `genTempDir` supported `globalStorage: true` option, it would:
- create isolated `$HOME` automatically
- set up `~/.rhachet/storage/` structure
- clean up after test

**impact**: helps all future global storage tests, not just this behavior.

**effort**: low — extend extant `genTempDir` function.

---

## recommendation

**defer factory improvement** — the test infrastructure is sufficient for this behavior.

if global storage becomes a repeated pattern, then extract `genTempDir({ globalStorage: true })` as a factory improvement.
