# self-review: has-questioned-requirements

## requirements questioned

### 1. storage path: `repo=$repo/role=$role/` vs flat global

**from wish**: `ROLE_STORAGE_DIR=$HOME/.rhachet/storage/repo=$repo/role=$role/.meter`

**question**: the wish mentions per-repo namespaced paths, but for a *global* deny, do we need this structure?

- if truly global (one deny for all repos), we could use `~/.rhachet/storage/.meter/` (flat)
- if per-repo-role global storage, we'd use `repo=$repo/role=$role/.meter/`

**my vision assumed**: flat `~/.rhachet/storage/.meter/` for global deny

**what if we didn't?**: if we used namespaced paths for global, we'd need to specify which repo/role the global applies to — but "global" means all repos, so namespace is contradictory

**conclusion**: flat path makes sense for global deny. the `repo=$repo/role=$role/` structure might be for future per-repo global state (not this feature). **should validate with wisher** — question #3 in vision.

---

### 2. toggle semantics for `del --global`

**from wish**: "`git.commit.uses del --global` would set the meter to 0 globally"

**question**: the wish says `del --global` *sets* a deny. but how do we lift it?

**my vision assumed**: `del --global` toggles (first call sets deny, second call lifts)

**alternatives**:
- `del --global` always sets deny; lift via manual file delete
- `del --global` always sets deny; lift via new command like `lift --global`
- toggle behavior (my assumption)

**what if toggle is wrong?**: users might accidentally lift a deny they wanted to keep. but explicit lift commands add surface area.

**conclusion**: toggle is simplest UX, but **should validate with wisher** — question #1 in vision.

---

### 3. "no global allows" interpretation

**from wish**: "lets only support global denies. no global allows -> each repo will still individually need to allow"

**question**: does "no global allows" mean:
- A) can't do `set --quant 5 --global` (grant quota globally)
- B) also can't lift a global deny?

**my interpretation**: A only. "lift global deny" is not a "global allow" — it just removes the override and lets local rules apply.

**what if interpretation B?**: you could never lift a global deny except by manual file delete. this seems unintended.

**conclusion**: interpretation A is correct. lift is allowed; global allow (grant quota) is forbidden. **validated.**

---

### 4. scope: is this feature necessary?

**question**: what if we didn't do this at all?

**alternative**: humans remember which repos have quota and visit each one.

**evidence for need**:
- humans work across many repos
- step-away scenarios are common
- "fire and forget" quota grants without recall is real
- no current way to pause all

**conclusion**: the need is real. **validated.**

---

### 5. scope: is it too large?

**question**: could we achieve the goal in a simpler way?

**simplest version**:
- `del --global` creates `~/.rhachet/.deny` (empty file)
- `git.commit.set` checks for `~/.rhachet/.deny` before local check
- if exists, blocked
- to lift, delete the file

**my vision added**:
- JSON file with `{ "denied": true }`
- `get` shows global + local + effective
- toggle semantics
- tree output style

**could simplify**: the empty file approach is simpler. JSON adds structure but no current benefit.

**conclusion**: vision might be slightly over-engineered. **could simplify to empty file marker.** but JSON allows future extension (e.g., deny reason, timestamp). trade-off is acceptable.

---

### 6. extension: `get` shows both local and global

**from wish**: not mentioned

**my vision added**: `get` shows global, local, and effective status

**question**: is this scope creep?

**argument for**: users need to understand why they're blocked. if global deny is active but hidden, they'll be confused.

**argument against**: adds complexity to `get` output

**conclusion**: this is a reasonable extension, not scope creep. **should note as enhancement** in vision (already in questions #2).
