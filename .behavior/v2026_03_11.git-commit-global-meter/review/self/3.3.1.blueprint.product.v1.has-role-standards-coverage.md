>
> `.how`: use early returns or throws for guard clauses; avoid if (...) else or deep nested blocks; include context objects in thrown errors for debug
>

**requirements extracted:**

3. clear error message

**git.commit.set.sh codepath (lines 63-76):**
```

│  ├─ read global blocker file
│  │  ├─ if file absent → not blocked, continue
│  │  ├─ if file is invalid json → BLOCKED (fail safe)
│  │  │  ├─ print_turtle_header "bummer dude..."
```

## gap found and fixed

### the gap

fail-fast rule (rule.require.fail-fast.md) says:
> include context objects in thrown errors for debug

the corrupt file case (lines 66-69 in original blueprint) had error message but lacked actionable instruction:
```
├─ if file is invalid json → BLOCKED (fail safe)
│  ├─ print_turtle_header "bummer dude..."
│  ├─ print_tree_error "global blocker file corrupt"
│  └─ exit 2                                          ← no instruction!
```

the blocked case (lines 70-74) had instruction:
```
└─ if { blocked: true }
   ├─ print_turtle_header "bummer dude..."
   ├─ print_tree_error "commits blocked globally"
   ├─ print_instruction "ask your human to lift:"     ← has instruction
   └─ exit 2
```

### the fix

added `print_instruction` for corrupt file case in blueprint:
```
├─ if file is invalid json → BLOCKED (fail safe)
│  ├─ print_turtle_header "bummer dude..."
│  ├─ print_tree_error "global blocker file corrupt"
│  ├─ print_instruction "check the file:" "  $ cat $GLOBAL_METER_FILE"
│  └─ exit 2
```

**rationale**: fail-fast requires actionable context. the blocked case tells human how to lift; the corrupt case should tell human where to look.

### fix applied

✅ **fix applied** to `.behavior/v2026_03_11.git-commit-global-meter/3.3.1.blueprint.product.v1.i1.md` lines 66-70
