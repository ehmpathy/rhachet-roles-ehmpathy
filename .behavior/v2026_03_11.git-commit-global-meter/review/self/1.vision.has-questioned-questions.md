## triage of open questions

### question 1: how to lift global deny?

**can I answer via logic?** yes, with code analysis.

**code analysis** (from git.commit.uses.sh):
- local `del` does NOT delete the state file — it writes `{uses: 0, push: "block"}`
- local `del` falls through to `set` logic (line 112: `;&`)
- so "del" means "set to zero", not "remove file"

**constraint from wish**: "no global allows — each repo must individually need to allow"
- this forbids `set --global --quant N`
- so we cannot use `set` to lift; we need a different mechanism

**the options**:
- option A (manual delete): user runs `rm ~/.rhachet/storage/.meter/...`
  - pro: no code needed
  - con: user must remember path, not discoverable
  - con: `get --global` must show the path for users to know what to delete
- option B (`lift --global`): new subcommand that deletes the file
  - pro: discoverable, symmetric with `del --global`
  - pro: follows git patterns (`stash`/`stash pop`, `add`/`reset`)
  - con: adds 4th subcommand
- option C (toggle): `del --global` again lifts
  - con: conflicts with `del` semantics (del writes, not toggles)
  - con: unexpected behavior

**decision via elimination**:
- option C rejected: toggle semantics are confusing
- option A acceptable but poor UX
- option B is cleanest given the constraints

**my answer**: `allow --global` — new `block`/`allow` commands added as symmetric pair:
- `block` (local) = same as `del`
- `block --global` = set global deny
- `allow` (local) = grant quota
- `allow --global` = lift global deny

the wisher confirmed this approach. `block`/`allow` are natural opposites.

**verdict**: **self-answered via wisher feedback**. use `allow --global` to lift.

### question 2: should `get` show both local and global?

**can I answer via logic?** yes.

- if user has local quota (3 uses) but global deny is active, they'd be confused: "I have uses left, why is commit blocked?"
- `get` MUST show why effective state differs from local state
- transparency improves UX and reduces confusion

**my answer**: yes, `get` should show both local and global, plus effective state.

**verdict**: **self-answered**. update vision to mark as decided.

### question 3: storage path format?

**can I answer via wish?** yes, on second review.

- wisher explicitly stated: `ROLE_STORAGE_DIR=$HOME/.rhachet/storage/repo=$repo/role=$role/`
- this is not a suggestion — it's the declared pattern
- for global, the natural sentinel is `repo=_global`
- using flat path would deviate from wisher's expressed structure

**code search confirms**: no other global storage patterns exist in codebase. this is new. follow the wisher's pattern.

**my answer**: namespaced path `~/.rhachet/storage/repo=_global/role=mechanic/.meter/git.commit.uses.jsonc`
- follows wisher's `repo=$repo/role=$role/` pattern exactly
- `repo=_global` is the natural sentinel for global state
- enables future expansion (e.g., per-user global per-role settings)

**verdict**: **self-answered**. use namespaced path per wisher's declared pattern.

### question 4: can human lift global from anywhere?

**can I answer via logic?** yes.

- global deny affects all repos — user shouldn't need to `cd` to a repo first
- `git config --global` works from anywhere (even outside git repos)
- a global command that requires git context is awkward UX

**my answer**: yes, `del --global` and `lift --global` should work from any directory.

**verdict**: **self-answered**. update vision to mark as decided.

### question 5: is global deny per-role or all-roles?

**can I answer via logic?** yes, on deeper review.

**the skill itself is role-specific**:
- skill name: `git.commit.uses`
- purpose: manage mechanic's commit quota
- no other role uses this skill

**the path pattern confirms per-role**:
- wisher's pattern: `repo=$repo/role=$role/`
- global path: `repo=_global/role=mechanic/`
- each role has its own namespace

**if all-roles were intended**:
- the path would be `repo=_global/.meter/` (no role)
- but wisher explicitly included `role=$role`

**my answer**: per-role (mechanic-only)
- the skill is inherently mechanic-specific
- the wisher's path pattern includes `role=$role`
- other roles (if they need global meters) would use their own skills

**verdict**: **self-answered**. per-role design follows from skill scope and wisher's path pattern.
