# self-review: has-questioned-assumptions

## issues found from code review

I read the extant `git.commit.uses.sh` and found two issues with the blueprint:

### issue 1: git repo check blocks global commands

**the code** (lines 23-27):
```bash
if ! git rev-parse --git-dir > /dev/null 2>&1; then
  echo "error: not in a git repository"
  exit 2
fi
```

**the problem**: this runs unconditionally at skill start. global commands (`block --global`, `allow --global`, `get --global`) should work from any directory, but this check will exit immediately if not in a git repo.

**the fix**: parse `--global` flag BEFORE the git repo check. if `--global` is present, skip the git repo requirement.

**blueprint update needed**: add a note that `--global` flag must be parsed early to bypass git repo check.

---

### issue 2: `@any` not supported

**the code** (lines 141-144):
```bash
if ! [[ "$QUANT" =~ ^[0-9]+$ ]]; then
  echo "error: --quant must be a non-negative integer"
  exit 2
fi
```

**the problem**: the blueprint says `allow` (local) should use `--quant @any`, but the code only accepts numeric values.

**options**:
1. add `@any` handling (special string that bypasses decrement)
2. use a large number like `999999` (simpler, no special logic)
3. use `-1` as sentinel (requires changing comparison logic)

**decision**: use `999999` for simplicity. the extant code already handles numeric quant. no new logic needed.

**blueprint update needed**: change `@any` to `999999` in the `allow` shorthand definition.
