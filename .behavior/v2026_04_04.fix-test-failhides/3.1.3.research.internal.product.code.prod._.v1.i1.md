## pattern.1 = rule.forbid.failhide

**path:** `src/domain.roles/mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.forbid.failhide.md.pt1.md`

**citation [1]:**
```
failhide = any scenario where you hide real errors

the most common patterns is

try/catch

---

the only cases where a trycatch is allowed is if the catch has an allowlist of the errors it catches and CAREFULLY handles those, while it throws the rest up

otherwise, its a failhide hazard, where real errors will be silently hidden, which leads to defects or hours of pointless debug

never failhide

always failfast

---

this is a mega blocker
```

**citation [2]:** from `rule.forbid.failhide.md.pt2.md`:
```
never try+catch just to hide the error

if an error is thrown

it should not be caught and hidden

it should be rethrown

---

if we want to use it for control flow logic, then we epxpect
- all errors by default to be thrown
- only errors that match very restricted conditions to be handled
```

**action:** [REUSE] — this is the core failhide concept. will create symmetric version for code.test.

## pattern.2 = rule.require.fail-fast

**path:** `src/domain.roles/mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.require.fail-fast.md`

**citation [3]:**
```
.tactic = flow:fail-fast

.what = enforce fail-fast logic via early exits and HelpfulError subclasses to reject invalid state or input immediately

.scope:
  - required in all stitched logic, business procedures, and service flows
  - applies to guard checks, validations, and branch paths

.why:
  - improves readability via early collapse of failure paths
  - eliminates nested branches and lets core logic shine
  - increases safety via clear documentation and halt on invalid input
  - makes bugs easier to debug with rich context via `helpful-errors`

.how:
  - use **early returns or throws** for all guard clauses
    - never use `if (...) else` or deep nested blocks to control flow
  - prefer `UnexpectedCodePathError.throw(...)` for internal invariant violations
    - these signal logic errors that "should never happen"
  - prefer `BadRequestError.throw(...)` to reject invalid input or user requests
    - these signal invalid calls, not system failures
```

**citation [4]:** from `rule.require.fail-fast.[seed].md`:
```
fail fast saves lives

when you fail fast, you detect fast, and you fix fast

this contributes to a pit of success
- enables tests to alert when unexpected behaviors occur, detected at build time
- enables logs to alert when unexpected behaviors occur, detected at monitor time
=>
- this allows us to more safely develop => as any surprises will be flagged asap, loud and proud
- this allows us to more rapidly maintain => as any changes live in production will be flagged asap, loud and proud

---

specifically, you should choose from

import { UnexpectedCodePathError } from 'helpful-errors';
throw new UnexpectedCodePathError(message, metadata)

or

import { BadRequestError } from 'helpful-errors';
throw new BadRequestError(message, metadata)
```

**citation [5]:** from `rule.require.fail-fast.[demo].shell.md`:
```
can failfast via UnexpectedCodePathError && BadRequestError in shell too

if [ ${#INFO_FILES[@]} -eq 0 ]; then

  exit 1
elif [ ${#INFO_FILES[@]} -gt 1 ]; then
  echo "✗ UnexpectedCodePathError: multiple tunnel info files found: ${INFO_FILES[@]}"
  exit 1
fi
```

**action:** [EXTEND] — rename `fail-fast` to `failfast` for consistency with `failhide`. create symmetric version for code.test.

## pattern.3 = rule.require.exit-code-semantics

**path:** `src/domain.roles/mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.require.exit-code-semantics.md`

**citation [6]:**
```
skills use semantic exit codes:

| code | semantics | when |
|------|-----------|------|
| 0 | success | operation completed as expected |
| 1 | malfunction | external error (gh failed, network, unexpected state) |
| 2 | constraint | user must fix (needs rebase, no PR, bad input) |

## .why

- exit 1 vs 2 tells caller whether to retry or fix
- constraint errors (2) mean user action required before retry
- malfunction errors (1) may be transient, retry might help
```

**citation [7]:**
```
test-fns provides `ConstraintError` and `MalfunctionError` with exit codes built-in:

import { ConstraintError, MalfunctionError } from 'test-fns';

// ConstraintError has error.code.exit = 2
throw new ConstraintError('needs rebase');

// MalfunctionError has error.code.exit = 1
throw new MalfunctionError('gh failed');
```

**action:** [EXTEND] — this rule already contains the failloud error class semantics. extract into dedicated `rule.require.failloud.md` for clarity.

## pattern.4 = rule.prefer.helpful-error-wrap

**path:** `src/domain.roles/mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.prefer.helpful-error-wrap.md`

**citation [8]:**
```
in these cases, its best to use HelpfulError.wrap instead
- it automatically instantiates a HelpfulError (or more specific variant if specified) for maximum observability

import { HelpfulError } from 'helpful-errors';

await HelpfulError.wrap(
  async () => await doTask({ userUuid }),
  {
    message: 'doTask.error',
    metadata: {
      userUuid,
    }
  }
)()
```

**action:** [REUSE] — this supports failloud patterns by wrap errors with context. no changes needed.
