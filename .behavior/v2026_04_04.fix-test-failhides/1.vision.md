# vision: eliminate test failhides

## the outcome world

### before

tests pass. ci is green. confidence is high.

but then production breaks. the api key wasn't configured. the test was never actually running the real behavior — it was silently taking a shortcut path that made the test pass without verifying anything.

```ts
// the hidden trap
if (result.exitCode === 2 && !hasApiKey) {
  expect(result.stdout).toContain('API key required');
}
expect([0, 1, 2]).toContain(result.exitCode);  // "passes" when key is missing
```

the mechanic who wrote this test thought they were being careful. handling the no-key case gracefully. but they created a failhide — a silent path where the test passes without testing anything.

hours of debug later: "wait, the api key was never set in ci?"

### after

tests that can't run their intended behavior fail loudly.

```ts
// the pit of success
if (!hasApiKey) {
  throw new ConstraintError('API key required for this test', {
    hint: 'run: source .agent/repo=.this/role=any/skills/use.apikeys.sh',
  });
}
const result = await runOperation();
expect(result.exitCode).toBe(0);
expect(result.stdout).toContain('expected output');
```

the test either runs and verifies behavior, or it fails fast with a clear message about why it can't run.

### the "aha" moment

> "oh, this test was never actually running in ci. we thought we had coverage, but we had a failhide."

the moment you realize a "passing" test was hiding the fact that it wasn't testing anything — that's when the value clicks. failhides in tests are worse than no tests at all, because they create false confidence.

## user experience

### usecases

1. **mechanic writes a new integration test**
   - test requires api key / database / external service
   - mechanic adds conditional logic: "if resource unavailable, skip"
   - rule catches this as a failhide pattern
   - mechanic refactors to either: skip the test explicitly, or fail fast if the resource is required

2. **mechanic reviews test changes**
   - pr has test modifications
   - behavior guard runs failhide review automatically
   - catches patterns like `expect([0, 1, 2]).toContain(exitCode)`
   - reviewer sees the blocker before merge

3. **ci discovers misconfiguration**
   - test that requires api key runs in ci without key
   - instead of silently "passing" via fallback path
   - test fails loud: `ConstraintError: API key required`
   - ci failure surfaces the misconfiguration immediately

## mental model

### how users describe it

> "tests either pass, fail, or skip. there's no fourth state where they 'pass without testing anything'."

> "if a test can't run its intended behavior, it should fail loud, not sneak through."

> "failhide is when a test passes for the wrong reason."

### analogies

**the smoke detector with dead batteries**
- it's installed, it looks like it works
- but it won't alert you when there's a fire
- a failhide test is a smoke detector with dead batteries

**the locked door that's actually unlocked**
- you think the door is secure
- but the lock is broken
- a failhide test gives false security

### terminology

| term | definition | scope |
|------|------------|-------|
| failhide | hide errors, continue as if ok | prod + test (forbid) |
| failfast | detect bad state early, throw immediately | prod + test (require) |
| failloud | use proper error class with full details, super clear about the defect | prod + test (require) |

---

**this is the most important rule you'll ever read.**

**the motto:** if it can't fulfill the contract, failfast.

**failhides are broken promises** — code that claims to do something but silently doesn't. they break trust. all other rules are worthless if your code lies about what it does.

---

failfast + failloud together: throw the right error class early, with full context about what went wrong.

three concepts, each required in both prod and test (6 rules total):


## evaluation

### how well does it solve the goals?

| goal | solution | confidence |
|------|----------|------------|
| eliminate test failhides | explicit rule + guard | high |
| symmetric terminology | failhide (forbid) + failfast + failloud (require), all for prod + test | high |
| automated detection | behavior guard rules | high |

### pros

- catches silent test failures before merge
- symmetric with extant prod rules
- automated via guards
- clear terminology

### cons

- some tests legitimately need conditional paths (but should use explicit skip, not silent pass)
- may flag false positives on intentionally flexible assertions (need good examples)

### edgecases & pit of success

| edgecase | guidance |
|----------|----------|
| test runs in multiple environments | use explicit skip (`it.skip`) when env lacks requirements |
| flaky external service | mock at boundary or skip with clear reason |
| acceptance test with optional features | separate test files per feature, skip file if feature disabled |

## open questions & assumptions

### assumptions

1. tests should either: run and verify, or fail/skip explicitly
2. conditional assertions that "pass anyway" are always bugs — **refined**: both branches must verify, not just one
3. `expect([0, 1, 2]).toContain(x)` patterns are suspicious when used to hide errors — legitimate when all exit codes are valid outcomes with verification

### pattern completeness note

the wish examples are not exhaustive. other failhide patterns may include:
- `expect.any(Object)` used to avoid verification
- `try { ... } catch { /* pass anyway */ }` in tests
- empty test bodies: `it('does task', () => {})`
- `beforeEach` that silently swallows setup errors

these should be enumerated in the criteria phase.

### questions triaged

| question | status | resolution |
|----------|--------|------------|
| separate vs unified rules? | [answered] | separate rules — aligns with `code.prod/` vs `code.test/` structure |
| guard scope (all tests vs subset)? | [answered] | all test files — failhides can occur in any test type |

| absent resources: fail or skip? | [answered] | fail fast always — absent resource = unacceptable |

## what is awkward?

### terminology friction

- `failfast` = detect bad state, exit immediately (structural)
- `failloud` = when fail, use clear error class and message (observability)
- both are complementary and required everywhere
- error classes: ConstraintError, MalfunctionError, UnexpectedCodePathError, BadRequestError

### conditional skip vs conditional pass

**the test:** does every code path either verify behavior or explicitly skip/fail?

| pattern | verdict | why |
|---------|---------|-----|
| `if (cond) { expect(a) } else { expect(b) }` | legitimate | both branches verify |
| `if (cond) { expect(a) }` | failhide | one branch does no verification |
| `if (!cond) { return }; expect(...)` | legitimate | early return is explicit (though `it.skip` is better) |
| `if (!cond) { expect(true).toBe(true) }` | failhide | verification is fake |
| `it.skip('requires api key', ...)` | legitimate | explicit skip |

**key insight:** silent pass-through is the failhide. every path must either verify or announce that it won't.

### guard scope

- current guard checks all prod code for failhides
- do we check all test code, or only certain patterns?
- checking all test code may be noisy; checking only `*.integration.test.ts` may miss unit test failhides
