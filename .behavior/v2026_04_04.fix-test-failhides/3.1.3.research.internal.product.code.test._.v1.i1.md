## pattern.1 = test-fns error classes

**path:** `src/domain.roles/mechanic/briefs/practices/code.test/consistent.contracts/ref.package.test-fns.[ref].md`

**citation [1]:**
```ts
given.runIf(onLocalMachine)('some test that should only run locally', () => {
  then.skipIf(onProduction)('some test that should not run against production', () => {
    expect(onProduction).toBeFalse()
  })
})
```

**note:** test-fns provides `.runIf()` and `.skipIf()` for explicit conditional tests — these are the legitimate alternatives to failhide patterns.

**citation [2]:** from prod exit-code-semantics (cross-reference):
```ts
import { ConstraintError, MalfunctionError } from 'test-fns';

// ConstraintError has error.code.exit = 2
throw new ConstraintError('needs rebase');

// MalfunctionError has error.code.exit = 1
throw new MalfunctionError('gh failed');
```

**action:** [REUSE] — test-fns already has the error classes needed for failloud. tests should use `throw new ConstraintError(...)` when a required resource is absent.

## pattern.4 = explicit skip vs silent skip

**path:** `src/domain.roles/mechanic/briefs/practices/code.test/consistent.contracts/ref.package.test-fns.[ref].md`

**citation [5]:**
```ts
given.runIf(onLocalMachine)('some test that should only run locally', () => {
  // ...
})
```

**note:** test-fns provides `.runIf()` and `.skipIf()` which make the skip explicit and visible in test output. this is the pit of success for conditional tests.

**failhide contrast:**
```ts
// 👎 failhide — silent pass when condition not met
if (!hasApiKey) {
  expect(true).toBe(true);
  return;
}
```

vs

```ts
// 👍 failfast — explicit failure when required resource absent
if (!hasApiKey) {
  throw new ConstraintError('API key required for this test', {
    hint: 'run: source .agent/repo=.this/role=any/skills/use.apikeys.sh',
  });
}
```

**action:** [NEW] — create failhide rule that distinguishes:
- legitimate: `given.runIf(condition)`, `then.skipIf(condition)`, `it.skip()`
- failhide: `if (!condition) { return }`, `if (!condition) { expect(true).toBe(true) }`

## failhide patterns identified in wish

| pattern | why it's a failhide |
|---------|---------------------|
| `if (!condition) { expect(true).toBe(true) }` | fake verification |
| `if (!hasResource) { return }` | silent skip |
| `expect([0, 1, 2]).toContain(exitCode)` | accepts errors as valid |
| `expect.any(Object)` | avoids actual verification |
| `it('does x', () => {})` | empty body, no verification |

## conclusion

the test codepath has strong patterns for structure (given/when/then) and classification (unit vs integration) but no explicit failhide/failfast/failloud rules. main gaps:

1. no `pitofsuccess.errors/` directory in code.test/
2. no explicit failhide rule for tests
3. no explicit failfast rule for tests (absent resource should throw, not skip)
4. no explicit failloud rule for tests

all changes are new additions — no replacements needed.
