brief.compress.sh

                    ├── looks up XAI_API_KEY from keyrack ← POTENTIAL: keyrack not unlocked

### flake candidates

| location | node | why it could flake | evidence |
|----------|------|-------------------|----------|
| prod | genContextBrain() | keyrack credentials not unlocked in CI | BrainChoiceNotFoundError thrown when XAI_API_KEY absent |
| prod | xai API call | rate limit or network timeout | 120s timeout, CI may hit rate limits |
| test | process.env inheritance | env vars not passed to spawnSync | spawnSync uses process.env (correct) |

### ground truth notes

- shared state found: **no** — each test uses genTempDir with unique slug
- time dependencies found: **yes** — 120s timeout, API call latency
- external i/o found: **yes** — real API call to xai/grok, requires XAI_API_KEY
- parallelism found: **no** — test runs serially within describe block
- order dependencies found: **no** — no state shared between tests

**root cause hypothesis:** keyrack credentials not unlocked in CI environment. the test makes a real API call to xai/grok which requires XAI_API_KEY. if keyrack.unlock is not run before integration tests, the credential lookup fails with BrainChoiceNotFoundError, which exits with code 2.

## flake 2: git.release [row-25] watch with transitions

### test codepath treestruct

```
git.release.p3.scenes.on_main.into_prod.integration.test.ts
├── describe: git.release.p3.scenes.on_main.into_prod
│   └── scene.5: on main, into prod
│       └── given: [row-25] release PR: merged, tags: inflight
│           └── when: [watch] --watch with transitions
│               └── then: exit 0: watch tag workflows
│                   ├── setupScene({ scene, slug })
│                   │   ├── genTempGitRepo({ branch: 'main' })
│                   │   ├── genMockBinDir() → .fakebin/
│                   │   ├── genStateDir() → .mockstate/
│                   │   └── writeMockGh({ releasePr: 'merged', tagWorkflows: 'inflight', transitions: true })
│                   │       └── writes watch_sequence.json with state progression
│                   ├── runSkill(['--watch'], { tempDir, fakeBinDir })
│                   │   └── spawnSync('bash', [skillPath, '--watch'])
│                   │       └── env: PATH=.fakebin:$PATH
│                   └── expect(result.status).toEqual(0) ← FAILS: received 1
```

### prod codepath treestruct

```
git.release.sh --watch
├── parse args: MODE=watch
├── detect branch: main
├── infer goal: main → prod
└── watch loop
    ├── call: gh pr view (mocked)
    │   └── mockGh.ts
    │       ├── get_pr_view_count() ← reads .mockstate/gh_pr_view_count
    │       ├── increment_pr_view_count() ← writes counter + 1
    │       └── reads watch_sequence.json[counter] ← returns state at index
    ├── call: gh run list (mocked)
    │   └── same counter mechanism
    ├── check terminal condition
    │   └── if tagWorkflows == 'passed' → exit 0
    │   └── if tagWorkflows == 'inflight' && timeout → exit 1 ← THIS IS THE FLAKE
    └── sleep + retry
```

### flake candidates

| location | node | why it could flake | evidence |
|----------|------|-------------------|----------|
| test | increment_pr_view_count() | file I/O race condition | bash read-increment-write not atomic |
| test | watch_sequence.json read | counter at wrong index | if counter incremented twice per iteration |
| prod | watch loop | poll interval vs state file flush | state file may not be flushed before next poll |
| prod | timeout logic | real timeout vs mock time | test uses real sleep intervals |

### ground truth notes

- shared state found: **yes** — counter file in .mockstate/, watch_sequence.json
- time dependencies found: **yes** — poll interval duration, bash sleep
- external i/o found: **yes** — file read/write for counter and sequence
- parallelism found: **no** — each test has unique tempDir with unique state dir
- order dependencies found: **possible** — counter must match expected sequence index

**root cause hypothesis:** the mock gh increments the counter on BOTH `gh pr view` AND `gh run list` calls. if the watch loop calls both in one iteration, the counter advances by 2 instead of 1, causing the sequence to reach terminal state early or late. alternatively, file I/O latency under CI load causes the counter read to return stale value.
