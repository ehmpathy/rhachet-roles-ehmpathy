## flake 1: brief.compress [case3] [t1]

### root cause hypotheses

| rank | hypothesis | probability | evidence | verification |
|------|------------|-------------|----------|--------------|
| 1 | keyrack not unlocked | high | BrainChoiceNotFoundError when XAI_API_KEY absent; test calls real xai/grok API | check CI workflow for keyrack.unlock before integration tests |
| 2 | API rate limit | low | 120s timeout; multiple tests call xai/grok in parallel | add rate limit error detection; check xai API response codes |
| 3 | network timeout | low | CI runners may have variable network latency | add timeout log; increase timeout to 180s |

### observability additions needed

- none needed — error is already clear (exit code 2 = constraint error)

### reproduction status

- [x] reproduced locally — root cause confirmed
  - ran `rhx keyrack status --owner ehmpath` → XAI_API_KEY not unlocked
  - ran integration test → exit code 2
  - ran `rhx keyrack unlock --owner ehmpath --env test` → XAI_API_KEY unlocked
  - ran integration test → exit code 0

### predicted root cause

**keyrack credentials not unlocked in CI environment**

the CI workflow runs integration tests that call `brief.compress --via bhrain/sitrep`. this invokes `genContextBrain({ choice: { atom: 'xai/grok/code-fast-1' } })` which looks up XAI_API_KEY from keyrack. if keyrack is not unlocked before integration tests, the key lookup fails with BrainChoiceNotFoundError, which yields exit code 2.

## flake 2: git.release [row-25] watch with transitions

### root cause hypotheses

| rank | hypothesis | probability | evidence | verification |
|------|------------|-------------|----------|--------------|
| 1 | counter double-increment | high | mockGh increments counter on BOTH `gh pr view` AND `gh run list`; watch loop calls both per iteration | trace counter values; add log at each increment site |
| 2 | file I/O race | medium | bash read-increment-write not atomic; CI load causes flush delays | add fsync after counter write; add read-after-write verification |
| 3 | sequence index mismatch | low | counter may skip indices or read stale value | log sequence length vs counter at each read |

### observability additions needed

- [ ] add counter value log in mockGh.ts at each increment point
- [ ] log watch_sequence.json length vs counter at each read

### reproduction status

- [ ] not reproduced locally — flake occurred only once in CI
- [ ] awaited observability — need to trace counter increments

### predicted root cause

**counter increments per gh call, not per poll cycle**

the mock gh implementation increments a counter every time ANY gh command is called. the watch loop in git.release.sh calls BOTH `gh pr view` AND `gh run list` per iteration. this means the counter advances by 2 per poll cycle instead of 1, which causes the watch_sequence.json to be read at wrong indices. if the sequence has N states, the loop exits early or reads past the end, which yields timeout (exit 1).

fix: use separate counters per gh subcommand, OR increment counter once per poll cycle instead of per gh call.
