# 1.vision: consistent release transports

## the outcome world

### before

the output varies by transport type. the mechanic must learn different output shapes for feature branches vs release branches vs release tags. mental overhead compounds.

```
$ rhx git.release
🐢 heres the wave...

🐚 git.release --to main --mode plan

🌊 release: feat(oceans): add reef protection
   ├─ branch: turtle/add-surfboards
   ├─ pr: #42
   ├─ checks: 3 passed, 1 inflight
   └─ automerge: disabled

$ rhx git.release --to prod
🐢 heres the wave...

🐚 git.release --to prod

🌊 release: chore(release): v1.2.3 🎉
   ├─ status: checks passed
   └─ tag: v1.2.3 (awaits merge)
```

the problem: each transport shows different fields, different structure. no unified mental model. plus, the flag is `--to` which is ambiguous (could mean a range).

### after

```
$ rhx git.release
🐢 heres the wave...

🐚 git.release --into main --mode plan

🌊 release: feat(oceans): add reef protection
   ├─ 👌 all checks passed
   ├─ 🌴 automerge unfound (use --apply to add)
   └─ hint: use --apply to enable automerge and watch

$ rhx git.release --into prod
🐢 lets see...

🐚 git.release --into prod --mode plan

🌊 release: feat(oceans): add reef protection
   ├─ 👌 all checks passed
   └─ 🌴 already merged

🌊 release: chore(release): v1.33.0 🎉
   ├─ 👌 all checks passed
   ├─ 🌴 automerge unfound (use --apply to add)
   └─ hint: use --apply to merge and watch tag workflows
```

every transport shows the same shape:
- `🌊 release: {title}` header
- check status: `👌 all checks passed` | `🐢 N check(s) in progress` | `⚓ N check(s) failed`
- automerge status: `🌴 automerge unfound` | `🌴 automerge enabled [found]` | `🌴 automerge enabled [added]` | `🌴 already merged`
- hint or watch state

the "aha" moment: when you see multiple `🌊 release:` blocks with identical status trees, you realize they're the same concept at different stages.

## cli interface

### flags

| flag | alias | description |
|------|-------|-------------|
| `--into main` | | merge feature branch to main (default) |
| `--into prod` | | full release: feat → main → prod → tag |
| `--from main` | | skip feature branch, act as if on main |
| `--from <branch>` | | specify source branch explicitly |
| `--mode plan` | | show status only (default) |
| `--mode apply` | `--apply` | enable automerge and watch |
| `--watch` | | watch CI without automerge |
| `--retry` | | rerun failed workflows before watch |
| `--dirty block` | | fail fast if uncommitted changes (default) |
| `--dirty allow` | | allow release with uncommitted changes |

### examples

```bash
rhx git.release                      # plan: feat → main
rhx git.release --into prod          # plan: feat → main → prod
rhx git.release --from main          # plan: main → prod (skip feat)
rhx git.release --apply              # apply: enable automerge, watch
rhx git.release --into prod --apply  # apply: full release to prod
rhx git.release --watch              # watch CI without automerge
rhx git.release --retry --apply      # retry failed, then apply
```

## user experience

### usecase 1: check status (plan mode)

```
$ rhx git.release
🐢 heres the wave...

🐚 git.release --into main --mode plan

🌊 release: feat(oceans): add reef protection
   ├─ 👌 all checks passed
   ├─ 🌴 automerge unfound (use --apply to add)
   └─ hint: use --apply to enable automerge and watch
```

no flags = plan mode = safe read-only.

### usecase 2: watch without automerge

```
$ rhx git.release --watch
🐢 heres the wave...

🐚 git.release --into main --watch

🌊 release: feat(oceans): add reef protection
   ├─ 🐢 1 check(s) in progress
   ├─ 🌴 automerge unfound (use --apply to add)
   └─ 🥥 let's watch
      ├─ 💤 1 left, Xs in action, Xs watched
      ├─ 👌 all checks passed
      └─ hint: use --apply to add automerge
```

`--watch` shows poll lines uniformly. each poll is a branch in the tree.

### usecase 3: apply with automerge

```
$ rhx git.release --apply
🐢 cowabunga!

🐚 git.release --into main --mode apply

🌊 release: feat(oceans): add reef protection
   ├─ 👌 all checks passed
   ├─ 🌴 automerge enabled [added] -> and merged already
   └─ 🥥 let's watch
      └─ ✨ done! Xs in action, Xs watched
```

`--apply` (alias for `--mode apply`) enables automerge and watches to completion.

### usecase 4: full release to prod

```
$ rhx git.release --into prod --apply
🐢 radical!

🐚 git.release --into prod --mode apply

🌊 release: feat(oceans): add reef protection
   ├─ 👌 all checks passed
   └─ 🌴 already merged

🫧 and then...
   └─ ✨ found it! Xs in action, Xs watched

🌊 release: chore(release): v1.33.0 🎉
   ├─ 👌 all checks passed
   ├─ 🌴 automerge enabled [added] -> and merged already
   └─ 🥥 let's watch
      └─ ✨ done! Xs in action, Xs watched

🫧 and then...
   └─ ✨ found it! Xs in action, Xs watched

🌊 release: v1.33.0
   └─ 🥥 let's watch
      ├─ 🫧 no runs inflight
      └─ ✨ done! publish.yml, Xs in action, Xs watched
```

chains through all three transports: feature PR → release PR → tag workflows.

### usecase 5: handle failures with --retry

```
$ rhx git.release --watch
🐢 heres the wave...

🐚 git.release --into main --watch

🌊 release: feat(oceans): add reef protection

$ rhx git.release --retry
🐢 heres the wave...

🐚 git.release --into main --retry

🌊 release: feat(oceans): add reef protection
   ├─ ⚓ 1 check(s) failed
   │  ├─ 🔴 test-unit
   │  │     ├─ https://github.com/test/repo/actions/runs/123
   │  │     ├─ failed after Xm Ys
   │  │     └─ 👌 rerun triggered

```

```
🐢 crickets...

```

## mental model

### how users describe it

> "git.release moves your code through transports - first feature branch to main, then release branch to prod, then tag workflows. each transport shows the same status shape. --watch follows it, --apply drives it."

### analogies

**assembly line**: code flows through stations (transports). each station has the same status board: wait, active, passed, failed, done.

**relay race**: the baton (code) passes through runners (transports). each handoff follows the same protocol.

### terms

| user says | system shows |
|-----------|--------------|
| "my PR" | `🌊 release: feat(...)` |
| "the release PR" | `🌊 release: chore(release): vX.Y.Z` |
| "the tag build" | `🌊 release: vX.Y.Z` |
| "it's active" | `🐢 N check(s) in progress` |
| "it passed" | `👌 all checks passed` |
| "it failed" | `⚓ N check(s) failed` |
| "it merged" | `🌴 already merged` |
| "turn on automerge" | `--apply` |
| "just check it" | (no flags) or `--watch` |

## evaluation

### how well does it solve the goals?

| goal | solved? |
|------|---------|
| consistent stdout across transports | ✅ yes - same `🌊 release:` shape |
| reusable domain operations | ✅ yes - emit_transport_status, emit_transport_watch |
| clear integration test coverage | ✅ yes - composable test matrix |
| reduced friction in usecases | ✅ yes - uniform flags, `--apply` alias |

### pros

1. **uniform output**: learn one pattern, apply everywhere
2. **composable tests**: test transport operations independently, compose for full flows
3. **predictable behavior**: same flags (`--watch`, `--apply`, `--retry`) work on all transports
4. **progressive disclosure**: plan mode is safe, watch adds observation, apply adds action
5. **ergonomic aliases**: `--apply` is shorter than `--mode apply`

### cons

1. **more verbose output**: multiple `🌊 release:` blocks take vertical space
2. **migration**: `--to` → `--into` change requires documentation update

### edgecases and pit of success

| edgecase | pit of success |
|----------|----------------|
| `--from main --into main` | ConstraintError: can't merge main into main |
| on main, no flags | implies `--from main --into prod` |
| feature branch not pushed | `🫧 no open branch pr` with hint to push |
| release PR doesn't exist yet | `🫧 and then...` polls until found |
| tag workflows don't exist | `🫧 no tag workflows found` |
| automerge already enabled | shows `🌴 automerge enabled [found]` (no "added") |
| multiple release PRs | ConstraintError: ambiguous |
| `--from turtle/feat` when on main | releases that branch to main |

## open questions & assumptions

### assumptions

1. **three transports are exhaustive**: feature-branch → release-branch → release-tag covers all repos. no fourth transport exists.

2. **check states are finite**: each transport's checks are in exactly one of [inflight, passed, failed]. automerge status is tracked separately: [unfound, enabled, merged]. these cover all github PR/workflow states.

3. **automerge is the only "apply" action**: we don't need other mutations (e.g., approve, label) in apply mode.

4. **`--retry` only reruns failed**: it doesn't restart passed workflows.

### questions to validate

1. **is `--into` the right term?** [answered]
   - alternatives: `--to`, `--target`
   - resolution: `--into` is correct — the wisher explicitly requested "replace `--to` with `--into`" and `--into` emphasizes destination while `--to` suggests range

2. **should `--apply` be the default alias?** [answered]
   - resolution: yes — the wisher explicitly requested "add alias `--apply` for `--mode = apply`" for ergonomics

3. **what is the watch timeout behavior?** [answered]
   - duration: 15 minutes, not configurable
   - on timeout: exit with error, show partial progress

### external research needed

none - this is internal tooling with known constraints.

## what is awkward?

### release-tag transport is different

tags don't have PRs. they have "workflow runs". the `🌊 release: vX.Y.Z` shape works but the semantics differ:

- PR transports: checks → automerge → merge
- tag transport: workflows → completion

we unify this by use of the same `🌊 release:` header and similar status indicators.

### `--into prod` from feat branch is a chain

when you run `rhx git.release --into prod` from a feature branch, it chains three transports. if the first fails, you don't reach the others. this is correct behavior.

the output shows each transport sequentially with `🫧 and then...` transitions between them.

### retry scope

`--retry` reruns failed workflows for the transport currently blocked. if feature-branch has a failure, retry that. if release-tag has a failure, retry that.

when combined with `--apply`, retry runs first on the blocked transport, then apply continues if retry succeeds.

### watch cycles must be visible

the wish mandates at least 3 watch poll cycles visible in tests. this means mocks must emit multiple states, and we need to capture stdout with time artifacts.

solution: use relative times (`Xs in action, Xs watched`) which we control via mock time, replaced in snapshots.

## summary

the vision is a world where:

1. every release transport shows the same `🌊 release: {title}` status tree
2. `--watch` adds poll lines, `--apply` (alias for `--mode apply`) adds automerge
3. `--into` replaces `--to` for destination clarity
4. integration tests cover the cartesian product via composition
5. the mechanic sees uniform output regardless of which transport is active

the "aha" moment: `git.release` is just `emit_transport_status` + `emit_transport_watch` in a loop over the relevant transports for your goal.
