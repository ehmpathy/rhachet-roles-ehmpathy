the `git.release` skill has 3 different release transports that it cares about

1 = the feature branch
2 = the release branch
3 = the release tag

- they each get watched
- they each can fail
- they each can succeed
- they each can have already succeded and be already merged
- they each can have already succeded and be awaiting merger

the core usecases of this feature are 


1. `rhx git.release` = check the status of the current branch
    - if on main, then of the 'from main' transports of [release branch, release tag]
        - if the latest release branch is already found merged, then check the release tag
        - if the latest release branch is not yet merged, then that's the target transport
    - if on feat, then of the 'from feat' transports of [feature branch]

these should uniformly and exactly the same show the states of 

failed
inflight
succeeded:with-automerge
succeeded:wout-automerge

---

2. `rhx git.release --mode apply`

this should apply automerge to whatever transports it finds and watch at the same time, to drive it through --to the release target

if `--from main` (either via specified explicitly OR via current branch)

then its implicitly `--to prod`

so it should find the `release branch` transport, if its not already merged or does not already have automerge, apply automerge, then watch it, then wait for the `release tag`, and watch it

if its already merged, then just waits for the release tag and watches it


if `--from feat` branch (i.e., --from was not specified && is not currently on main)

then its by definition atleast --to main,

so it should simply find the `feature branch` transport, findsert the automerge flag, and watch it

if --to prod was specified, then it should also invoke the release --to prod flow, same as `--from main` above

---

in all of these conditions

the stdout emitted for each branch should be exactly the same (reusable domain operation to stdout the desired state)
