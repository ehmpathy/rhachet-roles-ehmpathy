# 3.1.5.research.reflection.product.rootcause._.v1.i1

## what is the symptom?

**the observable problem**: git.release skill produces inconsistent stdout across the three release transports (feature-branch, release-branch, release-tag).

- **what is broken?** each transport type shows different output shapes, field names, and status indicators. mechanics and humans must learn different patterns for each.

- **how does it manifest?**
  - feature-branch shows PR checks + automerge status
  - release-branch shows different format for same data
  - release-tag shows workflow runs with yet another format
  - no unified mental model

- **when does it occur?** every time a user runs `rhx git.release` with different `--from`/`--to` combinations.

---

## 5 whys tree

### branch 1: inconsistent output across transports

1. why is output inconsistent across transports?
   → because each transport has its own output code path

2. why does each transport have its own output code path?
   → because the main flow functions (`release_to_main`, `release_to_prod`) are monolithic with inline output

3. why are the main flows monolithic with inline output?
   → because there was no explicit contract for uniform transport status

4. why was there no explicit contract for uniform transport status?
   → because the transports were not recognized as instances of a common abstraction

5. why were transports not recognized as a common abstraction?
   → **root cause: domain model did not capture "transport" as a first-class concept with unified interface**

### branch 2: test coverage gaps

1. why do tests not catch the inconsistency?
   → because tests focus on individual transport behaviors, not cross-transport uniformity

2. why do tests not compare cross-transport uniformity?
   → because there's no single test suite that exercises all transports with same state matrix

3. why is there no unified test matrix?
   → because the code paths themselves are separate, so tests followed that structure

4. why did tests follow non-unified code structure?
   → because tests were written incrementally as features were added, not to a spec

5. why were tests not written to a spec?
   → **root cause: no blackbox criteria defined uniform output as a requirement**

### branch 3: unclear flag semantics

1. why is `--to` ambiguous?
   → because "to" can mean either destination or range endpoint

2. why was "to" chosen originally?
   → likely shorthand for "release to main" or "release to prod"

3. why does ambiguity matter now?
   → because mechanics need to infer `--from` and `--to` from branch context, and the relationship is unclear

4. why is the relationship unclear?
   → because inference rules are implicit in code, not documented or tested as a matrix

5. why are inference rules implicit?
   → **root cause: flag design did not follow explicit contract-first approach**

---

## root cause summary

| layer | issue | is root cause? |
|-------|-------|----------------|
| output code | inline output in monolithic functions | no — symptom |
| domain model | no "transport" abstraction | **yes** |
| test structure | no unified test matrix | no — consequence |
| blackbox spec | no uniform output requirement | **yes** |
| flag design | implicit inference rules | **yes** |

three root causes converge:

1. **absent domain abstraction**: "transport" not modeled as reusable concept
2. **absent blackbox spec**: uniform output never mandated
3. **implicit flag semantics**: inference rules buried in code

---

## recommendation

**address root causes directly:**

1. **domain model fix**: extract `emit_transport_status` and `emit_transport_watch` as reusable operations. each transport (feature-branch, release-branch, release-tag) becomes an adapter that returns normalized state. output operations consume normalized state.

2. **blackbox spec fix**: the criteria matrix [2.2.criteria.blackbox.matrix] now mandates uniform output. tests will enforce.

3. **flag semantics fix**: `get_one_goal_from_input` becomes explicit operation with 12 documented inference scenarios. `--to` renamed to `--into` for clarity.

all three root causes are addressed in this behavior scope. no follow-up needed.
