# 3.1.5.research.reflection.product.audience._.v1.i1

## purpose

identify who this is for to ensure we build for real needs.

---

## who is the audience?

### primary: mechanics (AI agents)

mechanics invoke `rhx git.release` to watch and apply releases. they parse stdout to determine next actions and compose into larger workflows.

- need: uniform output structure across all transports
- context: automated workflows, no human in the loop
- constraint: must parse text reliably

### secondary: humans who delegate

humans grant commit/push quota and observe release progress. they read terminal output to verify release status.

- need: scannable output that answers "did it work?"
- context: quick check between other tasks
- constraint: limited attention, wants headlines not details

### tertiary: future skill developers

developers who extend git.release or build similar watch-based skills. they reference patterns for reuse.

- need: clean code with extractable patterns
- context: future maintenance and extension
- constraint: unfamiliar with original implementation

---

## why do they care?

| audience | pain relieved | goal enabled | frustration eliminated |
|----------|---------------|--------------|------------------------|
| mechanic | inconsistent output per transport | compose reliable release workflows | no more special-casing per transport type |
| human | unclear release status | quick status check | no more manual gh pr checks |
| developer | tangled code | extract patterns for other skills | no more full-context reads to grasp |

---

## how much do they care?

| audience | priority | urgency | stakes |
|----------|----------|---------|--------|
| mechanic | high | high | release workflows break without uniform output |
| human | medium | low | can fall back to gh CLI |
| developer | low | low | can read extant code |

---

## what do they care about?

### mechanic

- **success criteria**: same output shape for feature-branch, release-branch, release-tag
- **acceptable tradeoff**: more verbose output if structure is consistent
- **delight**: one parse pattern works for all transports
- **disappoint**: different output shapes require transport-specific parse

### human

- **success criteria**: status is clear in 2 seconds of scan
- **acceptable tradeoff**: less detail if headlines are clear
- **delight**: emoji indicators convey state at a glance
- **disappoint**: wall of text with buried status

### developer

- **success criteria**: extractable domain operations
- **acceptable tradeoff**: more files if responsibility is clear
- **delight**: drop-in reusable operations
- **disappoint**: monolithic functions with inline logic

---

## how will they experience it?

### mechanic journey

1. invoke `rhx git.release --apply`
2. parse `🌊 release:` blocks
3. extract check status, automerge status
4. wait for watch completion
5. check exit code for success/failure
6. proceed to next step in workflow

**friction**: variable output structure per transport
**smooth experience**: every transport emits same status tree

### human journey

1. grant commit quota to mechanic
2. observe terminal output
3. scan for emoji indicators (👌, 🐢, ⚓)
4. trust mechanic to handle details

**friction**: unclear what the mechanic is doing
**smooth experience**: headers show progress through transports

### developer journey

1. search for watch pattern
2. find emit_transport_watch operation
3. extract and adapt for new skill

**friction**: logic embedded in main flow
**smooth experience**: operations extracted to separate functions
