# 3.1.5.research.reflection.product.premortem._.v1.i1

## purpose

imagine failure modes before they occur to prevent them.

---

## the premortem

it is 2 weeks after the git.release refactor shipped. it failed. what went wrong?

---

## failure scenario 1: watch tests pass but don't verify 3+ poll cycles

### what happened

the wish mandates "show at least 3 poll cycles" for watch behavior. we wrote tests that use SEQUENCE mocks, but the tests only assert on final output — not that 3 cycles actually occurred.

### why it happened

- focused on terminal state (passed/failed) not poll count
- SEQUENCE mock returned final state on cycle 2
- no assertion checked cycle count

### how to prevent

- add explicit assertion: `expect(output.match(/💤.*left/g)?.length).toBeGreaterThanOrEqual(3)`
- mock sequences must have 4+ elements (3 inflight + 1 terminal)
- test helper that validates poll count

---

## failure scenario 2: --to removal breaks external scripts

### what happened

external scripts and documentation reference `--to prod`. the rename to `--into` breaks them without notice. users complain.

### why it happened

- no deprecation period for `--to`
- no backwards-compat alias
- no search for extant `--to` usage in org repos

### how to prevent

- search org repos for `--to` usage before removal
- add temporary `--to` alias that emits deprecation notice
- document migration path in changelog
- consider 1 release cycle with both flags

---

## failure scenario 3: extracted operations change subtle edge case behavior

### what happened

the main flow refactor from monolithic functions to composed operations introduced subtle behavior changes. an edge case that worked before now fails silently or with wrong output.

### why it happened

- extant code had implicit state (e.g., variable reuse across transport checks)
- extracted operations lost that implicit state
- tests covered happy path, not edge combinations

### how to prevent

- snapshot test all extant usecases before refactor
- run full test suite on refactored code, compare snapshots
- explicit state pass via function arguments, not closures

---

## failure scenario 4: timeout behavior differs from extant

### what happened

the watch loop timeout (15 min) was already in extant code, but the refactored version polls at different intervals or handles timeout differently. users experience different timeout behavior.

### why it happened

- extant code has: 5s poll for <60s, then 15s poll
- refactored code used uniform poll interval
- timeout edge case not explicitly tested

### how to prevent

- document extant poll interval logic
- test with mock time to verify intervals
- assert on poll count at timeout boundary

---

## failure scenario 5: transport adapter inconsistency

### what happened

feature-branch and release-branch transports use PR checks. release-tag transport uses workflow runs. the adapter abstraction leaks — tag transport shows different output shape or timing.

### why it happened

- PR checks and workflow runs have different API response shapes
- adapter normalized them, but edge cases diverged
- tag workflows have no "automerge" concept, but output tried to show it

### how to prevent

- explicit interface contract for transport adapters
- tag adapter explicitly returns automerge: n/a or merged
- integration test each adapter with same state matrix

---

## failure scenario 6: test flakiness from real gh calls

### what happened

some tests accidentally hit real gh CLI instead of mock. they passed locally but failed in CI (no auth) or caused flakiness from network variance.

### why it happened

- PATH mock setup failed for some edge case
- test cleanup incomplete, leaked real environment
- CI environment had different PATH

### how to prevent

- assert at test start that gh mock is active
- mock gh to print unique marker, verify in output
- isolate tests in clean tempdir with controlled PATH

| timeout behavior change | medium | medium | document + test poll intervals |
| adapter inconsistency | medium | high | explicit interface + test matrix |
| gh call flakiness | low | medium | assert mock active |
