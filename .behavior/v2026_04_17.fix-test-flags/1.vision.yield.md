# vision: guide away from raw --testNamePattern

## the outcome world

### before (today)

```bash
# someone wants to run specific test cases by name
rhx git.repo.test --what acceptance -- --testNamePattern "driver.route.bind|driver.route.failsafe"
```

problems:
- `--` breaks the turtle vibes abstraction layer
- raw jest flags leak implementation details
- inconsistent with `--scope` ergonomics
- no validation or helpful error messages
- users must know jest internals

### after (with this)

```bash
# ergonomic equivalent - filter by test name
rhx git.repo.test --what acceptance --scope 'name(driver.route.bind|driver.route.failsafe)'
```

failfast with guidance when raw flags detected:

```bash
rhx git.repo.test --what acceptance -- --testNamePattern "driver.route.bind"

🐢 hold up, dude...

🐚 git.repo.test --what acceptance
   └─ ✋ blocked: raw --testNamePattern detected

🥥 did you know?
   ├─ --scope 'foo' filters by file path
   ├─ --scope 'path(foo)' filters by file path (explicit)
   └─ --scope 'name(foo)' filters by test name
```

### the "aha" moment

when a user types `--` they escape the abstraction. the skill should either:
1. provide what they need so they don't escape, or
2. guide them back if they do

## user experience

### use cases

| goal | today | vision |
|------|-------|--------|
| filter by file path | `--scope getUserById` | `--scope 'getUserById'` (unchanged) |
| filter by file path (explicit) | `--scope getUserById` | `--scope 'path(getUserById)'` |
| filter by test name | `-- --testNamePattern "..."` | `--scope 'name(...)'` |

### timelines

1. user runs with `-- --testNamePattern`
2. skill detects this pattern in REST_ARGS
3. skill emits guidance and exits 2 (constraint)
4. user reruns with `--scope 'name(...)'`
5. skill works as expected

## mental model

### how users describe this

"--scope filters tests. wrap in path() or name() if you need precision."

### analogies

- `--scope 'invoice'` = grep for "invoice" everywhere
- `--scope 'path(invoice)'` = grep file paths only
- `--scope 'name(invoice)'` = grep test names only

### term map

| user term | jest term | our term |
|-----------|-----------|----------|
| "run tests with X" | (both flags) | `--scope 'X'` |
| "run this file" | --testPathPatterns | `--scope 'path(X)'` |
| "run this test" | --testNamePattern | `--scope 'name(X)'` |
| "raw mode" | -- | (blocked with guidance) |

## evaluation

### pros

- pit of success: users discover `--scope 'name(...)'` before they need raw args
- one flag: `--scope` handles both path and name filters
- better errors: skill can validate patterns, suggest fixes
- abstraction preserved: jest internals stay hidden
- backwards compatible: `--scope foo` behavior unchanged (path filter)

### cons

- users with muscle memory for `-- --testNamePattern` need to relearn
- edge cases where raw jest args are actually needed (rare)

### edge cases

| case | how handled |
|------|----------|
| `-- --verbose` | allow (not a filter flag) |
| `-- --testNamePattern` | block with guidance (use `--scope 'name(...)'`) |
| `-- --testPathPattern` | block with guidance (use `--scope 'path(...)'`) |
| `-- --coverage` | allow (not a filter flag) |

## open questions & assumptions

### assumptions

1. `--scope` with `path()` and `name()` qualifiers is intuitive
2. immediate block is safe (verified: no CI usage)
3. raw args for non-filter flags should still work

### questions [all answered]

| question | resolution |
|----------|------------|
| separate flag vs extend `--scope`? | extend `--scope` with qualifiers |
| block vs deprecate? | immediate block - no CI usage found |
| legitimate use cases lost? | none - `--scope 'name(...)'` covers all |

### research needed

- none; this is internal tool with full control

## what is awkward?

### block is opinionated

immediate block is opinionated. but:
- no CI usage means no breakage
- clear guidance with 🥥 tip makes migration easy
- matches skill's failfast philosophy

### the "escape hatch" problem

users may have legitimate reasons to pass raw jest args. block filter-related flags only (`--testNamePattern`, `--testPathPattern`), allow others through.

decision: block known footguns, allow unknown passthrough.
