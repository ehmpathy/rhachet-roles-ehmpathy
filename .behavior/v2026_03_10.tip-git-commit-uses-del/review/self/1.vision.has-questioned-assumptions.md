# self-review: has-questioned-assumptions

## assumption 1: `del` is intuitive

**the assumption:** users will understand `del` means "revoke quota" because of redis/key-value patterns.

**evidence:** get/set/del is common in caches, redis, etc.

**what if opposite?** users might think `del` means "delete the meter file" literally, not "revoke privileges". or confuse with `rm`.

**did wisher say?** wisher said `del`, so verb is confirmed. interpretation is assumed.

**verdict:** low risk. `del` removes quota — semantically correct. output says "revoked" which clarifies.

---

## assumption 2: users will run `set --quant 0` to revoke

**the assumption:** users explicitly revoke quota by run of the verbose command.

**what if opposite?** users might just close the session, never see the tip.

**evidence:** when mechanic asks "can i commit?", human might run `set --quant 0` to say no. plausible flow.

**verdict:** the tip is opportunistic. if users don't revoke explicitly, no harm — they just won't see the tip. acceptable.

---

## assumption 3: users read command output

**the assumption:** users notice the tip line in output.

**what if opposite?** users run commands and scroll past output.

**evidence:** turtle vibes output is short and structured. tip is a subbranch, draws attention via indent.

**verdict:** reasonable assumption. can't force users to read, but structure helps.

---

## assumption 4: tip is non-intrusive at dim/muted

**the assumption:** dim style makes tip ignorable after first read.

**what if opposite?** users find it noisy, want it gone.

**evidence:** one line, only on revoke, dim. minimal.

**verdict:** acceptable. if users complain, can add `--quiet` flag later.

---

## assumption 5: `del` should behave like `set --quant 0 --push block`

**the assumption:** `del` sets state to {uses: 0, push: "block"} rather than delete of file.

**did wisher say?** wisher said "does the same" as `set --quant 0 --push block`.

**what if opposite?** `del` could delete the state file entirely. but then `get` would say "no quota set" vs "revoked". different semantics.

**verdict:** confirmed by wish. `del` = `set --quant 0 --push block`, not file delete.

---

## assumption 6: tip only on `set` revoke, not on `get` with zero quota

**the assumption:** tip appears only when user actively revokes, not when they check state.

**what if opposite?** could show tip on `get` when quota is 0: "tip: use `del` to revoke"

**evidence:** on `get`, user checks state, not acts. tip about action would be out of place.

**verdict:** tip on `set` only is correct. `get` is read-only.

## assumption 7: output.sh already supports dim/muted style

**the assumption:** the tip can be printed with dim/muted color.

**what i found:** output.sh has NO color support. all print functions use plain text with tree characters (└─, ├─, │). there are no ANSI escape codes.

**impact:** implementation must ADD color support to output.sh or inline ANSI codes.

**did wisher say?** wisher confirmed "dim/muted" style. implementation detail is mine.

**verdict:** SCOPE CREEP. vision should note that output.sh needs a `print_tip` function with ANSI dim style. needs update.

## assumption 9: tip shows for both `set --quant 0` and `set --quant 0 --push block`

**the assumption:** since both result in same state {uses: 0, push: "block"}, both show tip.

**what i found:** current code defaults `--push` to `block` when `--quant 0`. so `set --quant 0` alone triggers same path.

**verdict:** confirmed. tip shows whenever result is equivalent to `del` semantics.

---

## assumption 10: tip does NOT show for `set --quant 0 --push allow`

**the assumption:** this edge case (no commits but can push) is NOT equivalent to `del`.

**what i found:** vision mentions this under "what is awkward?" but doesn't make it explicit.

**verdict:** needs explicit decision. `del` = quant 0 + push block. tip only shows when BOTH conditions met. update vision to be explicit.
