# self-review: has-questioned-requirements

## requirement 1: add `del` subcommand

**who said?** the wish from vlad, today.

**evidence:** `set --quant 0 --push block` is verbose for a common action (revoke). the get/set/del pattern is familiar from redis, key-value stores.

**what if we didn't?** users keep the verbose incantation. not catastrophic, but friction.

**scope:** minimal — one new subcommand that delegates to `set --quant 0 --push block`.

**simpler way?** could document-only without implementation. but then tip would mislead if `del` doesn't exist.

**verdict:** justified. minimal scope, clear value.

---

## requirement 2: show tip on verbose revoke

**who said?** the wish explicitly asks for this.

**evidence:** progressive disclosure is good ux — users learn shortcuts organically.

**what if we didn't?** `del` exists but nobody knows. defeats purpose.

**scope:** one line of output, dim/muted. non-intrusive.

**simpler way?** could just document. but tips are more discoverable than docs.

**verdict:** justified. this is the core ask.

**who said?** human confirmed just now.

**verdict:** justified.

## edge case questioned: `set --quant 0 --push allow`

**the case:** what if someone sets quant 0 but allows push?

**analysis:** this means "no new commits, but can push extant". weird edge case.

**should tip show?** no — `del` means "revoke all" (quant 0 + push block). if someone wants this weird config, they're explicit about it.

**verdict:** tip only shows when both quant=0 AND push=block. this is correct.

**too small?** no — solves the stated wish completely.

all requirements justified. scope is minimal and focused. no bloat.
