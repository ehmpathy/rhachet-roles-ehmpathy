# 1.vision.yield

## the outcome world

### before

driver runs `rhx declapract.upgrade exec` and it fails.

```
🐢 upgrade time!

📦 ensure node version
📦 upgrade declapract packages
✨ apply best practices
error: variable 'organizationName' is required but absent

<driver shrugs and proceeds to next stone>
```

the driver:
- sees "error" but not **BLOCKER**
- doesn't know where to look for help
- proceeds to 2.detect.hazards without completing the upgrade
- creates a PR with incomplete/broken state
- human reviews, finds the mess, wastes 30 minutes understanding why

### after

driver runs `rhx declapract.upgrade exec` and it fails.

```
🐢 bummer dude...

🐚 declapract.upgrade exec
   └─ ⛔ BLOCKER: upgrade failed

   the upgrade did not complete. you MUST fix this before you continue.

   see brief: howto.fix-declapract-upgrade-failures

   do NOT proceed to next stone. fix this failure first.
```

the driver:
- sees **BLOCKER** — unmistakable
- sees which brief to consult
- knows they must stop here
- fixes the issue, re-runs exec, succeeds, then continues

### the "aha" moment

when the driver sees the BLOCKER label and inline guidance, they think: "oh, i know exactly what to do" instead of "maybe i can skip this."

## user experience

### usecases

1. **happy path**: driver runs `exec`, it succeeds, driver proceeds
2. **common failure**: driver runs `exec`, it fails with known issue, driver consults pointed-to brief, fixes, re-runs, succeeds
3. **unknown failure**: driver runs `exec`, it fails with unknown issue, driver documents in artifact, escalates to human — but does NOT proceed

### contract inputs & outputs

**exec success output:**
```
🐢 shell yeah!
   └─ upgrade complete, now review what broke 🏄
```

**exec failure output:**
```
🐢 bummer dude...

🐚 declapract.upgrade exec
   └─ ⛔ BLOCKER: upgrade failed

   see brief: howto.fix-declapract-upgrade-failures

   do NOT proceed to next stone. fix this failure first.
```

### timeline

1. driver arrives at 1.upgrade.invoke stone
2. driver runs `rhx declapract.upgrade exec`
3. **if success**: driver proceeds via `--as passed`
4. **if failure**: driver sees BLOCKER output, consults briefs, fixes, re-runs exec (loop until success)
5. only after success does driver proceed

## evaluation

### how well does it solve the goals?

| goal | solution |
|------|----------|
| failure is unmissable | ⛔ BLOCKER label |
| briefs are discoverable | inline guidance in failure output |
| drivers don't proceed | language is explicit ("do NOT proceed") |
| common failures are handled | inline guidance in failure output |

### pros

- zero ambiguity — BLOCKER means BLOCKER
- zero search — briefs are directly linked
- consistent with rule.require.failfast and rule.require.failloud

### cons

- longer failure output (acceptable tradeoff for clarity)
- requires changes to exec.sh, stone content, and guard

### edgecases

| edgecase | pit of success |
|----------|----------------|
| driver ignores BLOCKER | willful negligence — unmistakable language makes this a conscious choice |
| driver can't fix | explicit escalation path in stone instructions |
| unknown failure type | driver documents and escalates, does NOT proceed |
| driver modifies dpdm exclude pattern | guard catches this during repair phase |

### dpdm guard requirement

drivers sometimes "fix" dpdm lint failures by addition of prod dependencies to `.dpdmrc.yaml` exclude array — this defeats the purpose of cycle detection.

**solution**: add a self-review guard to `3.1.repair.test.defects.guard` that checks for dpdm-exclude violations:

```yaml
- slug: has-respected-dpdm-exclude
  say: |
    did the driver modify .dpdmrc.yaml exclude array incorrectly?

    we avoid all cycles, even in dependencies — ON PURPOSE.
    we check node_modules for all deps (prod and dev) by default.

    the .dpdmrc.yaml file controls dpdm cycle detection:
    - exclude: [] means check all files, node_modules included
    - devDeps may be excluded as escape hatch if needed
    - prod dependencies must never be excluded
    - node_modules must never be excluded by default

    forbidden changes:
    - add prod dep to exclude array
    - add node_modules or ^node_modules to exclude
    - change default to exclude node_modules

    if you exclude a prod dependency or node_modules, you ship
    cycles to consumers and break their builds.
    see brief rule.forbid.dpdm-exclude-change.

    if git diff shows any forbidden change to .dpdmrc.yaml:
    - this is a BLOCKER
    - revert the change
    - fix the actual circular imports instead
```

### exec success guard requirement

drivers sometimes proceed past failed exec without resolution.

**solution**: add a self-review guard to `1.upgrade.invoke.guard` (create if absent) that verifies exec completed:

```yaml
- slug: has-completed-exec-successfully
  say: |
    did `rhx declapract.upgrade exec` complete successfully?

    check for:
    - "shell yeah!" in output = success
    - "bummer dude..." or error output = failure

    if exec failed:
    - this is a BLOCKER — do NOT proceed
    - see brief howto.fix-declapract-upgrade-failures
    - fix the issue and re-run exec
    - if unfixable after brief triage, mark stone as blocked

    only proceed to next stone after exec succeeds.
```

this guard catches all exec failure modes — the brief covers specific fixes.

## what is awkward?

### raw command failures

currently, if `pnpm add` fails, the raw pnpm error is shown. no wrapping, no guidance. this is the core problem.

**fix**: wrap each command in a function that catches failure and emits structured output with guidance.

### stone language

the current stone says "if the upgrade fails, check the appendix" — this is passive. it should say "if the upgrade fails, this is a BLOCKER. do NOT proceed. see brief for fixes."

### brief location

common failures guidance moves from stone appendix to brief. briefs are always accessible, stones are sealed once route is created.
