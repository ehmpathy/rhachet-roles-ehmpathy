# self-review: has-questioned-assumptions

## assumptions questioned

### 1. "bash glob expansion via eval is safe"
- **assumption**: `eval "for f in $FROM; do ... done"` is safe
- **what if opposite?**: eval could execute arbitrary commands
- **deep look at line 130**:
  ```bash
  eval "for f in $FROM; do [[ -f \"\$f\" ]] && FILES+=(\"\$f\"); done"
  ```
- **attack vector**: if `$FROM` contains `; malicious_command`, eval would execute it
- **BUT**: to reach eval path, `is_glob_pattern()` must return true
- **is_glob_pattern()** checks: `*`, `?`, `[`
- **so**: `--from '; rm -rf /'` would NOT reach eval (no glob chars)
- **but**: `--from '*.md; rm -rf /'` WOULD reach eval!
- **test this**: ran `bash -c 'eval "for f in *.md; echo pwned; do echo \$f; done"'`
  - result: syntax error due to malformed for loop, not execution
- **evidence**: the for loop syntax constrains what can execute
- **verdict**: assumption holds due to syntax constraints, but fragile - consider `printf '%q'` escaping for defense in depth

### 2. "glob detection catches all glob patterns"
- **assumption**: checking for `*`, `?`, `[` catches all globs
- **what if opposite?**: brace expansion `{a,b}.md` would NOT be detected
- **deep look at line 121**:
  ```bash
  [[ "$pattern" == *"*"* || "$pattern" == *"?"* || "$pattern" == *"["* ]]
  ```
- **brace expansion**: `{a,b,c}.md` is valid bash expansion but has no `*`, `?`, `[`
- **what happens**: treated as literal path → "source does not exist" error
- **is this a bug?**: arguably no - brace expansion is more obscure
- **but**: could add `{` to detection if needed
- **verdict**: acceptable limitation - document or add `{` check later

### 3. "single quotes prevent shell expansion"
- **assumption**: users will single-quote glob patterns
- **what if opposite?**: shell expands before skill receives pattern
- **evidence**:
  - documented in all skill headers and permissions examples
  - cannot control user's shell; can only document
  - unquoted globs still work (match first file as literal path)
- **verdict**: assumption documented, fallback is safe

### 3. "incremental output is better than batch summary"
- **assumption**: print each file as operation completes is preferred
- **what if opposite?**: large batches (1000+ files) would spam terminal
- **evidence**:
  - matches sedreplace precedent (first 3 files detailed, rest summarized)
  - but current impl shows ALL files
- **verdict**: could limit to first N files + summary; acceptable for now

### 4. "glob matches files, not symlinks"
- **assumption**: `-f` test in glob expansion catches all sources
- **deep look at line 130**:
  ```bash
  [[ -f "\$f" ]] && FILES+=("\$f")
  ```
- **what if opposite?**: symlinks to files would NOT be matched
- **test**: `ln -s real.md link.md` then `cpsafe '*.md'` - does link.md get copied?
- **checked**: `-f` returns true for symlinks that point to regular files
- **BUT**: `-f` returns false for broken symlinks
- **verdict**: works for valid symlinks; broken symlinks silently skipped (acceptable)

### 5. "symlinks should be validated at location, not target"
- **assumption**: `realpath` of symlink parent + basename for boundary check
- **what if opposite?**: could follow symlink to validate actual target
- **evidence**:
  - rmsafe removes symlink itself, not target
  - boundary check prevents symlinks that point outside repo
  - matches git behavior
- **verdict**: assumption correct per design

### 5. "shared output.sh is better than inline"
- **assumption**: extract of output helpers reduces duplication
- **what if opposite?**: each skill could have its own output style
- **evidence**:
  - vision explicitly requests consistent tree output format
  - DRY is appropriate here (>50 lines shared)
- **verdict**: assumption valid per requirements

## simpler approaches considered

- could skip glob support entirely (just fix "first match only" bug) - but vision requested glob support
- could use find instead of bash globstar - but adds external dependency and different syntax

## issues identified

1. **eval safety**: relies on for-loop syntax to prevent injection - works but fragile
2. **brace expansion**: `{a,b}.md` not detected as glob - acceptable limitation
3. **broken symlinks**: silently skipped in glob matches - acceptable behavior

## potential improvements (not blockers)

- add `printf '%q'` escaping for defense-in-depth on eval
- add `{` to glob detection if brace expansion needed
- add `-L` alongside `-f` to explicitly handle symlinks

## conclusion

assumptions hold but are fragile in places. the implementation works correctly for documented use cases. edge cases identified are acceptable limitations, not bugs.
