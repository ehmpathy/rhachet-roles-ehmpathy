# self-review: has-questioned-requirements

## requirements questioned

### 1. glob support for cpsafe

**who said this was needed?**
- the wish originated from observation that `cpsafe --from '*.md'` only copied `0.wish.md` when 20+ files matched
- this is a clear defect: the user's intent was to copy all matched files

**what evidence supports this?**
- the repro steps in the wish clearly demonstrate the bug
- the current code treats `FROM` as a single file path (line 109: `if [[ ! -e "$FROM" ]]`)

**what if we didn't do this?**
- users would need to run cpsafe N times for N files
- error-prone, tedious, defeats the purpose of a bulk operation tool

**verdict**: ✅ clearly needed, well-supported by evidence

### 2. glob support for mvsafe

**who said this was needed?**
- the wish says "we probably have this defect for all the fileops; fix them"
- code review of mvsafe confirms identical pattern: treats source as single file

**what evidence supports this?**
- code review: same structure as cpsafe (line 109: `if [[ ! -e "$FROM" ]]`)
- logical consistency: if cpsafe supports globs, mvsafe should too

**what if we didn't do this?**
- inconsistent behavior across related tools
- users expect `mv '*.md' dir/` to work like `cp '*.md' dir/`

**verdict**: ✅ needed for consistency

### 3. glob support for rmsafe — QUESTIONED

**who said this was needed?**
- the wish says "all the fileops" but doesn't explicitly mention rmsafe
- i assumed rmsafe was included

**what evidence supports this?**
- code review: same pattern (treats path as single file)
- consistency argument applies

**what if we didn't do this?**
- users run rmsafe multiple times (safer!)
- BUT: `git rm '*.md'` already supports globs, so precedent exists

**is the scope too large?**
- rmsafe with globs is more dangerous than cpsafe/mvsafe
- accidental `rmsafe --path '*.md'` deletes many files
- HOWEVER: all files are within git repo, all recoverable

**could we achieve the goal in a simpler way?**
- option 1: skip rmsafe glob support entirely (safest)
- option 2: add glob support with confirmation prompt (verbose)
- option 3: add glob support same as others (consistent)

**verdict**: ⚠️ needs wisher confirmation — should rmsafe support globs?

my recommendation: YES, add glob support for consistency. the repo boundary check + git recoverability provides sufficient safety net. but the wisher should confirm.

### 4. single-quote examples in docs/permissions

**who said this was needed?**
- the wish explicitly requests this: "update the examples to use `'...*'` to prevent unix shell glob expansion"

**what evidence supports this?**
- shell behavior: unquoted `*.md` expands before the skill sees it
- single quotes pass the literal string `*.md` to the skill
- double quotes may still allow expansion in some contexts

**what if we didn't do this?**
- documentation shows patterns that silently break
- users copy-paste examples and get unexpected behavior

**verdict**: ✅ clearly needed, explicitly requested
