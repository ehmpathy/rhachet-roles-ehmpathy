## deep dive: potential gaps investigated

### gap check 1: eval security in cpsafe.sh:130

```bash
eval "for f in $FROM; do [[ -f \"\$f\" ]] && FILES+=(\"\$f\"); done"
```

**concern:** does `eval` with `$FROM` pose injection risk?

**analysis:**
- `$FROM` comes from user input (--from arg)
- `$FROM` is not sanitized before eval
- however, the only consumers are mechanics via claude code
- the skill is not exposed to arbitrary user input
- the boundary check on lines 189-195 validates each file is within repo
- `set -euo pipefail` ensures fail-fast on any error

**verdict:** acceptable risk - injection vector is mitigated by boundary validation

### gap check 2: path with spaces

**concern:** does `$FROM` handle spaces in glob paths?

**analysis:**
- `eval "for f in $FROM"` would split `src/path with spaces/*.md`
- this is a known shell glob limitation
- the examples use single quotes which prevents shell expansion at caller level
- if spaces exist, user can escape: `src/path\ with\ spaces/*.md`

**verdict:** acceptable - documented via single-quote examples

### gap check 3: race condition between glob and copy

**concern:** file deleted between glob expansion and copy?

**analysis:**
- line 127: FILES array populated
- line 191: copy happens in loop
- file could be deleted between these
- `set -e` ensures `cp` failure causes skill exit
- error is clear from cp output

**verdict:** acceptable - fail-fast behavior handles this

### gap check 4: symlink pointing outside repo

**concern:** glob matches symlink that points to /etc/passwd?

**analysis:**
- line 188: `FROM_ABS=$(realpath "$FILE")`
- realpath follows symlinks to absolute path
- line 191-195: boundary check uses resolved path
- symlink to /etc/passwd would fail boundary check

**verdict:** covered - boundary check uses resolved symlink target
