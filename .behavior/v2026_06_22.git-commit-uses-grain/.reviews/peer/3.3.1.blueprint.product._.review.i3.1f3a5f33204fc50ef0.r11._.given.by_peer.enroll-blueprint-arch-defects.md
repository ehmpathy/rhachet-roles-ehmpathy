вФВ  вФВ  ### рЯФі blocker 1 вАФ `@all reset` stores redundant entries (state pollution + confusing `get --org`)
вФВ  вФВ  
вФВ  вФВ  **location**: blueprint codepath tree вЖТ `git.commit.uses.org.sh` вЖТ `allow/@all` + `block/@all`
вФВ  вФВ  
вФВ  вФВ  **defect**: the blueprint specifies `@all reset` as:
вФВ  вФВ  > "1. reset: set ALL orgs to 'blocked'; 2. default: set @all to 'blocked'"
вФВ  вФВ  
вФВ  вФВ  this produces a state file like:
вФВ  вФВ  ```jsonc
вФВ  вФВ  { "orgs": { "@all": "blocked", "ehmpathy": "blocked", "ahbode": "blocked" } }
вФВ  вФВ  ```
вФВ  вФВ  
вФВ  вФВ  `ehmpathy: blocked` and `ahbode: blocked` are **redundant** вАФ `@all: blocked` already covers them via the effective state algorithm. the entries add noise forever. worse, `get --org` would output redundant rows that imply explicit overrides where none exist.
вФВ  вФВ  
вФВ  вФВ  **correct semantics** (matches `radio.uses.org.sh` reference pattern too):
вФВ  вФВ  - `block --org @all` вЖТ delete all individual org entries, set `@all: blocked`
вФВ  вФВ  - `allow --org @all` вЖТ delete all individual org entries, set `@all: allowed`
вФВ  вФВ  - result: only meaningful overrides exist in state; `get --org` shows only explicit distinctions
вФВ  вФВ  
вФВ  вФВ  **impact**: state file grows unboundedly; `get --org` shows noise vs signal; `get --org ehmpathy` would show `ehmpathy: blocked (explicit)` when it's actually `(from @all)` вАФ misleading
вФВ  вФВ  
вФВ  вФВ  **fix**: revise `@all` codepath to clear all individual entries before writing new `@all` value.
вФВ  вФВ

вФВ  вФВ  ### рЯФі blocker 2 вАФ `check_org_blocker()` return signature mismatch
вФВ  вФВ  
вФВ  вФВ  **location**: blueprint codepath tree вЖТ `git.commit.operations.sh` вЖТ `check_org_blocker()`
вФВ  вФВ  
вФВ  вФВ  **defect**: blueprint says `check_org_blocker()` "returns blocked/allowed/unset" вАФ a **string return**. but the calling site in `git.commit.set.sh` uses:
вФВ  вФВ  ```bash
вФВ  вФВ  check_global_blocker || exit 2
вФВ  вФВ  check_org_blocker || exit 2     # вЖР this pattern requires exit codes, not strings
вФВ  вФВ  ```
вФВ  вФВ  
вФВ  вФВ  `check_global_blocker()` uses exit codes (0 = not blocked, 2 = blocked) + sets `GLOBAL_BLOCK_REASON`. `check_org_blocker()` must follow the same pattern. the "unset" state (when both `@all` and the org are unset) needs a **distinct error message** from "blocked" but must also exit 2 вАФ this is not expressible as just a return code. it requires `ORG_BLOCK_REASON` env var or equivalent.
вФВ  вФВ  
вФВ  вФВ  **impact**: if implemented as string-return, `check_org_blocker || exit 2` will always pass (bash strings are truthy), silently skipping the org check вАФ a silent security gap.
вФВ  вФВ  
вФВ  вФВ  **fix**: revise `check_org_blocker()` signature to match `check_global_blocker()`:
вФВ  вФВ  - exit 0 = allowed (continue)
вФВ  вФВ  - exit 2 = blocked or unset (halt)
вФВ  вФВ  - sets `ORG_BLOCK_REASON` with specific message ("org blocked" vs "org unset, failfast required")
вФВ  вФВ

вФВ  вФВ  ### рЯФі blocker 3 вАФ `get_org_from_keyrack()` placed in wrong domain file
вФВ  вФВ  
вФВ  вФВ  **location**: blueprint filediff tree вЖТ `git.commit.operations.sh` (proposed `[+] get_org_from_keyrack()`)
вФВ  вФВ  
вФВ  вФВ  **defect**: `git.commit.operations.sh` is a git behavioral commit operations file (behavioral commits, ancestor branch detection, global blocker check). reading `.agent/keyrack.yml` is a **keyrack domain concern**, not a git commit concern.
вФВ  вФВ  
вФВ  вФВ  critically: `keyrack.operations.sh` already exists in the same directory and handles keyrack reads (`fetch_github_token()`). placing `get_org_from_keyrack()` in `git.commit.operations.sh` creates a mixed-responsibility file and misses the obvious placement.
вФВ  вФВ  
вФВ  вФВ  **impact**: `git.commit.operations.sh` now has 3 unrelated domains (git history, global blocker, keyrack reads). `get_org_from_keyrack()` is not discoverable to future engineers looking for keyrack operations. violates bounded contexts.
вФВ  вФВ  
вФВ  вФВ  **fix**: move `get_org_from_keyrack()` to `keyrack.operations.sh`. update blueprint filediff/codepath accordingly (`[~] keyrack.operations.sh`).

вФВ  вФВ  ### рЯФі blocker 4 вАФ `del --org @all` behavior undefined (behavior hazard)
вФВ  вФВ  
вФВ  вФВ  **location**: blueprint codepath tree вЖТ `git.commit.uses.org.sh` вЖТ `del` subcommand
вФВ  вФВ  
вФВ  вФВ  **defect**: blueprint only covers `del --org $specific_org` вАФ "remove org entry, defer to @all". it says nothing about `del --org @all`.
вФВ  вФВ  
вФВ  вФВ  if a human runs `del --org @all`, the `@all` key is deleted from the state file. all subsequent commits will hit the "unset" failfast condition вАФ **every commit silently breaks** until a human notices and sets `@all` again.
вФВ  вФВ  
вФВ  вФВ  **impact**: behavior hazard вАФ `del --org @all` is a footgun with no protection or clear semantics.
вФВ  вФВ  
вФВ  вФВ  **fix**: explicitly handle in codepath tree:
вФВ  вФВ  - option A: `del --org @all` вЖТ error: "cannot del @all; use allow or block --org @all"
вФВ  вФВ  - option B: `del --org @all` вЖТ clears entire state file (full reset), with explicit warning output
вФВ  вФВ  
вФВ  вФВ  add test case: `given: human runs del --org @all вЖТ then: clear error or explicit semantics`.
вФВ  вФВ

вФВ  вФВ  ### рЯФі blocker 5 вАФ `--org` optional-value parsing unspecified in dispatcher
вФВ  вФВ  
вФВ  вФВ  **location**: blueprint codepath tree вЖТ `git.commit.uses.sh` вЖТ `[~] arg parse` вЖТ `[+] --org flag detection`
вФВ  вФВ  
вФВ  вФВ  **defect**: the dispatcher currently strips `--global` and passes remaining args through. adding `--org` introduces an **optional-value flag**:
вФВ  вФВ  - `git.commit.uses get --org` вЖТ `--org` has no value (show all)
вФВ  вФВ  - `git.commit.uses get --org ehmpathy` вЖТ `--org` has value `ehmpathy`
вФВ  вФВ  
вФВ  вФВ  the blueprint shows `[+] --org flag detection` but doesn't specify how the dispatcher extracts the optional value. naive `--global`-style detection (just check for presence of `--org`) would strip `--org` and lose the org value before dispatch.
вФВ  вФВ  
вФВ  вФВ  **impact**: `git.commit.uses.org.sh` would receive args without the org name, making `get --org ehmpathy` behave like `get --org` вАФ wrong org resolution.
вФВ  вФВ  
вФВ  вФВ  **fix**: specify in dispatcher codepath how `--org` with optional value is extracted and forwarded. example: detect `--org` presence for dispatch decision, but pass `--org <value>` through to `git.commit.uses.org.sh` intact (don't strip it like `--global`).
