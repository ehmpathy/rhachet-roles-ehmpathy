# blocker: 5.3.verification

## what blocks me

the route system treats peer review constraint errors (exit code 2) as blockers when they should be treated as "not applicable."

this behavior only modifies:
- shell scripts (`.sh`)
- markdown briefs (`.md`)
- permission configs (`.jsonc`)
- package configs (`.json`, `.yaml`)

no TypeScript files were changed. the peer reviews in 5.3.verification.guard are scoped to `**/*.ts` files:
- r1-r8: all scope to `--paths-with '**/*.ts' --join intersect`

when the intersection of `since-main` diff with `**/*.ts` yields zero files, the reviewer exits with code 2 (constraint). the route system interprets this as "blocked by constraints" instead of "passed (not applicable)."

## what i tried

1. ran `rhx route.stone.set --stone 5.3.verification --as passed` multiple times
2. ran `rhx route.stone.set --stone 5.3.verification --as arrived`
3. both fail with same error: `passage blocked` → `blocked by constraints` → `exit code: 2 ✋`

the level 3 enrolled reviews (r9-r11) all pass:
- r9: enroll-verif-snapshot-coverage → approved, 0 blockers, 0 nitpicks
- r10: enroll-verif-snapshot-blemishes → approved, 0 blockers, 0 nitpicks
- r11: enroll-verif-test-intent → approved, 0 blockers, 0 nitpicks

the level 1 reviews (r1-r8) all show `✋ constraint` which is correct - but the guard validation re-runs them and exits with code 2.

## what i need

human approval to bypass the guard, since the peer reviews correctly report "zero files to review" for their TypeScript scope, which does not apply to this shell-only behavior.

alternatively: fix the route system to treat exit code 2 (constraint) from reviews as "passed" when the constraint is "zero files match the scope."
