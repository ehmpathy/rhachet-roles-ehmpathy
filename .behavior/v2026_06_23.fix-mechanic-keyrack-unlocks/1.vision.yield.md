# vision: fix mechanic keyrack unlocks

## the outcome world

### before

```
🐢 hey, let me push this commit for you...

[skill runs: rhx keyrack unlock --owner ehmpath --env prep]

⏳ wait for AWS SSO browser auth...
⏳ ...still wait...
⏳ human, are you there?
⏳ ...
```

the skill only needs `EHMPATHY_SEATURTLE_GITHUB_TOKEN` (stored in os.secure, no browser auth required), but the full keyrack unlock also attempts to unlock aws.config keys that require SSO browser auth. the skill hangs, wait for a key it doesn't even need.

### after

```
🐢 hey, let me push this commit for you...

[skill runs: rhx keyrack unlock --owner ehmpath --env prep --key EHMPATHY_SEATURTLE_GITHUB_TOKEN]

✅ unlocked 1 key (os.secure, no browser auth)

🐢 cowabunga! pushed to origin/vlad/my-feature
```

the skill unlocks only what it needs. no wait. no browser auth popup. instant.

### the "aha" moment

when you realize the skill finished in 2 seconds instead of hung indefinitely wait for you to click "allow" in a browser you didn't even know was open.

## user experience

### usecases

| usecase | keys needed | storage type | browser auth? |
|---------|-------------|--------------|---------------|
| git.commit.push | EHMPATHY_SEATURTLE_GITHUB_TOKEN | os.secure | no |
| git.release | EHMPATHY_SEATURTLE_GITHUB_TOKEN | os.secure | no |
| git.repo.test (integration) | varies by repo | mixed | maybe |

### contract inputs & outputs

**current (broad unlock):**
```bash
rhx keyrack unlock --owner ehmpath --env prep
# unlocks ALL keys in prep, includes aws.config keys
# hangs if any key needs browser auth
```

**proposed (narrow unlock):**
```bash
rhx keyrack unlock --owner ehmpath --env prep --key EHMPATHY_SEATURTLE_GITHUB_TOKEN
# unlocks ONLY the specified key
# no hang if other keys need browser auth
```

### timeline

1. skill needs a credential
2. skill calls `rhx keyrack unlock --key SPECIFIC_KEY`
3. only that key unlocked (instant if os.secure)
4. skill uses credential
5. done

vs before:

1. skill needs a credential
2. skill calls `rhx keyrack unlock`
3. ALL keys attempted, aws.config keys trigger SSO
4. **hang wait for browser auth**
5. human notices popup, clicks allow
6. skill finally continues

## mental model

### how users would describe this

> "skills now unlock only the keys they actually need, so i don't get random browser auth popups for credentials the skill isn't even use."

### analogies

- **before**: ask for the whole keychain when you only need the car key
- **after**: ask for just the car key

- **before**: unlock every door in the build to get to one room
- **after**: unlock just the door you need

### terms

| our term | sense |
|----------|-------|
| narrow unlock | unlock only specific keys needed |
| broad unlock | unlock entire keyrack (all keys for env) |
| os.secure | credential stored in OS keychain, no browser auth |
| aws.config | credential that requires AWS SSO browser auth |

## evaluation

### how well does it solve the goals?

| goal | before | after |
|------|--------|-------|
| no unnecessary browser auth | fails - all keys unlocked | passes - only needed keys |
| skills run without hang | fails - hangs on aws.config | passes - skips unneeded keys |
| human doesn't need to babysit | fails - must click browser popup | passes - runs autonomously |

### pros

- skills run faster (no wait for unused credentials)
- no surprise browser auth popups
- clear about what credentials a skill actually needs
- self-document: skill code shows exactly which keys it depends on

### cons

- skills must declare their key dependencies explicitly
- if a skill needs a new key, must update the unlock call
- `git.repo.test` is harder: tests may need any credential (see open questions)

### edge cases

| edge case | pit of success |
|-----------|----------------|
| skill needs multiple keys | can pass `--key` multiple times: `--key KEY1 --key KEY2` |
| skill doesn't know which keys tests need | `git.repo.test` may need special treatment (see open questions) |
| key doesn't exist in keyrack | fails fast with clear error |

## what is awkward?

### git.repo.test tension

`git.repo.test` runs arbitrary integration tests. those tests may need any credential declared in the repo's keyrack.yml. we can't know ahead of time which keys.

options:
1. **keep broad unlock for tests** — accept that test runner is special
2. **read keyrack.yml and unlock all declared keys** — still unlocks aws.config keys
3. **add a test-keys declaration** — new mechanism, more complexity

the first option is simplest and probably fine — test runs are typically interactive anyway, human is present to click browser auth if needed.

### multiple keys

if a skill needs N keys, the unlock call gets verbose:
```bash
rhx keyrack unlock --owner ehmpath --env prep \
  --key KEY1 \
  --key KEY2 \
  --key KEY3
```

acceptable tradeoff for clarity about dependencies.
