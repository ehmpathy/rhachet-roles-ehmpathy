# vision: git.branch.rebase skills

## the outcome world

### day-in-the-life with git.branch.rebase

the human delegates a task to mechanic. mechanic works on a feature branch and creates several commits. meanwhile, main has moved forward. before push, mechanic needs to rebase onto main to keep history clean.

**before git.branch.rebase skills:**
```
human: "rebase onto main before you push"
mechanic: "i cannot run git rebase - it's a destructive operation.
           please run: git pull origin main --rebase"
human: *switches context, runs command, fixes conflicts*
human: "ok done, continue"
mechanic: *resumes work*
```

**after git.branch.rebase skills:**
```
human: "rebase onto main before you push"
mechanic: *runs git.branch.rebase begin --mode plan*
mechanic: "here's what the rebase will do: [preview]"
mechanic: *runs git.branch.rebase begin --mode apply*
mechanic: *encounters conflict, sees the diff, fixes it*
mechanic: *runs git.branch.rebase continue*
mechanic: "rebased 3 commits onto origin/main. ready to push."
```

### the "aha" moment

when mechanic can rebase on demand — human asks "rebase onto main", mechanic handles it all: fetch, rebase, conflict settlement, continue. no context switch for human.

**note**: rebase requires push permission grant. whether mechanic rebases proactively or when asked is not enforced — only that the grant exists.

**trust model**: the robot wrote the whole PR. we trust it to rebase successfully.

## user experience

### usecases

| usecase | command | when |
|---------|---------|------|
| rebase onto main before push | `git.branch.rebase begin` | feature complete, ready to merge |
| continue after conflict settlement | `git.branch.rebase continue` | mechanic fixed conflicts |
| abort if stuck | `git.branch.rebase abort` | stuck state |

### contract inputs & outputs

```bash
# begin a rebase onto origin/main (implicit target)
rhx git.branch.rebase begin                    # plan mode (preview)
rhx git.branch.rebase begin --mode apply       # execute rebase

# continue after mechanic fixes conflicts
rhx git.branch.rebase continue

# abort a stuck rebase
rhx git.branch.rebase abort
```

### what it does under the hood

`git.branch.rebase begin --mode apply` runs:
```bash
git pull origin main --rebase
```

if conflicts occur, mechanic:
1. sees the conflict markers in the files
2. edits files to settle the conflicts
3. stages the settled files
4. runs `git.branch.rebase continue`

### timeline

1. human grants push permission via `git.commit.uses set --quant N --push allow`
2. mechanic works, creates commits
3. mechanic runs `git.branch.rebase begin --mode plan` → sees preview
4. mechanic runs `git.branch.rebase begin --mode apply` → executes
5. if conflicts: mechanic sees them, fixes them, runs `git.branch.rebase continue`
6. mechanic pushes with force-with-lease (safe force push after rebase)

## mental model

### how users would describe this

> "the mechanic can rebase my branch onto main, just like it can commit and push. it even handles conflicts. i give it permission once and it keeps the branch clean."

### analogies

- like you hand your assistant the keys to reorganize the file cabinet (rebase = reorganize commits)
- plan mode = show you the reorganization plan before you move files
- apply mode = actually move the files
- conflict settlement = your assistant notices two files need the same spot and figures out where each goes

### terms

| user says | we say |
|-----------|--------|
| "rebase onto main" | `git.branch.rebase begin` |
| "continue the rebase" | `git.branch.rebase continue` |
| "nevermind, undo" | `git.branch.rebase abort` |

## evaluation

### how well does it settle the goals?

| goal | solution | quality |
|------|----------|---------|
| mechanic can rebase autonomously | git.branch.rebase begin/continue | ✅ full |
| human controls permission | tied to push permission | ✅ natural fit |
| safe by default | plan mode default | ✅ safe |
| handles conflicts gracefully | mechanic sees and fixes them | ✅ fully autonomous |

### pros

- **no context switch**: human doesn't leave their work to run git commands
- **safe defaults**: plan mode previews before destructive changes
- **permission-gated**: only when push is allowed (rebase rewrites history = needs push)
- **conflict settlement**: mechanic can see and fix conflicts autonomously

### edgecases and pit of success

| edgecase | how we handle it |
|----------|------------------|
| no push permission | reject: "rebase requires push permission" |
| main/master branch | reject: "cannot rebase main/master" |
| dirty work tree | reject: "unstaged changes exist" |
| no commits to rebase | report: "branch is up to date, no rebase needed" |
| network unavailable for fetch | report fetch error, suggest retry |
| rebase already in progress | detect and guide to continue/abort |
| conflicts in rebase | mechanic sees them, fixes them, continues |
| force push to shared branch | use `--force-with-lease` (safer) |

## open questions & assumptions

### assumptions

1. **push permission implies rebase permission** - makes sense since rebase requires force push
2. **plan mode is sufficient preview** - shows commits to be rebased, target
3. **plan output surfaces to human** - mechanic shares preview for transparency
4. **mechanic can fix conflicts** - mechanic sees conflict markers and settles them
5. **simple rebases only** - only `git pull origin main --rebase`, no arbitrary targets
6. **force-with-lease is safe enough** - prevents accidental overwrite of others' work
7. **fetch is implicit** - `git pull` fetches before rebase
8. **rebase doesn't auto-push** - mechanic decides when to push after rebase
9. **continue executes immediately** - no plan mode for continue; intent is clear
10. **merge commits flattened** - git default behavior; topology not preserved
11. **rebased commits signed by mechanic** - if repo requires signed commits, mechanic signs them

### questions resolved

all questions were self-answerable via wish, logic, or wisher clarification:

| question | source | answer |
|----------|--------|--------|
| permission model | wisher's words | tie to push permission |
| conflict settlement | wisher clarification | mechanic fixes conflicts |
| target specification | wisher clarification | implicit origin/main only, no --onto |
| command name | wisher's words | "git.branch.rebase begin" |
| continue plan mode | logic | no, execute immediately |

no questions require wisher input.

## what is awkward?

### feels off

- **name choice**: `git.branch.rebase begin` vs just `git.branch.rebase` - the "begin" feels redundant but distinguishes from "continue"

### trust model

- rebase is inherently dangerous - but the robot wrote the whole PR
- if we trust it to write the code, we trust it to rebase
- permission gate still required

### scope limitation

- only rebases onto origin/main (or base branch)
- no arbitrary `--onto` targets in v1
- can expand scope later if needed
