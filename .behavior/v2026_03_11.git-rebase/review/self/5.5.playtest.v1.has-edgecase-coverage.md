## what could go wrong?

### permission errors

| failure mode | playtest | step |
|--------------|----------|------|
| no push permission granted | yes | edge case 5 |
| permission revoked mid-rebase | no | out of scope (rare, manual recovery) |

### branch errors

| failure mode | playtest | step |
|--------------|----------|------|
| on main/master branch | yes | edge case 6 |
| on orphan branch | no | rare, git rejects anyway |
| branch deleted remotely | no | rare, git rejects anyway |

### work tree errors

| failure mode | playtest | step |
|--------------|----------|------|
| unstaged changes | yes | edge case 7 |
| staged but uncommitted | no | same guard as unstaged |
| untracked files | no | git allows rebase with untracked |

### rebase state errors

| failure mode | playtest | step |
|--------------|----------|------|
| no rebase in progress (continue) | yes | edge case 10 |
| no rebase in progress (abort) | yes | edge case 11 |
| rebase already in progress (begin) | yes | edge case 9 |
| conflicts not settled | yes | edge case (implicit in happy path 3) |

### git errors

| failure mode | playtest | step |
|--------------|----------|------|
| network unavailable | no | transient, simple error message |
| corrupt git state | no | rare, requires manual git recovery |
| merge commits in history | no | git handles, may require --rebase-merges |

## gap analysis

### not tested but acceptable

| gap | rationale |
|-----|-----------|
| permission revoked mid-rebase | rare, requires manual intervention anyway |
| orphan branch | rare, git rejects with clear error |
| branch deleted remotely | rare, git rejects with clear error |
| staged but uncommitted | same guard catches this |
| network unavailable | transient, simple error |
| corrupt git state | rare, requires git expert |
| merge commits | git-level behavior, not skill behavior |
| 100+ commits | performance test, not correctness |
| special chars in commit | git responsibility |

all gaps are either:
- rare conditions that git handles
- transient errors with simple messages
- performance concerns (not correctness)

- branch up to date → edge case 8 (line 195-215)
- single commit → covered by happy path 2 (same logic)

- 0 vs 1+ commits ahead → edge case 8 vs happy path 1,2
- 0 vs 1+ conflicts → happy path 2 vs happy path 3
