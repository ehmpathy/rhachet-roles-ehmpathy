## question: can git.branch.rebase.operations.sh be removed?

**examined.** blueprint lists 4 functions:
- is_rebase_in_progress
- get_conflict_files
- get_commits_left
- get_current_commit

i implemented 12 functions. the extra 8 are:
- get_total_commits (progress display)
- get_conflict_commit (show which commit caused conflict)
- get_commits_to_rebase (preview)
- get_behind_count (preview)
- get_current_branch (branch guard)
- is_base_branch (branch guard)
- is_worktree_dirty (worktree guard)
- is_push_allowed (permission guard)

**question:** should guards be inline in begin.sh instead?

**analysis:** only begin.sh uses is_push_allowed, is_base_branch, is_worktree_dirty. the codepath tree shows these as separate nodes under begin.sh, not as calls to operations.sh.

**verdict:** implementation diverged from blueprint but result is cleaner. operations.sh provides single source for all shared state queries. if we deleted operations.sh, we'd inline these in begin.sh, but they're reusable if we ever add more skills. keep.

## question: can git.branch.rebase.abort.sh be removed?

**examined.** the wish says:
> "git.branch.rebase begin and git.branch.rebase continue"

abort is NOT in the wish. it's added by the vision.

**question:** if we deleted abort and had to add it back, would we?

**analysis:**
- rebase can enter stuck states (corrupt .git/rebase-merge, unresolvable conflicts)
- without abort, human must manually run `git rebase --abort`
- defeats purpose of autonomous mechanic

**verdict:** yes, we'd add it back. keep abort.
