## inferences taken as requirements

### the wisher SAID

1. "similar to git.commit and git.add" — follow those patterns
2. "whenever git push is allowed, allow rebase" — tie permission to push
3. "git.rebase begin" and "git.rebase continue" — these TWO commands

### i INFERRED (not stated by wisher)

| what i added | why i added it | justification |
|--------------|----------------|---------------|
| plan/apply mode | pattern from git.commit.set | wisher said "similar to git.commit" |
| --onto flag | git rebase needs a target | essential for the command to work |
| abort command | rebase has abort state | implied by "continue" — if you can continue, you might need to abort |
| force-with-lease | safety for push after rebase | best practice, not stated |
| fetch before rebase | safety for stale refs | best practice, not stated |
| all edgecases | pit of success | not stated but good design |

### are these inferences valid?

**plan/apply mode**: wisher said "similar to git.commit" which has plan/apply. VALID inference.

**--onto flag**: rebase requires a target. wisher didn't specify syntax but it's essential. VALID inference.

**abort command**: wisher mentioned "continue", implying a multi-step process. abort is the escape hatch. VALID inference.

**force-with-lease, fetch, edgecases**: these are best practices, not requirements. wisher didn't ask for them. could be SCOPE CREEP.

### verdict

the core features (begin, continue, tied to push) match the wish.
the safety features (force-with-lease, fetch, edgecases) are defensive additions.
the wisher didn't ask for them but they prevent bad outcomes.

should we include them? yes — but flag as "best practices, not requirements".
