## deep questions before requirements

### why does mechanic need rebase at all?

the wish assumes mechanic should rebase. but what problem does this actually solve?

- human asks "rebase onto main" → mechanic says "i can't, please do it"
- is that actually a problem? human runs one command.
- **alternative**: mechanic could warn "your branch is behind main" and let human decide

**but**: the wish explicitly asks for this. the human wants the convenience. that's evidence enough to proceed.

### what happens when rebase goes wrong?

the vision shows happy path. but:
- what if rebase creates a mess and human can't fix conflicts?
- what if mechanic's continue fails after human's fix?
- what if the rebase creates unexpected merge results?

**gap identified**: should add guidance that if conflict settlement fails repeatedly, mechanic should **surface to human** rather than retry endlessly.

### is force-with-lease actually safe?

assumed it prevents overwrite. but:
- force-with-lease checks ref we last fetched, not current remote
- if remote changed while mechanic rebased, force-with-lease might still overwrite

**gap identified**: should `git fetch` before rebase to ensure we have fresh refs.

### why `--onto`?

used `--onto` because git uses it. but is it the clearest name?
- `--onto origin/main` — clear: rebase onto target
- `--base origin/main` — alternative, but less git-like
- positional arg — git allows this but flags are clearer

**verdict**: `--onto` is correct. matches git terminology, explicit, no ambiguity.

### does continue need plan mode?

what would plan mode show for continue?
- "will continue rebase" — not useful
- shows remaining commits — maybe useful

**gap identified**: plan mode for continue is questionable. could simplify by making continue always execute immediately. **flag for wisher**.

### what if rebase succeeds immediately?

the timeline assumes conflicts happen. but most rebases succeed cleanly.
- step 4: rebase → success → step 6: push
- conflicts are the exception, not the rule

**verdict**: vision is fine. conflicts are handled but not assumed.

### what if human runs git commands between begin and continue?

if human runs `git checkout other-branch` or `git stash` between begin and continue:
- mechanic's continue might fail or act on wrong state
- should we detect this?

**verdict**: this is an edge case. git itself handles this via `.git/rebase-merge/` state. rely on git's state machine.

### should abort require permission?

abort undoes work but doesn't push. is it safe?
- abort is a recovery mechanism
- blocking abort could trap mechanic in stuck state
- abort should always be allowed if rebase was allowed

**verdict**: abort does not need separate permission. if begin was allowed, abort is always safe.

## critique as if reviewing junior's work

### is the entire premise flawed?

**rebase is dangerous** — why should an AI rewrite git history? most senior devs are cautious about rebase. giving this power to mechanic feels like a trust leap.

**counter**: the wish explicitly asks for this. the human trusts mechanic with push, so they trust it with rebase. the permission gate protects against accidents.

**verdict**: premise holds if permission is explicitly granted.

### is the scope too large for v1?

the vision includes: begin, continue, abort, plan mode, permission checks, fetch, conflict handling, force-with-lease.

**minimum viable**: just `git.rebase --onto X --mode apply`. no plan mode. no continue. no abort.

**counter**: the wish asks for begin AND continue. both are required for the vision.

**verdict**: scope matches wish. cannot reduce without ignoring wish.

### are we solving the wrong problem?

**alternative framing**: the real problem might be push fails due to divergence, not "mechanic can't rebase".

**alternative solution**: when push fails, suggest rebase to human. don't give mechanic rebase power.

**counter**: the wish explicitly says "allow the robot to rebase". the human wants mechanic to have this power, not just suggest it.

**verdict**: we're solving the problem the wisher asked us to solve.

### why rebase instead of merge?

the vision assumes rebase is correct but doesn't justify why.

**answer**: rebase keeps linear history. merge creates merge commits. the choice is a team preference, not something we decide.

**verdict**: mechanic follows human's instruction. if human says "rebase", mechanic rebases. if human says "merge", that's a different skill.
