## deeper convention analysis

### why skill names convention holds

**extant pattern in git.commit:**
```
git.commit.set.sh   → verb "set" = mutate state
git.commit.push.sh  → verb "push" = send to remote
git.commit.bind.sh  → verb "bind" = establish constraint
git.commit.uses.sh  → noun "uses" = meter state
```

**blueprint pattern in git.branch.rebase:**
```
git.branch.rebase.begin.sh    → verb "begin" = start operation
git.branch.rebase.continue.sh → verb "continue" = resume operation
git.branch.rebase.abort.sh    → verb "abort" = cancel operation
```

**why it holds:** the verbs "begin", "continue", "abort" are explicit imperative actions. they describe what the skill does, not what it is. this matches the extant pattern where verbs describe the action.

### why test file names convention holds

**extant pattern:**
- `git.commit.set.integration.test.ts` (not `test.git.commit.set.ts`)
- collocation: test file lives next to source file
- suffix pattern: `{skill}.integration.test.ts`

**blueprint pattern:**
- `git.branch.rebase.begin.integration.test.ts`
- `git.branch.rebase.continue.integration.test.ts`
- `git.branch.rebase.abort.integration.test.ts`

**why it holds:** follows exact same structure. skill name first, then `.integration.test.ts` suffix. enables glob `*.integration.test.ts` to find all integration tests.

### why output format convention holds

**extant output.sh functions:**
```bash
print_turtle_header()    # 🐢 {message}
print_tree_start()       # 🐚 {command}
print_tree_branch()      # ├─ {label}: {value}
print_tree_leaf()        # └─ {label}: {value}
print_tree_error()       # └─ error: {message}
print_instruction()      # guidance text
```

**git.branch.rebase usage:**
```bash
source "$COMMIT_SKILL_DIR/output.sh"
print_turtle_header "righteous!"
print_tree_start "git.branch.rebase begin --mode apply"
echo "   ├─ rebase"
```

**why it holds:** git.branch.rebase sources output.sh from git.commit, uses the same print_* functions. manual echo lines for tree structure follow same indentation pattern (`   ├─`). no new output utilities introduced.

### why argument parse convention holds

**extant rhachet passthrough (git.commit.set.sh):**
```bash
    --repo|--role|--skill)
      # rhachet passthrough args - ignore
      shift 2
      ;;
```

**git.branch.rebase usage:**
```bash
    --skill|--repo|--role)
      shift 2
      ;;
```

**why it holds:** both shift 2 for these args (arg + value). order in case statement differs but behavior identical. rhachet passes these to all skills, skills must ignore them.

### why mode flags convention holds

**extant git.commit.set.sh:**
```bash
MODE="plan"  # default to plan mode
...
--mode)
  MODE="$2"
  shift 2
  ;;
...
if [[ "$MODE" != "plan" && "$MODE" != "apply" ]]; then
  echo "error: --mode must be 'plan' or 'apply'"
```

**git.branch.rebase.begin.sh:**
```bash
MODE="plan"  # default to plan mode
...
--mode)
  MODE="$2"
  shift 2
  ;;
```

**why it holds:** same default (`plan`), same parse, same value set (`plan|apply`). pattern directly reused.

### why turtle vibes phrases convention holds

**extant phrases from git.commit.set.sh:**
- `"heres the wave..."` — preview/plan mode
- `"righteous!"` — success
- `"cowabunga!"` — success with push
- `"bummer dude..."` — constraint error
- `"hold up dude..."` — validation nudge

**git.branch.rebase phrases:**
- `"heres the wave..."` — plan mode preview
- `"righteous!"` — success (apply mode clean)
- `"hold up dude..."` — guard errors, conflicts
- `"already up to date"` — no-op case
- `"no worries dude"` — abort success

**why it holds:** reuses extant phrases for same semantics. "heres the wave" for previews, "righteous" for success, "hold up" for errors. new phrase "no worries dude" for abort is consistent with chill seaturtle vibes.

## divergences analyzed

### divergence 1: "no worries dude" phrase

**extant:** not found in git.commit
**introduced:** git.branch.rebase.abort.sh

**analysis:** this is a new phrase but fits the seaturtle personality. abort is a unique case — user intentionally cancels, not an error. "no worries" is reassuring, consistent with turtle vibes.

**verdict:** acceptable new phrase for new semantic context.

### divergence 2: "already up to date" phrase

**extant:** not found in git.commit
**introduced:** git.branch.rebase.begin.sh (when no rebase needed)

**analysis:** git.commit.set doesn't have a "no work" case — it requires staged changes. rebase can be a no-op. phrase matches git's own "already up-to-date" output.

**verdict:** acceptable — new case requires new phrase.
