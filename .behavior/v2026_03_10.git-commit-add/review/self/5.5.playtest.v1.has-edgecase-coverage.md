# edge case coverage review: playtest

## review question

are edge cases covered?
- what could go wrong?
- what inputs are unusual but valid?
- are boundaries tested?

## what could go wrong

| failure mode | playtest coverage |
|--------------|-------------------|
| stage without permission | edgey path #1 — "stage not allowed" error |
| file does not exist | edgey path #2 — "file not found" error |
| path escapes repo | edgey path #3 — "path must be within repo" error |
| quota file absent | edgey path #4 — del succeeds (idempotent) |
| file already staged | edgey path #5 — re-stage is no-op |
| stage flag not specified | edgey path #6 — defaults to block |
| global blocker active | edgey path #7 — overrides local permission |

**all failure modes covered.**

## unusual but valid inputs

| unusual input | playtest coverage |
|---------------|-------------------|
| uses="infinite" string | happy path #2 — allow sets uses: "infinite" |
| multiple files at once | happy path #4 — stage two files |
| stage all with `.` | happy path #5 — stages all files in cwd |
| del with no quota file | edgey path #4 — succeeds silently |
| re-stage of staged file | edgey path #5 — no-op |

**unusual but valid inputs covered.**

## boundaries tested

| boundary | test |
|----------|------|
| permission boundary: allow vs block | happy path #1 (allow) vs edgey path #1 (block) |
| quota boundary: numeric vs infinite | happy path #7 (infinite does not decrement) |
| path boundary: inside vs outside repo | happy path #4 (inside) vs edgey path #3 (outside) |
| state boundary: file exists vs absent | happy path #6 (del removes) vs edgey path #4 (del idempotent) |
| precedence boundary: local vs global | edgey path #7 (global overrides local) |

**all boundaries tested.**

## additional edge cases considered

### not in playtest (intentionally)

| edge case | reason not included |
|-----------|---------------------|
| commit with stage: block | covered by automated tests; playtest focuses on stage skill |
| push after commit | separate skill (git.commit.push), not changed in this behavior |
| concurrent stage operations | race conditions are unit/integration concern, not UX concern |

### covered implicitly

| edge case | how covered |
|-----------|-------------|
| empty file list | edgey paths all include cleanup that removes test files |
| mixed valid/invalid files | fail-fast behavior; one invalid file = entire operation fails (covered by edgey #2) |

## conclusion

| category | edge cases | covered |
|----------|------------|---------|
| failure modes | 7 | 7 ✅ |
| unusual inputs | 5 | 5 ✅ |
| boundaries | 5 | 5 ✅ |

**edge case coverage complete.**
