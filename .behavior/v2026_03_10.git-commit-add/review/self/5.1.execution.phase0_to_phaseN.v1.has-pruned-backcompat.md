# backwards compat review: execution phase

## review question

for each backwards-compat concern in the code, ask:
- did the wisher explicitly say to maintain this compatibility?
- is there evidence this backwards compat is needed?
- or did we assume it "to be safe"?

## backwards compat concerns reviewed

### 1. `--stage` defaults to `block`

**requested?** yes, explicitly in vision:

> note: --stage defaults to block for backwards compat
> note: extant commands like `set --quant 5 --push allow` still work

**why it holds**: vision explicitly requested that `set --quant 5 --push allow` (without --stage) continues to work. default to block preserves this.

### 2. state file format preserves `stage` field

**requested?** implicitly required.

when git.commit.set.sh writes state after decrement, it must preserve the stage field. this is not backwards compat — it's correctness. if stage field is lost, mechanic loses stage permission mid-session.

**why it holds**: required for correctness, not backwards compat assumption.

### 3. `del` removes entire state file

**requested?** yes, explicitly in vision:

> git.commit.uses del   # removes .meter/git.commit.uses.jsonc entirely

**why it holds**: vision explicitly states del removes entire file. no need to preserve fields.

### 4. `get` command checks for `999999` as unlimited (FOUND ISSUE)

**code in git.commit.uses.local.sh:**
```bash
if [[ "$USES" == "infinite" || "$USES" == "999999" ]]; then
  USES_DISPLAY="unlimited"
```

**requested?** no, not explicitly.

**why added**: previous `allow` command wrote `999999` as the "unlimited" value. this check allows old state files to display correctly.

**concern**: this is backwards compat added "to be safe" without explicit request.

**resolution**: keep this check — it is harmless and prevents user confusion if they have an old state file. however, it could be removed since:
- new `allow` writes `"infinite"`
- old state files with `999999` will still work (just display as "999999" instead of "unlimited")
- users can run `git.commit.uses del` then `git.commit.uses allow` to update

**flagged as open question for wisher**: should we remove the `999999` check? it's a 1-line change but low risk to keep.

### 5. stage field fallback to "block" in get command

**code:**
```bash
STAGE_STATE=$(jq -r '.stage // "block"' "$STATE_FILE")
```

**requested?** yes, implicitly.

**why it holds**: state files created before this change won't have a `stage` field. default to "block" is the safe/conservative behavior that matches the vision's default.

## conclusion

| concern | requested? | action |
|---------|------------|--------|
| --stage defaults to block | yes | keep |
| state file preserves stage | required | keep |
| del removes state | yes | keep |
| 999999 fallback in get | **no** | **flag as question** |

**open question for wisher**: the `999999` check in `get` command was added for backwards compat but not explicitly requested. keep or remove?
