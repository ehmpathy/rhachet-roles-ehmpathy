# clear instructions review: playtest

## review question

are the instructions followable?
- can the foreman follow without prior context?
- are commands copy-pasteable?
- are expected outcomes explicit?

## analysis

### foreman context

the playtest assumes only:
- a git repository (the current repo)
- `npx rhachet` available (standard for this repo)
- no prior quota state (instructions include cleanup)

**no prior context required** — each step is self-contained with setup and cleanup.

### commands copy-pasteable

every command in the playtest is formatted as a bash code block:

```sh
npx rhachet run --skill git.commit.uses set --quant 3 --push allow --stage allow
```

**verified copy-pasteable:**
- no special characters that require escape sequences
- multi-line commands use explicit line breaks

### expected outcomes explicit

each step specifies:

| step | explicit outcome |
|------|------------------|
| grant stage permission | "output shows uses: 3, push: allowed, stage: allowed" |
| grant unlimited quota | "output shows uses: unlimited, state file shows \"uses\": \"infinite\"" |
| check quota | "output shows all three permissions" |
| stage files | "output shows tree-formatted list, files appear in staged area" |
| stage all with `.` | "all files staged, output shows staged: ." |
| revoke permissions | "output confirms revocation, file removed" |
| unlimited no decrement | "before commit: uses: unlimited, after commit: uses: unlimited" |

### edgey path outcomes explicit

each edge case specifies the expected error message:

| edge case | expected error |
|-----------|----------------|
| no permission | "stage not allowed" |
| file not found | "file not found: nonexistent-file.ts" |
| path outside repo | "path must be within repo" |

### verification commands

each happy path includes a `verify:` section with commands to confirm the outcome:

```sh
# example: verify files staged
git diff --cached --name-only | grep playtest
```

### cleanup commands

each step that creates test files includes cleanup:

```sh
git reset HEAD playtest-file1.ts playtest-file2.ts
rm playtest-file1.ts playtest-file2.ts
```

## conclusion

**instructions are followable:**

| check | result |
|-------|--------|
| no prior context required | ✅ each step is self-contained |
| commands copy-pasteable | ✅ all in bash code blocks |
| expected outcomes explicit | ✅ each step specifies observable result |
| error messages specified | ✅ each edge case lists expected error |
| verification commands included | ✅ `verify:` section for happy paths |
| cleanup commands included | ✅ each step cleans up test artifacts |
