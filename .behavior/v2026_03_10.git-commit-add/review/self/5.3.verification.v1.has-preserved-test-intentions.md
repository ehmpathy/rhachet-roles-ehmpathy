### blackbox/guardBorder.onWebfetch.acceptance.test.ts

**change made** (line 307):
```typescript
// before
env: { XAI_API_KEY: '' },

// after
env: { XAI_API_KEY: '', HOME: tempDir },
```

**what did this test verify before?**
- case7: when XAI_API_KEY is not configured, hook exits with code 2 and stderr instructs agent to ask human for key

**original assertions (untouched)**:
```typescript
// line 313-314: exit code check
then('exits with code 2', () => {
  expect(res.result.code).toBe(2);  // ← unchanged
});

// line 317-320: stderr message check
then('stderr instructs agent to ask human for XAI_API_KEY', () => {
  expect(res.result.stderr).toContain('XAI_API_KEY');  // ← unchanged
  expect(res.result.stderr).toContain('ask the human');  // ← unchanged
});
```

**does it still verify the same behavior after?**
- yes — `expect(res.result.code).toBe(2)` unchanged
- yes — `expect(res.result.stderr).toContain('XAI_API_KEY')` unchanged
- yes — `expect(res.result.stderr).toContain('ask the human')` unchanged
- the only change is env setup, not any of these assertions

**did you change what the test asserts?**
- no — all three assertions remain identical
- fix addresses why the test failed (env var override from user config), not what it tests

**why it holds**:

the test intention is: "when XAI_API_KEY is absent, the hook should fail and tell the agent to ask the human."

the test bug was: the hook sources `~/.config/rhachet/apikeys.env` at startup, which overrides the empty value we set. the test was correct in its intent, but its setup was incomplete — it set the env var to empty but did not prevent the hook from locating the user's config file.

the fix is: set `HOME: tempDir` so the hook looks for config in the temp directory (where no config exists), thus the empty `XAI_API_KEY` we set takes effect.

this is a **setup bug**, not an **assertion bug**:
- setup bug: test did not fully isolate the environment
- assertion bug: test expected the wrong outcome

the assertions are correct. the setup was incomplete. the fix preserves intent.

### lessons for next time

1. **setup bugs vs assertion bugs**:
   - when a test fails, first check if the assertions are correct
   - if assertions are correct, the bug is in setup or prod code
   - never change assertions to match broken output

2. **environment isolation**:
   - tests that depend on env vars must fully isolate the environment
   - set `HOME` to temp dir to prevent user config interference
   - this is a setup fix, not an assertion change

3. **type annotation changes**:
   - widening a type (e.g., `number` to `number | string`) is backwards-compatible
   - extant tests continue to work with original types
   - new tests can use the extended type
