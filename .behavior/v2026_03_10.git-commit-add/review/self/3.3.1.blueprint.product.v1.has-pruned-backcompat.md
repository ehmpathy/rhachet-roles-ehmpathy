## OPEN QUESTIONS FOR WISHER

per the guide: "if backwards compat was not explicitly requested: flag it as an open question for the wisher"

---

### OPEN QUESTION 1: should --stage default to block or allow?

**what we assumed:** --stage defaults to block for backwards compatibility.

**wisher did NOT explicitly request this.** the wish says "only if uses --stage allowed" — it does not specify what default should be.

**why we assumed this:**
- extant `set --quant 5 --push allow` commands would suddenly gain stage permission if default were `allow`
- vision commented "for backwards compat"
- conservative defaults are standard security practice

**the question for wisher:**

| option | behavior | consequence |
|--------|----------|-------------|
| default block | `set --quant 5 --push allow` = stage blocked | requires explicit `--stage allow`, more typing |
| default allow | `set --quant 5 --push allow` = stage allowed | breaks extant grants, less typing |

**our recommendation:** default to block. but wisher should confirm.

---

### OPEN QUESTION 2: should stage permission be in same state file as commit quota?

**what we assumed:** stage permission lives in `.meter/git.commit.uses.jsonc` alongside uses and push.

**wisher did NOT explicitly request this.** wish names `git.stage.add` as separate skill but doesn't specify state file structure.

**why we assumed this:**
- additive extension is simpler
- vision shows stage as permission within quota
- separate file would duplicate uses tracking

**alternative:**
- stage in `.meter/git.stage.uses.jsonc`
- completely separate from commit quota
- would require separate grants

**our recommendation:** keep in same file. but wisher should confirm.

## what would elimination look like?

### bc1 elimination: --stage defaults to allow

if we eliminated this backcompat assumption:
- `set --quant 5 --push allow` would grant stage permission by default
- extant grants would permit stage operations
- users would need to revoke and re-grant with explicit `--stage block`

**impact:** high — breaks extant workflows.

### bc3 elimination: separate state file for stage

if we eliminated this backcompat assumption:
- stage in separate `.meter/git.stage.uses.jsonc`
- commit quota and stage quota decoupled
- more files, more commands, more complexity

**impact:** medium — adds complexity but no breaking change.
