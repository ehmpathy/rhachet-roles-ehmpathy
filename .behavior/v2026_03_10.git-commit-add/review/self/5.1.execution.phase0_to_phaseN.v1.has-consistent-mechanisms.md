# consistent mechanisms review: execution phase

## review question

for each new mechanism in the code, ask:
- does the codebase already have a mechanism that does this?
- do we duplicate extant utilities or patterns?
- could we reuse an extant component instead of a new one?

## mechanisms reviewed

### 1. output.sh reuse

**git.stage.add.sh:**
```bash
COMMIT_SKILL_DIR="$(cd "$SKILL_DIR/../git.commit" && pwd)"
source "$COMMIT_SKILL_DIR/output.sh"
```

**status**: reuses extant mechanism from git.commit

**why it holds**: output.sh contains turtle vibes utils (print_turtle_header, print_tree_start, print_tree_error). git.stage.add reuses these instead of duplicating.

### 2. state file location

**git.stage.add.sh:**
```bash
METER_DIR="$REPO_ROOT/.meter"
STATE_FILE="$METER_DIR/git.commit.uses.jsonc"
```

**status**: consistent with git.commit.set.sh and git.commit.uses.local.sh

**why it holds**: all three skills use the same state file location.

### 3. global blocker check

**git.stage.add.sh:**
```bash
GLOBAL_METER_FILE="$HOME/.rhachet/storage/repo=$ROLE_REPO/role=$ROLE_SLUG/.meter/git.commit.uses.jsonc"
```

**status**: consistent with git.commit.set.sh

**why it holds**: uses same path pattern and check logic as commit skill.

### 4. unbound glob check

**git.stage.add.sh:**
```bash
if [[ "$GLOB_PATTERN" == "**"* || "$GLOB_PATTERN" == "*"* ]]; then
  if [[ "$GLOB_PATTERN" =~ ^(\*\*/|\*) ]]; then
    # error
```

**sedreplace.sh:**
```bash
if [[ -n "$GLOB_FILTER" && "$GLOB_FILTER" == \*\*/* ]]; then
  # error
```

**status**: functionally equivalent with minor difference

**why it holds**: both block patterns that start with `**` or `*`. git.stage.add is slightly more strict (also checks patterns that start with `*`), which is appropriate since we want bounded globs only. the difference is intentional per the vision which says "unbound globs (**/*.ts) are forbidden" — the implementation covers both `**/*.ts` and `*.ts`.

### 4b. glob expansion mechanism (considered alternative)

**git.stage.add.sh uses bash globstar:**
```bash
shopt -s globstar nullglob
EXPANDED_FILES=($GLOB_PATTERN)
shopt -u globstar nullglob
```

**sedreplace.sh uses git ls-files with magic pathspec:**
```bash
FILES=$(git ls-files --cached --others ":(glob)$GLOB_FILTER")
```

**status**: different mechanisms for different purposes (intentional)

**why different approaches are appropriate**:
- sedreplace operates on git-tracked content: uses git ls-files to find files git knows about
- git.stage.add operates on filesystem: uses bash globstar to find files on disk to stage

**considered using git ls-files**: could work, but bash globstar is simpler and more direct for the purpose. staging files that exist on disk is filesystem operation, not git operation. git add will handle the git-tracking part.

### 5. repo root check

**git.stage.add.sh:**
```bash
if [[ "$abs_path" != "$REPO_ROOT"* ]]; then
  # error: path must be within repo
```

**cpsafe.sh:**
```bash
if [[ "$SOURCE_PATH" != "$REPO_ROOT"* ]]; then
  # error: source must be within repo
```

**status**: consistent pattern with cpsafe/mvsafe/rmsafe

**why it holds**: uses same path containment check pattern.

### 6. permission check pattern

**git.stage.add.sh:**
```bash
STAGE_ALLOWED=$(jq -r '.stage // "block"' "$STATE_FILE")
if [[ "$STAGE_ALLOWED" != "allow" ]]; then
  # error
```

**git.commit.set.sh:**
```bash
PUSH_ALLOWED=$(jq -r '.push' "$STATE_FILE")
if [[ "$PUSH_ALLOWED" != "allow" && "$PUSH" == "yes" ]]; then
  # error
```

**status**: consistent pattern for permission check

**why it holds**: both use jq to read permission field and check against "allow".

## conclusion

no duplicate mechanisms found. git.stage.add.sh reuses:
- output.sh from git.commit
- state file location pattern
- global blocker check pattern
- repo path containment pattern
- permission check pattern

the unbound glob check is functionally equivalent to sedreplace with intentional strictness per vision.
