# self-review: has-questioned-deletables

## context

reviewed after rebase onto main. key architecture changes:
- `git.commit.uses.sh` is now a dispatcher (routes to local.sh or global.sh)
- `allow` command already uses infinite for unlimited (replaces infinite concept)
- `del` command already exists (fall-through to set --quant 0)

---

## component-by-component analysis

### q1: can we delete git.stage/ directory structure?

**considered:** place git.stage.add.sh in git.commit/ instead of new git.stage/ folder.

**decision:** keep git.stage/ separate.
- the wish explicitly names "git.stage.add" (new namespace)
- separation follows skill namespace pattern (git.commit vs git.stage)
- future git.stage operations (git.stage.rm?) would belong here

**verdict:** cannot delete — wish mandates new namespace.

---

### q2: can we delete --stage option entirely?

**considered:** always allow stage when quota exists.

**decision:** keep --stage as separate permission.
- wish says "only if uses --stage allowed"
- conservative default (block) preserves backwards compat
- some humans want control over what gets staged
- explicit is better than implicit

**verdict:** cannot delete — core wish requirement.

---

### q3: can we delete infinite skip-decrement in git.commit.set.sh?

**considered:** let unlimited decrement naturally.

**decision:** keep skip-decrement logic.
- without it, "unlimited" would become 999998, 999997...
- breaks user expectation of truly unlimited
- one `if` check — minimal complexity

**verdict:** cannot delete — breaks unlimited behavior.

---

### q4: can we delete bounded glob support from git.stage.add?

**considered:** only support explicit file paths.

**decision:** keep bounded glob.
- vision explicitly includes: "rhx git.stage.add --glob 'src/**/*.ts'"
- saves many invocations when many files match
- safety is preserved via unbound glob block
- follows sedreplace pattern

**verdict:** cannot delete — vision specification.

---

### q5: can we delete path validation from git.stage.add?

**considered:** trust user input.

**decision:** keep validation.
- security requirement: "path must be within repo"
- follows cpsafe skill pattern
- prevents stage of /etc/passwd or other sensitive files

**verdict:** cannot delete — security requirement.

---

### q6: can we delete global blocker check from git.stage.add?

**considered:** only check local stage permission.

**decision:** keep global blocker check.
- follows git.commit.set.sh pattern
- global blocker should block ALL git operations
- consistency across skills

**verdict:** cannot delete — consistency requirement.

---

### q7: can we delete tree output format?

**considered:** simple line-per-file output.

**decision:** keep tree format.
- vision says: "output: tree-formatted list of staged files (follows git.* skill pattern)"
- consistency with git.commit.* output
- reuses output.sh — no new code

**verdict:** cannot delete — vision specification.

---

### q8: can we simplify integration tests?

**considered:** fewer test cases.

**decision:** keep full test coverage.
- each test maps to a criterion in 2.1.criteria.blackbox.md
- coverage matrix in 2.2 shows no redundant tests
- test execution is fast

**verdict:** cannot simplify — would cause loss of criterion coverage.

---

## what WAS already simplified (via architecture)

1. **no infinite string logic** — reuse extant `allow` command (infinite)
2. **no new del command** — already exists in local.sh
3. **no --mode flag for git.stage.add** — vision says "apply-only"
4. **no quota decrement for stage** — vision says "does NOT decrement uses"
5. **no --all flag** — vision says "--all is forbidden"
6. **no separate output.sh for git.stage** — reuse from git.commit

## final check: simplest version that works?

the blueprint represents the minimal implementation:
- one new skill file (git.stage.add.sh)
- one new test file
- extensions to two extant files (local.sh, set.sh) with minimal changes
- reuse of extant patterns (output.sh, path validation, glob validation)

no additional simplification possible without violation of wish/vision.
