## failure analysis

### failures found: 1 (fixed)

#### pre-extant acceptance test failure

**file**: `blackbox/guardBorder.onWebfetch.acceptance.test.ts`
**case**: `[case7] XAI_API_KEY not configured`

**symptom**: test passed on CI but failed locally

**root cause**:
1. test set `XAI_API_KEY: ''` to simulate absent key
2. hook sources `~/.config/rhachet/apikeys.env` at startup
3. local machine has this file with real key — override happens
4. CI has no such file — test env's empty value is effective

**fix** (line 307):
```typescript
env: { XAI_API_KEY: '', HOME: tempDir },
```

**verification**: test now passes both locally and on CI

**lesson**: when test env var behavior, isolate home directory to prevent user config from test interference.

**why the fix is correct**:
1. minimal change — one additional env var in test
2. targeted — only case7 needs this isolation
3. no side effects — other tests unaffected
4. matches the intent — test verifies "key not configured" scenario
5. CI-parity — test now behaves the same locally and on CI

**why this failure was not "unrelated to my changes"**:
- although the failure exists in acceptance tests (not skill tests), the zero tolerance policy applies
- "it's unrelated to my changes" is not an excuse
- finding and fixing pre-extant failures is part of the work
- the failure was found during `npm run test` — it blocked the verification
