# vision: git.stage.add + unlimited quota

## the outcome world

### day-in-the-life: before

human wants mechanic to stage specific files before commit:

```sh
# human has to stage files manually, then grant commit quota
git add src/domain/invoice.ts src/domain/customer.ts
git.commit.uses set --quant 1 --push allow

# mechanic can only commit what human staged
# if mechanic needs to add more files, human must intervene again
```

human wants to give mechanic full autonomy for a session:

```sh
# human grants 10 commits, but what if more needed?
git.commit.uses set --quant 10 --push allow

# mechanic runs out, has to ask human for more
# friction, context switching
```

### day-in-the-life: after

human grants stage permission alongside commit quota:

```sh
# human grants stage + commit + push in one command
git.commit.uses set --quant 3 --push allow --stage allow

# mechanic can now stage files as needed
rhx git.stage.add src/domain/invoice.ts src/domain/customer.ts
rhx git.commit.set -m @stdin --mode apply
```

human gives mechanic unlimited autonomy for a session:

```sh
# human grants unlimited commits with full permissions (uses allow command)
git.commit.uses allow

# mechanic works freely without quota anxiety
# ... does many commits and pushes ...

# when session ends, human revokes
git.commit.uses del
```

### the "aha" moment

the value clicks when:
1. mechanic can stage files without human intervention — workflow is unblocked
2. `allow` means "trust me for this session" — no more "oops, ran out of quota"
3. `del` cleanly revokes all permissions — session ends cleanly

## user experience

### usecases

| goal | command | outcome |
|------|---------|---------|
| let mechanic stage specific files | `--stage allow` | mechanic uses `git.stage.add` |
| unlimited commits for a session | `allow` | uses=infinite, push=allow, stage=allow |
| limited commits with stage | `set --quant 5 --push allow --stage allow` | mechanic stages + commits up to 5 |
| end session, revoke all | `del` or `block` | meter cleared |

### contract inputs & outputs

**git.commit.uses set**
```sh
# new option
--stage allow|block   # control stage permission (default: block)

# note: --stage defaults to block for backwards compat
# note: extant commands like `set --quant 5 --push allow` still work

# examples
git.commit.uses set --quant 5 --push allow --stage allow
git.commit.uses set --quant 5 --push block --stage allow
git.commit.uses set --quant 5 --push allow             # --stage defaults to block
git.commit.uses set --quant 0  # revoke (defaults --push block --stage block)
```

**git.commit.uses allow**
```sh
# extant command, updated behavior
git.commit.uses allow   # sets uses=infinite, push=allow, stage=allow

# note: now includes stage=allow (previously only set uses+push)
```

**git.commit.uses del**
```sh
# extant command (already implemented)
git.commit.uses del   # removes .meter/git.commit.uses.jsonc entirely

# note: succeeds silently if no quota file exists (like rm -f)
```

**git.stage.add** (new skill)
```sh
# stage files for commit
rhx git.stage.add src/file1.ts src/file2.ts    # stage specific files
rhx git.stage.add src/**/*.ts                  # stage via shell glob
rhx git.stage.add .                            # stage all in current dir

# output: tree-formatted list of staged files (follows git.* skill pattern)
# mode: apply-only (no --mode flag — stage is reversible via git reset)

# permission & quota:
# - requires --stage allow in quota
# - does NOT decrement uses (uses counts commits only)

# error cases:
# - no stage permission: "error: stage not allowed"
# - no files specified: "error: no files specified"
# - file not found: "error: file not found: path/to/file.ts"
# - path outside repo: "error: path must be within repo"

# behavior:
# - idempotent: re-stage of already staged file is no-op
# - fail-fast: if any file not found, entire operation fails

# constraints:
# - paths validated to be within repo (same as safe* skills)
```

### timelines

**typical session:**
1. human: `git.commit.uses allow`
2. mechanic: works, stages files, commits, pushes freely
3. mechanic: completes task
4. human: `git.commit.uses del`

**cautious session:**
1. human: `git.commit.uses set --quant 3 --push allow --stage block`
2. human: manually stages files
3. mechanic: commits staged files, pushes
4. quota exhausted naturally

## evaluation

### how well does it solve the goals?

| goal | solved? | notes |
|------|---------|-------|
| mechanic can stage files | yes | via `git.stage.add` skill with `--stage allow` |
| unlimited quota | yes | `allow` sets uses=infinite, no decrement at that level |
| clean revocation | yes | `del` removes state file |
| backwards compatible | yes | current commands still work |

### pros

- minimal api surface: one new skill, one new option (`--stage`)
- clear permission model: `--stage allow` parallels `--push allow`
- `allow` already exists — just extend it to include stage
- `del` is idiomatic (matches `set`/`get`/`del` pattern)

### cons

- `--stage allow` defaults to block — requires explicit grant

### pit of success

| edgecase | how we handle |
|----------|---------------|
| stage without permission | fails with clear error: "error: stage not allowed" |
| no files specified | fails with: "error: no files specified" |
| file not found | fails with: "error: file not found: path/to/file.ts" |
| path outside repo | fails with: "error: path must be within repo" |
| file already staged | no-op (idempotent) |
| commit without staged files | current behavior: fails with "no changes to commit" |
| push without permission | current behavior: fails with "push not allowed" |
| `del` with no quota file | succeeds silently (like rm -f) |

## open questions & assumptions

### assumptions

1. `--stage allow` controls only `git.stage.add` skill, not `git add` directly
2. `allow` command sets stage=allow alongside uses=infinite and push=allow
3. `del` removes the entire state file (already implemented)
4. `--stage` defaults to `block` (conservative default)

### questions for wisher

**none** — key decisions made:

| area | decision |
|------|----------|
| permissions | `--unstaged include` remains independent of `--stage allow` |
| permissions | `--stage` defaults to `block` (backwards compat, conservative) |
| permissions | `git.stage.add` does NOT decrement `uses` (quota counts commits only) |
| revocation | `del` already exists and works as expected |
| unlimited | use explicit `infinite` string value |
| constraints | paths validated to be within repo (follows safe* skills) |
| output | tree-formatted output (follows git.* skill pattern) |
| mode | apply-only (no --mode flag — stage is reversible) |
| errors | fail-fast: partial failure = entire operation fails |
| behavior | idempotent: re-stage of staged file = no-op |
| skill name | `git.stage.add` (new namespace, per wish text) |
| stage all | use `.` to stage all files in cwd (shell glob expansion) |

## what is awkward?

### stage permission is separate from commit permission

could argue stage should be implied by commit permission, but:
- separation gives finer control
- some humans want to control exactly what gets staged
- explicit is better than implicit
