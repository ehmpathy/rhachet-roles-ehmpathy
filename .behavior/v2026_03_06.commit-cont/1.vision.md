# vision: cont-prefixed continuation commits

## definitions

- **behavioral commit**: a commit that changes behavior — `fix:` or `feat:` prefixed
- **first behavioral commit**: the first fix/feat commit on a branch — defines the PR's deliverable
- **continuation commit**: subsequent fix/feat commits — must use `cont:` or `cont(scope):` prefix
- **non-behavioral commit**: `chore:`, `docs:`, `style:`, etc. — exempt from `cont:` rule

## the outcome world

### before

```
CHANGELOG v1.27.0

- fix(api): validate input parameters
- fix(api): add missing test coverage
- fix(api): handle edge case in validation
- feat(auth): add oauth support
- fix(auth): correct token refresh logic
- fix(auth): update test snapshots
```

the release log is noisy. three "fixes" for api, but really it was one fix with follow-up work. same for auth — one feature, two "fixes" that were just iteration on that feature.

### after

```
CHANGELOG v1.27.0

- fix(api): validate input parameters
- feat(auth): add oauth support
```

the release log shows what actually shipped: one fix, one feature. the continuation commits (`cont: add test coverage`, `cont: handle edge case`) existed but don't pollute the changelog.

### the "aha" moment

when reviewing the changelog before a release, you see exactly what value was delivered — not the sausage-making of how you got there.

## user experience

### usecase: mechanic iterating on a fix

1. human creates branch `vlad/fix-validation`
2. human binds commit level: `rhx git.commit.bind set --level fix`
3. mechanic makes first commit: `fix(api): validate input parameters`
4. tests fail — mechanic fixes them
5. mechanic attempts: `fix(api): add test coverage`
6. **git.commit.set blocks**: "branch already has a fix commit. use `cont:` or `cont(api):` prefix"
7. mechanic commits: `cont(api): add test coverage`
8. pr merges — changelog only shows the first fix

### usecase: human continuing their own work

same flow applies. after the first feat/fix commit on a branch, all subsequent commits must be `cont:` prefixed.

### contract inputs

```sh
# first commit on branch — normal prefix
echo "fix(api): validate input" | rhx git.commit.set -m @stdin --mode apply

# subsequent commits — cont prefix required
echo "cont(api): add tests" | rhx git.commit.set -m @stdin --mode apply
echo "cont: polish docs" | rhx git.commit.set -m @stdin --mode apply
```

### contract outputs

- first commit: proceeds normally
- subsequent commits with `fix/feat`: **blocked** with clear error
- subsequent commits with `cont:` or `cont($slug):`: proceeds normally

## mental model

### how users describe this

> "the first commit is the 'real' one — it's what shows up in the changelog. everything after is just finishing the job."

### analogies

- **git squash, but semantic**: instead of squashing commits, you label them as continuations
- **pr = one deliverable**: the first commit names the deliverable, the rest are implementation details
- **headline vs body**: first commit is the headline, cont commits are the body paragraphs

### terms

| user term | our term |
|-----------|----------|
| "the main commit" | first behavioral commit |
| "follow-up commits" | continuation commits |
| "finishing touches" | `cont:` prefixed commits |

## evaluation

### pros

- **clean changelogs**: release notes show real deliverables, not iteration noise
- **clear intent**: readers know which commits are "the point" vs supporting work
- **enforced consistency**: can't accidentally pollute changelog with follow-up fixes
- **works with current flow**: just adds validation, doesn't change commit mechanics

### cons

- **learning curve**: new prefix to learn (`cont:`)
- **enforcement friction**: blocked commits require retry with different prefix
- **edge cases**: what if you legitimately have two fixes in one pr?

### pit of success

- **clear error message**: when blocked, tells you exactly what to do
- **simple rule**: first commit = normal, rest = cont
- **escape hatch**: if you truly need two fixes, split into two prs (better anyway)
- **fail fast, never failhide**: invalid repo states are blocked immediately with clear errors:
  - repo has no commits → blocked
  - no base branch (main/master/trunk) → blocked
  - on base branch → blocked (must create feature branch)

## open questions & assumptions

### assumptions

1. **one deliverable per pr**: a pr represents one logical change
2. **first behavioral commit = intent**: the first fix or feat commit message captures what the pr delivers
3. **changelog generator respects this**: the changelog generator filters out `cont:` commits (GIVEN — assumed to be handled separately)
4. **commits go through git.commit.set**: raw `git commit` already forbidden via permissions.jsonc (GIVEN)
5. **`cont:` is a valid commit type**: toolchain under our control, will be configured (GIVEN)

### self-answered questions

1. **should `cont:` commits appear in changelog?**
   - answer: NO — the whole purpose is clean changelogs
   - requires: changelog generator to filter `cont:` type

2. **what about `chore:` commits?**
   - answer: EXEMPT — chore doesn't appear in changelog, so no need to require `cont:`
   - the rule only applies to fix/feat after the first fix/feat

3. **what about rebasing?**
   - answer: topological position on branch, not timestamp
   - git compares branch to base; "first" is topologically first commit on branch

4. **scope inheritance?**
   - answer: scope is OPTIONAL — wish explicitly says "cont: or cont($slug):"
   - both `cont:` and `cont(scope):` are valid

### confirmed decisions

1. **first behavioral commit rule** (CONFIRMED)
   - after first behavioral commit (fix/feat), subsequent fix/feat commits need `cont:`
   - chore is exempt — not behavioral, doesn't appear in changelog

### answered questions (from wish analysis)

1. **scope**: git.commit.set only (wish line 3 specifically names it)
2. **changelog filter**: assumed to be handled separately (GIVEN)

## what is awkward?

### ~~"first commit" ambiguity~~ RESOLVED

- term: "first behavioral commit" = first fix/feat commit on branch
- chore is not behavioral, so it doesn't trigger the rule
- example: `chore: setup` → `fix(api): actual fix` → `cont: tests` ✓

### forced mental model

- users might want to say "fix: test coverage" because it feels like fixing the tests
- but semantically, it's continuing the original fix work
- the label "cont" might feel bureaucratic

### multi-concern prs

- some prs legitimately touch multiple things
- forcing everything into one first-commit label might feel constraining
- mitigation: encourage smaller, focused prs (this is good anyway)

### rebase complexity

- if someone rebases and reorders commits, the "first" commit changes
- do we track by timestamp? by position? by branch creation?
- likely answer: position in branch (first commit on branch compared to base)
