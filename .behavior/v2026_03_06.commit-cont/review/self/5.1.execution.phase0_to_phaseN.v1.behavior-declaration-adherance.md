## why the implementation holds

### the core logic is correct

the `get_behavioral_commits_on_branch()` helper (lines 109-143) correctly:
- calls `git log --reverse --format="%s" "$base_branch..HEAD"` to get commits on branch
- filters to `^(fix|feat)(\([^)]+\))?:` — exactly the behavioral patterns from vision line 5
- returns empty string when no behavioral commits exist → allows first fix/feat
- returns subjects when behavioral commits exist → triggers cont: requirement

the guard logic (lines 393-416) correctly:
- only checks when BEHAVIORAL_COMMITS is non-empty (has prior behavioral)
- only blocks when COMMIT_PREFIX is "fix" or "feat" (line 395)
- allows cont prefix because it's not in the (fix, feat) check
- extracts scope from header via sed (line 400) to suggest `cont($scope):`

### the exemptions are correct

the case statement (lines 286-290) skips validation for:
- chore, docs, refactor, test, ci, build, perf, style

this matches vision line 8: "non-behavioral commit: chore, docs, style, etc. — exempt"

### the fail-fast guards are correct

- NO_COMMITS (lines 358-366): matches vision line 117 "repo has no commits → blocked"
- NO_BASE (lines 369-377): matches vision line 118 "no base branch → blocked"
- ON_BASE (lines 380-390): matches vision line 119 "on base branch → blocked"

### the error messages match the spec

vision line 51 says error should be: "branch already has a fix commit. use cont: or cont(api): prefix"

implementation line 404: "branch already has a behavioral commit"
implementation lines 409-412: shows `cont:` or `cont($COMMIT_SCOPE):` suggestion
