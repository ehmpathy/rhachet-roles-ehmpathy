# 1.vision: git.release skill

## crystal clear

**`--mode apply` = enable automerge + WATCH until done**

the skill does NOT return until all checks are complete. it watches. it waits. it only exits when:
- all checks pass and PR merges (exit 0), or
- a malfunction error occurs (exit 1: gh failed, network error)
- a constraint error occurs (exit 2: needs rebase, no PR)

**`--to main` = just this branch**

enable automerge on the feature branch PR, watch it merge to main.

if PR needs rebase: failloud, tell caller to rebase, exit 1. don't even try automerge.

if no PR for branch: failloud, tell caller to push, exit 1. can't release what doesn't exist.

**`--to prod` = full journey: feature → main → release → tag**

leverages `--to main` first, then continues:

1. `--to main`: enable automerge on feature branch PR, watch it merge to main
2. find the release PR (semantic-release creates it), enable automerge, watch it merge
3. watch the tag workflows (publish.yml or deploy.yml) complete

each step is watched. each step uses automerge. the skill doesn't return until the whole journey is done.

## the outcome world

### day-in-the-life

a mechanic finishes implementing a feature. they commit with `git.commit.set --push`. now what?

**before**: the mechanic says "pushed! watch CI and let me know if anything fails." the human opens github, navigates to the PR, clicks Actions, watches the checks. if they pass, they click "merge." then they wait again for the main branch CI. then they tag for prod. then they wait again.

**after**: the mechanic says "pushed! let me ride the wave 🌊" and runs `git.release --to main`. the terminal shows the PR merging, watches CI pass, and confirms success — or surfaces errors immediately. the mechanic can then run `git.release --to prod` to tag and deploy, watching the full pipeline complete.

### before/after contrast

| before | after |
|--------|-------|
| human manually merges PR | mechanic runs `git.release --to main --mode apply` |
| human watches CI in browser | terminal shows live progress |
| human manually creates release tag | `git.release --to prod --mode apply` handles the full cycle |
| errors discovered minutes later | errors surfaced immediately inline |
| context switch between terminal and browser | everything stays in terminal |

### the "aha" moment

the moment the mechanic runs `git.release --to prod --mode apply` and watches the entire pipeline — merge to main, main CI, tag to prod, deploy/publish — complete successfully without touching a browser. that's when the value clicks: **shipping is now a single command**.

## user experience

### usecases

| goal | command |
|------|---------|
| preview what merge to main would do | `git.release --to main` |
| merge PR to main and watch CI | `git.release --to main --mode apply` |
| preview full release to prod | `git.release --to prod` |
| run full release: merge → main CI → tag → deploy | `git.release --to prod --mode apply` |

### contract inputs

```
git.release --to <target> [--mode plan|apply]

--to main     merge feature branch PR to main, watch test CI
--to prod     full release: merge to main → watch test CI → tag → watch deploy/publish CI

--mode plan   show what would happen (default)
--mode apply  execute the release
```

## evaluation

### how well does it solve the goals?

| goal | solved? |
|------|---------|
| reduce context switching | ✨ yes — everything in terminal |
| automate merge → deploy | ✨ yes — single command |
| surface errors quickly | ✨ yes — watch mode catches failures |
| match extant `git release` vibes | ✨ yes — same tree structure, turtle vibes |
| plan/apply pattern | ✨ yes — consistent with other skills |

### pros

- **unified**: one skill handles feature→main and main→prod
- **observable**: live progress in terminal
- **safe**: plan mode is default, apply requires explicit flag
- **composable**: can stop at main or go to prod
- **vibey**: turtle themes, tree output, same patterns

### cons

- **complexity**: full prod release involves multiple CI waits
- **timing**: relies on CI completing (network/service dependency)
- **permissions**: requires github token with merge/release access

### edgecases and pit of success

| edgecase | handled by |
|----------|------------|
| PR not mergeable (conflicts, reviews) | plan mode surfaces blockers before apply |
| CI fails mid-release | error surfaced immediately, release halts |
| wrong branch (on main trying to release) | guard: "already on main" error |
| no PR exists for branch | guard: "no PR found" error |
| tag already extant | guard: "version already released" error |

## open questions & assumptions

### assumptions made

1. **wrap extant alias**: skill wraps `git release` alias from `~/.bash_aliases` (`git_alias_release` function). adds turtle vibe headers and `--to main|prod` + `--mode plan|apply` contract on top.
2. **PR already extant**: `git.commit.push` finderts the PR, so we assume it extant
3. **CI workflow names**: assume `test.yml` and either `publish.yml` or `deploy.yml` extant
4. **semver tags**: assume prod releases use semver tags (v1.2.3)
5. **github token via keyrack**: fetch `EHMPATHY_SEATURTLE_PROD_GITHUB_TOKEN` from keyrack (same pattern as `git.commit.push`). use `GH_TOKEN="$token" gh ...` for all gh commands.
6. **"main" means default branch**: `--to main` targets whatever the repo's default branch is (main, master, etc.), not literally a branch named "main"
7. **interactive terminal**: assume tty available; should detect non-tty and simplify output

### questions for the wisher

1. **[answered] wrap vs reimplement**: wrap the extant `git release` alias. this bootstraps implementation, reuses battle-tested logic, and keeps the skill lightweight. the alias lives in `~/.bash_aliases` as `git_alias_release`.
2. **[answered] merge method**: follow repo settings. `gh pr merge` without `--squash` or `--rebase` uses repo defaults.
3. **[answered] tag version source**: skill doesn't increment versions. CI/CD handles that (standard-version/semantic-release creates release PR with version bump). skill just watches the release PR and tag workflows.
4. **[answered] which CI to watch**: detect dynamically via `gh workflow list`, fall back to common names.
5. **[answered] automerge**: enable automerge immediately when `--mode apply` is set, just like extant `git release --apply`. uses `gh pr merge --auto --squash`.
6. **[answered] branch protection**: plan mode surfaces blockers (e.g., "PR needs reviews") before apply mode attempts merge.

### external research needed

- **[answered]** stdout pattern: researched `~/.bash_aliases` `git_alias_release` function. pattern uses `🌊` header, tree structure with status emojis (`⚓` fail, `🐢` pending, `👌` pass, `🌴` merged, `🐚` rebase, `🥥` watch). vision updated to match.
- **[answered]** merge method: extant alias uses `gh pr merge --auto --squash`. always squash.
- **[answered]** release creation: alias doesn't call `gh release create`. CI/CD creates releases. alias just watches tag workflows via `gh run list --branch "$tag"`.
- **[answered]** watch multiple CI: alias polls `gh pr view --json statusCheckRollup` for PR checks, `gh run list --branch "$tag"` for tag runs. sequential poll loop with sleep intervals.

### ⚠️ blueprint prerequisite

**before blueprint**: re-read `~/.bash_aliases` and extract the full `git_alias_release` function tree. the alias on this machine is the source of truth. verify all [answered] items above match the actual implementation.

## what is awkward?

### potential friction points

1. **two targets only**: `--to main` and `--to prod` feel limiting. what about preprod? canary? but keeping it simple may be better.

2. **ci workflow names**: hardcoding `test.yml`, `publish.yml`, `deploy.yml` assumes naming conventions. what if repos use different names?

3. **version number**: where does the version come from for `--to prod`? auto-increment risks collisions. manual input breaks automation.

4. **long waits**: watch CI for 10+ minutes in terminal is awkward. maybe background mode? or progress polling?

5. **non-tty contexts**: turtle vibes and spinners assume interactive terminal. should detect non-tty and emit simpler output (no spinners, just status lines).

### where design fights user mental model

- **"release" vs "deploy"**: users might say "deploy to prod" but we say "release to prod". both are valid — release implies the whole cycle.

- **plan mode surprise**: if someone runs `git.release --to prod` expecting it to release, they get a preview instead. good for safety, but might confuse first-timers.

### no tradeoffs — replicate extant alias

no new decisions here. the skill replicates the exact `git release` workflow from `~/.bash_aliases`. all behavior, flags, and patterns come from the extant alias.

this skill brings the release pipeline into the terminal with turtle vibes. `git.release --to main` lands PRs. `git.release --to prod` ships to production. plan mode shows what would happen. apply mode does it. errors surface inline. the whole flow matches the extant git.commit and git.rebase skill patterns.
