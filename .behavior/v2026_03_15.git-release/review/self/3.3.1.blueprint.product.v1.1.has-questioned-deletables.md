## component analysis

### 1. git.release.operations.sh — KEEP

**can this be removed?** no.

**why keep?** the operations file contains gh CLI wrappers that are called multiple times throughout the skill. `_gh_with_retry` alone is called for every gh command. extracting to a separate file follows the pattern from git.commit.operations.sh.

**simplest version?** could inline into git.release.sh, but that would duplicate retry logic across every gh call.

### 3. --retry flag — KEEP

**can this be removed?** could be deferred.

**why keep?** extant `git release` alias has retry behavior. wrapping the alias means inheriting this. also, retry is in the BDD criteria (usecase.5).

## changes made

### issue 1: risk assessment section deleted

**what was wrong**: the blueprint had a risk assessment table (gh CLI rate limits, keyrack locked, watch timeout, etc.) that was not required by the vision or criteria. the mitigations were already present in the codepath tree (_gh_with_retry, clear error messages).

**how it was fixed**: removed the entire "risk assessment" section from the blueprint. the risks are self-evident from the implementation and don't need separate documentation.

**lesson for next time**: don't add defensive documentation that duplicates what's already in the codepath. if the mitigation is in the code tree, it doesn't need a separate table.

---

### issue 2: execution phases simplified

**what was wrong**: the blueprint had 6 phases (setup, core skill, --to main, --to prod, --retry, tests, polish) which was excessive for a skill that wraps an extant alias. too many phases create artificial boundaries.

**how it was fixed**: collapsed to 3 phases:
1. setup (directory, output.sh, operations.sh)
2. implementation (main skill, --to main, --to prod, --retry)
3. tests (integration tests, BDD verification, permissions)

**lesson for next time**: number of phases should match actual work batches, not logical decomposition. wrapping an alias is simpler than building from scratch.

## non-issues (why they hold)

### git.release.operations.sh — holds

the operations file contains reusable gh CLI wrappers (_gh_with_retry, get_pr_for_branch, etc.) that are called from multiple places. extracting to separate file follows the pattern from git.commit.operations.sh and prevents duplication.

### output.sh — holds

the output utils (print_release_header, print_check_status) are used across plan/apply modes and success/failure paths. release-specific emojis (🌊, 🫧, 🥥) differentiate from git.commit output.sh.

### --retry flag — holds

retry is explicitly in BDD criteria (usecase.5). the extant alias has this behavior. implementation is minimal (gh run rerun --failed).

### --to prod workflow — holds

explicitly requested in wish. the full release cycle (merge → main CI → tag → deploy) is the core value proposition.

### watch loop — holds

watch is the "ride the wave" experience from vision. without it, user still needs browser to verify CI.

### keyrack token fetch — holds

gh commands require auth for apply mode. reusing the battle-tested pattern from git.commit.push.sh avoids reinventing auth flow.
