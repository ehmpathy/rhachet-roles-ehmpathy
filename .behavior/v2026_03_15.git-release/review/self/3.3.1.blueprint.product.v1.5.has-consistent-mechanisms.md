**blueprint**: standard while-case loop for `--to`, `--mode`, `--retry`, with rhachet passthrough handle.

**extant**: git.commit.push.sh uses identical pattern (lines 47-92).

**duplicates?**: NO — this is standard bash argument parse. each skill needs its own because flags differ. git.release has `--to`, git.commit.push has `--mode apply` but not `--to`. reuse would require a shared library that adds complexity.

## non-issues (why they hold)

### argument parse — holds (not duplicate)

each skill has unique flags. argument parse is boilerplate but not shareable. git.release needs `--to` which git.commit.push doesn't have.

**why not extract to shared lib?**: argument parse is ~30 lines. a shared lib would add:
- dependency management
- version coordination
- import complexity
- test surface

the cost exceeds the benefit. copy-paste argument parse is the pragmatic choice.

**evidence**: all git.* skills have their own argument parse (git.commit.set, git.commit.push, git.rebase.begin). none share a library. this is established pattern.

### gh operation wrappers — holds (release-specific)

get\_pr\_status, enable\_automerge, get\_tag\_runs are release-specific. git.commit.push has PR lookup but for different purpose (existence check vs status check). different questions, different operations.

**detailed comparison**:

| operation | git.commit.push | git.release | same? |
|-----------|-----------------|-------------|-------|
| PR lookup | `gh pr list --head "$branch"` → number | `gh pr list --head "$branch"` → number | YES |
| PR status | N/A | `gh pr view --json statusCheckRollup` | NO |
| automerge | N/A | `gh pr merge --auto` | NO |
| run list | N/A | `gh run list --branch "$tag"` | NO |

only PR number lookup is shared. but:
- git.commit.push: checks if PR exists (boolean)
- git.release: gets PR number for status check (input to next operation)

**verdict**: small overlap doesn't justify extraction. release operations are a coherent set.

### print\_release\_\* functions — holds (new capability)

release-specific outputs (check status, automerge status, etc.) have no extant equivalent. correctly marked as new.

**extant output functions searched**:

| file | functions | relevant? |
|------|-----------|-----------|

**conclusion**: no extant release-specific output. the 6 new print\_release\_\* functions are genuinely new.

## reflection

### the reuse strategy taxonomy

this review revealed three categories of "reuse" that need explicit strategy:

| category | strategy | when to use | example |
|----------|----------|-------------|---------|
| per-skill pattern | copy boilerplate | flags differ per skill | argument parse |
| sister-skill share | source the file | output consistency needed | output.sh functions |
| alias wrap | reimplement logic | source unavailable at runtime | \_gh\_with\_retry |
| release-specific | create new | no extant equivalent | print\_check\_status |

### why strategies matter

the blueprint had correct "[←] reuse" markers but lacked HOW. this ambiguity leads to:

1. **implementation confusion**: builder doesn't know if they should copy, source, or extract
2. **inconsistent decisions**: different builders make different choices
3. **accidental duplication**: without strategy, copy is the default

the review fixed this by adding explicit strategies.

### what I searched for duplicates

to verify no duplicates exist, I searched:

| search | result |
|--------|--------|
| `git.commit.push.sh` PR operations | found PR list lookup, but different purpose |
| `show.gh.action.logs.sh` run operations | found run list, but for view not release |
| `git.commit/output.sh` functions | found shared turtle output (marked for source) |
| `git.rebase/*.sh` | no output.sh, no reusable operations |
| `~/.bash_aliases` | found alias functions (marked for reimplement) |

**key lesson**: "[←] reuse" is not actionable without strategy. specify: copy, source, or reimplement.

### what this review improved

1. blueprint now has explicit reuse strategies
2. builder can implement without ambiguity
3. future reviewers can verify reuse was done correctly
