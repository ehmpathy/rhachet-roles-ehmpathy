# review.self: has-pruned-backcompat

## summary

reviewed implementation for backwards compatibility concerns. none found.

## analysis

### why no backwards compat concerns

this is a **new skill**. there is no prior version to maintain compatibility with.

the skill intentionally uses a different interface from the extant `git release` alias:

| extant alias | new skill |
|--------------|-----------|
| `git release --watch` | `git.release --to main --mode apply` |
| `git release --apply` | `git.release --to main --mode apply` |
| `git release main --watch` | `git.release --to prod` |

the wish explicitly said to "base off of how my current git config alias works" for **behavior**, not **interface**. the skill reuses the internal logic patterns (poll intervals, status emojis, gh commands) but provides a cleaner `--to main|prod` + `--mode plan|apply` contract.

### backwards compat patterns NOT present

| pattern | present? | why |
|---------|----------|-----|
| deprecated flags | no | new skill |
| fallback behaviors | no | new skill |
| legacy API shims | no | new skill |
| "for now, keep old behavior" comments | no | new skill |
| conditional paths for old vs new | no | new skill |

## verdict

✓ no backwards compatibility code detected

this is a new skill with no prior version. the contract is clean and forward-only.
