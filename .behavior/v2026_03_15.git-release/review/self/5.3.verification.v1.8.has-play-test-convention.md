# self-review: has-play-test-convention

## question

are journey test files named correctly?

## analysis

### check for `.play.` convention in repo

ran `grep -r '\.play\.' src/` — no matches found.

### check extant test name conventions

examined test files in `src/domain.roles/mechanic/skills/`:

| skill | test file |
|-------|-----------|
| git.commit | `git.commit.set.integration.test.ts` |
| git.rebase | `git.rebase.begin.integration.test.ts` |

| mvsafe | `mvsafe.integration.test.ts` |
| rmsafe | `rmsafe.integration.test.ts` |
| **git.release** | `git.release.integration.test.ts` |

### repo convention

this repo uses `.integration.test.ts` suffix for journey tests, not `.play.test.ts`.

evidence:
- 0 files use `.play.` pattern
- 70+ files use `.integration.test.ts` pattern
- extant skill tests all use `.integration.test.ts`

### fallback convention used

the guide says: "if not supported, is the fallback convention used?"

yes — this repo's fallback convention is `.integration.test.ts`, and the git.release skill follows that convention:

```
src/domain.roles/mechanic/skills/git.release/git.release.integration.test.ts
```

### location verification

| aspect | expected | actual |
|--------|----------|--------|
| directory | colocated with skill | ✅ `skills/git.release/` |

## why it holds

1. **repo does not use `.play.` convention** — zero files match pattern

## conclusion

holds — repo uses `.integration.test.ts` convention, not `.play.`. git.release follows extant convention.
