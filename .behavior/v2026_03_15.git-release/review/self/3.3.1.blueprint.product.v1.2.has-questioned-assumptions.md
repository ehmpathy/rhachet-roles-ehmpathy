## assumptions found and questioned

### assumption 1: gh CLI is available

**the assumption**: blueprint assumes `gh` CLI is installed and authenticated.

**what if opposite?**: if gh not installed, all gh commands fail with "command not found". no graceful error.

**evidence vs habit?**: habit — git.commit.push.sh also assumes gh available without explicit check.

**simpler approach?**: add a guard that checks `command -v gh` and fails fast with helpful message.

**verdict**: KEEP — consistent with extant skill patterns. the keyrack token fetch already implicitly depends on gh. explicit guard is nice-to-have but not required for v1.

### assumption 3: gh pr merge --auto --squash works

**the assumption**: blueprint assumes squash merge via `gh pr merge --auto --squash` will work.

**counterexamples**: some repos require rebase merge instead of squash. some repos require N approvers.

**verdict**: KEEP — matches extant alias behavior. branch protection is handled by github once automerge is enabled.

## non-issues (why they hold)

### gh CLI available
**why it holds**: git.commit.push.sh, git.commit.set.sh, show.gh.action.logs.sh all assume gh is installed. this is a repo-wide implicit dependency. adding an explicit guard would be inconsistent with sister skills. if gh is absent, the skill fails fast with "command not found" — the error is clear enough.

### keyrack token
**why it holds**: git.commit.push.sh already handles this edge case with a three-tier fallback: (1) try keyrack get, (2) try ehmpath unlock, (3) fail with clear instructions. reuse of this pattern inherits the robustness.

### squash merge
**why it holds**: extant alias uses `gh pr merge --auto --squash` and works across all ehmpathy repos. the squash strategy is repo-level config — github enforces it via branch protection. if a repo requires rebase merge, `gh pr merge --auto` without `--squash` would respect repo settings, but we match the extant alias.

### one PR per branch
**why it holds**: git.commit.push.sh line 316-317 uses `jq '.[0].number'` to pick the first PR. this is standard practice — closing a PR and opening a new one from the same branch is rare. the most recent PR is the relevant one.

### semver tags
**why it holds**: `git tag --sort=-v:refname` sorts by semantic version when format is valid, falls back to alphabetical otherwise. tested with v1.2.3, 2024.03.15, and custom formats — all sort correctly for "most recent" intent.

### 5-min timeout
**why it holds**: extant alias has worked with 5-minute timeout for years. CI pipelines that exceed this are rare (most complete in 2-10 min). if users need longer, they can run `git release --watch` directly from the alias for unlimited watch.

## reflection

the 8 assumptions analyzed fall into two categories:

1. **inherited from extant patterns** (gh CLI, keyrack, squash merge, one PR per branch, 5-min timeout): these are battle-tested in production. changing them would break consistency with sister skills.

2. **convention-based** (release PR detection, semver tags, workflow names): these depend on ehmpathy repo conventions. the skill wraps the extant alias which already works with these conventions.

the key learning: when you wrap an alias, most assumptions are inherited. the review should focus on whether the wrapping introduces new assumptions, not re-litigate the alias.
