## non-issues (why they hold)

### extant alias compatibility — holds

wish explicitly says "base off of how my current git config alias of `git release` works". this is not backwards compat we assumed — it is the core requirement.

**why this distinction matters**: if the wish said "build a release skill", then we would ask "should we match the alias?". but the wish says "wrap the alias", so match is mandated. the blueprint correctly treats this as a requirement, not a backcompat assumption.

**evidence**: wish line 14: "they should base off of how my current git config alias of `git release` works"

### output format — holds

wish says "same vibes and structures". match emoji/tree format is required, not assumed.

**why this distinction matters**: output format could be a "nice to have" if the wish was silent on it. but wish explicitly says "same stdout pattern" and "same vibes and structures". the blueprint honors this as a requirement.

**evidence**: wish line 30: "we should use the same stdout pattern as `git release` does. same vibes and structures"

### gh pr merge flags — holds

follows from "wrap alias" requirement. if we change this, we break the requirement to match extant behavior.

**why this is not backwards compat**: we are not "preserve old behavior to avoid break". we are "implement specified behavior". the alias uses `--auto --squash`. the skill matches it. this is the spec.

**evidence**: research verified alias uses `gh pr merge --auto --squash` (see 3.1.3.research.internal.product.code.prod).

### keyrack pattern — holds (not backwards compat)

internal implementation reuse. users never see token handle. no external contract.

**why this is not backwards compat at all**: keyrack fetch is an internal detail. the skill's external contract is `git.release --to main --mode apply`. how we get the token is implementation. we could change to env var, keychain, or credential helper without break to users.

**key insight**: backwards compat concerns apply to external contracts, not internal implementation choices.

### timeout — holds

matches alias. could be made configurable later, but defaults match extant behavior per requirement.

**why this is low concern**: the timeout is a behavior detail of the watch loop. if we changed it to 10 minutes, no user's workflow breaks — they just wait longer (or less). this is a tune, not a contract.

**future consideration**: could add `--timeout` flag if users need configurability. but this is enhancement, not backcompat.

### --retry flag — holds (not backwards compat)

new explicit feature that exposes alias capability. additive.

**why this is not backwards compat**: the flag is additive. skill works without it. we add it to expose functionality that alias has. this is feature parity, not backcompat.

**key distinction**: backwards compat asks "can old users do what they could before?". --retry is new capability. no old user relied on it.

### tree utils reuse — holds (not backwards compat)

internal code organization. no external contract.

**why this is not backwards compat at all**: how we organize code (reuse output.sh functions vs inline) has zero impact on users. they see the same output regardless. this is code hygiene, not compatibility.

## reflection

### key distinction: requirements vs assumptions

this review revealed a critical distinction:

| category | definition | example |
|----------|------------|---------|
| **backwards compat** | preserve old contracts when we could break them | "users relied on X, so keep it" |
| **requirement to match** | explicitly required by wish | "base off of how my alias works" |

the first is assumption-driven. the second is requirement-driven.

the blueprint has no assumption-driven backcompat. every "match extant behavior" choice traces to the explicit wish requirement.

### why this distinction matters

backwards compat becomes bloat when:
1. we assume users need it (without evidence)
2. we "play it safe" by preserve old behavior
3. we fear break even when no contract exists

the git.release blueprint avoids these traps by:
1. wrap = explicit requirement
2. format match = explicit requirement
3. internal details (keyrack, tree utils) = not external contracts

### taxonomy of concerns analyzed

| type | count | examples |
|------|-------|----------|
| explicit requirements | 2 | alias wrap, output format |
| implicit (flow from explicit) | 2 | squash merge, timeout |
| internal patterns (not backcompat) | 2 | keyrack, tree utils |
| additive features (not backcompat) | 1 | --retry flag |

**key learning**: 4 of 7 concerns were "not backwards compat at all" — they were internal or additive. the review discipline helped separate true backcompat concerns from unrelated patterns.

### lesson for next time

when wish says "wrap X", match X's behavior is a requirement, not an assumption. no need to flag these as open questions.

when a concern is internal (users never see it), it is not a backwards compat concern. categorize it as implementation choice.

when a feature is additive (new capability), it is not backwards compat. it is feature parity or enhancement.
