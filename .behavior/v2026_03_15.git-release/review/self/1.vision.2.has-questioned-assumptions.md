## issues fixed

### issue 1: "main" branch name assumption

**what i found**: `--to main` was assumed to target a branch literally named "main". but some repos use "master" or other names.

**why this matters**: if someone has a repo with "master", the command `git.release --to main` would be unclear. does it target a branch named main, or the default branch?

**the deeper issue**: the wish says `--to main` but this is *conceptual* (the main trunk), not *literal* (a branch named "main"). i conflated the two.

**what i fixed**: added assumption #5 to the vision's "assumptions made" section:

```
5. **"main" = default branch**: `--to main` targets whatever the repo's default branch is (main, master, etc.), not literally a branch named "main"
```

**how to remember**: when you name CLI args, ask: is this a *literal value* (e.g., `--branch feature/xyz`) or a *concept* (e.g., `--to main` = "the primary branch")? if conceptual, document that it maps to different concrete values.

**impact of not fixed**: someone with a `master` branch would expect `--to master` to work, but we'd only have `--to main`. they'd be confused or blocked.

---

### issue 2: non-tty environment

**what i found**: the vision describes fancy turtle vibes with spinners and tree output. it assumes someone watches a terminal.

**why this matters**: in CI/CD pipelines or non-interactive scripts, there's no tty. spinners would produce garbled output. ANSI codes would be visible as escape sequences.

**the deeper issue**: the wish focuses on human-directed output, but skills run in varied contexts. i didn't think about where else this skill might run.

**what i fixed**: added two changes to the vision:

1. assumption #6 to "assumptions made":
```
6. **interactive terminal**: assume tty available; should detect non-tty and simplify output
```

2. friction point #5 to "what is awkward":
```
5. **non-tty contexts**: turtle vibes and spinners assume interactive terminal. should detect non-tty and emit simpler output (no spinners, just status lines).
```

**how to remember**: for any terminal tool, always ask: "what if stdout is not a tty?" answers include: detect with `[ -t 1 ]`, simplify output, skip spinners, avoid ANSI codes if not supported.

**impact of not fixed**: the skill would produce ugly or broken output in CI logs. someone would file a bug, and we'd have to fix it reactively instead of proactively.

## deeper reflection

what did i learn from this review?

**pattern 1: literal vs conceptual names**. CLI args like `--to main` can be misread as literal branch names. documentation must clarify when names are conceptual.

**pattern 2: context diversity**. skills run in more places than we imagine. interactive terminals, CI pipelines, cron jobs, IDE terminals, ssh sessions. each has different capabilities.

**pattern 3: the hidden assumption trap**. easy assumptions come from personal experience. "main" is my default branch. i have a terminal. these aren't universal truths.

2. **linear workflow**: explicit in the wish.
3. **GitHub platform**: implied by the wish's references to `gh` and github workflows.
4. **sync watch**: correct for MVP, background mode is future enhancement.
5. **tag triggers deploy**: matches the wish's description.
