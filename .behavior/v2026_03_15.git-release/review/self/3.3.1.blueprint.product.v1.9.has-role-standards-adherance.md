## potential issues examined

### issue 1? enable_automerge vs set_automerge

**what was questioned**: `enable_automerge` uses imperative verb not get/set/gen pattern. should it be `set_automerge` or `set_pr_automerge`?

**examination**:

the rule.require.get-set-gen-verbs states:

> **scope**: applies to all operations in domain.operations/; applies to all dao methods
> **exempt**: contract/cli entry points; imperative action commands (e.g., dispatchTask, enqueueTask)

`enable_automerge` is:
1. an imperative action (tells github to enable a feature)
2. a CLI-adjacent operation (wraps `gh pr merge --auto`)
3. not a domain operation in typescript sense

**verdict**: NOT AN ISSUE — falls under exempt categories.

---

### issue 2? rerun_failed_workflows imperative verb

**what was questioned**: `rerun_failed_workflows` uses imperative verb. same pattern as enable_automerge.

**examination**:

this function wraps `gh run rerun "$run_id" --failed`. it is:
1. an imperative command to github
2. a side effect operation
3. not a domain retrieval or mutation in our codebase

**verdict**: NOT AN ISSUE — imperative command exempt from get/set/gen.

### issue 3? git.release.sh single file vs verb suffix

**what was questioned**: extant skills use `git.commit.set.sh`, `git.commit.push.sh` with verb suffix. should this be `git.release.run.sh` or `git.release.do.sh`?

**examination**:

reviewed in has-consistent-conventions (review 6). conclusion:

| skill | has verb suffix? | why? |
|-------|-----------------|------|
| git.commit.set.sh | yes | commit has variants (set, push, uses) |
| git.commit.push.sh | yes | commit has variants |
| git.rebase.begin.sh | yes | rebase has variants (begin, continue, abort) |
| show.gh.action.logs.sh | no | single-purpose skill |
| sedreplace.sh | no | single-purpose skill |

git.release is single-purpose (no variants planned). `release` is the verb implicit in the name.

**verdict**: NOT AN ISSUE — follows single-purpose skill pattern.

### issue 5? plan/apply vs PLAN/APPLY case

**what was questioned**: blueprint uses lowercase `plan|apply`. the forbidden term rule shows `'PLAN' | 'APPLY'` uppercase. which is correct?

| skill | mode values |
|-------|-------------|

| git.commit.set.sh | `--mode plan`, `--mode apply` |

bash conventions use lowercase flags. typescript enums use UPPERCASE.

blueprint is for bash skill, so lowercase is correct.

**verdict**: NOT AN ISSUE — bash uses lowercase, typescript uses uppercase.

### issue 6? fetch_github_token in operations vs separate file

**what was questioned**: blueprint puts token fetch in `git.release.operations.sh`. should auth be separate (e.g., `git.auth.sh`)?

**examination**:

reviewed has-consistent-mechanisms (review 5). conclusion was:

> extract keyrack token fetch logic into git.release.operations.sh as `fetch_github_token` operation. this avoids duplicate code while also avoiding fragile source of git.commit.push.sh.

decision was deliberate to avoid:
1. duplication (copy from git.commit.push.sh)
2. fragile link (source git.commit.push.sh directly)
3. premature abstraction (shared git.auth.sh before 3+ users)

**verdict**: NOT AN ISSUE — follows wet-over-dry and mechanism consistency decisions.

### PATH injection for test mocks — holds

blueprint specifies PATH injection mock strategy per howto.mock-cli-via-path lesson.

**evidence**: lesson documents this pattern for CLI tool mocks in integration tests.

### no else branches in dispatch — holds

blueprint shows `--to main` and `--to prod` as separate dispatch targets, not else branches. implementation should use bash case statement:

```bash
case "$TO" in
  main) ... ;;
  prod) ... ;;

```

**evidence**: bash case statements are the idiomatic way to handle dispatch without else branches.

### integration test classification — holds

blueprint specifies `.integration.test.ts` not `.test.ts` for unit tests. this is correct because:
1. tests invoke bash skill via spawnSync (process boundary)
2. tests use PATH injection mocks (not pure function calls)
3. tests verify stdout output (system boundary)

per rule.forbid.remote-boundaries, unit tests cannot cross process boundaries.

**evidence**: the test file touches external CLI (`gh`), which requires integration test classification even with mocks.

### print\_ function prefix — holds

output functions use `print_` prefix instead of `get_` or `set_`. this is correct because:
1. print functions are output utilities, not domain operations
2. rule.require.get-set-gen-verbs explicitly exempts utilities
3. `print` is the action verb (write to stdout)

**evidence**: git.commit/output.sh uses same `print_` prefix pattern.

### argument parse boilerplate — holds

blueprint does not extract shared argument parser. each skill has own parse block. this follows rule.prefer.wet-over-dry:

> wait for 3+ usages across contexts before abstraction

git.release has unique flags (`--to`, `--retry`). sharing parse logic would require:
1. dependency management
2. version coordination
3. import complexity

cost exceeds benefit. copy-paste is pragmatic choice.

**evidence**: all git.* skills have own argument parse. none share a library.
