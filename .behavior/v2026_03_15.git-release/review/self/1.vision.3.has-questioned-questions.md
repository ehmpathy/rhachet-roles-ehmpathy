### question 2: merge method

**original**: squash, merge commit, or rebase? or follow repo settings?

**can logic answer now?** yes!

**the logic**: repos have branch protection settings that specify allowed merge methods. the skill should follow repo settings (via `gh pr merge` default behavior), not override them.

**what if we hardcode one method?** we'd break repos that only allow squash (or only allow merge commits). follow repo settings = pit of success.

**verdict**: [answered] — follow repo settings. `gh pr merge` without `--squash` or `--rebase` flag uses repo defaults.

### question 4: which CI to watch

**original**: hardcode `test.yml` + `publish.yml|deploy.yml`? or detect from repo?

**can logic answer now?** yes!

**the logic**: repos have different workflow names. hardcoded names are fragile. `gh workflow list` can enumerate workflows. we can:
1. detect workflows dynamically
2. filter by trigger (e.g., `on: push to main`, `on: release`)
3. fall back to common names if detection fails

**what if we hardcode names?** we'd break repos with different names (e.g., `ci.yml`, `build.yml`, `release.yml`).

**verdict**: [answered] — detect dynamically via `gh workflow list`, fall back to common names.

### question 5: automerge

**original**: should `--to main` enable automerge and wait? or actually merge?

**can logic answer now?** partially.

**the logic**:
- "enable automerge and wait" = gh pr merge --auto, then wait for checks
- "actually merge" = wait for checks, then gh pr merge

the difference: `--auto` delegates to github's automerge. direct merge is explicit.

**can docs/code answer now?** no. depends on desired behavior.

**should research answer later?** yes. we should research how `gh pr merge --auto` behaves.

**does only wisher know?** yes. the wisher must decide the behavior.

**verdict**: [wisher] + [research] — research `gh pr merge --auto` behavior, then ask wisher.

---

### question 6: branch protection

**original**: if main requires reviews, what happens?

**can logic answer now?** yes!

**the logic**: if main requires reviews and the PR doesn't have them, `gh pr merge` will fail. plan mode should surface this *before* apply mode attempts the merge.

**implementation**: plan mode checks PR status via `gh pr view --json reviewDecision,statusCheckRollup`. if blocked, show "PR needs reviews" instead of "ready to ride".

**verdict**: [answered] — plan mode surfaces blockers before apply mode attempts merge.
