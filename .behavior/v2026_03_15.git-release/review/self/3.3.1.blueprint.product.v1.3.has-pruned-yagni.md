## the question

> YAGNI = "you ain't gonna need it"
>
> for each component in the blueprint, ask:
> - was this explicitly requested in the vision or criteria?
> - is this the minimum viable way to satisfy the requirement?
> - did we add abstraction "for future flexibility"?
> - did we add features "while we're here"?
> - did we optimize before we knew it was needed?

## reflection

this blueprint benefits from several constraints:

1. **wrap, don't reimplement**: the vision says "wrap extant alias" which naturally limits scope. we don't design new features, just expose extant behavior via a new contract.

2. **BDD criteria define scope**: test cases come from criteria, not from "nice to have" ideas.

3. **pattern reuse**: reuse of git.commit patterns (output.sh, operations.sh, keyrack) means less new code to add.

the key lesson: when a blueprint wraps an extant implementation, YAGNI risk is low because scope is pre-defined by what already exists.
