### skill file names

**extant patterns**:
- `git.commit.set.sh` — verb follows skill name
- `git.commit.push.sh` — verb follows skill name
- `git.rebase.begin.sh` — verb follows skill name

**blueprint**:
- `git.release.sh` — no verb suffix

**ISSUE FOUND**: blueprint uses `git.release.sh` but extant pattern is `git.{skill}.{verb}.sh`.

**options**:
1. rename to `git.release.run.sh` (add verb)
2. keep as `git.release.sh` (single-purpose skill)
3. rename to `git.release.main.sh` (add clarifier)

**analysis**: git.release is invoked as single command unlike git.commit which has set/push/uses variants. the skill IS the verb.

**verdict**: KEEP — git.release.sh is acceptable because:
- skill has no variants (unlike git.commit.set, git.commit.push)
- `release` is the verb (unlike `commit` which is noun)
- precedent: show.gh.action.logs.sh has no verb suffix

## non-issues (why they hold)

### git.release.sh vs git.release.{verb}.sh — holds

the extant pattern is `git.{skill}.{verb}.sh` (e.g., git.commit.set.sh). but git.release has no variants — the skill IS the action.

**evidence searched**:
| file | has verb suffix? | why? |
|------|-----------------|------|
| git.commit.set.sh | yes | commit has variants (set, push, uses) |
| git.commit.push.sh | yes | commit has variants |
| git.rebase.begin.sh | yes | rebase has variants (begin, continue, abort) |
| show.gh.action.logs.sh | no | single-purpose skill |
| sedreplace.sh | no | single-purpose skill |

**pattern extracted**: verb suffix needed when skill has variants. git.release is single-purpose.

**why this is not divergence**: the underlying pattern is "skill name reflects purpose". git.release.sh reflects purpose as clearly as git.commit.set.sh. no ambiguity exists.

### --to flag — holds (new but consistent)

`--to main|prod` is new. but:
- follows `--{word}` pattern
- requires value like `--mode`
- semantic: "release to target"

**extant flag patterns**:
| flag | value? | pattern |
|------|--------|---------|
| --mode | yes (plan\|apply) | enum |
| --push | no | boolean |
| --unstaged | yes (ignore\|include) | enum |
| --output | yes (tree\|json) | enum |

**--to fits**: `--to main|prod` follows enum pattern like `--mode`. consistent.

**why not --target?**: `--to` is more natural English ("release to main" not "release target main"). semantic clarity over consistency with imagined alternative.
