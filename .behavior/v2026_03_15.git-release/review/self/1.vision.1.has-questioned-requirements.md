### 3. wrapping vs replacing `git release` alias

**who said?** the wish implies this: "they should base off of how my current git config alias of `git release` works"

**what i assumed**: the skill would be a new implementation with turtle vibes.

**re-reading the wish**:
```
`git.release --to main` should run `git.release --watch [--apply]`
`git.release --to prod` should run `git release --apply --watch && ...`
```

**issue found**: 🔴 the wish describes `git.release` calling the extant `git release` alias. my vision describes a new skill that implements the release logic directly.

**resolution**: the vision should clarify whether we:
1. wrap the extant `git release` alias (lighter, less code, depends on alias)
2. reimplement the logic (heavier, more control, no dependency)

**what happens if we wrap?**
- simpler implementation
- requires the alias to extant
- less control over output format

**what happens if we reimplement?**
- more code
- full control over vibes
- works without alias

**verdict**: ✅ resolved. wisher decided to wrap the extant alias. this bootstraps implementation and reuses battle-tested logic.

### 6. same stdout pattern as `git release`

**who said?** the wish: "we should use the same stdout pattern as `git release` does, same vibes and structures"

**what i assumed**: turtle vibes with tree structure, matched to other skills.

**issue found**: 🟡 i hadn't seen the current `git release` output. my vision assumed it follows turtle patterns.

**resolution**: ✅ researched `~/.bash_aliases` `git_alias_release` function. discovered the actual pattern:
- `🌊` = release header (not `🐢`)
- `🫧` = no PR found
- `⛈️` = checks failed
- `🐢` = checks in progress (not header)
- `👌` = all passed
- `🌴` = automerge/merged
- `🐚` = needs rebase (not skill name)
- `🥥` = watch mode

**verdict**: ✅ resolved. vision updated to match real pattern.

## issues fixed

### issue 1: wrap vs reimplement ambiguity

**what i found**: re-reading the wish, line 26 says `git.release --to main` should run `git.release --watch`. line 30 says `--to prod` should run `git release --apply --watch && ...`. this references the extant `git release` alias, not a new implementation.

**why this matters**: if we wrap the alias, we inherit its behavior but depend on it. if we reimplement, we control everything but write more code.

**what i fixed**: i added a new question (#1) to the "questions for the wisher" section in 1.vision.md:
```
1. **wrap vs reimplement**: should this skill wrap the extant `git release` alias, or reimplement the logic with full control? wrap = lighter but depends on alias. reimplement = heavier but independent.
```

**how to remember**: always re-read the original wish when writing the vision. look for references to extant tools or aliases that might imply wrapping rather than reimplementing.
