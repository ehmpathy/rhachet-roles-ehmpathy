## non-issues (why they hold)

### semantic-release PR format — holds

blueprint specifies `chore(release): vX.Y.Z` pattern. this matches the vision [answered] section which says semantic-release creates the release PR. the format is standard semantic-release convention.

**evidence**: vision line confirms "standard-version/semantic-release creates release PR with version bump"

### timeout implicit — holds

blueprint says "exit on complete or timeout" without explicit timeout value. this is acceptable because:

1. blueprint explicitly says "[←] wrap _git_release_watch from extant alias"
2. the extant alias has the 5-minute timeout
3. the timeout is an implementation detail, not a behavioral contract

### test mock strategy — holds

blueprint specifies PATH injection for mock of gh CLI. vision and criteria don't mention test approach. this is acceptable because:

1. test implementation is not part of the behavioral spec
2. PATH injection is a standard pattern (documented in howto.mock-cli-via-path lesson)
3. mock strategy is an implementation choice, not a user-visible contract

### reimplement vs wrap terminology — holds

blueprint uses "[←] reimplement" for some items and "[←] wrap" for others:

| item | marker | why correct |
|------|--------|-------------|
| _gh_with_retry | reimplement | cannot source ~/.bash_aliases at runtime |
| fetch_github_token | reimplement | copy logic from git.commit.push.sh |
| watch loop | wrap | calls extant alias function |

terminology reflects the actual reuse strategy. has-consistent-mechanisms review validated this taxonomy.

### operations file split — holds

operations are separated into git.release.operations.sh because:

1. follows extant pattern: git.commit has git.commit.operations.sh
2. operations are reusable across potential future skills
3. keeps main file focused on flow, not implementation

**evidence**: git.commit.operations.sh has same structure (helper functions for gh operations).

### output.sh source path — holds

`source "$SKILL_DIR/../git.commit/output.sh"` is correct because:

1. git.release/ and git.commit/ are peer directories under skills/
2. SKILL_DIR points to git.release/
3. `../git.commit/output.sh` resolves to sister skill's output.sh

**evidence**: filediff tree shows both under `src/domain.roles/mechanic/skills/`.

## methodology

### approach taken

this adherance review followed these steps:

1. **read blueprint section by section** — 14 sections total
2. **for each section, find vision/criteria reference** — traced back to spec
3. **compare exact phrases** — checked for semantic match
4. **flag potential deviations** — 6 items flagged for examination
5. **verify each flagged item** — all 6 verified as non-deviations
6. **build adherance matrix** — 14x3 cross-reference table

### questions asked per section

| question | purpose |
|----------|---------|
| what does blueprint say? | establish blueprint claim |
| what does vision say? | find authoritative spec |
| what does criteria say? | find behavioral requirement |
| do they match semantically? | detect deviation |
| if mismatch, is it intentional? | classify deviation vs design choice |

### why this matters

adherance review catches:
- misinterpretations of spec
- scope creep beyond spec
- accidental deviations from vision
- incorrect implementation of criteria

this blueprint passed all checks — it stays within bounds.
