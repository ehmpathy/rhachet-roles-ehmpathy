## analysis

### mechanism: output.sh (turtle vibes)

**extant mechanism**: `git.commit/output.sh` provides base turtle vibe functions:
- `print_turtle_header()`
- `print_tree_start()`
- `print_tree_branch()`
- `print_tree_leaf()`

**git.release approach**: sources the extant mechanism and extends it

```bash
# git.release/output.sh line 21
source "$SKILL_DIR/../git.commit/output.sh"
```

then adds release-specific functions:
- `print_release_header()` — 🌊 release: branch → target
- `print_check_status()` — 👌/🐢/⚓ status
- `print_automerge_status()` — 🌴 automerge
- `print_watch_status()` — 🥥 watch mode

**verdict**: ✓ consistent — reuses extant, extends appropriately

### mechanism: keyrack token fetch

**extant mechanism**: `git.commit.push.sh` lines 225-289 has inline token fetch with:
- temp file for output capture
- JSON parse via jq
- fallback unlock to ehmpath owner
- detailed error messages

**git.release approach**: reimplemented in `operations.sh` as `fetch_github_token()`:
- simpler implementation (no temp file)
- same fallback pattern (try keyrack, unlock ehmpath, retry)
- returns token or empty

**why duplication?**
the blueprint explicitly prescribed this:
> "reimplement in git.release.operations.sh as fetch_github_token (same logic as git.commit.push.sh lines 225-289)"

the blueprint chose to reimplement rather than extract a shared utility because:
1. git.commit.push.sh has more complex error output (shows full keyrack error)
2. git.release has simpler needs (token or empty)
3. extract shared utility would change git.commit.push.sh behavior

**future opportunity**: extract `fetch_github_token()` to a shared location (e.g., `shared/keyrack.operations.sh`) that both skills can use. not done now because:
- blueprint did not request it
- change to git.commit.push.sh was out of scope
- would require test for both skills

**verdict**: ✓ intentional duplication per blueprint — flagged as future refactor opportunity
