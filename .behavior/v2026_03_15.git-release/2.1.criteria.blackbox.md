# 2.1.criteria.blackbox: git.release skill

## crystal clear

**`--mode apply` = enable automerge + WATCH until done**

- skill does NOT return until all checks complete
- watches, waits, only exits when checks pass and PR merges, or fails

**exit codes**

- exit 0: success
- exit 1: malfunction error (gh failed, network error, unexpected state)
- exit 2: constraint error (needs rebase, no PR, user must fix something)

**`--to main` = just this branch**

- enable automerge on feature branch PR, watch it merge
- if PR needs rebase: failloud, tell caller to rebase, exit 2 (constraint error)
- if no PR for branch: failloud, tell caller to push, exit 2 (constraint error)

**`--to prod` = full journey (leverages `--to main`)**

1. `--to main`: enable automerge on feature branch PR, watch it merge
2. find release PR, enable automerge, watch it merge
3. watch tag workflows complete

**decomposition**

`--to main` and `--to prod` maximally decompose to recompose common operations (PR lookup, automerge, watch loop).

---

## usecase.1 = release to main (plan mode)

```
given a feature branch with an open PR
  when git.release --to main
    then show turtle vibe header
    then show skill invocation line
    then show release status with current checks
    then show automerge status
    then show hint to use --mode apply
      sothat user knows how to proceed

given a feature branch with no PR
  when git.release --to main
    then failloud "no open pr for branch" to stderr
    then show hint to use git.commit.push
    then exit 2 (constraint error)
      sothat user knows to push first

given user is on main branch
  when git.release --to main
    then show release PR status if one extant
    then show latest tag status if no release PR
      sothat user sees current release state
```

## usecase.2 = release to main (apply mode)

```
given a feature branch with an open PR
  when git.release --to main --mode apply
    then show turtle vibe header (cowabunga)
    then enable automerge via gh pr merge --auto --squash
    then watch CI checks until complete
    then show merged status when complete
      sothat PR lands on main automatically

given a feature branch with failed CI
  when git.release --to main --mode apply
    then show turtle vibe header (bummer dude)
    then show failed checks with links
    then show hint to use --retry
      sothat user can rerun failed workflows

given a feature branch that needs rebase
  when git.release --to main --mode apply
    then failloud "needs rebase" to stderr
    then exit 2 (constraint error)
    then do not enable automerge
      sothat user fixes conflicts first before release

given automerge fails to enable (gh error)
  when git.release --to main --mode apply
    then gh error goes direct to stderr (not captured/reformatted)
    then exit 1 (external error)
      sothat user sees exact gh error and can diagnose

given PR ready to merge immediately (clean status)
  when git.release --to main --mode apply
    then gh pr merge --auto returns "clean status" error
    then suppress the error (not a real failure)
    then check if PR actually merged
    then show "automerge enabled [added] -> and merged already"
    then exit 0 (success)
      sothat PR merges immediately without watch loop
```

## usecase.3 = release to prod (plan mode)

```
given feature branch with open pr
  when git.release --to prod
    then run --to main flow first
    then stop after --to main status shown
      sothat user sees feature branch status before prod

given feature branch without pr
  when git.release --to prod
    then show "hold up" turtle vibe header
    then show "no open pr for branch"
    then show "did you git.commit.push to create the pr yet?"
    then exit 1
      sothat user knows to push first

given user on main branch
  when git.release --to prod
    then show current release pr status (if extant)
    then show what --mode apply would do
      sothat user previews full release cycle

given no release pr extant (on main)
  when git.release --to prod
    then show latest tag status
    then show tag workflow status
      sothat user sees last release state
```

## usecase.4 = release to prod (apply mode)

```
given feature branch with open pr
  when git.release --to prod --mode apply
    then run --to main flow first
    then watch feature pr merge
    then only proceed to release pr after feature pr merged
      sothat full journey is: feature → main → release → tag

given feature branch without pr
  when git.release --to prod --mode apply
    then fail fast with "did you git.commit.push" message
    then exit 1
      sothat user knows to push first

given a release pr extant (on main)
  when git.release --to prod --mode apply
    then show turtle vibe header (radical)
    then enable automerge on release pr
    then watch release pr merge
    then watch tag workflows (publish.yml or deploy.yml)
    then show completion status
      sothat full release ships unattended

given release pr already merged
  when git.release --to prod --mode apply
    then show "already merged" status
    then watch tag workflows
      sothat user can still verify deploy status

given tag workflow fails
  when git.release --to prod --mode apply
    then show failed workflow with link
    then show hint to use --retry
      sothat user can rerun failed deploy
```

## usecase.5 = retry failed workflows

```
given failed CI checks on PR
  when git.release --to main --mode apply --retry
    then rerun failed workflows via gh run rerun --failed
    then show "rerun triggered" confirmation
    then continue watch loop
      sothat flaky tests get another chance

given failed tag workflows
  when git.release --to prod --mode apply --retry
    then rerun failed deploy/publish workflows
    then show "rerun triggered" confirmation
    then continue watch loop
      sothat failed deploys can recover
```

## usecase.6 = watch mode

```
given --mode apply with checks in progress
  when checks are still in progress
    then poll check status periodically
    then show elapsed time
    then show pending check count
      sothat user sees progress without browser

given --mode apply with all checks passed
  when automerge completes
    then show completion status
    then exit successfully
      sothat user knows release succeeded

given --mode apply with checks that fail mid-watch
  when failure detected
    then show failed checks immediately
    then show hint to --retry or investigate
    then exit with error
      sothat user can act on failures fast

given --mode apply with rebase required mid-watch (race condition)
  when another PR merges to main while we watch
    then detect mergeStateStatus changed to BEHIND
    then show "needs rebase (another PR merged while we watched)"
    then show hint to rebase onto main and retry
    then exit 2 (constraint error)
      sothat user knows to rebase before release can proceed
```

## boundary conditions

```
given github token not available
  when git.release --to main --mode apply
    then fetch token from keyrack
    then show unlock instructions if keyrack locked
    then fail gracefully with clear error
      sothat user knows how to fix auth

given non-interactive terminal (no tty)
  when git.release with any flags
    then emit simple status lines (no spinners)
    then preserve all status information
      sothat automation contexts work correctly

given rate limit or network error
  when gh api call fails
    then retry with backoff (per extant alias)
    then fail gracefully if retries exhausted
      sothat transient errors don't break releases
```
