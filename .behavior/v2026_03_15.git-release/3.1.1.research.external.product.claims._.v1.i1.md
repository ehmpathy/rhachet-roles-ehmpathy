# 3.1.1.research.external.product.claims: git.release skill

## summary

this research documents claims — facts, assumptions, questions, and opinions — relevant to the `git.release` skill. the skill wraps the extant `git release` alias to provide terminal-based release automation with plan/apply semantics.

---

## claims

### [FACT] manual releases are error-prone

> "Manual releases are error-prone and time-consuming—you might forget to update the changelog, tag the wrong commit, or publish an incomplete artifact."
> — [1] OneUptime Blog: How to Automate Releases with GitHub Actions

this supports the vision's goal of automated release via terminal.

**citation [1]**: [How to Automate Releases with GitHub Actions](https://oneuptime.com/blog/post/2026-01-25-automate-releases-github-actions/view) [blog] (2026-01)

## anti-patterns

### frequent poll
> "If you set up a job running every minute that polls an API looking for code commits... that's over 500,000 queries per year."
> — [15] webhooks vs polling

extant alias avoids this by: poll 5-15s intervals, cap at 5 minutes.

### no progress indication
> "Make an effort to display meaningful status updates for actions that take time."
> — [13] Evil Martians

extant alias avoids this by: show check count, elapsed time, status updates.
