# vision: fix `--from main` footgun in git.release

## the outcome world

### day-in-the-life

a clone finishes a feature, pushes, and wants to ship to prod. it reaches for the
release skill. today it sometimes types `--from main --into prod` because that
phrase appears in the skill header usage block and "feels" like the way to release
to prod. that's the footgun.

in the outcome world:

- the clone reads the skill header and sees `--into prod` is THE default way to
  release. `--from main` is fenced off as a special-circumstance escape hatch.
- if the clone (a non-human) still reaches for `--from main` while on a feature
  branch, the skill stops it cold with a ConstraintError: "can you just use
  `--into prod`? it's safer. if not, tell me `--why`."
- the clone re-reads, realizes `--into prod` is exactly what it wanted, and ships
  correctly.

### before / after contrast

| moment | before | after |
|--------|--------|-------|
| clone wants prod release | guesses `--from main --into prod` | uses `--into prod` (the documented default) |
| clone on feat runs `--from main` | silently targets the *prior* release PR (stale) | blocked with ConstraintError unless `--why` given |
| human runs `--from main` | works | still works (TTY = human, no friction) |
| skill header reading | `--from main` looks like a first-class path | `--from main` flagged as special-circumstance only |

### the "aha" moment

the value clicks when a clone types `--from main --into prod` out of habit and
gets:

```
🐢 hold up dude...

🐚 git.release --from main --into prod
   └─ ✋ blocked: --from main needs a --why

   you're on a feature branch. --from main skips your branch and
   releases whatever is already on main.

   ➤ if you just want to ship your work, that's:
     rhx git.release --into prod

   ➤ if you really need --from main (hotfix, recovery), say why:
     rhx git.release --from main --into prod --why "recover from broken release-please state"
```

the clone realizes: "oh — `--into prod` is what i actually wanted." pit of success.

## user experience

### usecases & goals

| user | goal | path |
|------|------|------|
| clone (non-human) on feat | ship my merged work to prod | `rhx git.release --into prod --mode apply --watch` |
| clone on feat, habitually | release to prod (wrong instinct) | `--from main` → blocked → corrected to `--into prod` |
| clone with genuine need | hotfix/recover from broken main state | `--from main --into prod --why "<reason>"` |
| human (TTY) | anything, including `--from main` | works without `--why` friction |

### contract inputs & outputs

proposed contract change (additive — no breaking changes to extant flags):

```
rhx git.release [--into main|prod] [--from main|<branch>] [--why "<reason>"] \
                [--watch] [--apply] [--mode plan|apply] [--retry] [--dirty block|allow]
```

new input:
- `--why "<reason>"` — articulation required when a non-human uses `--from main`
  from a feature branch. free-text string explaining why `--into prod` won't work.

new behavior (the guard):
- **when** `--from main` is set
- **and** the caller is on a feature branch (not on `main`/default branch)
- **and** the caller is non-human (stdin not a TTY, and `__I_AM_HUMAN != true`)
- **and** `--why` is absent or empty
- **then** ConstraintError (exit 2): explain the footgun, suggest `--into prod`,
  show how to articulate `--why` if truly needed.

outputs:
- exit 2 + stderr ConstraintError when guard trips
- unchanged behavior in all other cases (human, on-main, `--why` provided, no `--from`)

### what leveraging it looks like

```bash
# the happy path (what most clones actually want)
rhx git.release --into prod --mode apply --watch

# the blocked path (clone habit → corrected)
rhx git.release --from main --into prod
# → exit 2, "can you just --into prod?"

# the escape hatch (genuine special circumstance)
rhx git.release --from main --into prod --why "release-please PR stuck, manual recovery"

# human at a terminal — no friction
rhx git.release --from main --into prod   # works (TTY detected)
```

### timeline

1. clone reads skill header → sees `--into prod` is the default
2. clone runs `--into prod` → ships correctly (most common)
3. OR clone runs `--from main` out of habit → blocked in <1s with a clear redirect
4. clone either switches to `--into prod` or adds `--why`

## mental model

### how users describe it to a friend

> "the release tool has a guard rail. if a bot tries to use the advanced
> `--from main` shortcut without a reason, it stops and asks 'don't you just
> want the normal prod release?' — because 9 times out of 10, you do."

### analogies / metaphors

- **the seatbelt buzzer**: you *can* drive without it, but the tool nags non-humans
  into the safe default.
- **a `sudo` reason prompt**: the dangerous path stays available, but you must
  justify it out loud.
- **a fork in the trail with a sign**: "main road → prod. side trail (`--from main`)
  → experts only, state your reason."

### terms

| we say | users might say |
|--------|-----------------|
| ConstraintError | "it blocked me" / "it yelled at me" |
| `--from main` footgun | "the wrong release command" |
| articulate `--why` | "give it a reason" |
| pit of success | "it nudged me to the right thing" |

## evaluation

### how well it solves the goals

- **goal 1 (docs clarity)**: solved by rewriting skill header + `howto.git-release`
  brief so `--into prod` is THE default and `--from main` is fenced as special-only.
- **goal 2 (guard non-humans)**: solved by the `--why`-gated ConstraintError on the
  `feat + --from main + non-human` path.

### pros

- additive, non-breaking: `--why` is new; all prior invocations still parse
- reuses the established human-detection pattern (`! -t 0 && __I_AM_HUMAN != true`)
  already used by `git.commit.uses.*`
- reuses the established ConstraintError shape (exit 2, turtle "hold up dude...")
- keeps the escape hatch open — doesn't forbid `--from main`, just gates it

### cons / tradeoffs

- adds a new flag (`--why`) to an already flag-heavy contract
- `--why` content is unvalidated free-text — a clone could write `--why "because"`
  to bypass. mitigation: the goal is to force a *pause + articulation*, not to
  semantically judge the reason. the friction itself is the value.
- the human-vs-non-human signal is TTY-based, which is a heuristic, not a proof.

### edgecases & pit-of-success

| edgecase | behavior |
|----------|----------|
| human on feat, `--from main`, no `--why` | allowed (TTY = human) |
| non-human on feat, `--from main`, `--why "x"` | allowed (articulated) |
| non-human on feat, `--from main`, no `--why` | **blocked** (the target case) |
| non-human ON main, `--from main` | allowed — already on main, no feat to skip; this is the legitimate main-release path |
| `--from main --into main` | unchanged: extant ConstraintError ("already on main") |
| `--from <some-branch>` (not main) | open question — see below |
| `--why` given WITHOUT `--from main` | accepted but inert (no-op); `--why` only applies as a `--from main` modifier. not an error — avoids a surprise for a caller who over-supplies it |
| integration tests | use `__I_AM_HUMAN=true` bypass, consistent with extant tests |

## open questions & assumptions

### assumptions

1. **human = TTY or `__I_AM_HUMAN=true`.** assumed the same detection used by
   `git.commit.uses.local.sh:135` is the right signal here. verified the pattern
   is present and tested.
2. **the guard scope is `feat + --from main` only.** when already on `main`,
   `--from main` is the legitimate way to release main→prod, so no `--why` needed.
   the wish says "if someone's on a branch and runs `--from main`" — i read
   "on a branch" as "on a feature branch (not main)".
3. **`--why` is free-text, unvalidated.** the value is the forced articulation,
   not semantic validation.

### open questions (triaged)

1. **[wisher]** should `--from <non-main-branch>` (e.g. `--from turtle/other-feat`)
   also require `--why`, or is the guard strictly for the literal `--from main`
   footgun? *(my default: guard only `--from main`, since that's the documented
   footgun in the wish. only the wisher knows whether to widen scope.)*
2. **[wisher]** should `--why` be **persisted** anywhere (e.g. logged to `.log/` for
   audit), or is it purely an in-the-moment articulation gate? *(my default:
   in-the-moment only, no persistence, to keep scope tight. a preference call only
   the wisher can settle.)*
3. **[answered]** is a minimum-length / non-empty check on `--why` desired? **yes —
   resolved via logic now.** an empty/whitespace `--why` defeats the articulation
   purpose, so the guard requires `--why` to be non-empty after trim. no wisher input
   needed; the requirement follows directly from the wish's intent ("articulate why").
4. **[answered]** which branch signal detects "on a feature branch"? **resolved via
   code:** `CURRENT_BRANCH != DEFAULT_BRANCH` (`git.release.sh:561-562`), NOT
   `goal_from` (which collapses to "main" whenever `--from main` is passed — see
   internal research).
5. **[answered]** where in the skill does the guard fire? **resolved via code:** it
   must run right after arg parse (≈ line 130), before the dirty/permission checks
   (172-242), which requires a move of `get_current_branch` up from line 561 (see
   review r3.has-questioned-assumptions). a sequence constraint for the blueprint.
6. **[research]** none — no external research items remain (no external dependencies;
   the release-please race described in the wish is motivation, not an integration).

### internal research

verified against extant behavior (cited):

- **flag parsing**: `git.release.sh:72-129` — main arg loop; `--from`/`--into`
  captured into `FROM`/`TO`. unknown args → exit 2. i'll add `--why` here.
- **goal inference**: `git.release._.get_one_goal_from_input.sh:40-89` — derives
  `from`/`into`; already has a ConstraintError for `from=main && into=main`
  (lines 73-84). the new guard fits naturally near here or in the main skill after
  branch detection.
- **branch detection**: `git.release.sh:561-578` — `CURRENT_BRANCH`,
  `DEFAULT_BRANCH`, and `RELEASE_BRANCH` resolution. the "on a feature branch"
  signal = `CURRENT_BRANCH != DEFAULT_BRANCH`.
  - **correction (caught on review r2, fixing a wrong r1 note):** the guard's
    "on a feature branch" condition MUST use `CURRENT_BRANCH != DEFAULT_BRANCH`
    directly — NOT `goal_from`. reason: `get_one_goal_from_input.sh:50-51` sets
    `goal_from="main"` whenever `--from main` is passed, *regardless of physical
    branch*. so `goal_from` collapses "on feat + --from main" and
    "on main + --from main" into the same value and cannot tell them apart. an
    earlier r1 note suggested preferring `goal_from` as the feat-signal — that was
    incorrect and is hereby retracted. the physical-branch check is the only
    reliable signal for "did this clone skip its own feature branch."
- **human detection**: `git.commit.uses.local.sh:133-142` — pattern
  `[[ ! -t 0 && "${__I_AM_HUMAN:-}" != "true" ]]` = non-human. tested in
  `git.commit.uses.integration.test.ts` (case20, case21, case30).
  - **caution (caught on review r2):** `git.release.sh` already defines its own
    `IS_TTY` at lines 165-168 using `-t 1` (stdout), purely for output formatting.
    that is a DIFFERENT signal from the human-detection `-t 0` (stdin) convention.
    the new guard MUST use the stdin `-t 0` + `__I_AM_HUMAN` convention (the
    human signal), NOT reuse `IS_TTY` (the output signal). conflating them would
    misclassify a human who pipes output (e.g. `| tee`) as non-human. the blueprint
    must keep these two TTY checks distinct.
- **ConstraintError shape**: exit 2 + turtle "hold up dude..." header to stderr
  (`get_one_goal_from_input.sh:74-84`) and stdout variants
  (`git.release.sh:175-189`). i'll match the established shape.
- **briefs to update**: `howto.git-release.[lesson].md:47-65` already calls
  `--from main` a footgun — i'll strengthen it and make `--into prod` THE default.
  also the skill header usage block (`git.release.sh:10-39`) which currently lists
  `--from main` prominently.
- **test scenes**: `git.release.p3.scenes.on_feat.from_main.integration.test.ts`
  covers the feat+`--from main` matrix today (all currently succeed). new guard
  means these extant scenes must pass `--why` or `__I_AM_HUMAN=true` to keep
  working — that's the regression-impact surface.
- **dedicated test suite (new, separate file)**: this behavior gets its OWN
  integration test file — not just edits to extant scenes. it exhaustively covers
  the guard's truth table:
  - non-human + feat + `--from main` + no `--why` → exit 2 (blocked, snapshot the
    ConstraintError stdout/stderr)
  - non-human + feat + `--from main` + `--why "x"` → allowed
  - human (`__I_AM_HUMAN=true`) + feat + `--from main` + no `--why` → allowed
  - non-human + on-main + `--from main` → allowed (legitimate main release)
  - `--why ""` / whitespace-only → blocked (per assumption 3, pending wisher)
  - snapshots per `rule.require.snapshots` + explicit exit-code + message assertions.
- **stdout/vibes**: turtle treestruct format per
  `rule.require.treestruct-output` and `howto.write.skills-stdout`.

## what is awkward?

1. **`--why` as free-text is "soft".** a determined clone can satisfy it with junk.
   it fights the instinct that a guard should be airtight. but the wish explicitly
   frames this as "ask them to articulate" — the pause is the point, not enforcement.
   feels slightly uncomfortable but matches the wish's intent.

2. **TTY-based human detection is a heuristic.** in an IDE-integrated or piped
   human session, a real human might be classified non-human. mitigation: `--why`
   is always a valid escape, so a misclassified human just adds one flag. low harm.

3. **flag sprawl.** `git.release` already has 6+ flags. adding `--why` (only
   meaningful alongside `--from main`) is contextual and could confuse readers who
   see it standalone. mitigation: document it as a sub-modifier of `--from main`,
   not a top-level concept.

4. **scope ambiguity on `--from <branch>`.** the wish names `--from main`
   specifically, but `--from <other-branch>` shares the same "skips your current
   work" hazard. guarding only `--from main` is narrower but matches the literal
   wish; widening is a judgment call flagged above for the wisher.
