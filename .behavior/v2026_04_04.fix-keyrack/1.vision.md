# vision: keyrack-powered border guard

## the outcome world

### before

mechanic tries to use WebFetch. border guard fails:
```
🚫 webfetch blocked: border guard not configured

the XAI_API_KEY environment variable is required to enable webfetch.
please ask the human to add XAI_API_KEY to their environment...
```

human has to manually configure `~/.config/rhachet/apikeys.env` — a path that:
- isn't documented in keyrack.yml
- isn't consistent with how git.commit.push fetches credentials
- requires human to know this secret path exists

### after

mechanic tries to use WebFetch. border guard checks keyrack:
```
🔐 XAI_API_KEY locked

run: rhx keyrack unlock --owner ehmpath --env prep
```

mechanic runs unlock. WebFetch works. same pattern as git.commit.push.

### the "aha" moment

"oh, border guard uses keyrack just like all other skills. one unlock command, all credentials available."

## user experience

### usecases

1. **mechanic runs WebFetch** — keyrack provides XAI_API_KEY to border guard
2. **mechanic runs git.commit.push** — keyrack provides EHMPATHY_SEATURTLE_GITHUB_TOKEN
3. **human unlocks once** — `rhx keyrack unlock --owner ehmpath --env prep` unlocks both

### contract

**option A: SDK (human suggested)**
```typescript
// guardBorder.onWebfetch.ts
import { keyrack } from 'rhachet/keyrack';

const xaiKey = await keyrack.get({
  key: 'XAI_API_KEY',
  owner: 'ehmpath',
  env: 'prep'
});
```

**option B: CLI (same pattern as git.commit.push)**
```bash
# posttooluse.guardBorder.onWebfetch.sh
XAI_API_KEY=$("$repo_root/node_modules/.bin/rhachet" keyrack get \
  --key XAI_API_KEY \
  --owner ehmpath \
  --env prep \
  --allow-dangerous \
  --json | jq -r '.grant.key.secret // empty')
export XAI_API_KEY
```

**decision:** use SDK (option A). human said "if typescript, use sdk" and the credential fetch code will be in TypeScript.

### timeline

1. mechanic attempts WebFetch
2. border guard calls `keyrack.get({ key: 'XAI_API_KEY', ... })`
3. if locked: emit unlock instructions, exit 2
4. if unlocked: proceed with content inspection

## mental model

### how users describe it

"keyrack holds all the keys. unlock once, use everywhere."

### analogies

- keyrack = keychain (macOS) — unlock device, all passwords available
- keyrack = ssh-agent — add key once, use for all ssh operations

### terms

| user term | our term |
|-----------|----------|
| "unlock the keyrack" | `keyrack unlock` |
| "add my XAI key" | `keyrack fill` |
| "check if key is available" | `keyrack get` |

## evaluation

### pros

- **consistent** — same pattern as git.commit.push, git.release
- **discoverable** — keyrack.yml documents all required keys
- **secure** — credentials fetched at runtime, not hardcoded in env files
- **debuggable** — `keyrack status` shows what's unlocked

### cons

- **requires SDK import** — TypeScript code needs `import { keyrack } from 'rhachet/keyrack'`
- **async** — keyrack.get() is async, changes control flow slightly

### pit of success

- keyrack.yml lists required keys — humans know what to fill
- unlock instructions are actionable — mechanic tells human exactly what to run
- fail-fast — border guard blocks immediately if key absent, doesn't proceed and fail later

## what is awkward?

### the shell wrapper

`posttooluse.guardBorder.onWebfetch.sh` currently sources apikeys.env, then delegates to TypeScript. after this fix:
- shell wrapper omits credential logic entirely
- TypeScript handles credentials via keyrack SDK
- shell wrapper becomes pure delegation

this is cleaner, but the shell wrapper feels vestigial. consider: could the hook be pure TypeScript?

### env naming

`EHMPATHY_SEATURTLE_PROD_GITHUB_TOKEN` has `_PROD_` in the name but lives in `env.prep`. rename to `EHMPATHY_SEATURTLE_GITHUB_TOKEN` fixes this inconsistency.
