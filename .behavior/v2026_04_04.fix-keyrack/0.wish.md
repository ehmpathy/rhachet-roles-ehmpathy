# wish: fix mechanic keyrack for border guard

## .what

the mechanic's keyrack lacks `XAI_API_KEY` and the border guard doesn't use keyrack to fetch credentials.

## .problems

### 1. XAI_API_KEY absent from keyrack.yml

`src/domain.roles/mechanic/keyrack.yml` only defines:
```yaml
env.prep:
  - EHMPATHY_SEATURTLE_PROD_GITHUB_TOKEN
```

but `posttooluse.guardBorder.onWebfetch.sh` requires `XAI_API_KEY` to function.

### 2. border guard uses hardcoded apikeys.env path

current approach in `posttooluse.guardBorder.onWebfetch.sh`:
```bash
if [[ -z "${XAI_API_KEY+x}" && -f ~/.config/rhachet/apikeys.env ]]; then
  source ~/.config/rhachet/apikeys.env
fi
```

this is a hardcoded path. should use keyrack SDK instead (TypeScript skill).

### 3. TypeScript CLI should use keyrack SDK

`src/contract/cli/guardBorder.onWebfetch.ts` checks `process.env.XAI_API_KEY` directly:
```ts
if (!process.env.XAI_API_KEY) {
  // fail
}
```

rhachet exports a keyrack SDK from `'rhachet/keyrack'` that should be used instead:
```ts
import { keyrack } from 'rhachet/keyrack';

const xaiKey = await keyrack.get({ key: 'XAI_API_KEY', env: 'prod' });
```

## .fix

1. add `XAI_API_KEY` to `src/domain.roles/mechanic/keyrack.yml` under appropriate env
2. update `guardBorder.onWebfetch.ts` to use keyrack SDK from `'rhachet/keyrack'`
3. remove hardcoded apikeys.env source from the shell hook (let TypeScript handle it)

## .why

- keyrack is the canonical credential management system
- hardcoded paths are fragile and inconsistent
- other skills (git.commit.push, git.release) already use keyrack properly
- border guard should follow the same pattern
