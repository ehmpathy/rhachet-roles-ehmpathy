# vision: fix rmsafe trash cache for gitignored files

## the outcome world

### before
```
mechanic: rhx rmsafe -r node_modules
# node_modules (500MB) → moves to .agent/.cache/.../trash/node_modules
# next declapract scan: 134+ seconds crawling trash with nested node_modules
```

### after
```
mechanic: rhx rmsafe -r node_modules
# node_modules is gitignored → deleted directly, no trash
# next declapract scan: fast, trash contains only recoverable files
```

### the "aha" moment
trash is a recovery cache for files worth recovering. gitignored files are regeneratable by definition — they don't need recovery. keeping them out of trash keeps the repo clean and tools fast.

## user experience

### usecases

| usecase | behavior |
|---------|----------|
| delete `src/foo.ts` | → trash (git-trackable, recoverable) |
| delete `node_modules/` | → direct delete (gitignored, regeneratable) |
| delete `.cache/` | → direct delete (gitignored, regeneratable) |
| delete `dist/` | → direct delete (gitignored, regeneratable) |
| delete `build/*.tmp` via glob | → each file checked individually |

### contract

**input**: unchanged — same CLI interface
```bash
rhx rmsafe 'path/to/file'
rhx rmsafe -r 'path/to/dir'
rhx rmsafe --path 'src/**/*.bak'
```

**output**: slight change in behavior
- git-trackable files: trash + coconut hint (as today)
- gitignored files: direct delete, no coconut hint

**stdout example for gitignored file**:
```
🐢 sweet

🐚 rmsafe
   ├─ path: node_modules
   ├─ type: directory (gitignored)
   └─ removed
      └─ node_modules/
```

**stdout example for git-trackable file** (unchanged):
```
🐢 sweet

🐚 rmsafe
   ├─ path: src/old.ts
   ├─ files: 1
   └─ removed
      └─ src/old.ts

🥥 to restore, run:
   └─ rhx cpsafe '.agent/.cache/.../trash/src/old.ts' './src/old.ts'
```

### timeline

1. user runs `rhx rmsafe -r node_modules`
2. rmsafe checks: `git check-ignore node_modules` → ignored
3. rmsafe deletes directly: `rm -rf node_modules`
4. output shows "(gitignored)" — no coconut hint
5. trash stays clean, declapract stays fast

## mental model

### how users would describe this
> "rmsafe only trashes files that matter. throwaway stuff like node_modules gets deleted directly."

### analogy
like a recycling bin that only accepts recyclables. you don't put food waste (regeneratable stuff) in the recycling — it goes straight to compost (direct delete).

### terms
| user term | internal term |
|-----------|---------------|
| "trash" | trash cache at `.agent/.cache/.../trash/` |
| "throwaway" | gitignored files |
| "worth keeping" | git-trackable files |

## evaluation

### how well does it solve the goals?

| goal | solution |
|------|----------|
| declapract 134s → fast | trash no longer contains node_modules |
| preserve recovery for important files | git-trackable files still go to trash |
| no breaking changes | same CLI interface |

### pros
- declapract scans stay fast
- trash contains only meaningful recovery targets
- no change to user workflow
- simple heuristic: gitignored = regeneratable = no trash

### cons
- gitignored files cannot be recovered from trash (by design — they're regeneratable)
- directory deletion is slower due to per-file enumeration (acceptable tradeoff)

### edgecases

| edgecase | behavior |
|----------|----------|
| file both tracked AND in .gitignore (rare) | `git check-ignore` determines → if ignored, direct delete |
| nested: `src/` (trackable) with `.cache/` (ignored) | enumerate files: trackable → trash, gitignored → direct delete |
| glob `**/*.log` matching both tracked and ignored | each file checked individually |
| symlink to gitignored location | check symlink path itself, not resolved target |

### pit of success
- default behavior is safe (gitignored = regeneratable assumption)
- no new flags needed
- `git check-ignore` is the source of truth

### test coverage boundaries

expect various subsets — thorough test coverage required on these combinations:

| target type | content | expected behavior |
|-------------|---------|-------------------|
| file | trackable | → trash |
| file | gitignored | → direct delete |
| dir | all trackable | all → trash |
| dir | all gitignored | all → direct delete |

## open questions & assumptions

### assumptions
1. **gitignored = regeneratable** — files in .gitignore don't need recovery
2. **git check-ignore is authoritative** — we trust git's ignore rules
3. **no override needed** — confirmed by wisher: no `--force-trash` flag

### questions (triaged)

1. **[answered]** should we add `--force-trash` flag for explicit trash of gitignored files?
   - answer: no — keep it simple, no override needed
   - if users want to recover gitignored files, they can use version control or backups

2. **[answered]** should we log differently for mixed directories (some trashed, some direct)?
   - yes — show "(gitignored)" in output for gitignored files
   - already reflected in stdout examples above

3. **[answered]** should symlinks follow target or check symlink path itself?
   - check the symlink path, NOT the resolved target
   - reason: user is about to delete the symlink; its location determines track/ignore status
   - if symlink `src/link` points to `node_modules/foo`:
     - `src/link` is not gitignored → goes to trash
     - the target `node_modules/foo` being ignored is irrelevant
   - implementation note: do NOT use `realpath` for ignore check

4. **[answered]** how do we handle mixed directories (trackable dir with gitignored subdirs)?
   - example: delete `src/` where `src/.cache/` is gitignored
   - answer: ask git which files are trackable (not gitignored)
   - trackable = not gitignored, even if not yet tracked
   - trackable files → trash (recoverable)
   - gitignored files → direct delete (no trash)
   - handles nested gitignored dirs properly without special cases

5. **[answered]** what should stdout look like for mixed directories?
   - for directory deletion: show directory name only, not per-file breakdown
   - "(gitignored)" label applies to the top-level target, not subdirs
   - this keeps output clean — no per-file enumeration needed

## what is awkward?

### the mixed directory case (resolved)
when deleting a directory that contains both trackable and gitignored files:
- enumerate files via git
- trackable files → trash (recoverable)
- gitignored files → direct delete (no trash)

not atomic — each file checked individually. this keeps trash clean at all levels.

### symlinks (resolved)
if `foo` is a symlink to `node_modules/bar`:
- check `foo` (the symlink path) — this is what user wants to delete
- NOT `node_modules/bar` (the target)

decision: check symlink path itself. implementation must avoid `realpath` for ignore check.

### stdout verbosity (resolved)
for directory deletion: show directory name only, not per-file breakdown.
"(gitignored)" label applies to the top-level target.
no per-file enumeration needed — keeps output clean.
