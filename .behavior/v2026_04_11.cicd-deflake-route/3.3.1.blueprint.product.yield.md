# blueprint: cicd.deflake skill

## summary

create a route-based workflow skill for structured deflake of CI test flakes. the skill follows the same pattern as `declapract.upgrade`:

- entry point: `cicd.deflake.sh` with subcommand dispatch
- `init` subcommand: creates route with stones/guards, binds to branch
- templates: 9 stones + 6 guards for the deflake workflow

the route enforces:
1. evidence gather from main-branch CI runs (detect → research → yield)
2. diagnosis research with codepath treestruct guard (test + prod codepaths)
3. diagnosis rootcause with hypothesis guard
4. plan with test-intent preservation guard + protect src/**/*
5. execution with peer-review + self-review guards (failhide/failfast detection)
6. verification with 3x pass requirement and rewind enforcement
7. itemize repairs traceably
8. reflect for institutional memory (with human approval)
9. institutionalize lessons as briefs

### 2.1.diagnose.research.stone

```markdown

.why = understand the full execution path before you predict root cause.

---

## how to research

for each flake, trace BOTH the test codepath AND the prod codepath to identify all spots where the flake COULD have come from.

### step 1: trace the test codepath

use behavior blueprint style treestruct to map the test execution:

\`\`\`
testName.integration.test.ts
├── given: [setup]
│   ├── createTestData()
│   │   └── daoUser.upsert() ← potential: db connection delay
│   └── mockExternalService()
│       └── jest.mock() ← potential: mock not reset between runs
├── when: [action]
│   └── invokeFeature()
│       └── {prod function called here — trace in step 2}
└── then: [assertion]
    └── expect(result).toEqual() ← potential: snapshot drift
\`\`\`

### step 2: trace the prod codepath

for the prod code invoked by the test, trace the execution:

\`\`\`
src/domain.operations/doFeature.ts
├── doFeature(input, context)
│   ├── validateInput(input) ← potential: validation race
│   ├── getRelatedData(input.id, context)
│   │   └── context.daoEntity.findByRef() ← potential: db timeout
│   ├── computeResult(data)
│   │   └── pureTransform() ← unlikely: deterministic
│   └── persistResult(result, context)
│       ├── context.daoResult.upsert() ← potential: db contention
│       └── context.svcNotify.emit() ← potential: network timeout
\`\`\`

### step 3: annotate flake candidates

for each node in BOTH codepaths, mark potential flake sources:

| location | node | why it could flake |
|----------|------|-------------------|
| test | createTestData() | db connection delay |
| test | mockExternalService() | mock not reset |
| prod | getRelatedData() | db timeout |
| prod | persistResult() | db contention |

### step 4: gather ground truth

for each flake candidate, gather concrete evidence:

1. **read the code** — what does it actually do?
2. **check shared state** — any globals, singletons, closures?
3. **check time dependencies** — setTimeout, Date.now, timestamps?
4. **check external i/o** — network, filesystem, database?
5. **check parallelism** — Promise.all, concurrent tests?
6. **check order dependencies** — test isolation, cleanup?
```

### 3.plan.stone

```markdown
propose a plan to deflake each test.

.why = concrete steps with verification criteria.

---

## how to plan repairs

for each diagnosed flake, map the repair to the codepath node that causes the flake:

### repair patterns by flake type

| flake type | repair pattern | example |
|------------|---------------|---------|
| race condition | add mutex or serialization | `await lock.acquire()` before shared access |
| time dependency | use frozen time or relative assertions | `jest.useFakeTimers()` or `expect(elapsed).toBeLessThan(threshold)` |
| shared state | isolate state per test | `beforeEach(() => state = fresh())` |
| network timeout | increase timeout or add retry | `{ timeout: 30000 }` or `withRetry(fetch, 3)` |
| parallelism | run in band or add isolation | `--runInBand` or resource locks |
| snapshot drift | mask dynamic values | `result.replace(/\d{4}-\d{2}-\d{2}/g, '$DATE')` |
| cleanup failure | add explicit cleanup | `afterEach(() => cleanup())` |

### verification criteria

for each repair, define:
1. **local verification**: how to prove the fix works locally
2. **ci verification**: what ci must show to confirm stability
3. **regression guard**: what prevents reintroduction

---

## plan per flake

for each diagnosed flake:

### flake: {test name}

**root cause confirmed:** {from diagnosis}

**repair steps:**
1. [ ] {codepath node}: {change to make}
2. [ ] {codepath node}: {change to make}

- local: run test 10x with `npm run test:integration -- {test} --runInBand`
- ci: 3x consecutive green runs on main
- guard: {what prevents reintroduction — e.g., code review rule, test isolation pattern}

**tips for future travelers:**
- {is it inherently probabilistic? → use `when.repeatedly` pattern}
- {is there a time issue? → consider frozen time or relative thresholds}
- {is it a snapshot issue? → mask the dynamic part with stability regex}
```

- who added it? (not for blame, for context)
