# vision: cicd.deflake thought route

## the outcome world

### day-in-the-life: before

monday morning. you merge a pr. ci fails on main. you check — it's a flaky test that passed on your branch but failed on main. you re-run. it passes. you move on.

two weeks later, the same test flakes again. and again. each time someone re-runs, shrugs, forgets. the flake persists. trust in ci erodes. "just re-run it" becomes the answer.

eventually someone gets fed up and skips the test. or wraps it in a retry. or accepts the flake as "known". the test still runs, but it no longer catches the bug it was meant to catch.

### day-in-the-life: after

monday morning. you notice ci has flaked a few times this week. you run:

```sh
rhx cicd.deflake init
```

the route initializes. you drive:

```sh
rhx route.drive
```

stone by stone, you:
1. **gather evidence** — run `rhx cicd.deflake detect` to scan main-branch ci runs, enumerate every flake: which test, how often, what error
2. **diagnose root causes** — for each flake, you articulate what makes it flaky, propose hypotheses, rank probabilities
3. **propose a plan** — concrete steps to fix each flake, with verification criteria
4. **execute** — make the repairs (code or infra), release
5. **verify** — run ci 3x in a row, confirm zero flakes
6. **itemize** — document each repair: true root cause, true fix, verification
7. **reflect** — distill lessons: how did these flakes get introduced? what systemic changes prevent future flakes? (human approval required)
8. **institutionalize** — create briefs or upgrade handoffs from approved lessons

no human approval needed. the route guards itself: every flake from step 1 must appear in step 2's diagnosis. every repair must preserve test intent — no skips, no "accept failure". peer-review guards catch failhide patterns.

### the "aha" moment

the value clicks when you realize: **deflaking is no longer a one-off heroic effort — it's a repeatable, documented process with institutional memory**.

each deflake cycle produces a reflection document. over time, these accumulate into a knowledge base: "these are the patterns that cause flakes in our codebase. here's how we prevent them."

## user experience

### usecases and goals

| goal | how cicd.deflake helps |
|------|----------------------|
| eliminate specific flakes | structured diagnosis + verified fix |
| prevent flake debt | reflection documents capture systemic lessons |
| maintain test intent | guards forbid skips or accepted failures |
| build trust in ci | prove stability with 3x verification |
| onboard new devs | reflection docs explain why tests are written this way |

### contract inputs & outputs

**inputs:**
- ci provider (github actions, etc.)
- branch to analyze (default: main)
- time range (default: last 30 days)
- test scope (unit, integration, acceptance, all)

**outputs:**
- `1.evidence.yield.md` — enumerated flakes with frequency and error messages
- `2.diagnosis.yield.md` — root cause analysis for each flake
- `3.plan.yield.md` — repair steps with verification criteria
- `4.execution.yield.md` — what was changed, pr links
- `5.verification.yield.md` — 3x ci run results
- `6.repairs.yield.md` — itemized repairs: cause, fix, verification
- `7.reflection.yield.md` — systemic lessons and prevention strategies
- `8.institutionalize.yield.md` — briefs or handoffs created from approved lessons

## mental model

### how users describe this to a friend

> "it's like a checklist for fixing flaky tests, but the checklist enforces itself. you can't skip steps, and it remembers where you left off."

> "instead of playing whack-a-mole with flakes, you go through a structured process that forces you to actually understand why things flake."

> "it's like a post-mortem process, but for test flakiness instead of incidents."

### analogies and metaphors

| concept | analogy |
|---------|---------|
| the route | a flight checklist — each item must be verified before proceeding |
| the guards | co-pilot who catches if you try to skip a step |
| evidence gathering | detective work — collect all the clues before hypothesizing |
| 3x verification | landing gear test — check three times before trusting |
| reflection document | incident post-mortem — prevent recurrence, not just fix |

### terms: user vs system

| user says | system calls it |
|-----------|-----------------|
| "flaky test" | flake evidence |
| "why does it fail?" | root cause hypothesis |
| "fix it" | repair execution |

## evaluation

### how well does it solve the goals?

| goal | rating | notes |
|------|--------|-------|
| eliminate specific flakes | strong | forced diagnosis prevents band-aids |
| prevent flake debt | strong | reflection docs create institutional memory |
| maintain test intent | strong | guards forbid skips and accepted failures |
| build trust in ci | strong | 3x verification proves stability |
| reduce re-run culture | strong | flakes get fixed, not re-run |

### pros

- **no skipping steps** — guards enforce completeness
- **minimal human approval** — only reflection stone requires human sign-off on proposed lessons
- **paper trail** — every decision documented
- **rewindable** — if verification fails, route driver must rewind to plan
- **institutional memory** — reflection docs accumulate knowledge
- **preserves test intent** — guards catch failhide patterns

### cons

- **time investment** — proper diagnosis takes time; quick-fix culture may resist
- **ci api dependency** — must be able to scrape flake history
- **observability gaps** — some flakes require adding logging to diagnose
- **false positives** — may flag infrastructure issues as test flakes

### edgecases and pit-of-success

| edgecase | how we handle it |
|----------|------------------|
| flake is actually infra issue | diagnosis phase distinguishes test flakes from infra flakes |
| test is inherently probabilistic | plan phase provides patterns: `when.repeatedly`, timing guards |
| flake requires observability to diagnose | route allows "add observability, re-run, continue" workflow |
| repair accidentally removes test intent | peer-review guard catches failhide patterns |
| verification fails | route driver must rewind to plan, not execution |
| no flakes found | route completes with "zero flakes detected" evidence |

## open questions & assumptions

### assumptions we've made

1. **ci history is accessible** — we can scrape main-branch runs via api (gh, etc.)
2. **flakes are detectable** — failed-then-passed-on-retry pattern identifies flakes
3. **3x verification is sufficient** — three consecutive passes proves stability (wisher-specified, configurable)
4. **reflection is valuable** — teams will read and apply systemic lessons
5. **lookback period is 30 days** — [answered] default to 30 days, configurable; gh retains logs 90 days
6. **environment changes are valid fixes** — [answered] covered by "code or infra" language
7. **past reflections via index file** — [answered] for v1, an index file lists reflection docs with summaries
8. **reflection format is post-mortem style** — [answered] root causes, why introduced, how to prevent

### questions that require research

1. **how do we detect flakes vs legitimate failures?** — [research] extant flake detection approaches needed
2. **how do we handle flakes that only reproduce under load?** — [research] parallelism flake patterns
3. **gh api for workflow runs** — [answered] extant skills `rhx git.release` and `rhx show.gh.test.errors` already query workflow runs
4. **extant flake detection tools** — [research] jest --detectOpenHandles, quarantine patterns
5. **common flake patterns** — [research] time-based, parallelism, snapshot, environment drift

### what must we validate with the wisher?

1. **scope of "flake"** — [wisher] is it strictly "passed on retry"? or also "sometimes fails, sometimes passes across prs"?
2. **compound flakes** — [wisher] should diagnosis support multiple root causes per flake?

## what is awkward?

### what feels off or forced?

- **"zero human approval"** — [resolved] humans review PRs before merge is allowed; route autonomy is within a branch, merge still requires PR review
- **rewind to plan on verification failure** — [resolved] correct design; execution must pass at least once before verification begins; typos should be caught in execution phase, not verification
- **3x verification** — arbitrary number; some flakes only appear 1-in-100 runs; configurable addresses this
- **domain knowledge gap** — [resolved] expert briefs are baked into repo; reflect stone proposes new `.agent/repo=.this/role=any/briefs` to institutionalize lessons for future travelers
- **compound flakes** — [resolved] diagnosis supports multiple root causes per flake; each cause gets its own repair steps and verification

### where does the design fight the user's mental model?

- **users expect "fix the flake"** — route demands "diagnose, plan, verify, reflect" which feels heavyweight for "obvious" flakes
- **users expect to iterate** — route is linear with rewind; may feel rigid

### what tradeoffs feel uncomfortable?

| tradeoff | discomfort | resolution |
|----------|------------|------------|
| thoroughness vs speed | proper deflake takes time; users may want quick fixes | fast-path for obvious flakes is possible |
| autonomy vs oversight | route is autonomous within branch | PR review required before merge; edge cases caught there |
| completeness vs flexibility | every flake from evidence must appear in diagnosis | guards enforce this; prevents flakes from escape |

### potential mitigations

- **fast-path for obvious flakes** — if diagnosis is trivial, allow condensed workflow
- **human checkpoint option** — optional "review before execute" for high-stakes repos
- **configurable verification threshold** — 3x default, but allow 5x or 10x for paranoid teams
