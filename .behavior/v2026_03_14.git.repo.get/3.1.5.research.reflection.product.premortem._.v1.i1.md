# research: premortem reflection

## premortem: imagine we shipped and it failed

we shipped git.repo.get, and it failed badly. why?

### most plausible failure scenarios

1. **permission prompts still appear** — the skill isn't pre-approved in permissions, so every invocation triggers a prompt. mechanic still interrupted; humans still frustrated.

2. **local clone paths mismatch** — skill looks in `~/git/org/repo/` but user has clones at `~/code/`, `~/projects/`, or org-less structures like `~/git/repo/`. skill reports "no local repos found" despite clones present.

3. **gh cli auth fails silently** — mechanic tries cloud fallback, gh cli isn't authenticated, error message unclear. mechanic says "repo not found" when it actually exists on github.

4. **github rate limits hit** — mechanic runs multiple searches (especially with `--repos` glob), hits 10 req/min code search limit. subsequent searches fail or return partial results.

5. **stale local data misleads** — local clone hasn't been fetched in months, mechanic returns outdated code, human implements based on stale patterns that were already refactored.

6. **glob pattern confusion** — user expects `--repos 'sql-*'` to find `sql-code-generator` but glob semantics don't match their mental model. empty results when repos exist.

---

## what obvious signs did we ignore?

look back from the imagined failure — what signs were present that we dismissed?

- **sign**: no way to configure local clone root (`~/git/` hardcoded)
- **sign**: no validation that gh auth is active before cloud operations
- **sign**: no alert about local clone age/staleness
- **sign**: no documentation about glob pattern semantics
- **sign**: no rate limit awareness or backoff logic
- **sign**: no test coverage for permission pre-approval verification

---

## what assumptions could turn out wrong?

| assumption | what if wrong? | likelihood |
|------------|----------------|------------|
| clones live at `~/git/org/repo/` | many clones unfindable; skill reports empty | medium |
| gh cli is authenticated | cloud fallback fails; error unclear | medium |
| local clones are recent enough | stale results mislead humans | medium |
| glob semantics are intuitive | users get unexpected empty results | low |
| 10 req/min is enough | heavy use hits rate limit | low |
| permission pre-approval works | still get prompts (whole point fails) | low |

## what is the nightmare scenario?

**the trust violation scenario:**

mechanic uses git.repo.get to search repos, returns results from a stale local clone. human implements based on a pattern that was deprecated or has security issues. code ships to production. incident occurs. postmortem reveals the pattern came from a year-old local clone that nobody fetched.

**why this is nightmare:**
- trust in mechanic is damaged
- human blames the tool
- recovery requires audit of all mechanic-informed decisions
- adoption of git.repo.get halted

**second nightmare: prompt fatigue remains**

the whole point of git.repo.get is to eliminate permission prompts. if the permission pre-approval doesn't work (e.g., wildcard not configured, rhachet path mismatch), every invocation prompts. humans disable the skill or stop repo search delegation. the friction we aimed to remove persists.

---

## what mitigations to consider?

| risk | mitigation | cost |
|------|------------|------|
| permission prompts persist | verify pre-approval in integration test; document exact permission entries | low |
| local clone path mismatch | env var `GIT_REPO_ROOT` override; config file support | medium |
| gh auth fails silently | check `gh auth status` before cloud ops; clear error message | low |
| rate limits hit | local-first by default; alert on cloud search count | low |
| stale local data | show last fetch date in output; optional `--fresh` to fetch first | medium |
| glob confusion | document slug model and glob semantics; show expanded matches | low |
| huge file overwhelms | default radius (21 lines); alert on large file reads | low |

---

## summary

| category | primary risk |
|----------|--------------|
| core value | permission prompts still appear |
| reliability | gh auth unclear failures |
| accuracy | stale local data misleads |
| usability | local clone paths mismatch |

**top mitigation priorities:**
1. verify permission pre-approval works (integration test)
2. clear gh auth error messages
3. show local clone source (local/cloud) and last fetch date
4. support configurable local clone root
