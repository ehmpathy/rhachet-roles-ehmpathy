# research: external product claims

## overview

this document investigates claims relevant to the git.repo.get skill — what facts, assumptions, and hypotheses can inform the design and validate the approach?

## claim.1 = local-first provides near-zero latency [FACT]

### the claim

local-first tools provide superior user experience via near-zero latency reads.

### evidence

> "local-first apps keep all data on users' devices, which provides a smooth, ultra-low-latency experience unrestricted by internet connectivity issues." [1]

> "local-first apps aim to provide near-zero latency responses via local database queries instead of server round-trips." [2]

> "once a user experiences a zero latency app, they cannot go back to API waits." [3]

### citations

[1] [practitioner] RxDB - Why Local-First Software Is the Future, https://rxdb.info/articles/local-first-future.html
[2] [practitioner] TechFinder - Local-First Software: The Silent Revolution, https://www.techfinder.io/post/local-first-software-the-silent-revolution-in-app-development
[3] [practitioner] PowerSync - Developer Benefits of Local-First, https://www.powersync.com/blog/developer-benefits-of-local-first

### convergence

sources [1], [2], [3] all agree: local data access eliminates network latency and improves UX.

## claim.3 = github code search has severe rate limits [FACT]

### the claim

github code search api has restrictive rate limits that hinder automated workflows.

### evidence

> "code search rate limits are separate from other search types, with a code search category limited to 10 requests per minute." [8]

> "the search API limits results to the first 100 entries." [9]

### convergence

sources [8], [9], [10] all agree: github code search has tight limits that constrain practical use.

## claim.11 = local-first with cloud fallback is optimal pattern [KHUE]

### the hypothesis

combination of local-first with cloud fallback provides the best of both worlds.

### rationale

> local provides speed and offline capability; cloud provides completeness and freshness. together, they cover all scenarios.

### support

> "users report a 30% increase in efficiency when they use local-first features." [30]

> "local-first delivers snappy UX by default while cloud provides comprehensive access." [31]

### citations

[30] [practitioner] CloudDataInsights - The Local-First Software Revolution, https://www.clouddatainsights.com/from-the-cloud-to-the-edge-exploring-the-local-first-software-revolution/
[31] [practitioner] TechBuzzOnline - Local-First Software Architecture Guide, https://techbuzzonline.com/local-first-software-architecture-guide/

### hypothesis basis

[KHUE] key hypothesis: the @all scope (local-first with cloud fallback) will satisfy most usecases with optimal latency.
