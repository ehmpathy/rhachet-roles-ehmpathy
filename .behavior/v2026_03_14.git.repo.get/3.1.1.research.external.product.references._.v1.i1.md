# research: external product references

## overview

this document cites authoritative references that ground the git.repo.get skill design — foundational concepts, established patterns, and proven approaches from the field.

## reference.1 = git object model (official)

### the concept

git stores content as four object types: blobs (file contents), trees (directories), commits (snapshots), and tags (named refs).

### key quote

> "git manages your codebase via four primary object types: blobs, trees, commits, and tags. a blob stores the contents of a file. a tree represents a directory and contains entries that each include a file name, permissions, and a reference to a blob or another tree." [1]

### relevance

git.repo.get uses `git ls-tree` and `git show` which operate on these object types directly without checkout.

### citations

[1] [official] Git SCM - Git Internals: Git Objects, https://git-scm.com/book/en/v2/Git-Internals-Git-Objects

## reference.2 = git show ref:path syntax (official)

### the concept

`git show <ref>:<path>` reads file content at any ref without worktree changes.

### key quote

> "if the path contains a colon (:), git interprets the part before the colon as a commit and the part after as the path in the tree, denominated by the commit." [2]

### relevance

this is the core mechanism for `lines` read mode — file access without checkout.

### citations

[2] [official] Git SCM - git-show Documentation, https://git-scm.com/docs/git-show

## reference.3 = git ls-tree for file enumeration (official)

### the concept

`git ls-tree -r --name-only <ref>` lists all files at a ref recursively.

### key quote

> "git ls-tree lists the contents of a given tree object. the tree-ish parameter refers to a specific commit, branch, or tag. with -r, recurse into sub-trees." [3]

### relevance

this is the core mechanism for `files` command — enumerate files at any ref.

### citations

[3] [official] Git SCM - git-ls-tree Documentation, https://git-scm.com/docs/git-ls-tree

## reference.4 = git grep at ref (official)

### the concept

`git grep <pattern> <ref>` searches content at any ref without checkout.

### key quote

> "you can use git grep <pattern> <branch> to search in the files of the specified branch. git grep accepts a tree-ish as an argument to specify which version of the repository to search — you don't need to checkout branches." [4]

### relevance

this is the core mechanism for `lines` search mode — grep across refs without worktree changes.

### citations

[4] [official] Git SCM - git-grep Documentation, https://git-scm.com/docs/git-grep

## reference.6 = glob pattern specification (official)

### the concept

glob patterns use wildcards (`*`, `?`, `[...]`) for path matches.

### key quote

> "most characters in a glob are treated literally, but * matches 0 or more characters, ? matches precisely one character, and [...] matches any single character in a specified set." [6]

### relevance

`--paths` filter uses glob patterns for file selection; `--repos` uses globs for repo selection.

### citations

[6] [official] Linux Man Pages - glob(7), https://man7.org/linux/man-pages/man7/glob.7.html

## reference.7 = grep context lines (-A, -B, -C) (official)

### the concept

grep provides -A (after), -B (before), -C (context) flags for lines around matches.

### key quote

> "-A NUM prints NUM lines of context that trail each match. -B NUM prints NUM lines of context that lead each match. -C NUM prints NUM lines both before and after." [7]

### relevance

`--radius` parameter mirrors grep -C for context around matches.

### citations

[7] [official] GNU Grep Manual - Context Line Control, https://www.gnu.org/s/grep/manual/html_node/Context-Line-Control.html

## reference.13 = github api rate limits (official)

### the concept

github REST API has rate limits: 60/hr unauthenticated, 5000/hr authenticated, 10/min for code search.

### key quote

> "the primary rate limit for unauthenticated requests is 60 requests per hour. all authenticated requests count toward a personal rate limit of 5,000 requests per hour. code search has a rate limit of 10 requests per minute." [13]

### relevance

local-first design avoids rate limits; cloud fallback respects these constraints.

### citations

## reference.19 = bash pathname expansion (official)

### the concept

bash expands globs before command execution.

### key quote

> "bash scans each word for *, ?, and [. if these appear and are not quoted, bash replaces the word with a sorted list of filenames that match the pattern." [19]

### relevance

`--repos` and `--paths` patterns must be quoted to prevent shell expansion.

### citations

[19] [official] GNU Bash Manual - Filename Expansion, https://www.gnu.org/software/bash/manual/html_node/Filename-Expansion.html

## anti-patterns noted

### gh search code limitations

source [15] warns: 384KB file limit, 10 req/min rate limit, no sort control. prefer local git grep when possible.

### unquoted globs

source [19] warns: shell expands unquoted globs before command sees them. always quote `--repos` and `--paths` patterns.
