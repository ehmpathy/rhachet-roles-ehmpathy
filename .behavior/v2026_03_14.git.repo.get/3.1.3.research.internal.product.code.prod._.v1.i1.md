## pattern.1 = turtle vibes output [REUSE]

### source

`src/domain.roles/mechanic/skills/git.commit/output.sh`

### the pattern

```bash
print_turtle_header() {
  local phrase="$1"
  echo "🐢 $phrase"
  echo ""
}

print_tree_start() {
  local command="$1"
  echo "🐚 $command"
}

print_tree_branch() {
  local label="$1"
  local value="$2"
  echo "   ├─ $label: $value"
}

print_tree_leaf() {
  local label="$1"
  local value="$2"
  echo "   └─ $label: $value"
}
```

### application

git.repo.get will source output.sh and use these functions for consistent turtle vibes output:
- `print_turtle_header "far out"` for success
- `print_turtle_header "crickets..."` for empty results
- `print_turtle_header "bummer dude"` for errors

## pattern.3 = named arg parse [REUSE]

### source

`src/domain.roles/mechanic/skills/git.commit/git.commit.set.sh`

### the pattern

```bash
while [[ $# -gt 0 ]]; do
  case $1 in
    # rhachet passes these - ignore them
    --skill|--repo|--role)
      shift 2
      ;;
    --message|-m)
      MESSAGE="$2"
      shift 2
      ;;
    --push)
      PUSH="true"
      shift
      ;;
    --help|-h)
      show_usage
      exit 0
      ;;
    *)
      echo "unknown argument: $1"
      exit 1
      ;;
  esac
done
```

### application

git.repo.get will use this pattern for `--repos`, `--repo`, `--paths`, `--words`, `--radius`, `--ref` arguments.

## pattern.12 = local clone path convention [EXTEND]

### source

`src/domain.roles/mechanic/briefs/practices/` (implicit in clone references)

### the pattern

local clones live at `~/git/<org>/<repo>/`:
```
~/git/
  ├── ehmpathy/
  │   ├── domain-objects/
  │   ├── sql-code-generator/
  │   └── rhachet-roles-ehmpathy/
  └── ahbode/
      └── protools-api/
```

### application

git.repo.get will scan `~/git/` to enumerate local repos:
```bash
find ~/git -maxdepth 2 -type d -name ".git" | while read gitdir; do
  repo_path="$(dirname "$gitdir")"
  org="$(basename "$(dirname "$repo_path")")"
  repo="$(basename "$repo_path")"
  echo "@local/$org/$repo"
done
```

### extension needed

- what if clones have different structures?
- should support `GIT_REPO_ROOT` env var override?
- or config file `~/.rhachet/git.repo.get.yml`?

## pattern.13 = gh api for cloud content [EXTEND]

### source

`src/domain.roles/mechanic/skills/show.gh.action.logs.sh` (uses gh api)

### the pattern

```bash
gh api repos/{owner}/{repo}/contents/{path} --jq '.content' | base64 -d
```

### application

git.repo.get will use gh api for cloud file reads:
```bash
# list files via git trees api
gh api repos/ehmpathy/domain-objects/git/trees/main?recursive=1 \
  --jq '.tree[] | select(.type == "blob") | .path'

# read file content
gh api repos/ehmpathy/domain-objects/contents/src/index.ts \
  --jq '.content' | base64 -d
```

### extension needed

- handle rate limits (10/min for code search)
- handle pagination for large trees (>1000 files)
- handle base64 decode for binary files
