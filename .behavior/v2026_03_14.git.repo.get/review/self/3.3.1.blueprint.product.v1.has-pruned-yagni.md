## potential YAGNI candidates

### output.sh symlink

**was it requested?** not explicitly. vision says "turtle vibes format" but doesn't say "reuse from git.commit".

**is it minimum viable?** yes — reuse is simpler than duplicate. to write new output functions would be more code.

**did we add for future flexibility?** no — we chose symlink because the functions already exist and match our needs.

**verdict**: keep. reuse is simpler, not more complex.

**why it holds**: the alternative would be to write new output functions from scratch. that's MORE code, not less. symlink means zero new code for output. this is the opposite of YAGNI — it's using extant code instead of new code.

### phases 0-5 structure

**was it requested?** no — vision doesn't prescribe phases.

**is it minimum viable?** yes — phases are a roadmap for implementation order, not extra code.

**did we add for future flexibility?** no — phases exist as a plan for implementation, not as abstractions.

**verdict**: keep. phases are plan artifacts, not shipped code.

**why it holds**: phases don't add any shipped code. they're a roadmap for implementation order. if anything, they REDUCE complexity by constraining scope per phase. phase 1 = only local repos. phase 2 = only files. etc. this prevents "while we're here" creep by explicit scope boundaries.

### integration test file

**was it requested?** yes — criteria section has explicit test cases.

**is it minimum viable?** yes — tests verify behavior.

**verdict**: keep.

**why it holds**: the criteria document has explicit test cases (repos with local clones, files in local repo, search in local repo, etc.). tests are not extras — they're verification of the criteria. without tests, we'd ship unverified behavior. the test file exists because criteria prescribe the test cases.
