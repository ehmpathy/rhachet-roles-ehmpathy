### header format â€” FINDING

**extant pattern (git.commit.set.sh, git.rebase.begin.sh):**

**git.repo.get.sh:**
```bash

#
# .what
#   multi-line content
#   on next line
#
# .when
#   - bullet points
#
# .how

```

**difference**: git.repo.get uses `.what`/`.when`/`.how` sections without `=` assignment, while extant skills use `.what = `/`.why = ` format.

**rationale**: the vision document prescribed this format specifically for git.repo.get because:
1. the skill has multiple subcommands needing detailed docs
2. `.when` describes usecases (different from `.why`)
3. `.how` documents the subcommands

**verdict**: minor divergence, documented in vision, serves the skill's complexity. not a blocker.

### subcommand dispatch

**extant pattern**: git.commit uses separate executables (git.commit.set.sh, git.commit.push.sh)

**git.repo.get pattern**: single skill with subcommand dispatch via `cmd_repos`, `cmd_files`, `cmd_lines`

**why single-file is valid here**: git.repo.get's subcommands share:
- operations (lookup_repo, get_files_at_ref, output format)
- state (REPO_SLUG, PATHS_GLOB, WORDS_PATTERN)
- output styling (same turtle vibes header/tree format)

git.commit uses separate files because git.commit.set and git.commit.push have different lifecycles and guards. git.repo.get's repos/files/lines are simpler data queries that benefit from a unified entry point.

this is not a divergence â€” it's appropriate design for this skill's scope

- exit code 2 (vs 1) follows convention that 2 = usage error

this is enhancement, not divergence â€” it could be backported to other skills later
