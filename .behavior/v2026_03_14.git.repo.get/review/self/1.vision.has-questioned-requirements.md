## requirement: local-first, github fallback

| question | answer |
|----------|--------|
| who said? | wish: "from local if local (checkout origin/main version first)" |
| evidence? | faster, no rate limits, no network dependency |
| if we didn't? | always hit GitHub — slower, rate limited |
| scope? | adds complexity (discovery, fetch, checkout) |
| simpler? | **yes — see issue below** |

### issue found: "checkout origin/main" is awkward

the wish says "checkout origin/main version first" but this implies:
- modify worktree state
- what if user has uncommitted changes?
- what if multiple mechanics access same clone?

**simpler approach**: use low-level git commands that don't modify worktree:

```sh
# read file at origin/main without checkout
git show origin/main:path/to/file.ts

# search at origin/main without checkout
git grep "pattern" origin/main -- '*.ts'

# list directory at origin/main without checkout
git ls-tree --name-only origin/main path/to/dir/
```

this achieves "read from origin/main" without worktree modification.

**fix applied**: updated vision's "awkward" section to note this. the contract shape remains same from user perspective — implementation detail.

## deeper review: issues found on second pass

### issue: "aha" moment promises multi-repo search

**location**: line 23 — "searched across 5 repos instantly"

**problem**: the contract only supports `--repo` singular. there's no `--repos` or way to search multiple repos at once. the "aha" moment is misleading.

**options**:
- A. add multi-repo search to contract: `--repos ehmpathy/a,ehmpathy/b` or `--org ehmpathy`
- B. fix the "aha" moment to match reality: "searched this repo instantly"
- C. both: add multi-repo later, fix "aha" now

**verdict**: flag to wisher. the vision promises more than the contract delivers.

---

### issue: "fresh" claims are unsubstantiated

**location**: lines 97, 129 — "fetched fresh", "always fresh"

**problem**: if we use `git show origin/main:file`, we need `git fetch origin` first for origin/main to be current. the vision claims freshness but never specifies when/how fetch occurs.

**options**:
- A. auto-fetch before every operation (slow but fresh)
- B. fetch on first access per session (balance)
- C. never auto-fetch, rely on user's manual fetch (stale risk)
- D. fetch with TTL (cache for N minutes)

**verdict**: add to open questions. freshness mechanism undefined.

---

### issue: "accessible" is ambiguous

**location**: line 38 — "check access" shows "cached index of accessible repos"

**problem**: what does "accessible" mean?
- repos the gh token can read? (api-accessible)
- repos that exist locally? (clone-accessible)
- repos registered in index? (config-accessible)

**verdict**: define "accessible" clearly. likely means "registered in index AND either locally cloned OR gh-readable".
