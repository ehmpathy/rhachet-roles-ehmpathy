## assumption 1: local clones live at ~/git/org/repo

**what we assume**: repos are cloned at `$HOME/git/<org>/<repo>/`

**what if opposite?** some users clone to:
- `~/code/`
- `~/projects/`
- `~/dev/`
- org-less: `~/git/repo-name/`

**evidence**: i reviewed the codebase — `GIT_REPO_ROOT` env var exists in get_git_root. configurable.

**exception**: org-less clones (`~/git/repo-name/`) not covered. would need `find -maxdepth 3` or smarter heuristics.

**verdict**: assumption holds with env var. org-less is edge case, defer.

**why it holds**: the env var `GIT_REPO_ROOT` allows users to override the default path. this is the standard escape hatch pattern — provide sensible default, allow override. org-less clones are rare enough that we can address them later without upfront complexity. the 80/20 principle: handle the common case first.

## assumption 2: gh cli is authenticated

**what we assume**: `gh auth status` succeeds before cloud operations.

**what if opposite?** gh not installed or not authenticated. cloud fallback fails.

**evidence**: blueprint mentions "check gh auth status" in get_all_cloud_repos. but what's the error message?

**simpler approach?** just try the gh command and let it fail with gh's own error.

**verdict**: remove explicit auth check. let gh fail naturally. add note to error output: "run `gh auth login` to enable cloud access"

## assumption 3: origin/main is the default ref

**what we assume**: all repos use `origin/main` as default branch.

**what if opposite?** some repos use `master`, `develop`, or custom default.

**evidence**: blueprint has get_default_branch operation via `git remote show origin | grep HEAD`.

**verdict**: assumption addressed. get_default_branch handles this.

**why it holds**: the blueprint already includes get_default_branch which queries `git remote show origin | grep HEAD` for local repos and `gh repo view --json defaultBranchRef` for cloud repos. this handles main, master, develop, and any custom default. the assumption was questioned and found to be already addressed — good sign the design is sound.

---

## assumption 4: git grep works on ref without checkout

**what we assume**: `git grep pattern ref` works without worktree modification.

**what if opposite?** some git versions require checkout.

**evidence**: git grep has supported `git grep pattern <tree-ish>` since git 1.5. standard behavior.

**exception**: sparse checkout or shallow clone might behave differently.

**verdict**: assumption holds. shallow clone is edge case, document in errors.

**why it holds**: `git grep pattern <tree-ish>` has been standard git behavior since 2006. any system with git installed supports this. shallow clones are the only edge case, and they're rare for local clones (more common in CI). the design leans on established git semantics rather than inventing new behavior.

## assumption 5: subcommand structure is correct

**what we assume**: three subcommands (repos, files, lines) is the right split.

**what if opposite?** could be:
- single command with mode flag
- four commands (repos, files, search, read)
- two commands (repos, get)

**simpler approach?** two commands:
- `repos` — enumerate repos
- `get` — get files or lines with smart mode detection

**evidence**: vision document established three subcommands. user tested the mental model.

**exception**: `get` could unify files and lines, but loses clarity: "get what?"

**verdict**: three subcommands provide clear mental model. keep.

**why it holds**: i considered merging files+lines into a single `get` command with mode detection. but "get what?" lacks clarity. the three-command model mirrors the user's mental model: (1) what repos exist? (2) what files are in this repo? (3) show me this file or search for a pattern. each command has a single purpose. this follows the unix philosophy of small, focused tools.

## assumption 6: output.sh can be symlinked from git.commit

**what we assume**: output functions are reusable across skills.

**what if opposite?** git.repo.get needs different output functions.

**evidence**: reviewed git.commit/output.sh — has print_turtle_header, print_tree_start, print_tree_branch, print_tree_leaf. generic tree output.

**exception**: multi-repo output may need nested tree functions not in git.commit.

**verdict**: start with symlink. extend if needed (add functions to output.sh, not fork).

**why it holds**: reuse over duplication is a core principle. git.commit/output.sh has print_turtle_header, print_tree_start, print_tree_branch, print_tree_leaf — all generic tree output functions. if git.repo.get needs nested tree functions for multi-repo output, we extend output.sh (one place) rather than fork it (two places to maintain). the symlink is a forcing function for code sharing.

## assumption 7: glob match via bash

**what we assume**: `filter_by_glob` can use bash pattern match.

**what if opposite?** complex globs like `**/*.ts` need recursive expansion.

**evidence**: bash `[[ $string == $pattern ]]` handles `*` and `?`. for `**`, need different approach.

**simpler approach?** use `find` with `-name` or explicit glob expansion.

**verdict**: for repo slugs, simple globs suffice (`org/*`, `name-*`). `**` not needed for slug match. for `--paths` file globs, `git ls-tree | grep` or `git ls-files` handles it.

**why it holds**: the two glob contexts are different. repo slugs are flat (`org/repo`), so `*` and `?` suffice — no need for `**` recursive match. file paths are hierarchical, but git handles that: `git ls-tree -r` already recurses, and we filter with grep or fnmatch. the complexity of recursive glob is delegated to git, not reinvented in bash.

## assumption 8: 21-line default radius is correct

**what we assume**: 21 lines of context is the right default for search results.

**what if opposite?** too much: overwhelms output. too little: loses context.

**evidence**: 21 lines = 10 above + match + 10 below. similar to `grep -C 10`. reasonable.

**exception**: for multi-match files, 21 lines per match could be excessive.

**simpler approach?** 5 lines default, user increases with --radius.

**verdict**: keep 21 as declared in vision. wisher specified this default explicitly.

**why it holds**: the vision explicitly states "21 lines radius by default". while 10 might be cleaner, a change to the default would diverge from declared behavior. the wisher chose 21 for a reason — perhaps to match their workflow. we respect their intent.
