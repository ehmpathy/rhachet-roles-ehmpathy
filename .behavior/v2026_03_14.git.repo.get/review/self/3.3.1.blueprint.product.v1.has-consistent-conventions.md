### convention 5: subcommand dispatch

**what blueprint says**: `cmd_repos`, `cmd_files`, `cmd_lines` functions

**extant convention**: inline `case "$COMMAND" in ... esac` with logic directly in case branches. actual code from git.commit.uses.local.sh:
```bash
case "$COMMAND" in
  block)
    # 10 lines of logic
    QUANT=0
    PUSH="block"
    VIA_DEL=true
    ;;
  allow)
    # 8 lines of logic
    QUANT=999999
    PUSH="allow"
    ;;
  get)
    # 15 lines of logic
    ...
    ;;
esac
```

**verdict**: divergence (justified).

**why justified**:

1. **extant subcommands are small**: `git.commit.uses` has 3 subcommands (block/allow/get) with 8-15 lines each. inline works fine.

2. **blueprint subcommands are large**: `cmd_repos` has slug parse + local enumerate + cloud enumerate + glob filter + output. `cmd_files` has repo lookup + file list + path filter + content filter + output. `cmd_lines` has repo lookup + search mode OR read mode + context expansion + output. each is 50-100 lines.

3. **function extraction is standard practice**: when a case branch exceeds ~20 lines, extract to a function. this follows the single-responsibility principle.

4. **the pattern is self-explanatory**: `cmd_repos`, `cmd_files`, `cmd_lines` clearly show what each function does. the dispatch remains simple:
   ```bash
   case "$SUBCOMMAND" in
     repos) cmd_repos ;;
     files) cmd_files ;;
     lines) cmd_lines ;;
   esac
   ```

**open question for wisher**: if wisher prefers inline logic, we can consolidate. flagged for visibility.
