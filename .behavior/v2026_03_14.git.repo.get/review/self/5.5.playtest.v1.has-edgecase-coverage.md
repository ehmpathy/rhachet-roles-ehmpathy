## edge case evidence from playtest

### step 8: no repos match

```sh
rhx git.repo.get repos --repos 'nonexistent-org-xyz/*'
```

expected outcome from playtest:
- header shows `crickets...`
- output shows `found: 0 repos`

**why it holds**: this pattern will never match any org because `nonexistent-org-xyz` is a made-up name. the skill correctly returns empty results with the `crickets...` header instead of an error.

### step 9: no search matches

```sh
rhx git.repo.get lines --in ehmpathy/domain-objects --words 'XYZNONEXISTENTPATTERN123'
```

expected outcome from playtest:
- header shows `crickets...`
- output shows `found: 0 matches`

**why it holds**: this pattern is a random string that will never appear in any source file. the skill correctly returns empty results with the `crickets...` header.

### step 10: repo not found

```sh
rhx git.repo.get files --in ehmpathy/unknown-repo-xyz
```

expected outcome from playtest:
- header shows `bummer dude`
- shows error message about repo not found

**why it holds**: `ehmpathy/unknown-repo-xyz` is a made-up repo that does not exist locally or on github. the skill correctly returns an error with the `bummer dude` header.

### step 11: absent required flag

```sh
rhx git.repo.get repos
```

expected outcome from playtest:
- error message about absent --repos flag
- exit code 2

**why it holds**: the `repos` subcommand requires `--repos` flag. omit the flag and the skill fails contract validation. exit code 2 signals validation failure (not runtime error).

## semantic analysis: crickets vs bummer dude

### the distinction matters

the playtest tests two different empty-result semantics:

| header | semantic | user action | playtest step |
|--------|----------|-------------|---------------|
| `crickets...` | valid query, no results | broaden search | steps 8, 9 |
| `bummer dude` | invalid query, cannot execute | fix input | step 10 |

**why this distinction matters**: a foreman needs to know whether to:
- broaden the search (crickets = no items matched)
- fix the input (bummer dude = input was wrong)

### crickets = valid query, empty universe

step 8 tests `repos --repos 'nonexistent-org-xyz/*'`:
- the glob pattern is syntactically valid
- the org just happens to not exist
- result: empty set, not an error
- skill returns exit code 0 (success with zero results)

step 9 tests `lines --words 'XYZNONEXISTENTPATTERN123'`:
- the search pattern is syntactically valid
- the pattern just happens to not appear in any file
- result: empty set, not an error
- skill returns exit code 0 (success with zero results)

### bummer dude = invalid query, cannot execute

step 10 tests `files --in ehmpathy/unknown-repo-xyz`:
- the repo slug is syntactically valid
- but the repo does not exist anywhere (local or cloud)
- skill cannot execute the query without a repo
- result: error, skill returns exit code 1

step 11 tests `repos` (no --repos flag):
- the command is syntactically incomplete
- required argument is absent
- skill cannot execute without the argument
- result: validation failure, exit code 2

### exit code semantics

| exit code | semantic | example |
|-----------|----------|---------|
| 0 | success (even if empty results) | steps 1-9 |
| 1 | runtime error (repo not found, etc.) | step 10 |
| 2 | validation error (bad args) | step 11 |

**why it holds**: the playtest covers all three exit code semantics. a foreman can verify that:
- exit 0 = query ran (even if no results)
- exit 1 = query failed at runtime
- exit 2 = query failed at validation

## issue found and fixed: step 14 reference

### issue detected

the playtest does not have an explicit cloud fallback step because:
1. cloud fallback requires a repo that exists on github but not locally
2. foreman may not have such a repo available
3. the behavior is tested in integration tests (case14)

### resolution

the cloud fallback behavior is:
- tested in integration tests (case14 in git.repo.get.integration.test.ts)
- documented in playtest notes section
- not tested in manual playtest steps (acceptable because depends on foreman's local state)

updated my understanding: "step 14" was an error in the has-vision-coverage review. the correct reference is "integration test case14" not "playtest step 14".
