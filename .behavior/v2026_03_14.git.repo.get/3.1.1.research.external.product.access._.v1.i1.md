# research: external product access

## overview

this skill requires access to two remote systems:
1. **local git repositories** — via git CLI commands
2. **github cloud repositories** — via gh CLI / GitHub API

---

## lesson.1 = read file at specific ref without checkout

### the pattern

use `git show <ref>:<path>` to read file contents without modifying the worktree.

> "If the path contains a colon (:), git interprets the part before the colon as a commit and the part after it as the path in the tree, denominated by the commit." [1]

> "You can use `git show` to view the content of a specific file at any commit or branch without switching HEAD." [2]

### citations

[1] [official] Git Documentation - git-show, https://git-scm.com/docs/git-show
[2] [tutorial] GeeksforGeeks - How To Retrieve Single File From A Specific Revision, https://www.geeksforgeeks.org/git/how-to-retrieve-single-file-from-a-specific-revision-in-git/
[3] [practitioner] juplo - Cat Any File in Any Commit With Git, https://juplo.de/cat-any-file-in-any-commit-with-git/

### convergence

sources [1], [2], [3] all agree: `git show ref:path` is the standard way to read files without checkout.

## lesson.2 = search at specific ref without checkout

### the pattern

use `git grep <pattern> <ref>` to search file contents at a specific branch/commit.

> "You can use `git grep <pattern> <branch>` to search in the files of the specified branch." [4]

> "The key insight is that git grep accepts a tree-ish (e.g., a branch name, commit hash, or tag) as an argument to specify which version of the repository to search. This means you don't need to checkout branches—you can search them directly by name!" [5]

## lesson.3 = list files at specific ref without checkout

### the pattern

use `git ls-tree -r --name-only <ref>` to list files.

> "The `git ls-tree` command allows you to view the contents of a specific commit or a tree object within the Git repository and lists the files and directories contained in the specified commit or tree." [7]

> "Unlike other methods, `git ls-tree` lists all files present in the commit, not just those modified by the commit." [8]

## lesson.4 = detect default branch

### the pattern (local)

use `git remote show origin` to detect HEAD branch.

> "origin/HEAD represents the default branch on the remote, which is defined automatically when you clone a repository from the internet. origin/HEAD is a symbolic ref that points to the default branch of the origin remote." [10]

### the pattern (cloud)

use `gh repo view --json defaultBranchRef` to get default branch.

> "The `gh repo view` command includes `defaultBranchRef` as one of the available JSON fields that can be output." [11]

### citations

[10] [practitioner] Jamie Tanna - Work with a Git remote's default branch, https://www.jvt.me/posts/2021/03/31/git-origin-head/

## lesson.5 = github API rate limits

### the limits

> "The primary rate limit for unauthenticated requests is 60 requests per hour. All authenticated requests count towards a personal rate limit of 5,000 requests per hour." [13]

> "The code search API has a rate limit of 10 requests per minute, which is separate from other search types." [14]

### best practices

> "Subscribe to webhook events instead of poll the API for data to help your integration stay within the API rate limit." [15]

> "Implement exponential backoff—if you hit the limit, wait progressively longer before retry (1s, 2s, 4s, 8s etc.)." [16]

> "Utilize ETag headers to fetch data only if it has changed." [17]

### citations

[13] [official] GitHub Docs - Rate limits for the REST API, https://docs.github.com/en/rest/using-the-rest-api/rate-limits-for-the-rest-api
[14] [official] GitHub Changelog - Changes to the code search API, https://github.blog/changelog/2023-03-10-changes-to-the-code-search-api/
[15] [official] GitHub Docs - Best practices for use the REST API, https://docs.github.com/en/rest/using-the-rest-api/best-practices-for-using-the-rest-api
[16] [practitioner] Lunar.dev - A Developer's Guide: Manage Rate Limits for the GitHub API, https://www.lunar.dev/post/a-developers-guide-managing-rate-limits-for-the-github-api
[17] [practitioner] the-pi-guy.com - GitHub API rate limit and best practices, https://the-pi-guy.com/blog/github_api_rate_limiting_and_best_practices/

### convergence

## summary: recommended access patterns

| operation | local (@local) | cloud (@cloud) |
|-----------|----------------|----------------|
| list repos | scan ~/git/ directories | `gh repo list <org>` |
| list files | `git ls-tree -r --name-only origin/main` | `gh api repos/{owner}/{repo}/git/trees/{sha}?recursive=1` |
| read file | `git show origin/main:path/to/file` | `gh api repos/{owner}/{repo}/contents/{path}` |
| search lines | `git grep -n pattern origin/main` | `gh search code pattern --repo owner/repo` (limited) |
| default branch | `git remote show origin \| grep 'HEAD branch'` | `gh repo view owner/repo --json defaultBranchRef` |

## conflicts detected

### gh search code vs local git grep

sources [4], [14], [23] conflict on search capabilities:
- local `git grep` supports regex, line numbers, context lines
- `gh search code` has 10 req/min limit, no sort, 384KB file limit, no context lines

**recommendation**: prefer local `git grep` when clone exists; fall back to `gh search code` with awareness of limitations.

## anti-patterns to avoid

1. **do not use `git archive --remote`** — disabled on most HTTPS hosts [24]
2. **do not poll GitHub API** — use conditional requests with ETags [15]
3. **do not use unauthenticated requests** — 60/hr vs 5000/hr limit [13]
4. **do not rely on code search for comprehensive results** — limited engine [14], [23]
