# research: root cause analysis

## what is the symptom?

**observable problem:**
mechanic cannot search or browse repos without permission prompts and failed searches.

**how it manifests:**
- mechanic runs `gh search code` or `git grep` → permission prompt appears
- human must approve → flow state broken
- search sometimes fails silently or returns incomplete results
- mechanic says "i cant access that repo" despite it is accessible

**when it occurs:**
- every session where mechanic needs cross-repo context
- when human asks "how did X repo do Y?"
- when mechanic wants to find patterns across repos

---

## 5 whys tree

### branch 1: permission prompts

**symptom**: permission prompts interrupt mechanic

1. **why?** → claude code prompts for every `gh` and external `git` command
2. **why?** → these commands are not in the pre-approved permission allowlist
3. **why?** → no skill exists that wraps repo access with pre-approval
4. **why?** → repo access is ad-hoc via raw commands, not via a skill
5. **why?** → **[root cause A]**: no unified skill abstracts repo access with pre-approved permissions

### branch 2: search reliability

**symptom**: searches fail or return incomplete results

1. **why?** → mechanic uses raw `gh search code` which has limitations
2. **why?** → gh search code has 10 req/min limit and 384KB file limit
3. **why?** → no local-first fallback implemented
4. **why?** → no logic to check for local clones before cloud search
5. **why?** → **[root cause B]**: no skill implements local-first search with cloud fallback

### branch 3: discoverability

**symptom**: mechanic doesn't know what repos exist

1. **why?** → mechanic must guess repo names or ask human
2. **why?** → no command to enumerate accessible repos
3. **why?** → local clones scattered; cloud requires gh repo list
4. **why?** → no unified view across local and cloud
5. **why?** → **[root cause C]**: no skill provides unified repo discovery

### branch 4: context loss

**symptom**: human loses focus when mechanic needs help

1. **why?** → permission prompts require human attention
2. **why?** → each prompt is a context switch
3. **why?** → context switches cost 23+ minutes to recover (research)
4. **why?** → claude code permission model requires per-command approval
5. **why?** → **[root cause D]**: permission model not designed for autonomous operation

---

## root cause summary

| layer | issue | is root cause? |
|-------|-------|----------------|
| permission | raw commands not pre-approved | no (symptom) |
| abstraction | no skill wraps repo access | **yes** |
| reliability | no local-first fallback | **yes** |
| discovery | no unified repo enumeration | **yes** |
| permission model | per-command approval | out of scope |

**core root cause**: no unified skill exists that:
1. abstracts repo access (repos/files/lines)
2. implements local-first with cloud fallback
3. can be pre-approved in permissions

---

## recommendation

### address root cause: build git.repo.get skill

create a skill that:
- unifies repo access via subcommands (repos, files, lines)
- implements local-first with cloud fallback
- can be pre-approved in claude code permissions
- provides consistent interface regardless of source

**why this addresses the root causes:**
- **root cause A** (no skill): git.repo.get is the skill
- **root cause B** (no local-first): local-first is core design principle
- **root cause C** (no discovery): `repos` subcommand provides unified view

### out of scope: permission model

the claude code permission model (per-command approval) is upstream. we work within it by:
- create a pre-approvable skill
- document exact permission entries
- verify pre-approval works in tests

**follow-up**: none needed. this is to work with the system, not to change it.
