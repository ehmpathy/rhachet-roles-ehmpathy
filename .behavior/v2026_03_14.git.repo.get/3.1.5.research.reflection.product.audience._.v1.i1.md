# research: audience reflection

## who is the audience?

### primary: the mechanic (AI agent)

the mechanic is an AI assistant that works on code tasks delegated by a human. the mechanic operates within claude code sessions and needs to access code across repos to fulfill requests.

**context constraints:**
- runs inside a claude code session
- has access to bash, git, gh cli
- bound by permission settings
- works on behalf of a human patron

### secondary: the human patron

the human who delegates work to the mechanic. they grant permissions and respond to prompts when the mechanic lacks approval for an action.

**context constraints:**
- engaged in deep work (flow state)
- interrupted by permission prompts
- wants to trust mechanic with repo access
- prefers autonomous execution

### tertiary: other mechanics (future)

other mechanics who might use git.repo.get as a pattern or who benefit from its pre-approved permission model.

---

## why do they care?

### mechanic

| pain | relief |
|------|--------|
| permission prompts break flow | pre-approved skill = frictionless access |
| search fails silently | local-first + cloud fallback = reliable results |
| unclear what repos exist | `repos` command shows accessible repos |
| must clone repo to search | git show/grep at ref = no checkout needed |
| github rate limits | local-first avoids api limits |

### human patron

| pain | relief |
|------|--------|
| interrupted by "allow gh?" prompts | pre-approved skill = no prompts |
| mechanic says "i cant access that" | local-first + cloud = transparent access |
| context switch costs (23+ min) | frictionless repo access preserves flow |
| uncertain if mechanic can help | mechanic shows what repos it can access |

### other mechanics

| pain | relief |
|------|--------|
| implement repo access per-skill | reusable pattern to follow |
| reinvent local/cloud logic | shared operations to import |

## what do they care about?

### mechanic criteria

| aspect | success criteria |
|--------|------------------|
| speed | local-first for instant results |
| reliability | cloud fallback when local absent |
| discoverability | can enumerate available repos |
| consistency | same interface for repos/files/lines |
| clarity | output shows source (local/cloud) |

### human patron criteria

| aspect | success criteria |
|--------|------------------|
| autonomy | mechanic can search without prompts |
| trust | skill is read-only (safe) |
| observability | output shows what was accessed |
| control | can limit to @local or @cloud if needed |

### tradeoffs they would accept

- **accept**: slower cloud fallback for repos not cloned locally
- **accept**: manual git clone to get local speed
- **reject**: any write operations (must be read-only)
- **reject**: obscured source (must show local vs cloud)

### what would delight vs disappoint

| delight | disappoint |
|---------|------------|
| "it just found it instantly" | "it says repo not found but i have it" |
| "i didnt have to clone first" | "i had to clone first to search" |
| "no permission prompt" | "it prompted me for gh access" |
| "i can see all ehmpathy repos" | "i dont know what repos it sees" |

## how will they experience it?

### mechanic journey

1. **trigger**: human asks "how did X repo implement Y pattern?"
2. **discover**: `rhx git.repo.get repos --repos 'ehmpathy/*'` → see available repos
3. **search**: `rhx git.repo.get lines --repos 'ehmpathy/*' --words 'pattern'` → find matches
4. **narrow**: `rhx git.repo.get lines --repo ehmpathy/found-repo --words 'pattern'` → focus
5. **read**: `rhx git.repo.get lines --repo ehmpathy/found-repo --paths 'src/file.ts'` → full file
6. **deliver**: mechanic synthesizes answer for human

### friction points

| step | friction | mitigation |
|------|----------|------------|
| discover | no local clones | cloud fallback |
| search | gh rate limit | local-first |
| narrow | repo not found | suggest broader search |
| read | file too large | radius/pagination? |

### smooth experience

```
human: "how does domain-objects implement the entity pattern?"

mechanic: (no prompt)
$ rhx git.repo.get lines --repo ehmpathy/domain-objects --words 'DomainEntity'

   │  └─ ... context ...
   │
   └─ found: 1 match

mechanic: "the pattern is at src/DomainEntity.ts:15 — it extends DomainObject and adds..."
```
