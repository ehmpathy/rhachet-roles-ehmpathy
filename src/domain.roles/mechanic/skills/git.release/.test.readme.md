# git.release integration tests

## critical: tests must be instant

all tests must complete in under 3 seconds. if any test takes longer:

1. **failfast** — do not let it continue
2. **track it down** — the mock or skill has a bug that triggers real waits
3. **root cause** — usually absent `GIT_RELEASE_TEST_MODE=true` or mock did not intercept a call

the skill uses `GIT_RELEASE_TEST_MODE=true` to:
- skip all sleep calls
- exit watch loops after 3 iterations (not 15 min elapsed time)
- the spawnSync timeout of 3s is a safety net, not a fix

if a test hits the 3s timeout, there is a bug — investigate, do not increase the timeout.

## overview

this file documents lessons for tests of shell skills that interact with external CLIs like `gh`.

## key patterns

### path injection for mocking

mock external CLIs by prepending a fake bin directory to PATH:

```typescript
const fakeBinDir = path.join(tempDir, '.fakebin');
fs.mkdirSync(fakeBinDir, { recursive: true });

fs.writeFileSync(path.join(fakeBinDir, 'gh'), ghMockContent);
fs.chmodSync(path.join(fakeBinDir, 'gh'), '755');

const result = spawnSync('bash', [skillPath, ...args], {
  env: { ...process.env, PATH: `${fakeBinDir}:${process.env.PATH}` },
});
```

### mock dispatch by command signature

use `case` statement to dispatch based on command + subcommand:

```bash
CMD_KEY="$1 $2"
case "$CMD_KEY" in
  "pr list")
    # handle pr list variants
    ;;
  "pr view")
    # handle pr view
    ;;
  "run list")
    # handle workflow runs
    ;;
esac
```

### pattern matching within command args

distinguish variants of the same command using grep on `$ALL_ARGS`:

```bash
ALL_ARGS="$*"

if echo "$ALL_ARGS" | grep -qF "chore(release)"; then
  # release PR lookup
elif echo "$ALL_ARGS" | grep -qF -- "--head"; then
  # feature branch PR lookup
fi
```

### detecting jq filter endings

when jq filter contains multiple field references, check what it ENDS with:

```bash
# BAD: .title appears in both filters (in select clause)
if echo "$ALL_ARGS" | grep -qF ".title"; then  # matches both!

# GOOD: check the ending
if echo "$ALL_ARGS" | grep -qE '\\.number$'; then
  echo "99"  # return PR number
else
  echo "chore(release): v1.33.0"  # return PR title
fi
```

### state-based responses

mock different states (open vs merged) based on args:

```bash
if echo "$ALL_ARGS" | grep -qF -- "--state open"; then
  echo ""  # no open PR
elif echo "$ALL_ARGS" | grep -qF -- "--state merged"; then
  echo "99"  # found via merged lookup
fi
```

## common pitfalls

### 1. rebuild after source changes

tests run against built output, not source files. always rebuild:

```bash
npm run build
```

### 2. debug logs in wrong location

use `.temp/` within the test directory, not `/tmp/`:

```bash
# bad
echo "DEBUG" >> /tmp/debug.log

# good
echo "DEBUG" >> "$TEMP_DIR/.temp/debug.log"
```

### 3. missing fallback in operations

when implementing fallback patterns (e.g., open then merged), verify both code paths execute:

```bash
# add debug output to verify fallback triggers
if [[ -z "$result" ]]; then
  echo "DEBUG: result empty, calling fallback" >> "$DEBUG_LOG"
  result=$(fallback_call)
  echo "DEBUG: after fallback, result='$result'" >> "$DEBUG_LOG"
fi
```

### 4. jq filter contains unexpected patterns

the jq filter `.[] | select(.title | startswith("chore(release):")) | .number` contains `.title` in the select clause but asks for `.number`. naive grep for `.title` matches incorrectly.

### 5. empty output vs whitespace

bash `echo ""` outputs a newline. when captured with `result=$(...)`, trailing newlines are stripped. verify with `${#result}` (string length).

## test structure

follow BDD pattern with `given`, `when`, `then`:

```typescript
given('[case3b] release to prod from feat (plan mode)', () => {
  when('[t3] feature PR merged and release PR also merged', () => {
    then('shows both merged status', () => {
      // test implementation
    });
  });
});
```

## debugging workflow

1. add debug output to operations file (in source)
2. rebuild: `npm run build`
3. clear debug log: `printf '' > .temp/debug.log`
4. run single test: `npm run test:integration -- -t "case3b.*t3"`
5. check debug log: `cat .temp/debug.log`
6. remove debug output before commit

## key insight

the hardest bugs to diagnose are when mock logic doesn't match operation behavior. when a test fails unexpectedly:

1. verify the mock receives expected args (log `$ALL_ARGS`)
2. verify the mock returns expected output
3. verify the operation handles that output correctly

trace from mock output through operation logic to final result.
